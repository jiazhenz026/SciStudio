# ADR-038 drift audit
Date: 2026-05-15
Auditor: no-context audit agent (D38-3.1a)
Repository state: `git log -1 --oneline origin/track/adr-038/lineage-db` → `c6e9c69 Merge pull request #951 from zjzcpj/feat/issue-948/d38-2-5-polish`

## Summary

Static walk of the unified-lineage implementation on `track/adr-038/lineage-db` against the accepted ADR-038 text. **13 findings total — 4 P1 (blocking), 6 P2 (should-fix), 3 P3 (nits).**

The four-table schema is correctly created and the recorder is wired into `ApiRuntime.start_workflow`. The blocking issues are concentrated in three classes:

1. **`block_executions` rows are written with empty strings for non-`BLOCK_DONE` terminations** because the scheduler's BLOCK_SKIPPED / BLOCK_CANCELLED / BLOCK_ERROR emitters do not carry `block_type` / `block_version` / `config` / `input_object_ids` / `output_object_ids` payload — only `workflow_id` (+ `error` for ERROR). The recorder coalesces missing values to `""` / `{}` then writes them into the `NOT NULL` columns, breaking ADR §3.7 Q3 ("which blocks ran") for any non-completed block.
2. **`block_version` "unknown" default is not removed** as ADR §3.3 requires. `_resolve_distribution_version` (`registry.py:726`) still returns `"unknown"` on resolution failure, and its docstring (lines 672-677) explicitly notes this contradicts ADR §3.3's "fail loudly" directive.
3. **`FrameworkMeta.lineage_id` is still permanently `None`** despite ADR §5.2 / §6 Phase 2 saying "populated with `block_execution_id`". The `with_lineage_id` helper exists but is never invoked anywhere in `src/scieasy`.

The other findings are dormant artifacts (legacy `LineageRecord` shell with `input_hashes`/`output_hashes`/`partial_output_refs`/`environment` fields still re-exported), stale field defaults (`workflow_dirty` / `workflow_git_commit` / `size_bytes` / `mtime_at_write` / `parent_run_id` columns are never populated despite being load-bearing for ADR §3.6 re-run validation), and a doc-vs-impl gap where ARCHITECTURE.md §4.4 describes a write flow that does not match the recorder's actual terminal-event-driven design.

| `src/scieasy/cli/main.py:177-197` | The CLI workflow runner (`scieasy run …`) constructs `DAGScheduler` without `lineage_recorder=`. The actual implementation writes `block_executions` only on the terminal event in `LineageRecorder._on_terminal` (`recorder.py:135-177`) — i.e. AFTER the block completes. Either rewrite ARCHITECTURE.md §4.4 write flow to match the implementation, or align the implementation with the docs by pre-allocating `block_execution_id` at dispatch time (which would also unblock §5.2 / §6 Phase 2 — see P1 row for `FrameworkMeta.lineage_id`). | §3.2 (write flow), ARCHITECTURE.md §4.4 |
