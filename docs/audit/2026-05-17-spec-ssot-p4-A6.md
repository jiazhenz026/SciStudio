## Part 2 ‚Äî Key findings

### 2.1 Finding A6-P1: `validate_connection` ignores `get_effective_*_ports` (b-code-wins confirmed)

**Priority**: P1 (code is authoritative; spec update required)
**Issue**: #889
**Label**: b-code-wins

`routes/blocks.py::validate_connection_route` (line 236‚Äì261) looks up ports by name from `spec.output_ports` / `spec.input_ports` ‚Äî these are the **static** port lists from the BlockSpec, not the per-instance effective ports returned by `block_instance.get_effective_input_ports()` / `get_effective_output_ports()`.

For dynamic blocks (LoadData, SaveData, variadic blocks configured with a specific core_type), the REST validation uses the wrong port types. The `validate_connection` function from `scieasy.blocks.base.ports` is called with static spec ports rather than instance-computed ports. The route does synthesize a permissive `DataObject` port for variadic blocks that lack a static spec entry (lines 250‚Äì256), which is a partial mitigation.

**ARCH ¬ß5.2 / draft v1 ¬ß2 (port-system)** describes `validate_connection` as taking actual `InputPort`/`OutputPort` objects ‚Äî the call itself is correct, but the inputs are wrong for dynamic blocks. The SSOT must note: *REST `POST /api/blocks/validate-connection` uses static spec ports; does not instantiate the block to call `get_effective_*_ports`. Fix is code-side.*

**Action**: b-code-wins already recorded in draft v1 ¬ß10. CONFIRMED correct label. Phase 5 fix note: the endpoint would need to instantiate a block with the target `config` to call `get_effective_*_ports`, or accept the current limitation and document it.

### 2.2 Finding A6-P1: `LineageRunDetail` wire shape mismatch (b-code-wins, upgrade severity)

**Priority**: P1 (frontend type is wrong relative to live backend wire shape)
**Issue**: needs GitHub issue (currently un-issued)
**Label**: b-code-wins

`frontend/src/types/lineage.ts::LineageRunDetail` (line 91‚Äì96) expects:
```typescript
interface LineageRunDetail {
  run: LineageRunSummary;
  blocks: LineageBlockExecution[];           // field name: "blocks"
  environment_snapshot: Record<string, string>;
  workflow_yaml_snapshot: string | null;

```

`src/scieasy/api/routes/runs.py::get_run` (lines 189‚Äì192) returns:

    "run": run,

The draft v1 label of "b-code-wins" was pre-assigned but the field name mismatch (`blocks` vs `block_executions`) was not explicitly called out.

**Action**: Draft v1 ¬ß10 must be updated to note the field name mismatch. b-code-wins: frontend type must use `block_executions` not `blocks`. Phase 5 fix: update `LineageRunDetail` interface:
```typescript

  block_executions: LineageBlockExecution[];  // correct field name
  // environment_snapshot and workflow_yaml_snapshot: remove (backend does not populate)
}
```
OR backend adds the missing wrapper ‚Äî manager decision needed.

**New GitHub issue required**: open as `fix(lineage): LineageRunDetail.blocks field name mismatch (should be block_executions)`.

ai_pty_module.unregister_ai_pty_subscriber(_on_ai_pty_message)
```

`block_pty_opened` and `block_pty_closed` are delivered via the ai_pty internal broadcaster, NOT via the EventBus subscription in `_OUTBOUND_EVENTS`. The `blocks.reloaded` event also bypasses the EventBus ‚Äî it is emitted directly as an EngineEvent but with a string constant defined in `ws.py` (`BLOCKS_RELOADED = "blocks.reloaded"`) and also in `projects.py` (`BLOCKS_RELOADED_EVENT_TYPE = "blocks.reloaded"`).

Both these constants exist because `scieasy.engine.events` is frozen by ADR-035/036 hard-scope rules. This is an intentional design choice with a known inconsistency: the pattern for `blocks.reloaded` (still goes through EventBus, just with a locally-defined string constant) is different from `block_pty_*` (pure ai_pty broadcaster, never hits EventBus).

**SSOT must document both routing paths explicitly.**

### 3.4 Finding A6-P2: `block_user_cancel` writes `mark_done.json` not a cancel signal (b-docs-wins confirmed)

**Priority**: P2 (behavioral documentation gap)
**Issue**: flagged in draft v1 ¬ß11
