## Detailed evidence

### M05-1 REST workflow/block schema (A)
- `WorkflowNode`, `WorkflowEdge`, `WorkflowCreate`, and `WorkflowResponse` provide explicit stable REST shapes.
- Block contract responses (`BlockSummary`, `BlockSchemaResponse`) are typed and include structured block metadata and constraints.
- Use of `Field(default_factory=...)` for collection defaults in API schema layer supports predictable request/response behavior.

Assessment: manager label **A** is consistent with implementation.

### M05-2 WS/SSE frame typing strength (B)
- Inbound WS frames are parsed from raw JSON and dispatched via string comparisons (`cancel_block`, `cancel_workflow`, `interactive_complete`, etc.).
- Payload validation is branch-local (`if not block_id or not workflow_id: warning; continue`) rather than model-level schema validation.
- Unknown types are logged and ignored; no typed protocol object boundary is enforced.
- Outbound frames are normalized through `serialise_event(...)`, but still dictionary conventions.

Assessment: manager label **B** is justified; protocol hardening remains needed.

### M05-3 API/runtime cross-layer coupling (C)
- WS layer performs engine-adjacent side effects through `_handle_block_user_signal(...)`.
- Function lazily imports `scieasy.api.routes.ai_pty`, resolves runtime run directory by `block_run_id`, and writes signal files under `<run_dir>/signals`.
- This transport endpoint behavior embeds runtime/worker coordination concerns directly in API module logic.

Assessment: manager label **C** is justified due to contract-boundary risk and coupling.

### M06-1 Workflow graph SoT + validate entrypoint (A)
- `WorkflowFileModel` formalizes top-level YAML ownership, converting into runtime `WorkflowDefinition` through explicit model methods.
- `validate_workflow(workflow, registry=None)` is the single semantic validation entrypoint.
- Structural checks (duplicate IDs, edge format/reference, cycle detection) are centralized in validator.

Assessment: manager label **A** is consistent with code.

### M06-2 Validation depth varies by registry availability (C)
- `validate_workflow(...)` executes structural checks always, then immediately returns if `registry is None`.
- When registry is present, additional checks apply: type compatibility, dangling required inputs, variadic cardinality, and extension conflicts.
- Unknown block types in registry-dependent path degrade into warning strings and skipped checks.

Assessment: manager label **C** is supported: diagnostics are materially context-dependent.

## Conclusion
All five Phase-3 manager labels for M05/M06 are **confirmed** by direct code evidence. No relabel recommendation.
