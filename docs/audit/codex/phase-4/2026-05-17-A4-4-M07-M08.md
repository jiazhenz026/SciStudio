## Evidence check details

### M07-1 — Frontend as editor/viewer (not runtime truth)
**Manager draft label:** A

**Code evidence observed**
- Frontend orchestration is API-mediated (`apiFetch<T>`, grouped API methods for workflow/data/block validation), indicating backend-owned execution truth rather than direct frontend-owned execution semantics.
- Execution/event state is consumed as messages (`WorkflowEventMessage`, `consumeEvent`) and projected into UI slices, consistent with viewer/editor role.
- Store persistence logic explicitly treats runtime-ish state as recoverable/derived (rehydration guardrails; reset/downgrade behavior for volatile tab/runtime-like states).

**Assessment**
- Evidence supports a stable architecture principle, matching **A**.

### M07-2 — Watcher/protocol reverse-coupling risk
**Manager draft label:** B (docs-fix)

**Code evidence observed**
- Multi-surface event/state contracts exist across `types/api.ts`, websocket hook, and store slices.
- Union/status-style schemas (`WorkflowEventMessage`, `AiBlockStatus`, tab/runtime state unions) are distributed across multiple files and update points.
- This is functional, but increases drift risk when event values evolve unless a stronger single normative contract is maintained.

**Assessment**
- Risk is real but does not by itself prove runtime contract breakage in current code.
- **B (docs-fix)** remains appropriate.

### M08-1 — CLI install / mcp-bridge contract
**Manager draft label:** A

**Code evidence observed**
- `install.py` defines explicit target/scope contract, stable server key (`MCP_SERVER_NAME = "scieasy"`), and deterministic command anchor (`python -m scieasy mcp-bridge`).
- Codex/Claude config writing paths/keys are explicit and idempotent-oriented; project scope env pin (`SCIEASY_PROJECT_DIR`) is encoded.
- `mcp_bridge.py` resolves project dir from `SCIEASY_PROJECT_DIR`, uses attached/standalone fallback strategy, and returns explicit non-zero status on missing env.

**Assessment**
- Contract shape is explicit and operationally coherent; **A** is supported.

### M08-2 — Scaffold conformance verification
**Manager draft label:** B (docs-fix)

**Code evidence observed**
- Scaffold flow is present (`main.py` subcommand registration -> `_scaffold.py` generation path and template rendering).
- However, auditable “must-pass conformance checklist” semantics for generated artifact validity are not clearly centralized as a normative acceptance contract in the inspected module-level code.

**Assessment**
- This fits a **B-level** hardening need (normative acceptance criteria/documentation), not a demonstrated C-level contract failure.
