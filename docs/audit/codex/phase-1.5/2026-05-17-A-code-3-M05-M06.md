## Findings by module

### M05 — API Surface & Realtime

#### 1) API data contracts (`src/scieasy/api/schemas.py`)

**Workflow payload models**
- `WorkflowNode`
  - fields: `id: str`, `block_type: str`, `config: dict[str, Any]={}`, `execution_mode: str|None`, `layout: dict[str, float]|None`
- `WorkflowEdge`
  - fields: `source: str`, `target: str`
- `WorkflowCreate`
  - fields: `id`, `version="1.0.0"`, `description=""`, `nodes: list[WorkflowNode]`, `edges: list[WorkflowEdge]`, `metadata: dict[str, Any]={}`
- `WorkflowResponse(WorkflowCreate)`
  - same envelope as create; intended read model

**Execution control models**
- `WorkflowExecutionResponse`
  - fields: `workflow_id: str`, `status: str`, `message: str`
- `ExecuteFromRequest`
  - fields: `block_id: str`
- `ExecuteFromResponse(WorkflowExecutionResponse)`
  - fields added: `reused_blocks: list[str]=[]`, `reset_blocks: list[str]=[]`

**Block/schema discovery models**
- `BlockPortResponse`
  - fields: `name`, `direction`, `accepted_types[]`, `required`, `description`, `constraint_description`, `is_collection`
- `TypeHierarchyEntry`
  - fields: `name`, `base_type`, `description`, `ui_ring_color`
- `BlockSummary`
  - fields: `name`, `type_name`, `base_category`, `subcategory`, `description`, `version`, `input_ports[]`, `output_ports[]`, `direction`, `source`, `package_name`, `variadic_inputs`, `variadic_outputs`
- `BlockListResponse`
  - fields: `blocks: list[BlockSummary]=[]`
- `BlockSchemaResponse(BlockSummary)`
  - fields added: `config_schema`, `type_hierarchy[]`, `dynamic_ports`, `direction`, `allowed_input_types[]`, `allowed_output_types[]`, `min/max input/output ports`

**Connection and data I/O models**
- `BlockConnectionValidation`
  - fields: `source_block`, `source_port`, `target_block`, `target_port`
- `ConnectionValidationResponse`
  - fields: `compatible: bool`, `reason: str=""`
- `DataUploadResponse`, `DataMetadataResponse`, `DataPreviewResponse`
  - common shape: `ref`, `type_name`, plus object-specific `metadata`/`preview`

**Project and cancellation models**
- `ProjectCreate`: `name`, `description`, `path`
- `ProjectUpdate`: optional `name`, `description`
- `ProjectResponse`: `id`, `name`, `path`, `description`, `last_opened`, `workflow_count`, `workflows[]`, `current_workflow_id`
- `CancelBlockRequest`: `block_id`
- `CancelWorkflowRequest`: empty body model
- `CancelPropagationResponse`: `cancelled_blocks[]`, `skipped_blocks[]`, `skip_reasons{}`
- `ErrorResponse`: `detail`, `error_code`

**Schema conventions observed**
- Request/response contracts consistently use Pydantic `BaseModel`.
- List/map defaults are generally provided via `Field(default_factory=...)` (not shared mutable literals) in API models.
- Workflow model in API (`WorkflowNode` / `WorkflowEdge`) mirrors workflow core dataclasses closely (M06).

#### 2) WebSocket realtime protocol (`src/scieasy/api/ws.py`)

**Outbound event surface**
- `serialise_event(event: EngineEvent) -> dict[str, Any]`
  - normalized frame keys: `type`, `block_id`, `workflow_id`, `data`, `timestamp`
- `_OUTBOUND_EVENTS` includes engine lifecycle/cancellation/interactive/workflow/git + string literal `BLOCKS_RELOADED`

**Inbound message conventions (string `type`)**
- `cancel_block`
  - required payload keys: `block_id`, `workflow_id`
  - emitted event: `CANCEL_BLOCK_REQUEST`
- `cancel_workflow`
  - required key: `workflow_id`
  - emitted event: `CANCEL_WORKFLOW_REQUEST`
- `interactive_complete`
  - payload keys: `block_id`, `data`
  - emitted event: `INTERACTIVE_COMPLETE`
- `block_user_marked_done` / `block_user_cancel`
  - both mapped to run-dir signal file writing via `_handle_block_user_signal(...)`

**Realtime integration conventions**
- `websocket_handler(websocket, event_bus)` runs bidirectional loops (inbound/outbound) and multiplexes:
  - EventBus events
  - AI PTY broadcaster messages
- Disconnection and task cancellation paths are swallowed (best-effort teardown).
- Unknown inbound message types are logged (warning), not hard-failed.

#### 3) SSE log streaming (`src/scieasy/api/sse.py`)

**Interface**
- `sse_handler(request: Request) -> StreamingResponse`
- Query-param filters:
  - `workflow_id`
  - `block_id`
  - `level`

**Protocol conventions**
- Event format: `event: log` + JSON `data:` payload.
- Keepalive heartbeat: comment line `: keepalive` every timeout interval.
- Subscriber lifecycle: subscribe on connect, always unsubscribe in `finally`.

#### 4) FastAPI composition surface (`src/scieasy/api/app.py`, routes)

**Application factory behavior**
- `lifespan(app)` initializes shared state (`ApiRuntime`, `ProcessRegistry`) and starts supporting async services.
- Root and static/SPA mount + API router registration occur centrally in app module.
- Realtime endpoints are integrated through WS handler and SSE handler.

**Route module inventory (`src/scieasy/api/routes/`)**
- Route domains present: `ai`, `ai_pty`, `blocks`, `data`, `filesystem`, `git`, `lint`, `projects`, `runs`, `workflows`, `workflow_watcher`.
- Convention: modular router split by domain under shared runtime/deps.

### M06 — Workflow Definition & Validation

#### 1) Canonical workflow dataclasses (`src/scieasy/workflow/definition.py`)

**`NodeDef`**
- fields: `id`, `block_type`, `config: dict[str, Any]`, `execution_mode: str|None`, `layout: dict[str, float]|None`

**`EdgeDef`**
- fields: `source`, `target`
- convention: both are string refs in `node_id:port_name` format

**`WorkflowDefinition`**
- fields: `id=""`, `version="1.0.0"`, `description=""`, `nodes[]`, `edges[]`, `metadata{}`

#### 2) YAML/schema models and transformation (`src/scieasy/workflow/schema.py`)

**`NodeModel`**
- fields mirror `NodeDef`
- validator: `id` must be non-empty after `strip()`
- methods:
  - `to_node_def() -> NodeDef`
  - `from_node_def(node: NodeDef) -> NodeModel`

**`EdgeModel`**
- fields: `source`, `target`
- validator on both fields: exact `node_id:port_name` two-segment colon format
- methods:
  - `to_edge_def() -> EdgeDef`
  - `from_edge_def(edge: EdgeDef) -> EdgeModel`

**`WorkflowModel`**
- fields mirror `WorkflowDefinition`
- methods:
  - `to_definition() -> WorkflowDefinition`
  - `from_definition(wf: WorkflowDefinition) -> WorkflowModel`

**`WorkflowFileModel`**
- top-level wrapper: `workflow: WorkflowModel`

**Schema conventions observed**
- Parsing boundary is explicit: YAML object -> `WorkflowFileModel` -> `WorkflowDefinition`.
- Serialization boundary mirrors parse flow in reverse via `WorkflowModel.from_definition`.

#### 3) Workflow semantic validator (`src/scieasy/workflow/validator.py`)

**Core helpers**
- `_parse_port_ref(ref: str) -> tuple[str, str] | None`
- `_find_port(ports, name) -> InputPort | OutputPort | None`
- `_effective_ports_for_node(registry, node, spec) -> (input_ports, output_ports)`
  - attempts real block instantiation; falls back to static `BlockSpec` ports on failure

**Main interface**
- `validate_workflow(workflow: WorkflowDefinition, registry: BlockRegistry | None = None) -> list[str]`
- return type convention: flat list of human-readable diagnostics (errors + warnings as text)

**Validation checks encoded in function**
1. structural duplicates (`node.id` uniqueness)
2. edge format (`node_id:port_name`)
3. edge references to existing node ids
4. cycle detection through DAG builder/topological sort
5. port type compatibility (`validate_connection`) when registry exists
6. dangling required input ports when registry exists
7. variadic cardinality checks (min/max limits)
8. duplicate output extension checks for variadic-output AppBlock case

**Operational conventions**
- If `registry is None`: only structural/edge/cycle checks run; registry-dependent checks are skipped.
- Unknown block type in registry-dependent pass yields warning strings and continues.
- Per-node effective ports are cached to avoid repeated block instantiation.

#### 4) YAML persistence and path normalization (`src/scieasy/workflow/serializer.py`)

**Public interfaces**
- `load_yaml(path) -> WorkflowDefinition`
- `save_yaml(workflow, path) -> None`
- `relativify_paths(config, project_dir, config_schema) -> dict`
- `absolutify_paths(config, project_dir, config_schema) -> dict`

**Path-field convention source**
- `_path_config_keys(config_schema)` scans schema `properties.*.ui_widget` values and treats:
  - `file_browser`
  - `directory_browser`
  as path-bearing config keys.

**Serialization conventions**
- Save: excludes `None`, preserves field ordering (`sort_keys=False`), writes UTF-8 YAML.
- Load: strict model validation by `WorkflowFileModel.model_validate(raw)`.
- Portability: relative path storage uses forward slashes via `PurePosixPath`.

## Open risks

1. **Dual schema defaults style across layers**
   - API models mostly use `Field(default_factory=...)`, but workflow Pydantic models in `schema.py` use literal `{}` / `[]` defaults on several fields. This is usually handled safely by Pydantic, but the style divergence may cause future maintenance confusion on mutability expectations.

2. **Validation result channel is untyped text list**
   - `validate_workflow` emits mixed errors/warnings as `list[str]` with string-prefix conventions (“Warning: ...”). Lacking structured severity/code fields may complicate stable client behavior for filtering/presentation.

3. **Realtime protocol is convention-by-string**
   - WS inbound message kinds and payload keys are stringly-typed and validated ad hoc in handler branches; no dedicated Pydantic frame schema is visible in this module.

4. **Edge format strictness vs extensibility**
   - Port reference parsing enforces exactly one colon split into two non-empty parts. Any future need for richer address forms would require coordinated changes across schema + validator + API consumers.
