- 读取范围：`docs/architecture/ARCHITECTURE.md` + `docs/` 下接口/契约相关文档（如 block-contract、cli-integration、PROJECT_TREE、ADR 索引）。

# Findings by module

## M01 Core Data Foundation

**接口/Schema清单（文档声明）**
- 核心六类数据对象（`Array / Table / Model / Collection / Scalar / Any`）及 typed 引用传递原则（运行时以引用/句柄而非大 payload 在块间流动）。
- `Array.axes` 采用“实例级 axes + 类级约束 schema”模型；文档给出 `required_axes / optional_axes / max_ndim` 语义与 OME 轴命名惯例。
- lineage 统一为 `<project>/.scieasy/lineage.db` 四表（`runs / block_executions / data_objects / block_io`）及写入流（engine 进程写库、worker 不直连）。
- lineage 与 git 的关系契约：两套历史体系分离，仅通过 `runs.workflow_git_commit` 关联。

**Convention清单（文档声明）**
- lineage 不是内容缓存；可覆写中间产物，长期保留“可重放配方”记录。
- 运行时真相在后端，前端不拥有数据真相；数据对象与血缘存储属于 Layer 1/3 的后端契约。

---

## M02 Block Contracts & Registry

**接口/Schema清单（文档声明）**
- Block 基本契约：typed input/output ports + `config_schema` + `run(inputs, config) -> outputs`。
- 变长端口机制：`dynamic_ports` 描述符 + `get_effective_input_ports()` / `get_effective_output_ports()` 作为实例级端口解析接口。
- AppBlock 手工审阅契约：通过文件交换 + `PAUSED` 生命周期实现人工介入，不存在单独“manual-review block class”。
- CodeBlock v2 / 脚本块约定：脚本入口 `run()`；可选 `configure()` 提供配置 schema；前端根据约定自动抽取端口/配置。
- IOBlock 格式契约：`supported_extensions`（文档已定义，且注明当前为文档先行、实现后续补齐）。

**Convention清单（文档声明）**
- 配置 UI 注入规则（variadic 端口编辑字段通过基类 schema 注入到子类）。
- 块作者仅实现块内逻辑，lineage 与 orchestration 写路径由 engine 负责（block-transparent）。

---

## M03 Execution Engine

**接口/Schema清单（文档声明）**
- 调度与终态传播契约：block terminal state 由调度器最终化并向下游传播取消/跳过语义。
- checkpoint / pause 路径与生命周期语义（`.scieasy/pause/`）在架构文档中与运行态联动描述。
- lineage 写流契约（engine-owned writes）属于引擎职责边界的一部分。

**Convention清单（文档声明）**
- 引擎保持运行时事实源；worker 执行与 orchestrator 视图解耦但语义对齐。
- 不把执行语义挪到前端状态；前端是编辑器/观察器。

---

## M04 AI Runtime Services

**接口/Schema清单（文档声明）**
- AI 能力边界：AI 可提议图/参数/块，但必须经 runtime schema + 规则验证后执行。
- MCP 工具分组（workflow 设计与执行等）在架构文档中以工具目录形式声明。
- 系统提示词三层来源（builtin / project / task）及职责边界在架构文档中定义。

**Convention清单（文档声明）**
- 计划先行、最小改动、验证后陈述等 agent 行为约束（生产模式工作原则）。
- AI 不得绕开 contract、lineage、execution policy。

---

## M05 API Surface & Realtime Channels

**接口/Schema清单（文档声明）**
- REST 主接口簇：blocks/workflows/runs 等读取与写入端点在架构文档有汇总表与示例路径。
- `GET /api/blocks/{type}` 等 block schema 获取接口被明确为前端端口/配置渲染来源。
- MCP bridge 契约：CLI 通过 `.scieasy/mcp.json` 连接 bridge，再连 FastAPI 进程。

**Convention清单（文档声明）**
- API 层负责“契约暴露”；运行时校验与执行语义仍在后端核心/引擎。
- URL 路由约定由架构文档统一说明，避免前后端各自命名。

---

## M06 Workflow Definition & Validation

**接口/Schema清单（文档声明）**
- 工作流图是 source of truth（定义、状态、lineage、执行语义归后端）。
- 工作流读写/校验/运行接口集合（`get_workflow / write_workflow / validate_workflow / run_workflow / cancel_run / get_run_status`）在文档中成组给出。
- 工作流 YAML 快照进入 run recipe（lineage runs）作为可重放输入之一。

**Convention清单（文档声明）**
- 前端不能成为 workflow 真相；仅编辑与可视化。
- 执行时遵循“引用传递 + 惰性物化”总原则。

## M07 Frontend State & Orchestration

**接口/Schema清单（文档声明）**
- 前端依赖 block schema API 渲染端口与配置；`ui_priority` 规则决定节点内联展示参数与完整配置面板展示。
- 类型颜色解析沿类型层级映射（typeColorMap + runtime schema）作为 UI 约定。
- PROJECT_TREE 文档声明前端 `types/workflow.ts`, `types/blocks.ts` 镜像后端 schema 的职责。

**Convention清单（文档声明）**
- 前端状态以“后端契约投影”为原则，不定义执行真相。
- 编辑态/展示态组件应服从统一 API typing 与实时通道语义。

---

## M08 CLI & Project Scaffolding

**接口/Schema清单（文档声明）**
- CLI 初始化/桥接契约：项目级 MCP 配置文件与桥接进程协同。
- block 脚手架约定：生成带 entry-point group 与最小 block 合同注释的模板（文档在架构章节与 block-dev 文档均有描述）。
- commit 前缀约定（`auto:` / `agent:` / 无前缀人工）作为 CLI + 历史视图共同契约。

**Convention清单（文档声明）**
- 生成物需对齐核心 block contract，不应绕开 schema/registry 机制。
- 自动化提交与人工提交需要可区分，以支持历史筛选与审计。

---

## M09 Plugin/Test Harness Boundary

**接口/Schema清单（文档声明）**
- 插件生态入口：三类 entry points（blocks / types / apps）与 package metadata 契约在架构文档与 block-development 文档中定义。
- 测试/契约验证职责：PROJECT_TREE 明确 `testing/harness` 与 `tests/architecture` 用于契约校验与边界保护。
- block-development 文档族（testing/publishing/custom-types）构成外部插件遵循的文档合同。

**Convention清单（文档声明）**
- “Everything is connectable, not everything is native”：插件是扩展层，core 保持小而稳。
- 扩展能力通过标准合同接入，不把插件逻辑回灌 core。

# Ambiguities

1. **M02 vs M06 边界有交叠**：Block 层动态端口与 Workflow 层图校验在文档中均涉及“端口有效性”，但“最终权威校验点”在多文档之间表述颗粒度不同，后续 Phase 2 需统一口径。
2. **M04 与 M08 的 `agent:` 前缀约定分散**：ARCHITECTURE 与 `cli-integration.md` 都提及约定，但“谁强制、谁仅建议”在文档层仍需做单点声明。
3. **M05 实时通道文档密度不均**：REST 汇总较完整，WS/SSE 的事件级 schema 在架构总览中颗粒较粗，需在后续分类时确认是否已有独立协议文档可作为权威。
4. **M09 插件契约分散在多处**：架构总览、block-development、ADR 索引都涉及插件约束，但缺少一份“插件接口总表”式文档作为单一引用入口。
5. **文档声明与实现状态的混合表达**：如 `supported_extensions` 被明确标注“文档已定义、实现待补齐”；后续分类阶段需逐条拆分“规范存在”与“实现存在”，避免同一条目混级。
