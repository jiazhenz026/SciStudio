## Findings by module

### M07 Frontend State & Orchestration

#### 1) API 调用层约定（`frontend/src/lib/api.ts`）
- `ApiError`：统一异常类型，字段：`name="ApiError"`, `status:number`, `message:string`。
- `apiFetch<T>(path, init?)`：
  - 非 2xx：尝试解析 `detail`（string 或 `{message, errors}`）并抛出 `ApiError`。
  - `204`：返回 `undefined as T`。
  - 其余：按 JSON 反序列化为 `T`。
- `api` 对象暴露编排接口（按职责分组）：
  - Project: `listProjects/createProject/openProject/updateProject/deleteProject/getProjectTree`。
  - Workflow: `listWorkflows/importWorkflowFile/importWorkflowFromPath/createWorkflow/getWorkflow/updateWorkflow/deleteWorkflow/executeWorkflow/pauseWorkflow/resumeWorkflow/cancelWorkflow/cancelBlock/executeFrom`。
  - Data: `uploadData/getDataMetadata/getDataPreview`。
  - Filesystem/UI: `browseFilesystem/revealInExplorer/openNativeDialog`。
  - Block schema: `listBlocks/getBlockSchema/validateConnection`。
  - Git/Lineage（同文件内并入前端编排边界，供 store 与 UI 调用）。
- 查询参数约定：`getDataPreview` 支持 `{slice,page,pageSize,sortBy,sortDir}`，并将 `pageSize -> page_size`、`sortBy -> sort_by`、`sortDir -> sort_dir` 做 URL 参数映射。

#### 2) 前端 API Schema（`frontend/src/types/api.ts`）
- Workflow：`WorkflowNode`, `WorkflowEdge`, `WorkflowResponse`, `WorkflowExecutionResponse`, `ExecuteFromResponse`。
- Block palette/schema：`BlockSummary`, `BlockSchemaResponse`, `BlockPortResponse`, `TypeHierarchyEntry`, `DynamicPortsConfig`。
- Data：`DataUploadResponse`, `DataMetadataResponse`, `DataPreviewResponse`, `DataPreviewQuery`。
- Runtime event/log：`WorkflowEventMessage`, `LogEntry`。
- Filesystem/tree：`FilesystemBrowseResponse`, `TreeResponse`。
- Git API 类型簇：`GitStatus`, `GitCommit`, `GitBranch`, `GitDiff`, `GitMergeResult`, `GitStashEntry`, `GitRestoreResult`, `GitStashApplyResult`, `GitCommitResponse`。

#### 3) UI / 状态层 schema（`frontend/src/types/ui.ts`, `frontend/src/store/types.ts`）
- Bottom 面板枚举：`BottomTab = "ai" | "config" | "logs" | "lineage" | "git"`。
- 节点数据约定：`BlockNodeData`, `AnnotationNodeData`, `GroupNodeData`。
- Store 顶层切片接口：
  - `ProjectSlice`, `WorkflowSlice`, `ExecutionSlice`, `UISlice`, `PreviewSlice`, `PaletteSlice`, `TabSlice`, `TerminalTabsSlice`, `LineageSlice`, `GitSlice`（后两者在聚合 store 中组合）。
- Tab 联合类型约定：`TabState = WorkflowTab | FileTab`，用 `kind` 作为判别字段。
- Terminal/AI block 生命周期约定：
  - `AiBlockStatus = "running" | "paused" | "done" | "error" | "cancelled"`。
  - `TerminalTab.state = "setup" | "running" | "closed"`。
  - `source?: "user" | "ai-block"`, `blockRunId?`, `blockStatus?`。

#### 4) Zustand 编排约定（`frontend/src/store/*.ts`）
- `store/index.ts`
  - 通过 `persist` 将 UI、terminal、file-tab 元数据持久化到 `scieasy-studio-ui`。
  - 文件 tab 持久化白名单：仅元数据；`content/dirty/loading` 在 rehydrate 时重置。
  - rehydrate 修正：非法 `activeBottomTab` 回退到 `lineage`；panel size 小于阈值时回默认；`running` terminal tab 降级为 `closed`。
- `workflowSlice.ts`
  - 工作流编辑指令：`addNode/updateNodeConfig/updateNodeLayout/connectNodes/removeNode/removeEdge`。
  - 撤销栈约束：`workflowHistory` 最多保留 40 条快照。
  - `setWorkflow` 在空 `id` 场景回退 `workflowName="Untitled"`。
- `executionSlice.ts`
  - 维护 `blockStates/blockOutputs/blockErrors/blockErrorSummaries` 与 `isRunning`。
  - 暴露 `consumeEvent/appendLog/resetExecution/setInteractivePrompt`。
- `tabSlice.ts`
  - `openTab/switchTab/closeTab/syncActiveTab/openFileTab` 等；最大标签数约束 50。
  - `openFileTab` 用 `file:<path>` 或 `source:<path>` 去重。
- `uiSlice.ts`
  - 面板状态与计数器：`unreadLogsCount`, `projectTreeRefreshCounter`。
  - `setActiveBottomTab("logs")` 时清零未读日志。

### M08 CLI & Project Scaffolding

#### 1) CLI 主入口与子命令边界（`src/scieasy/cli/main.py`）
- Typer 主应用暴露的核心命令域：
  - 项目/运行：`run`, `api`, `studio`。
  - 脚手架：`scaffold`（由 `_scaffold.py` 提供逻辑）。
  - 安装/桥接：`install`, `mcp-bridge`。
- 约定：CLI 层做参数解析与调用编排，具体副作用分发到独立模块函数。

#### 2) 脚手架接口约定（`src/scieasy/cli/_scaffold.py`）
- 脚手架模板来源：`src/scieasy/cli/templates/block_package/*`。
- 主要函数/约定：
  - 目标目录存在性检查与冲突处理。
  - 模板渲染写入（`*.tpl` 到目标文件）。
  - 包名/块名/模块名规范化（snake/camel 风格转换在本模块函数内完成）。
- 产物约定：生成可安装 block package 的最小结构（`pyproject`, `README`, `blocks.py`, `tests` 等模板项）。

#### 3) MCP 安装命令约定（`src/scieasy/cli/install.py`）
- 数据结构：`InstallResult`（`target/scope/path/action/detail`）。
- 目标支持：`claude`, `codex`, `skill`；作用域支持：`user`, `project`。
- 关键契约：
  - MCP server 名固定 `MCP_SERVER_NAME = "scieasy"`。
  - 命令锚定当前 Python：`{sys.executable} -m scieasy mcp-bridge`。
  - Claude 配置写入：`~/.claude.json`（user）或 `<cwd>/.mcp.json`（project），字段 `mcpServers.scieasy`。
  - Codex 配置写入：`~/.codex/config.toml`（user）或 `<cwd>/.codex/config.toml`（project），块 `[mcp_servers.scieasy]` + `args` + 可选 `env`。
  - Project scope 下附加环境变量：`SCIEASY_PROJECT_DIR=<cwd>`。
- 行为约定：安装/移除均幂等；JSON/TOML 异常时拒绝覆盖并抛错。

#### 4) MCP 桥接运行时约定（`src/scieasy/cli/mcp_bridge.py`）
- 入口函数：`run(socket: str | None) -> int`。
- 工程目录发现：`SCIEASY_PROJECT_DIR`（缺失时返回退出码 `2`）。
- 模式切换：
  - Attached mode：尝试连接 `<project>/.scieasy/mcp.sock`（Windows 走 `.port` 文件 + TCP 回环）。
  - Standalone mode：连接失败时启本进程 in-process MCP server。
- I/O 协议：stdio 与 socket 间做双向代理（line-delimited JSON-RPC 帧由 server 侧约束）。
- 平台约束：stdin 读取采用 executor 线程方案（兼容 Windows asyncio 限制）。

## Open risks
- 前端 API types 与后端 Pydantic schema 分离维护，字段重命名或可空性变化时存在漂移风险（需靠测试/契约校验兜底）。
- `TabState`、`TerminalTab`、`WorkflowEventMessage` 等联合/事件 schema 在多入口更新（slice + websocket + components），新增状态值时易出现未覆盖分支。
- CLI `install` 对外部配置文件（JSON/TOML）采取“保守写入”策略，格式异常会 hard-fail；对用户而言可见为安装阻断。
- `mcp-bridge` 依赖 `SCIEASY_PROJECT_DIR` 与 socket/port 文件约定；环境变量缺失或端口文件损坏时仅能回退/报错，排障依赖日志可见性。
