# Phase 1.5 Audit — A-code-1 (M01 + M02)

## Summary
- Scope: **code-only audit** for `M01 Core Data Foundation` and `M02 Block Contracts & Registry`.
- Method: enumerated concrete interfaces/schemas/conventions at class/function/field level from runtime code under `src/scieasy/core/types` and `src/scieasy/blocks`.
- This document intentionally avoids decision conclusions (no A/B/C/D judgement).

## Findings by module

### M01 — Core Data Foundation

#### 1) Core object contract (`DataObject`, `TypeSignature`)
- `TypeSignature` schema:
  - `type_chain: list[str]`
  - `slot_schema: dict[str, str] | None`
  - `required_axes: frozenset[str] | None`
- `TypeSignature.matches(other)` convention:
  - Prefix-compatible `type_chain` matching.
  - If `other.slot_schema` exists, each slot/type must match exactly.
- `TypeSignature.from_type(data_type)` convention:
  - Builds chain from MRO (`DataObject` lineage only).
  - Reads `expected_slots` when present.
  - Reads non-empty `required_axes` class var when present.

- `DataObject.__init__` public constructor slots:
  - `framework: FrameworkMeta | None`
  - `meta: BaseModel | None`
  - `user: dict[str, Any] | None`
  - `storage_ref: StorageReference | None`
  - Legacy shim: `metadata: dict[str, Any] | None` (deprecated path mapped to `user`).
- `DataObject` conventions:
  - Three-slot metadata model (`framework`, `meta`, `user`) is first-class.
  - `user` is validated as JSON-safe payload.
  - `_transient_data` is explicit non-serialized memory slot.

#### 2) Type registry and dynamic loading (`TypeRegistry`, `TypeSpec`)
- `TypeSpec` schema fields:
  - `name`, `module_path`, `class_name`, `base_type`, `description`.
- `TypeRegistry` core APIs:
  - `register(name, spec)`
  - `register_class(cls)`
  - `resolve(name: str) -> TypeSpec`
  - `resolve(type_chain: list[str]) -> type | None`
  - `load_class(name) -> type`
  - `is_instance(obj, type_name) -> bool`
  - `all_types() -> dict[str, TypeSpec]`
- Registration-time convention:
  - `Meta` model validation is enforced by `_validate_meta_class` for classes declaring `Meta`.
  - Expected constraints include Pydantic inheritance and JSON round-trip safety checks.

#### 3) Serialization/reconstruction hooks across canonical data families
- Pattern present in core families: class methods
  - `_reconstruct_extra_kwargs(metadata: dict[str, Any]) -> dict[str, Any]`
  - `_serialise_extra_metadata(obj: DataObject) -> dict[str, Any]`
- Family-level field conventions observed:
  - `Array`: shape/axes/dtype-like structural metadata + optional in-memory data slot.
  - `DataFrame`: `columns`, `row_count`, `schema`, optional transient `data`.
  - `Series`: `index_name`, `value_name`, `length`, optional transient `data`.
  - `Text`: `content`, `format`, `encoding`.
  - `Artifact`: `file_path`, `mime_type`, `description`.
  - `Collection` / `CompositeData`: collection/item or slot-oriented semantics with type-aware contract support.

#### 4) Immutability-style metadata update convention
- Core data families override/implement `with_meta(**changes)` pattern:
  - Requires existing typed `meta` (otherwise raises error).
  - Derives a new framework identity lineage (`framework.derive()` path).
  - Returns new instance preserving family-specific fields.

### M02 — Block Contracts & Registry

#### 1) Block base contract (`Block`)
- Mandatory abstract execution signature:
  - `run(self, inputs: dict[str, Collection], config: BlockConfig) -> dict[str, Collection]`
- Optional hooks and defaults:
  - `validate(inputs) -> bool`
  - `postprocess(outputs) -> dict[str, Collection]`
- State machine conventions:
  - `BlockState` enum: `IDLE/READY/RUNNING/PAUSED/DONE/ERROR/CANCELLED/SKIPPED`.
  - Explicit transition table enforced by `transition(target)`.
- Class-level metadata contract:
  - `name`, `description`, `version`, `subcategory`
  - `input_ports`, `output_ports`
  - `execution_mode`, `terminate_grace_sec`
  - `config_schema`, `key_dependencies`

#### 2) Port schema and compatibility conventions (`Port`, `InputPort`, `OutputPort`)
- Base `Port` schema:
  - `name`, `accepted_types`, `is_collection`, `description`, `required`.
- `InputPort` extensions:
  - `default`, `constraint`, `constraint_description`.
- Key compatibility helpers:
  - `port_accepts_type(port, data_type)` (inheritance-aware; Collection-transparent behavior)
  - `port_accepts_signature(port, signature)` (TypeSignature-based)
  - `validate_connection(source_port, target_port)`
  - `validate_port_constraint(port, value)`

#### 3) Variadic/dynamic port conventions
- Variadic class vars:
  - `variadic_inputs`, `variadic_outputs`
  - `allowed_input_types`, `allowed_output_types`
  - `min_input_ports`, `max_input_ports`, `min_output_ports`, `max_output_ports`
- Effective-port resolution APIs:
  - `get_effective_input_ports()` / `get_effective_output_ports()`
  - Config shape for runtime ports: list of `{"name": str, "types": list[str]}` (consumed by `ports_from_config_dicts`).
- Dynamic-port descriptor convention (`dynamic_ports`):
  - Must include `source_config_key`.
  - Must include exactly one mapping key:
    - `output_port_mapping` **or** `input_port_mapping`.
  - Nested shape: `port_name -> enum_value -> [type_name...]`.

#### 4) Block config payload contract (`BlockConfig`)
- Pydantic model with `extra="allow"`.
- Canonical field:
  - `params: dict[str, Any]`.
- Access convention:
  - `get(key, default)` checks `params` first, then pydantic extra fields.

#### 5) Block registry descriptors and discovery (`BlockRegistry`, `BlockSpec`)
- `BlockSpec` schema fields include:
  - identity/source: `name`, `module_path`, `class_name`, `source`, `type_name`, `package_name`
  - UX/categorization: `description`, `version`, `base_category`, `subcategory`, `direction`
  - ports/config: `input_ports`, `output_ports`, `config_schema`, `dynamic_ports`
  - file tracking: `file_path`, `file_mtime`
  - variadic constraints: `variadic_*`, `allowed_*`, `min/max_*`
- Registry conventions:
  - Descriptor-first storage (`_registry`, `_aliases`), class object resolved at load time.
  - Multi-source scan pipeline:
    - built-ins
    - tier1 local drop-in files
    - tier2 entry points
    - optional monorepo package scan
  - Dynamic-port descriptor validation runs during scan.

## Open risks
- Type-name fallback behavior in `ports_from_config_dicts` resolves unknown type strings to base `DataObject`; this may weaken strictness when config type names drift.
- Variadic/dynamic port behavior relies on config shape correctness (`input_ports`/`output_ports` dictionaries); malformed runtime payloads can move failures to validation-time instead of authoring-time.
- `with_meta` pattern depends on typed `meta` presence; mixed usage between typed and untyped instances can produce runtime exceptions at update call sites.
- Registry scan/import pipelines (tier1/tier2) are sensitive to packaging/import metadata consistency; partial plugin metadata can degrade discoverability.
