# Phase 1.5 差异审计（A-diff）— M01..M09

日期：2026-05-17  
范围：仅基于仓库内代码与文档进行“潜在不一致点”记录（事实差异，不做 ABCD 裁决）

## Summary

- Phase 1 中定义的 M01..M09 路径在当前仓库均可定位到实体目录/文件；未发现“模块完全缺失”。
- 潜在差异主要集中在三类：
  1) **文档宣称的“强约束/单一事实源”与代码中的“兼容性降级路径”并存**（例如 registry 缺失时的 warning/fallback）。
  2) **架构分层边界在实现中出现跨层耦合点**（例如 API 路由触发 runtime registry refresh、workflow watcher 与前端协议字段绑定）。
  3) **文档覆盖面与实现广度不完全同步**（例如部分行为以代码注释/ADR 引用为主，用户向文档可见度不足）。

## Potential mismatches by module

### M01 Core Data Foundation

- 文档强调“数据对象以引用/持久化工件流动”为主；代码侧存在多处对运行时对象的直接解析/重建逻辑（如 worker 中类型注册表解析与重建路径），显示实现同时支持“引用语义 + 进程内重建”双轨。两者并非直接冲突，但约束强度存在差异。  
- M01 与 M03 的边界在实现中较紧：engine runner 直接依赖 core type serialization registry，实际耦合度高于“核心最小化契约”叙述的直观预期。

### M02 Block Contracts & Registry

- 文档要求 block contract 稳定、可验证；workflow validator 中存在“spec-only registry entry”与实例化失败时的降级处理（继续按 spec 校验/给 warning），说明系统在“严格运行时可实例化”之外保留了兼容路径。  
- 文档将 registry 视为契约真源之一；实现中 registry 在不同入口（engine / API / workflow validator）被以不同强度依赖（hard dependency vs optional dependency），存在“同名契约、不同强制等级”现象。

### M03 Execution Engine

- 架构层描述强调 DAG 调度与状态语义统一；实现中 scheduler 同时承担事件封装、block version stamping fallback、PTY 交互桥接等职责，模块责任面较宽，可能与“模块职责窄化”原则存在张力。  
- 引擎对 registry 缺失提供 fallback 路径（例如 mock-runner 场景），与“运行时统一强校验”叙述之间存在执行语义梯度。

### M04 AI Runtime Services

- 文档原则要求“AI propose, runtime validate”；代码中 AI/PTY/MCP 相关路径与 engine/API 之间采用 HTTP loopback + in-process fallback 组合，体现了可运行性优先的双通道实现。与“单一路径可审计”目标相比，审计面更复杂。  
- `src/scieasy/ai/agent/mcp` 存在，但部分 AI 运行时行为的权威说明更多散落在 ADR 引用与代码注释，而非统一用户文档章节。

### M05 API Surface & Realtime Channels

- 文档将 REST/WS/SSE 作为统一入口层；实现上除标准 routes 外，`workflow_watcher` 对前端字段（如 commit SHA 语义）有显式协议耦合注释，说明 API 协议与前端实现细节耦合度较高。  
- API 路由中出现 runtime 级动作（例如分支切换后 refresh block registry），从分层角度看属于“API 编排 runtime 状态刷新”的跨层连接点。

### M06 Workflow Definition & Validation

- 文档强调 workflow graph 为 source of truth；实现中 validator 在 registry 可用与不可用场景下行为不同（严格端口/类型校验 vs warning/fallback），导致“事实源严格性”受运行上下文影响。  
- 这意味着“同一图定义”在不同执行环境下可得到不同级别诊断结果（可验证深度差异），属于文档需显式化的语义差异点。

### M07 Frontend State & Orchestration

- 文档声明前端不是事实源；但代码注释显示后端 watcher/WS 协议会直接参考前端 hook 的字段约定，形成“后端为兼容前端而固定消息形状”的反向约束。  
- 前后端契约目前更像“协同演进接口”，而非“后端单向定义、前端被动消费”的纯分层关系。

### M08 CLI & Project Scaffolding

- 模块路径存在且与架构中“脚手架/agent 接入”定位一致；但在当前可见文档中，CLI 与 agent_provisioning 的边界契约（哪些生成物必须对齐核心 contract）描述粒度有限，更多依赖实现内约定。  
- 对“生成物合规性检查”机制的公开文档化程度，低于架构愿景中的可追踪性要求。

### M09 Plugin/Test Harness Boundary

- 文档强调插件生态与回归验证边界；代码/测试目录均存在，但“插件契约失败时的标准诊断分层（contract fail vs runtime fail）”在用户向文档中不够集中，分散于测试与 ADR 语境。  
- `tests/plugins`、`tests/testing`、`tests/architecture` 并存，体现多维验证；但模块地图中对各测试层的职责分割与准入标准未形成单一索引页，存在认知成本差异。

## Data-quality caveats

- 本次仅执行“代码+文档静态差异审计”，未运行端到端流程、未拉取 CI 历史、未对行为做动态复现实验。  
- “潜在不一致点”按可见证据记录，部分条目属于**约束强度差异**而非功能错误。  
- 本文不包含优先级判定、不包含整改方案、不包含 ABCD 裁决，仅提供 Phase 1.5 输入事实。
