## Implementation rationale

The planning doc told me to call `materialise_to_file(obj, exchange_dir
/ "inputs", extension)`. The non-obvious finding: `SaveData`'s
`supported_extensions` is a flat union (`{".npy", ".csv", ".parquet",
...}`) shared across all core types, so the materialiser's
"first-declared extension wins" default picks `.npy` for *everything*,
which then causes `_save_dataframe` to raise `ValueError("Unsupported
DataFrame file extension '.npy'")`. The bridge has the type
information at the call site, so the right fix is a small per-core-type
default-extension map in `bridge.py` â€” local, declarative, and easy to
extend per plugin if a future block wants to override.

Per-port format/extension selection from "port config" (the second
half of the planning doc's bullet "Resolve target format/extension
(from port config or saver default)") is **out of scope** for this PR
since AppBlock does not currently expose a per-port extension override
on its config schema. The bridge falls back to the per-type default
unconditionally; per-port overrides can be added when CodeBlock v2
(ADR-041) needs them.
