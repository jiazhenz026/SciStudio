## Implementation rationale

- **Module-level dispatch functions can't call `self._detect_format` directly.** ADR-028 Addendum 1 §C9 forces the per-format implementations to be module-level private functions, not helper classes. The `IOBlock._detect_format` method is bound to an instance. The cleanest faithful integration was to thread `block: LoadData | None` through `_load_array`/`_load_dataframe`/`_load_series`/`_load_text` (matching the pre-existing `_load_series(cfg, block, output_dir)` pattern), and provide a module-level `_resolve_format(path, block)` helper that delegates to `block._detect_format` when a block is in hand and falls back to walking the `LoadData.supported_extensions` ClassVar otherwise. Same approach for the saver side (`_resolve_save_format`).
- **Format identifiers are stable strings, not enum values.** I used short, lowercase identifiers (`'npy'`, `'parquet'`, `'csv'`, `'text'`, etc.) that match the spirit of the existing hardcoded suffix strings. Both `.parquet` and `.pq` map to `'parquet'` so downstream `if fmt == 'parquet'` covers both. Both `.pkl` and `.pickle` map to `'pickle'` but the pickle path is gated by the separate `_check_pickle_allowed` / `_check_pickle_gate` helpers (unchanged) so the format identifier is informational only at that branch.
- **LoadData and SaveData supported_extensions are an exact mirror.** The spec test asserts `SaveData.supported_extensions == LoadData.supported_extensions`. The internal `_save_text` `supported` frozenset keeps the historical extras (`.markdown`, `.htm`) because they have no `LoadData` counterpart and removing them would be a silent behavior change; the registry-visible ClassVar surfaces only the canonical mirror set so `BlockRegistry.find_loader` / `find_saver` (issue #1077) see a consistent menu.
- **Text branch retains its `_TEXT_FORMAT_MAP`.** That map controls `Text.format` field population (`"plain"` vs `"markdown"` vs `"yaml"` etc.) and is internal to the Text loader; the new dispatch uses `fmt == 'text'` as the gate but `_TEXT_FORMAT_MAP[suffix]` still drives the format-name attribute.
- **Error messages reference `sorted(LoadData.supported_extensions.keys())` rather than a per-core-type subset.** Per the spec verbatim: "Replace hardcoded `Supported: .npy, .npz, ...` substring in `ValueError` messages with `sorted(self.supported_extensions.keys())`." Per-core-type subset hints (e.g. "Array does not support `.csv`") are still implied by the calling core_type, so the error remains diagnostic; the change makes the supported list self-consistent with the registry-visible declaration.

## Deviations from the spec

None substantive. Two minor notes:

1. The spec describes "format-keyed dispatch" generically; my implementation keeps the existing per-core-type private functions (`_load_array` / `_save_array` etc.) as the dispatch backend and uses the format identifier (e.g. `'npy'`, `'parquet'`) only as the inner switch. This matches the literal spec line "per-format `_load_<fmt>` / `_save_<fmt>` methods stay as the implementation backend" by treating "per-format" as the existing per-core-type method, since the existing code structure pairs core_type with a private function not a per-suffix function.
2. SaveData's `_save_text` retains its broader internal `supported` frozenset (containing `.markdown`, `.htm`) for backward compatibility. The ClassVar (registry-visible) excludes them so the mirror with LoadData is exact. The error-message body explicitly cites both the canonical ClassVar AND the Text-specific superset so users diagnosing a bogus suffix see both surfaces.
