# Implementation record: #1078

## Files modified

- `src/scieasy/core/materialisation.py` (NEW) — public helpers
  `materialise_to_file(obj, dest_dir, extension=None, *, filename_stem,
  registry) -> Path` and `reconstruct_from_file(path, target_type,
  extension=None, *, registry) -> DataObject`. Both consult
  `BlockRegistry.find_saver` / `find_loader` (#1077). The materialiser
  also threads a `core_type` config kwarg through when the resolved
  saver's `config_schema` declares it (so the dynamic-port
  `LoadData` / `SaveData` blocks pick the right per-core-type dispatch
  table). The reconstructor has a documented `Artifact` fallback and
  a dynamic-port second-pass against `find_loader(DataObject, ext)` so
  the core IO loader is reachable for concrete target types whose
  static `accepted_types` would otherwise reject it.
- `src/scieasy/utils/fs.py` (NEW) — cross-platform `mount_pathlike`
  (POSIX symlink, Windows directory junction via `_winapi.CreateJunction`,
  Windows file hardlink via `os.link`). Raises `OSError` when no
  primitive succeeds so callers can fall back to a byte copy.
- `tests/core/test_materialisation.py` (NEW) — 12 tests covering
  round-trip DataFrame (.csv) and Array (.npy), Artifact fallback,
  LookupError for concrete-type miss, missing-file error,
  pass-through invocation (with mock + side-effect to satisfy the
  downstream existence assertions), pass-through skip when extensions
  differ, default-extension selection via a module-level custom saver
  (`_TextDotXyz`), LookupError when no saver matches the type, and
  the extension-without-leading-dot normalisation.
- `tests/utils/test_fs.py` (NEW) — 6 tests: POSIX symlink (skipif
  win32), Windows hardlink with `st_ino` equality, Windows directory
  junction with content surfacing, `FileNotFoundError` on missing
  source, `FileExistsError` on existing destination, and parent
  directory auto-creation.

## Implementation rationale

- **Pass-through link policy.** The planning doc called for a
  `mount_pathlike` link when the source file already matches the
  target extension. `materialise_to_file` checks `obj.storage_ref.path`
  case-insensitively against the requested extension. On link failure
  the helper silently falls through to a saver round-trip (debug
  log), so the optimisation never breaks the correctness path.
- **Dynamic-port fallback in `reconstruct_from_file`.** `LoadData`
  declares `output_ports[0].accepted_types=[DataObject]` because its
  concrete output type is decided per-instance from `config['core_type']`.
  `BlockRegistry.find_loader` therefore (correctly per #1077's
  loader-output-IS-A-target rule) returns `None` for queries like
  `find_loader(DataFrame, ".csv")`. The reconstructor's second pass
  asks `find_loader(DataObject, ext)` and only accepts the result
  when the candidate's `config_schema` declares `core_type` — at which
  point it injects `core_type=target_type.__name__` and routes the
  load through the dynamic-port dispatch. This keeps the public
  `find_loader` contract narrow (no `dynamic_ports` awareness) while
  preserving the round-trip the planning doc asked for.
- **`mount_pathlike` on Windows: junction vs symlink.** Windows true
  symlinks need Developer Mode or admin privileges. Directory
  junctions (CreateJunction, NTFS reparse points) and hardlinks
  (NTFS-only) are both available on stock Windows without
  elevation, so the helper prefers those. Cross-volume hardlinks
  surface as `OSError` and the caller falls back to a byte copy —
  documented behaviour, not a bug.
- **Registry kwarg.** Both helpers accept an optional pre-built
  `BlockRegistry`. Tests construct an isolated registry via the
  same `_register_spec` + `_spec_from_class` pattern used by
  `tests/blocks/test_registry_find.py` (avoids the entry-point
  manifest dependency). The default-path constructs and scans a
  fresh registry on first call, acceptable for AppBlock prepare /
  restore which is once-per-session, not per-event.
- **Mock target for `mount_pathlike`.** `_try_pass_through` does
  `from scieasy.utils.fs import mount_pathlike` at function scope, so
  the test patches `scieasy.utils.fs.mount_pathlike` (the
  authoritative definition), not the importing module. Common
  `unittest.mock` gotcha, called out in the test docstrings.

## Deviations from the spec

- **Reconstructor signature.** Added an explicit `*, registry` kwarg
  on both helpers. The planning doc did not specify it; I introduced
  it because (a) the unit tests need to inject an isolated registry
  and (b) hot-path callers will want to avoid the per-call entry-point
  scan. The positional surface matches the planning doc exactly.
- **`filename_stem` kwarg on `materialise_to_file`.** Defaults to
  `"data"`. The planning doc said "`<obj-id-or-port>{extension}`";
  the helper exposes a kwarg so AppBlock (the call site in #1079)
  can pass the port name. Default keeps unit tests terse.
- **Reconstructor's second pass for dynamic-port loaders.** Not
  spelled out in the planning doc; required to make the
  round-trip-DataFrame test pass. Documented in
  `reconstruct_from_file`'s body and in the rationale above. The
  alternative (extending `find_loader` to consult `dynamic_ports`)
  was out-of-scope per the dispatch's frozen-files list
  (`src/scieasy/blocks/base/`).
