## Implementation rationale

- **Caching `supported_extensions` on `BlockSpec` vs lazy import.**
  Both options are valid. Caching trades a few bytes per spec for
  avoiding an import round-trip on every dispatch query. Since the
  query is called from setup-time UI code (port-editor format
  dropdown) AND from the runtime dispatch path (when LoadData /
  SaveData resolve a concrete loader/saver per ADR-028 Addendum 1),
  the caching version is the better default. The class still has to
  be imported when the caller wants the class object, but that
  import happens once per spec lookup rather than on every extension
  filter.
- **Type-compatibility direction.** The spec phrasing in #1077 is
  "compatible with *dtype*", which is direction-ambiguous. ADR-027
  resolves it via `TypeSignature.matches`: `A.matches(B)` is True
  when `A`'s chain has `B`'s chain as a prefix, i.e. `A` IS-A `B`.
  For a **loader**, we want the loader's output to be assignable
  to `dtype` — so `accepted_sig.matches(dtype_sig)` (output IS-A
  dtype). For a **saver**, we want `dtype` to be assignable to the
  saver's input port — so `dtype_sig.matches(accepted_sig)`. This
  is the classical covariance / contravariance flip. The unit tests
  (`test_subtype_covariance_direction_array_loader_does_not_match_image`
  and
  `test_saver_input_type_compat_uses_contravariant_direction`)
  exercise both directions explicitly.
- **Disambiguation: longest type chain wins, first-registered-wins
  on tie.** `_best_specificity` returns the longest matching chain
  length, and the main loop sorts by `-specificity` (descending).
  Python's sort is stable, so insertion order is preserved on ties
  — and `dict.values()` in Python 3.7+ is iterated in insertion
  order. This makes "first registered wins" a free property of the
  data structure rather than something I have to implement.
- **`dtype=None` shortcut on `find_loader`.** The spec body says
  callers may know only the file extension. When `dtype is None`,
  the type filter is skipped entirely and `_best_specificity` is
  not consulted (specificity defaults to 0, all matches tie, first-
  registered wins). The unit test `test_dtype_none_skips_type_filter`
  registers a CSV loader BEFORE a TIF loader and verifies that
  `find_loader(None, ".tif")` still returns the TIF loader (the
  filter narrows by extension).
- **Import failure tolerance.** `_resolve_class` catches every
  exception and returns `None`. The choice mirrors the existing
  `_scan_tier1` / `_scan_tier2` `try/except` tolerance (a single
  broken plugin should not poison query results). Failures are
  logged at DEBUG level (not WARNING) because query traffic is
  expected to hit a wide variety of plugins, including ones that
  failed to import during the scan; the corresponding scan-time
  warning is already on the record.
- **Empty `accepted_types` semantics.** A port with
  `accepted_types=[]` historically means "accepts anything"
  (see `port_accepts_type` in `src/scieasy/blocks/base/ports.py`).
  `_best_specificity` honors that by returning 1 (the chain length
  of `DataObject`), so an empty-list block is treated as the most
  general match. This makes `find_io_blocks_for_type` enumerate
  catch-all loaders/savers when they exist, while still allowing
  the more-specific ones to outrank them in `find_loader` /
  `find_saver`.

## Deviations from the spec

None of substance. The spec's pseudo-signature
`def find_loader(self, dtype: type[DataObject] | None, extension: str) -> type[IOBlock] | None`
is honored as-is. The implementation signature drops the explicit
`type[IOBlock]` constraint on the return type (uses `type | None`)
because the method also resolves Tier-1 drop-in blocks whose class
objects are not statically known to be `IOBlock` subclasses at type-
check time (they pass the direction filter at runtime).
