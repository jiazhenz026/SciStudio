## Implementation rationale

* **TC-1F.3 severity policy** — ADR-043 §4.5 table marks Phase 1 as
  "report" / Phase 3 as "hard gate". `TQMUT-below-threshold` therefore
  emits `WARNING` (matches `test_first_check.py` default; Phase 3
  cleanup-track work will promote to `ERROR`). `TQMUT-baseline-regression`
  emits `INFO` — it's a trend signal, not actionable on its own.
  `TQMUT-unavailable-platform` / `TQMUT-tool-missing` / `TQMUT-no-target`
  also emit `INFO` (informational about the environment, not failures).
  `TQMUT-run-failed` / `TQMUT-parse-failed` emit `WARNING` (tool-level
  errors the user should investigate).

* **TC-1F.3 rule-ID namespace** — `TQMUT-<slug>` mirrors the
  `TQAST-<slug>` / `TQTF-<slug>` pattern from Sub-PR 1 so SARIF /
  annotation consumers can filter unambiguously. The full set is
  exposed as `mutation_runner.RULE_IDS: frozenset[str]` for downstream
  documentation generators.

* **TC-1F.3 subprocess seam** — `_run_capture(args)` is the single
  subprocess gateway, identical to `test_first_check._run_capture`.
  Unit tests patch this one symbol via a `_Recorder` callable that
  scripts the (rc, stdout, stderr) triples returned across `mutmut
  run` + `mutmut results` invocations. Coverage: 100% line coverage
  on `mutation_runner.py`.

* **TC-1F.3 parser tolerance** — `_parse_results_payload` accepts both
  the modern JSON form (`{"killed": …, "survived": …, "timeout": …}`)
  and the historical plaintext form (`Killed: N / Survived: M /
  Timeout: K`), and tolerates the `timed_out` / `timed out:` aliases
  used by older mutmut versions. Garbled output returns `None`; callers
  emit a `TQMUT-parse-failed` finding rather than crashing.

* **TC-1F.3 platform handling** — `_is_windows()` is a one-line wrapper
  around `sys.platform == "win32"` so tests can monkeypatch the platform
  check without touching `sys`. The dev-extras entry uses the
  `sys_platform != 'win32'` PEP-508 marker so Windows pip installs
  succeed (no install-time failure for contributors who never run
  mutation testing).

* **TC-1F.4 property scope** — Property tests target only the **pure
  helpers** of `mutation_runner` (threshold lookup, score arithmetic,
  results-parsing primitives). The subprocess-bound `run_targeted`
  orchestration is NOT property-tested — it's side-effectful glue, and
  unit tests in `test_mutation_runner.py` cover its branches.
  Retrofitting property tests onto 1F.1 (`ast_lint`) and 1F.2
  (`test_first_check`) public surface is intentionally **out of scope**
  for this sub-PR per cascade plan; Phase 3 cleanup-track work will
  retrofit.

* **TC-1F.5 dual-write** — The skill body is written to TWO locations:
  - `src/scieasy/_skills/qa/test-author/SKILL.md` (canonical source —
    will be the input for the 1H sub-PR 3 cross-runtime installer).
  - `.claude/skills/test-author/SKILL.md` (force-added past `.gitignore`
    so the current Claude Code session sees the skill immediately).
  The two files MUST stay byte-identical until 1H sub-PR 3 ships
  `agent_provisioning.qa_skills`; this is enforced by
  `test_source_and_mirror_byte_identical` in `tests/qa/test_test_
  author_skill.py`.

## §4.4 verbatim compliance

The skill body reproduces the 7-step protocol from ADR-043 §4.4 in the
exact order (Identify the contract → Write the assertion first → Run
pytest [fail] → Write minimum implementation → Run pytest [pass] → Add
edge cases → Add property test). The "Forbidden patterns" section
copies the ADR's bulleted list. The §4.5 mutation-score table is added
as a quick reference; the ADR-043 §4.7 stub for `run_targeted` matches
this PR's signature exactly.

## Tests prove what

Specific behaviors asserted (one bullet per behavior):

- A module under `src/scieasy/qa/**` resolves to mutation-score target
  0.90; a module under `src/scieasy/ai/**` resolves to 0.70; a module
  outside the §4.5 four families resolves to `None`.
- Score arithmetic is bounded in [0, 1] for any non-negative
  `(killed, survived, timeout)` triple (property test, 200 examples).
- `(killed + survived + timeout) == total` always holds (property test,
  200 examples).
- A 100%-killed mutation run scores exactly 1.0 (property test).
- A module scoring below its §4.5 target emits exactly one
  `TQMUT-below-threshold` finding with WARNING severity.
- A module exactly at its target emits no finding.
- A module above target but BELOW baseline emits one
  `TQMUT-baseline-regression` finding with INFO severity.
- A module below target AND below baseline emits BOTH findings
  (dashboard distinguishes "stable below target" from "still falling").
- Windows platform emits exactly one `TQMUT-unavailable-platform`
  finding (INFO) and SKIPS subprocess invocation entirely.
- Missing `mutmut` binary emits exactly one `TQMUT-tool-missing`
  finding (INFO) and SKIPS the per-module loop.
- A garbled `mutmut results` payload yields a `TQMUT-parse-failed`
  finding instead of an uncaught exception.
- `mutmut run` returning rc=2 (some survived) is accepted as "produced
  results"; any other non-zero rc emits `TQMUT-run-failed`.
- `mutmut results` returning rc=2 is also accepted; non-{0,2} emits
  `TQMUT-run-failed`.
- A malformed baseline JSON file emits one `TQMUT-parse-failed`
  finding and the runner CONTINUES with an empty baseline (does not
  abort the whole evaluation).
- Multiple modules in one invocation are assessed independently — each
  gets its own threshold lookup and its own (run, results) pair.
- Both the canonical source and the `.claude/` mirror of the skill
  exist and are byte-identical.
- The skill frontmatter declares `name: test-author`,
  `priority: P0`, `allowed-tools: [Read, Write, Edit, Bash]`,
  `source_adr: ADR-043`, and a `source_section` field.
- All seven §4.4 protocol step headings are present in the body.
- The "Forbidden patterns" section is present.
- The §4.5 mutation-score table is present with all four path-family
  rows.
- The `TODO(#1145)` followup pointing at 1H sub-PR 3 is present in
  both mirrors.
