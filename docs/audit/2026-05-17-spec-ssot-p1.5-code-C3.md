## collection-transport

### `Collection` — Homogeneous ordered container of DataObjects for inter-block transport
- **Kind**: class (plain, no ABC/Pydantic)
- **Source**: `src/scieasy/core/types/collection.py:L15-L84`
- **Public?**: yes
- **Inherits / implements**: `object` (NOT a DataObject subclass — ADR-020 explicitly forbids it)
- **Signature**:
  ```python
  class Collection:
      __slots__ = ("_item_type", "_items")
      def __init__(
          self,
          items: list[DataObject] | None = None,
          item_type: type | None = None,
      ) -> None: ...
  ```
- **ClassVars / attributes**:
  - `_items: list[DataObject]` — the ordered item list (slot)
  - `_item_type: type` — immutable element type (slot); inferred from `items[0]` when not supplied; falls back to `DataObject` for the `or DataObject` branch (line 50, reached only when `items` is non-empty and `item_type` was already set — in practice `item_type` is always non-None by line 50)
- **Invariants**:
  - Empty collection without explicit `item_type` → `TypeError` (ADR-020-Add6, line 34)
  - All items must be `isinstance(item, item_type)` — homogeneity enforced at init (lines 41-47)
  - `item_type` is set once; no setter exists
- **Properties**:
  - `item_type -> type` (read-only)
  - `length -> int` (mirrors `__len__`)
  - `storage_refs -> list[Any]` — returns `[item.storage_ref for item in self._items]`; note: attribute is `storage_refs` (plural), **not** `storage_ref` — callers reading ARCHITECTURE.md should note the name
- **Methods**:
  - `__iter__() -> Any` — returns `iter(self._items)`; yields DataObject instances (not materialised native data)
  - `__len__() -> int`
  - `__getitem__(index: int | slice) -> Any` — slice path manually iterates via `index.indices(len(self._items))`; int path returns `self._items[index]` directly (no bounds check beyond list's own IndexError)
  - `__class_getitem__(cls, item_type: type) -> Any` — enables `Collection[Image]` annotation syntax; returns `cls` (not a generic alias), so runtime isinstance checks against `Collection[X]` don't work
  - `__repr__() -> str`
- **Raises**:
  - `TypeError` on empty collection without `item_type`
  - `TypeError` on heterogeneous items
- **Notes**:
  - `storage_refs` property calls `item.storage_ref` (singular) on each item — depends on `DataObject.storage_ref` existing (cross-module dep: `scieasy.core.types.base.DataObject`)
  - Port matching: `port_accepts_type()` in `blocks/base/ports.py` checks `Collection.item_type` directly when a `Collection` instance is passed — Collection is transparent to the port system (ADR-020). See `ports.py:L52-L56`.
  - `__class_getitem__` returning bare `cls` (not a `_GenericAlias`) means `Collection[Image]` can be used in type hints but **not** for `isinstance` checks — intentional limitation, not a bug.

### `LazyList` — Memory-safe list-like wrapper for Collection items in CodeBlock
- **Kind**: class (plain)
- **Source**: `src/scieasy/blocks/code/lazy_list.py:L38-L105`
- **Public?**: yes (part of CodeBlock auto-unpack API)
- **Inherits / implements**: `object`
- **Signature**:
  ```python
  class LazyList:
      __slots__ = ("_collection",)
      def __init__(self, collection: Collection) -> None: ...
  ```
- **ClassVars / attributes**:
  - `_collection: Collection` — the backing collection (slot)
- **Methods**:
  - `__iter__() -> Iterator[Any]` — yields `collection[i].view().to_memory()` one at a time via `_load_item`; O(1) peak memory per item; prior items are GC-eligible
  - `__getitem__(index: int) -> Any` / `__getitem__(index: slice) -> list[Any]` — overloaded; normalises negative int indices; raises `IndexError` on out-of-range; slice returns list of materialised objects
  - `__len__() -> int` — delegates to `len(self._collection)`, no storage access
  - `to_list() -> list[Any]` — materialises all items; emits `ResourceWarning` when count > 100 (`_ITEM_COUNT_WARNING_THRESHOLD`)
  - `__repr__() -> str`
- **Raises**:
  - `IndexError` on out-of-range int index
  - `ResourceWarning` (warning, not exception) from `to_list()` above threshold
- **Notes**:
  - Materialisation path: `_load_item(collection, i)` → `collection[i].view().to_memory()`. This means `LazyList` depends on `DataObject.view()` returning a view proxy that implements `to_memory()`. This is a cross-module dep on `scieasy.core.types.base.DataObject` (view/to_memory contract).
  - Module-level private helper `_load_item(collection, index)` — not part of public surface.
  - Module-level constant `_ITEM_COUNT_WARNING_THRESHOLD = 100` — not exported.
  - ADR-020-Add4: CodeBlock decides between single native object (length=1) vs `LazyList` (length>1) during auto-unpack. The decision logic lives in CodeBlock, not here.

## Module summary — collection-transport

**Total public interfaces**: 6 (Collection, LazyList, set_output_dir, get_output_dir, clear, plus pack/unpack/map_items/parallel_map/_auto_flush as cross-module deps on Block)

**Breakdown by kind**:
- class: 2 (Collection, LazyList)
- function: 3 in flush_context (set_output_dir, get_output_dir, clear)
- private helper: 1 (_load_item)
- module-level state: 1 (_output_dir)
- Block static methods (cross-module): 5 (pack, unpack, unpack_single, map_items, parallel_map) + 1 private (_auto_flush)

**Cross-module deps**:
- `scieasy.core.types.base.DataObject` — Collection items must be DataObject instances; LazyList materialises via `.view().to_memory()`
- `scieasy.blocks.base.block.Block` — all pack/unpack/map/flush utilities live on Block, not in this module
- `scieasy.core.storage.backend_router.get_router` — used by `_auto_flush` to find the storage extension
- `scieasy.core.storage.ref.StorageReference` — referenced in `_auto_flush` and `persist_array`/`persist_table`

**d-class hints**:
- Collection's `storage_refs` property name (plural) differs from `DataObject.storage_ref` (singular) — potential naming confusion in docs/specs
- `flush_context.py` is a module-level global singleton; it is not a class and has no concept of "context manager" scoping — the name `flush_context` implies something like a `with` block but the API is purely procedural
- `_output_dir` is not thread-safe; this is intentional (subprocess workers are single-threaded) but should be noted in specs
- `Block.pack` auto-flushes items before wrapping — this means items without `storage_ref` are written to disk as a side effect of pack(); callers must have a flush context configured or items stay in-memory silently

### `BlockRegistry` — Central catalogue of available block types
- **Kind**: class (plain)
- **Source**: `src/scieasy/blocks/registry.py:L86-L576`
- **Public?**: yes
- **Inherits / implements**: `object`
- **Signature**:
  ```python
  class BlockRegistry:
      def __init__(self) -> None: ...
  ```
- **ClassVars / attributes** (instance):
  - `_registry: dict[str, BlockSpec]` — keyed by display name (`spec.name`)
  - `_aliases: dict[str, str]` — maps `spec.type_name` → `spec.name`
  - `_scan_dirs: list[Path]` — Tier 1 scan directories
  - `_packages: dict[str, PackageInfo]` — package metadata keyed by package name (only Tier 2 / monorepo packages that provide `PackageInfo`)

**Public methods**:

#### `scan`

  Calls `_scan_builtins()` → `_scan_tier1()` → `_scan_tier2()` → optionally `_scan_monorepo_packages()`. Order matters: builtins first, monorepo last (monorepo skips already-registered names).

#### `hot_reload`
  ```python
  def hot_reload(self) -> None: ...
  ```
  Removes stale Tier 1 entries (deleted files) then re-runs `_scan_tier1()`. Does not re-scan Tier 2 or builtins.

#### `get_spec`
  ```python

  ```
  Resolves by display name first, then by `type_name` alias. Returns `None` if not found.

#### `instantiate`
  ```python
  def instantiate(self, name: str, config: dict[str, Any] | None = None) -> Any: ...
  ```
  Resolves spec, re-imports the module (mtime-aware for Tier 1), stamps `_scieasy_file_path` on class, returns `cls(config=config)`. Raises `KeyError` when name not registered, `ImportError` when module cannot be loaded.

#### `packages`
  ```python
  def packages(self) -> dict[str, PackageInfo]: ...
  ```
  Returns copy of `_packages`. Only includes packages that provided `PackageInfo` via `(PackageInfo, list)` return convention.

#### `specs_by_package`
  ```python
  def specs_by_package(self) -> dict[str, list[BlockSpec]]: ...
  ```
  Groups all specs by `package_name`. Builtins and Tier 1 blocks (no package name) appear under key `""`.

#### `all_specs`
  ```python
  def all_specs(self) -> dict[str, BlockSpec]: ...
  ```
  Returns copy of `_registry`.

**Private methods**:

#### `_scan_builtins`
  ```python
  def _scan_builtins(self) -> None: ...
  ```
  Imports and registers: `LoadData`, `SaveData`, `AIBlock`, `SubWorkflowBlock`. Source tag `"builtin"`. Base classes and non-functional blocks (Merge, Split, etc.) are explicitly excluded from the palette.

#### `_scan_tier1`
  ```python
  def _scan_tier1(self) -> None: ...
  ```
  Iterates `_scan_dirs`, globs `*.py` (skips `_`-prefixed), loads via `importlib.util.spec_from_file_location` with synthetic mtime-stamped module name. Stamps `_scieasy_file_path` on concrete non-abstract Block subclasses defined in that file (via `__module__` guard — avoids re-registering imported base classes). Catches all exceptions per-file.

#### `_scan_tier2`
  ```python
  def _scan_tier2(self) -> None: ...
  ```
  Reads `scieasy.blocks` entry-points. Each entry-point callable may return:
  - `(PackageInfo, list[type[Block]])` — preferred form
  - `list[type[Block]]` — backward-compatible plain list
  - A direct `type[Block]` class (legacy fallback)
  
  Abstract Block subclasses are skipped with a warning. Bad entry-points are logged and skipped.

#### `_scan_monorepo_packages`
  ```python
  def _scan_monorepo_packages(self) -> None: ...
  ```
  Dev fallback. Globs `packages/scieasy-blocks-*` under repo root, adds `src/` to `sys.path`, imports module, calls `get_block_package()` (preferred) or `get_blocks()`. Skips already-registered blocks. Only runs when `include_monorepo=True`.

#### `_register_spec`
  ```python
  def _register_spec(self, spec: BlockSpec) -> None: ...
  ```
  Registers under `spec.name`; also registers `spec.type_name` → `spec.name` alias when `type_name` is non-empty.

#### `_validate_dynamic_ports` (staticmethod)
  ```python
  @staticmethod
  def _validate_dynamic_ports(cls: type) -> None: ...
  ```
  Validates that `cls.dynamic_ports` is `None` or a dict with `source_config_key` (non-empty str) and exactly one of `output_port_mapping` or `input_port_mapping` (not both). Each mapping must be `{port_name: {enum_value: [TypeName, ...]}}`. Raises `ValueError` on violation.

- **Raises**:
  - `KeyError` (from `get_spec` path in `instantiate`) when block not registered
  - `ImportError` when Tier 1 module cannot be loaded at instantiation time

#### `_validate_meta_class`

- **Raises**:

**d-class hints**:
- `BlockSpec.source` is a de-facto `Literal["builtin", "tier1", "entry_point", "monorepo"]` but typed as plain `str` — callers doing `spec.source == "tier1"` string comparisons risk silent drift if a new source string is added
- All 14 core `scieasy.blocks` entry-points point at classes directly, not callables returning `(PackageInfo, list)`, so the `_packages` dict is always empty for core blocks — this means `packages()` and `specs_by_package()` won't show named package metadata for builtins
- `BlockSpec.input_ports` / `output_ports` are typed `list[Any]` but hold `InputPort` / `OutputPort` instances — loose typing avoids circular imports but hides the actual contract
- `TypeRegistry` has no `has(name: str) -> bool` method — callers must catch `KeyError` from `resolve(str)` or call `name in all_types()`
- `TypeSpec.base_type` records only the immediate parent's `__name__` (one level of MRO), not the full type chain — the full `type_chain` is reconstructed at serialisation time from `DataObject.dtype_info.type_chain`, not from `TypeSpec`
- `_resolve_distribution_version` makes `scieasy.__version__` the fallback for all in-repo and monorepo blocks — if the `scieasy` package is not properly installed, the fallback chain reaches the final `BlockRegistrationError` raise, potentially breaking all block registrations in a dev environment
- `BlockTestHarness.smoke_test` does not auto-wrap raw data into Collections — the dispatch prompt note "auto-wrapping" in the task description is inaccurate; no such wrapping exists in the current implementation
