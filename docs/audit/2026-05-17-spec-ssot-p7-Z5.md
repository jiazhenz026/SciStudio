## 5. Cross-cutting observations

### CC-1: SSOT confuses SQL column type with Python dataclass type

Multiple ¬ß8 entries (RunRecord, BlockExecutionRecord, DataObjectRow) type Python dataclass fields as `str` because the SQL column is TEXT-encoded JSON. In code, the dataclass holds the *Python object* (dict, bool) and serialization to TEXT happens inside `store.py.insert_*()` via `json.dumps()`. Phase 8 should split documentation: **table column types** (TEXT) vs **dataclass field types** (dict/bool/etc.).

### CC-2: SSOT method names are aspirational ("put_/get_") while code uses "insert_/upsert_/list_/finalize_"

The `LineageStore` API mismatch is the largest single drift in ¬ß8. Either Phase 8 reground SSOT to current code OR Phase 9 ships a rename PR. The current SSOT cannot be extracted as a CI-checkable contract.

### CC-3: SSOT GitEngine surface is aspirational

The 22-method enumeration was useful as a discovery scaffold (it forced auditors to enumerate the API surface) but the *signatures* shown are an idealized API that doesn't exist. Code returns dicts where SSOT returns dataclasses; code accepts `*, kw=None` patterns where SSOT shows positional. Phase 8 must rewrite from `git_engine.py` literally.

### CC-4: 4 SQL CREATE TABLE statements are the **strongest part** of ¬ß8

These are the contract that machine-extractable validation can rely on TODAY. They are correctly enumerated, with correct types/constraints/FKs. Phase 8 should preserve them and consider them the gold-standard pattern for other contracts.

### CC-5: c-drop entries (`git_author.json`, `watcher.py`) are correctly identified

Both confirmed by directory listing + grep. Phase 8 doc-side fixes (PROJECT_TREE.md + ARCH ¬ß4.6 + ARCH ¬ß10) are uncontroversial.

### CC-6: ARCH ¬ß5.3 anchor is wrong for LineageRecorder

SSOT cites `ARCHITECTURE.md ¬ß4.4, ¬ß5.3 (EventBus subscription matrix)`. The matrix is at ¬ß6.1 (verified L1929). ¬ß5.3 is "Block categories". Phase 8 must fix.
