### `lineage-db/LineageRecorder` — event-driven lineage writer
- **Kind**: class
- **Source**: `src/scieasy/core/lineage/recorder.py:L63-L360`
- **Public?**: yes
- **Inherits / implements**: none
- **Signature**:
  ```python
  class LineageRecorder:
      def __init__(
          self,
          event_bus: EventBus,
          lineage_store: LineageStore | None,
          run_id: str,
      ) -> None: ...

      @property
      def run_id(self) -> str: ...

      def record_start(self, block_id: str) -> None: ...
      def begin_run(self, run: RunRecord) -> None: ...
      def finalize_run(self, *, status: str) -> None: ...
      def dispose(self) -> None: ...
      async def _on_terminal(self, event: EngineEvent) -> None: ...
  ```
- **ClassVars / attributes**:
  - `_event_bus: EventBus`
  - `_store: LineageStore | None`
  - `_run_id: str`
  - `_start_times: dict[str, datetime]` — captures block start time for duration calculation
  - `_exec_ids: dict[str, str]` — `block_id → block_execution_id` dedup map
  - `_subscribed: bool`
- **Notes**:
  - Subscribes to `"block_done"`, `"block_error"`, `"block_cancelled"`, `"block_skipped"` on construction.
  - All DB errors are caught and logged as warnings (non-fatal). A lineage outage cannot break a workflow.
  - `dispose()` unsubscribes from all 4 events; is idempotent. Must be called from `ApiRuntime._finalize_lineage_run` to prevent fan-out accumulation across multiple workflow runs.
  - `_on_terminal` is an `async` method; DB writes are synchronous (SQLite is serialized internally).
  - Event-type → termination mapping: `block_done="completed"`, `block_error="error"`, `block_cancelled="cancelled"`, `block_skipped="skipped"`.

**Wire-payload → row translation:**

`_on_terminal` reads from `event.data`:
- `data["block_type"]`, `data["block_version"]`, `data["config"]` → `BlockExecutionRecord`
- `data["input_object_ids"]` / `data["output_object_ids"]` — `{port_name: [object_id, ...]}` already-unrolled Collections → `block_io` rows
- `data["inputs"]` / `data["outputs"]` — raw wire-format `{port_name: <dict|list>}` → `data_objects` rows

**Collection unrolling:** `_wire_items_for_port()` checks for `value.get("_collection") == True`; if so, flattens `value["items"]` list. Otherwise returns `[value]` for a single item. This produces the per-item rows that populate `block_io.position`.

**`wire_payload` JSON shape in `data_objects`:** The full wire dict per item, structured as:
```json
{
  "metadata": {
    "framework": {
      "object_id": "<uuid>",
      "derived_from": "<uuid>|null",
      ...
    },
    "type_chain": ["DataObject", "Image"]
  },
  "backend": "zarr",
  "path": "<storage_path>",
  ...
}
```
For Collections the root wire dict has shape `{"kind": "collection", "item_type": "Image", "items": [...]}`.

### Module summary — lineage-db

**Classification recommendation**: **a** (well-defined, stable, fully implemented)

The 4-table SQLite schema is fully specified and matches ADR-038 §3.1. The `LineageStore` write/read API is complete and public. Row dataclasses mirror tables precisely. `LineageRecorder` handles event subscription, start-time capture, Collection unrolling, `wire_payload` extraction, and `dispose()` idempotency. `EnvironmentSnapshot.capture()` is implemented with `uv pip freeze` / `pip freeze` fallback. `render_methods_markdown()` is implemented and covers all 4 ADR-038 §3.7 questions. `RunContext` ContextVar is implemented.

**Key gaps / notes for spec comparison:**
- No re-run validation methods (input size/mtime drift, env drift) are implemented in this module. ADR-038 §3.6a mentions this; no code exists for it in `lineage/`.
- No `ancestors()` / `descendants()` methods on `LineageStore` itself — that logic lives in the `MetadataStore` shim (delegate). The spec may describe these as `LineageStore` methods.
- `size_bytes` and `mtime_at_write` columns exist on `data_objects` but are never populated by `LineageRecorder._persist_io_entry()` — these fields are `None` in practice (only `DataObjectRow` dataclass carries them as optional).
- `conda_env` field on `EnvironmentSnapshot` is reserved but never set by `capture()`.
- `DB_PATH_CONSTANT` is not a named module constant; it is inline in `__init__`.

      def __init__(self, path: Path) -> None: ...
      # Attributes:
      path: Path         # absolute path to git executable
      version: str | None  # cached "git --version" string (best-effort)

      @classmethod

          # 1. SCIEASY_GIT_BUNDLE_ROOT env var override
          # 2. PyInstaller: sys.executable.parent (+ ../Resources on macOS)

          args: list[str],
          *,
          cwd: Path | None = None,
          check: bool = True,

### Module summary — versioning-git

**Classification recommendation**: **d** (public API documented only in docstrings; not in ARCHITECTURE.md)

The S1 flag is confirmed. `GitEngine` has 20+ public methods, all of which are documented only via inline docstrings and ADR-039 section/line references scattered across the codebase. There is no dedicated reference page in `docs/architecture/` or `docs/specs/` that enumerates the full method list, parameter names, and return shapes in one place — reviewers must reconstruct the API from the source.

**Additional gaps / notes for spec comparison:**
- **`git_author.json` does not exist.** The dispatch prompt and possibly existing specs describe a `git_author.json` identity cache. No such file, class, or module is present in `src/scieasy/core/versioning/`. Identity is hardcoded as `_DEFAULT_AUTHOR_NAME = "SciEasy User"` / `_DEFAULT_AUTHOR_EMAIL = "noreply@scieasy.local"`. This is a spec-vs-code delta that the comparison phase must flag.
- **`GitTimeoutError` does not exist.** `subprocess.TimeoutExpired` is not wrapped.
- **`author_cache.py` does not exist.** The glob in the dispatch prompt lists it as a potential file; it is absent.
- **`no_git` file path constant** is inlined in `runtime.py`, not in the versioning package.
- **`GitChangeWatcher`** was collapsed into `scieasy.api.routes.workflow_watcher._GitHeadHandler` per D39-3.2 (#968). The `versioning/__init__.py` notes this but the watcher is no longer part of the versioning package's public surface.
- **`SCIEASY_GIT_BUNDLE_ROOT`** env var override is the only runtime configuration hook for the binary path (plus the standard bundle detection and `shutil.which` fallback).
- **Bundle path lookup priority**: `SCIEASY_GIT_BUNDLE_ROOT` > PyInstaller `sys.executable` parent > developer checkout `<repo>/desktop/` > `shutil.which("git")`.
