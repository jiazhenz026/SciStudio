### F12.2 [P1] `MCPContext` Protocol gap — b-code-wins decision confirmed, but Protocol doc comment is MISLEADING

**Severity**: P1  
**Finding**: The `MCPContext` Protocol in `_context.py` declares exactly 3 structural members:
- `block_registry: BlockRegistry`
- `type_registry: TypeRegistry`
- `project_dir: Path | None` (property)

Code across the four tool modules accesses 4 additional attributes via `getattr` fallback:
1. `scheduler` — not declared in Protocol
2. `event_bus` — not declared in Protocol; accessed via `getattr(ctx, "event_bus", None)` in `tools_workflow.py:79`
3. `workflow_runs` — not declared in Protocol; accessed via `getattr(ctx, "workflow_runs", None)` in `tools_workflow.py:655, 700, 403`
4. `ai_block_run_dir` — not declared in Protocol; accessed via `getattr(ctx, "ai_block_run_dir", None)` in `tools_workflow.py:776`

The draft v1 §12 decision "b-code-wins — Code is current behavior" is factually correct: code does use `getattr` defensively and the Protocol is intentionally minimal (as stated in `_context.py:29` "The Protocol is structural: callers only need attributes / methods the tools actually reach for").

However, the `_context.py` module docstring says "T-ECA-202..205. The 25 MCP tools need to talk..." — the tool count in the docstring is wrong; there are 26 tools. This is a minor stale reference.

The **real P1** is that the `MCPContext` Protocol docstring says "Production: implemented by `scieasy.api.runtime.ApiRuntime`" but gives no guidance that `workflow_runs`, `ai_block_run_dir`, `event_bus`, and `scheduler` must be present on `ApiRuntime` for the write-class tools to function. An agent-provisioning unit test with a minimal stub would silently drop these attributes and cause silent failures on `cancel_run`, `get_run_status`, `get_block_output`, and `finish_ai_block`.

**Required fix for SSOT**: The Phase 5 manager-fix must document that:
> "The Protocol declares the MINIMUM surface. `ApiRuntime` additionally provides `workflow_runs: dict`, `event_bus`, `scheduler`, `ai_block_run_dir` — accessed via `getattr` fallback in the tool implementations. Test stubs that implement only the 3-attribute Protocol will silently break the 4 write-execution tools."

This is a documentation gap, not a code bug. No code change needed; the Phase 6 SSOT entry for `MCPContext` must enumerate all 7 accessed attributes and their source.

The `TODO(#1012)` in `runtime.py:69-74` correctly tracks the `ai_block_run_dir` gap on `StandaloneMCPRuntime`. Protocol is acceptable as-is per b-code-wins; the SSOT must document the full access surface.

## Cross-cutting observations

### OBS-1: `hook_deny_scieasy_cli.py` regex coverage

The regex `r"^\s*(?:\S*/)?scieasy(?:\s|$)"` correctly anchors at the start of the command string. It will match `scieasy list-blocks` and `/path/to/scieasy list-blocks`. It will NOT match `python -c 'import scieasy'` (acceptable — Python-level imports are not CLI invocations). Edge case: `SCIEASY=1 python foo.py` is not matched (correct — setting env vars is not a CLI call). This is an appropriate heuristic per ADR-040 §3.6 defense-in-depth framing.

### OBS-2: `update_block_config` categorized under `inspection` not `authoring`

`update_block_config` carries tag `category:inspection` (not `category:authoring`). This is a deliberate design choice (it modifies the workflow YAML, not a block source file). The SKILL.md static fallback lists it under Inspection. Consistent.

### OBS-3: `run_block_tests` is tagged `write` in authoring

Correctly classified as `write` since it invokes a subprocess. The subprocess path does not go through the MCP validated write path — it calls pytest directly. This is acceptable for local test feedback. The distinction is `write` in the MCP mutation sense, not the filesystem sense. No issue.

### OBS-4: `finish_ai_block` — `FinishAIBlockError` union return type

`finish_ai_block` returns `FinishAIBlockOK | FinishAIBlockError` (discriminated by `status` field). FastMCP will serialize whichever branch the runtime selects. The error branch uses `status="error"` as discriminator. The SSOT should enumerate both return shapes, not just the OK path.
