## Notes on judgment calls

### 1. Authority hierarchy application
Where ARCHITECTURE.md is silent and only ADRs speak, ADR-aligned + code-aligned = a + note. Example: LineageStore methods are enumerated in ADR-038 but not ARCHITECTURE.md §4.4 body; however ARCHITECTURE.md §4.4 shows the SQL schema which is the structural contract, so full a-class is appropriate.

### 2. ProcessBlock setup/teardown
Row 5 is b-class because ARCHITECTURE.md §6.2 pseudocode (primary authority) is stale, even though §5.1 (also ARCHITECTURE.md, same doc) correctly documents the 3-arg lifecycle. Within-ARCH inconsistency treated as b per classification rules.

### 3. Block.allowed_input_types default
Row 4: `[DataObject]` vs `[]` is classified b because ARCH is PRIMARY and it explicitly shows `[DataObject]` with `# no constraint` comment. Code comment says "empty = accept any DataObject subclass" — semantically equivalent but form differs. Front-end/tools reading the default value will see different behavior. This is a real disagreement.

### 4. versioning-git classified as d overall (with one c)
All GitEngine method signatures, error types, and binary resolution are code-only. The single c-class item is `git_author.json` — docs describe a file that does not exist.

### 5. rest-api endpoints
Classifying the REST endpoint surface as b (row 138) rather than splitting into a/d per route, to keep the table compact. The a-subset (documented routes) and d-subset (undocumented routes) are clear from xcheck, but a single row per module-level grouping is appropriate given the scope.

### 6. mcp-tools finish_ai_block
Classified c (row 149) because the interface is fully implemented in code and the docs actively omit it (ADR-035 §3.5 path a is documented in an ADR but ARCHITECTURE.md §7.2 category table was not updated).

### 7. agent-provisioning module itself
Classified d (row 166) because the `agent_provisioning` Python package is not listed in ARCHITECTURE.md's layer diagram or PROJECT_TREE.md module inventory, even though §10.2 describes its behavior. The interface exists in code without a matching architectural registration in the primary authority document.
