### ws-sse-protocol

**WS endpoints**: 2 — `/ws` (bidirectional engine+ai_pty, any origin) and `/api/ai/pty/{tab_id}` (PTY bidirectional).

**Internal HTTP (IPC)**: 2 — `POST /api/ai/pty/internal/request-tab` and `POST /api/ai/pty/internal/notify`. Both protected by `SCIEASY_ENGINE_IPC_TOKEN` header.

**SSE endpoints**: 1 — `GET /api/logs/stream`.

**Outbound WS message types** (server→client on `/ws`):
- Via EventBus: `block_ready`, `block_running`, `block_paused`, `block_done`, `block_error`, `block_cancelled`, `block_skipped`, `workflow_started`, `workflow_completed`, `interactive_prompt`, `workflow.changed`, `blocks.reloaded`, `git.head_changed`
- Via ai_pty broadcaster (bypasses EventBus): `block_pty_opened`, `block_pty_closed`

**Inbound WS message types** (client→server on `/ws`):
- `cancel_block` → CANCEL_BLOCK_REQUEST engine event
- `cancel_workflow` → CANCEL_WORKFLOW_REQUEST engine event
- `interactive_complete` → INTERACTIVE_COMPLETE engine event
- `block_user_marked_done` → writes `signals/mark_done.json` file
- `block_user_cancel` → writes `signals/mark_done.json` file (same as marked_done; full cancel propagation deferred)

**PTY message protocol** (`/api/ai/pty/{tab_id}`):
- Frame format: JSON text frames (not binary)
- Client→Server: `{"type": "stdin", "data": str}`, `{"type": "resize", "cols": int, "rows": int}`
- Server→Client: `{"type": "stdout", "data": str}`, `{"type": "exit", "code": int}`, `{"type": "error", "message": str}`

**EventBus → WS bridge pattern**: `ws.py` subscribes `_on_event` callback for each event type in `_OUTBOUND_EVENTS`. On engine event, `serialise_event()` produces `{type, block_id, workflow_id, data, timestamp}` and pushes to `asyncio.Queue`. Outbound loop drains the queue. The `block_pty_opened`/`block_pty_closed` messages bypass this path — they are pushed directly to the same queue via `_on_ai_pty_message`.

**TypeScript WS counterpart**: `frontend/src/types/api.ts:L216-L222` — `WorkflowEventMessage`. This type covers the EventBus-originated messages. The `block_pty_opened`/`block_pty_closed` messages have no dedicated TS type — they are likely handled as `WorkflowEventMessage` with `type` discriminated at runtime.

**Notable design points**:
- The ai_pty broadcaster (`_ai_pty_subscribers` set in `ai_pty.py`) is a second pub/sub layer that bypasses EventBus to avoid adding new event types to `engine/events.py` (per ADR-035 hard-scope constraint). The WS handler registers one subscriber per connection.
- `block_user_cancel` and `block_user_marked_done` both write the same signal file (`mark_done.json`) — full CANCEL semantics are deferred. This is noted as a follow-up in `ws.py:L248-L254`.
- `INTERACTIVE_COMPLETE` events do NOT have a WS outbound forwarding path — they are pure inbound. This is correct; the engine handles them internally.
- `PROCESS_SPAWNED`, `PROCESS_EXITED`, `CHECKPOINT_SAVED` are NOT forwarded to the WS client — internal engine events only.
