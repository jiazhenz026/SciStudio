### 5.1 `collection-transport.Collection` — Status: a → **DRIFT**

**SSOT signature** vs **actual code** (`src/scieasy/core/types/collection.py`):

| Aspect | SSOT claim | Actual code | Severity |
|---|---|---|---|
| `__slots__` | `("items", "item_type")` | `("_item_type", "_items")` (underscore-prefixed) | **P1** — port-matching invariant relies on this name; downstream code reading the SSOT will write to the wrong attribute |
| `items: list[DataObject]` (public field) | Listed as field | Exposed only via `__iter__`/`__getitem__`; no `.items` attribute. The slot is `_items`. | **P1** |
| `item_type: type` (public field) | Listed as field | Read-only `@property` (L52-55), backed by `_item_type` slot | **P2** — semantically the same but contract is "property" not "field" |
| `storage_refs()` method | Function signature `def storage_refs(self) -> list[StorageReference]` | `@property` returning `list[Any]` (L80-83); NO `StorageReference` type used | **P1** — caller code written from SSOT will do `c.storage_refs()` and get `TypeError: 'list' object is not callable` |
| `__getitem__(self, index: int)` | int only | Accepts `int \| slice`; returns `list` for slice (L68-71) | **P2** |
| `__class_getitem__` | Documented | Confirmed (L73-75) | OK |
| `__init__` invariants | NOT documented at all | ADR-020-Add6: empty Collection without `item_type` raises `TypeError`; homogeneity validated; auto-infers `item_type` from first item | **P2** — a load-bearing invariant is silently missing |
| Missing helpers | — | `length` property (L57-60), `__repr__` (L77-78), `__init__` constructor signature | **P3** |

**Verdict**: P1 drift. The §5 module preamble brags about "verified by C3 — uses `__slots__`" but then writes the SSOT slot names wrong.

**Fix direction**: rewrite this entry. The contract is:
```python
class Collection:
    __slots__ = ("_item_type", "_items")
    def __init__(self, items: list[DataObject] | None = None, item_type: type | None = None) -> None: ...
    @property
    def item_type(self) -> type: ...
    @property
    def length(self) -> int: ...
    @property
    def storage_refs(self) -> list[Any]: ...
    def __getitem__(self, index: int | slice) -> DataObject | list[DataObject]: ...
    def __iter__(self) -> Iterator[DataObject]: ...
    def __len__(self) -> int: ...
    def __class_getitem__(cls, item_type: type) -> type: ...
```

## Cross-cutting observations

1. **§6 module preamble is too confident about C3 verification**. The preamble says "14 of 14 core scieasy.blocks entry-points are DIRECT CLASS REFERENCES, not (PackageInfo, list[Block]) callables — BlockRegistry._packages is therefore always {} for core blocks." This is correct, but the same confidence in C3 verification didn't extend to actually re-reading `registry.py` to capture BlockSpec's real field list or BlockTestHarness's real method list. Suggests C3 audit was selective.

2. **Source path collision between `registry.py` and `base/spec.py`**. SSOT was probably written assuming a planned refactor that hasn't happened — BlockSpec was perhaps meant to be extracted to `base/spec.py` (next to `package_info.py`). It hasn't. SSOT must match the current repo, not the planned refactor. Either (a) update SSOT to point at `registry.py:L42-L83`, or (b) open an issue for the extraction and TODO-tag both sides.

3. **BlockTestHarness sub-label `d-document` was probably the wrong call**. The class has 4 public methods (`validate_block`, `validate_package_info`, `validate_entry_point_callable`, `smoke_test`). All four are well-tested and well-docstring'd in the source. ARCH §12 just references ADR-026 instead of enumerating them. The right action is "lift the actual class contract from the source verbatim into SSOT and reference back from ARCH §12", not "imagine a `run()` method that auto-wraps." `d-document` is fine as the sub-label; the SSOT entry needs to be rewritten to actually document what's there.

4. **§5 cross-listed entries are a useful pattern but need real line ranges**. The cross-listing convention works in principle (avoids duplication; keeps module surface complete) but L268-L332 truncates the cross-referenced symbol set. Either widen the range or use `multi-range: L268-L332 + L437` notation.

5. **`source` field is `str` not `Literal`** — SSOT's b-code-wins call is correct, but worth noting that adding `Literal["builtin", "tier1", "entry_point", "monorepo"]` will cascade through `_spec_from_class` (line 800, `source=source` passed as a kwarg) and every test fixture that constructs `BlockSpec(source="something")`. Open a tracking issue before changing.
