### Per-entry findings — Module 3

#### F3-01 — CompositeData `slot_types()` sub-label needs additional note (P2)

**Entry**: `data-types.CompositeData` — `b-code-wins`
**Issue**: The draft correctly calls this b-code-wins because `slot_types()` returns `dict(self.expected_slots)` (schema-declared types) rather than `{k: type(v) for k, v in self._slots.items()}` (per-instance actual types) as D1 describes. However, the draft's key-decision note (Module 3 "Key decisions") says only `slot_types() returns declared schema not per-instance` without capturing the full behavioral consequence: **if a slot was set with a proper subtype (e.g. `FluorImage` where `expected_slots` says `Array`), `slot_types()` will report `Array`, not `FluorImage`**. This is a semantically non-trivial discrepancy for port-constraint callers.

**Code truth** (`composite.py:L66-L68`):
```python
def slot_types(self) -> dict[str, type]:
    """Return the expected slot-type mapping declared on this class."""
    return dict(self.expected_slots)
```

**Recommended fix**: Phase 5 should add a behavioral note in the SSOT entry: "`slot_types()` returns the **declared schema** (`expected_slots` ClassVar copy), not the types of currently-stored instances. Port constraint callers must use `expected_slots` for schema checks and `type(self.get(name))` for runtime type."

**Severity**: P2 — behavioral clarification needed; label is correct.

### Per-entry findings — Module 4

#### F4-01 — StorageReference label conflict: draft says `b-docs-wins`, M2 says `b-code-wins` (P1)

**Entry**: `storage-backends.StorageReference`
**Issue**: This is a material label conflict between the draft and M2.

- **Draft v1 Module 4 key decisions**: "StorageReference: b-docs-wins (C2: docstring calls it 'immutable' but dataclass NOT `frozen=True`. Fix is code-side: add `frozen=True` to match documented intent)"
- **M2 §4** (M2:L968-L983): Sub-label is `b-code-wins`. M2 rationale: "Not `frozen=True` despite docstring saying 'immutable'; `format` values in practice are `"zarr"/"parquet"/"composite"/"filesystem"` not `"zarr/arrow/file"` per D1. Code wins."

These are OPPOSITE labels. `b-docs-wins` means code should be fixed to match docs (add `frozen=True`). `b-code-wins` means docs should be updated to remove the "immutable" claim.

**Code truth** (`ref.py:L9-L27`):
```python
@dataclass
class StorageReference:
    backend: str
    path: str
    format: str | None = None
    metadata: dict[str, Any] | None = field(default=None)

    def __post_init__(self) -> None:
        self.path = self.path.replace("\\", "/")
```

The dataclass is NOT `frozen=True`. The `storage_ref` setter on `DataObject` (`base.py`) actively writes to `_storage_ref`. Making `StorageReference` frozen would break the setter pattern. C2's notes (C2:L582) confirm: "Not `frozen=True` — `storage_ref` setter on `DataObject` writes it."

**Analysis**: Making StorageReference `frozen=True` is NOT safe without also removing the `DataObject.storage_ref` setter, which is a larger architectural change. The code pattern (mutable ref, writable setter) is intentional. M2's `b-code-wins` is the correct label. The draft's `b-docs-wins` (directing a code fix) conflicts with M2 and is architecturally wrong.

**Recommended fix**: Phase 5 must change label from `b-docs-wins` to `b-code-wins`. The fix is docs-side: update ARCH/docstring to remove "immutable" claim, describe `StorageReference` as "lightweight pointer" without the immutability guarantee. Also note the `metadata: dict[...]` field is NOT deep-copied (mutation possible — C2:L582).

**Severity**: P1 — label is wrong direction (docs vs code), creates conflicting fix instructions.

#### F4-02 — CompositeStore and its `iter_chunks` / `write` non-atomic: label inconsistency (P1)

**Entry**: `storage-backends.CompositeStore`
**Issue**: The draft key-decisions say:
- "CompositeStore.iter_chunks: b-docs-wins (silently ignores `chunk_size`, yields `(slot_name, data)` tuples — semantically inconsistent with other backends; fix code-side OR document the exception)"
- "CompositeStore.write non-atomic: b-docs-wins (manifest written last — fix code-side or document the gap)"

But the roster entry for CompositeStore in the draft shows label `a` — these b-findings are in the key-decisions prose but NOT reflected in the entry's label.

**Code truth** (`composite_store.py:L150-L163`):
```python
def iter_chunks(self, ref: StorageReference, chunk_size: int) -> Iterator[Any]:
    """Yield slots one at a time from the composite at *ref*."""
    # chunk_size parameter is accepted but ignored
    for slot_name, slot_info in manifest["slots"].items():
        ...
        yield slot_name, backend.read(slot_ref)
```

The `chunk_size` parameter is silently ignored. The docstring says "Yield slots one at a time" — not "yield fixed-size chunks" like other backends. This IS a semantic inconsistency with `StorageBackend` Protocol intent. C2 confirms this (C2:L735): "chunk_size silently ignored" and "iter_chunks semantics differ from other backends."

The write non-atomicity is explicitly documented in code (`composite_store.py:L63-L75`) with `.. warning::` and `.. todo::`.

**Analysis**: CompositeStore should be labeled `b-docs-wins` (or `b-code-wins` depending on resolution) not `a`. The two specific sub-behaviors (iter_chunks semantics + non-atomic write) are real discrepancies. Since the non-atomic write has `.. warning::` in code, and since fixing `iter_chunks` semantics would be a behavioral change affecting correctness, both are `b-docs-wins` (document the divergence explicitly in SSOT spec, don't silently omit). This is what the key-decisions prose already says — but the label `a` needs to change to `b`.

**Recommended fix**: Phase 5 change CompositeStore entry from `a` to `b-docs-wins`. The Phase 6 SSOT entry must document:
1. `iter_chunks` yields `(slot_name, slot_data)` tuples, NOT sized chunks; `chunk_size` is ignored.
2. `write` is NOT atomic; manifest written last; a crash mid-write leaves partial output.
Both are code behaviors the spec must accurately describe.

**Severity**: P1 — the label `a` contradicts the key-decisions prose and misrepresents a known behavioral gap.

**Recommended fix**: Phase 6 SSOT must source from M2/C2's 6-method signature. Phase 5 add a note: "Phase 6 source: C2 verbatim, NOT D1. D1's 4-method summary is outdated relative to ADR-020 Add5."

**Severity**: P2 — not a label error but a source-of-record ambiguity that could silently produce an incomplete SSOT.

### Missing entries — Module 4

#### M4-01 — ArrowBackend missing atomicity-asymmetry behavior note (P3)

**Entry**: `storage-backends.ArrowBackend` — `a`
The draft and M2 both mark ArrowBackend as `a` and note "Atomic write (caveat)" under CompositeStore but do NOT explicitly note that ArrowBackend is the only backend WITHOUT atomic write. ZarrBackend uses `tempfile.mkdtemp` + `Path.rename()`. FilesystemBackend uses `tempfile.mkstemp` + `os.replace()`. ArrowBackend calls `pq.write_table(table, ref.path)` directly — no temp/rename pattern.

C2 calls this out (C2:L663): "No atomic write (unlike ZarrBackend and FilesystemBackend)." This asymmetry is load-bearing: a crash during `ArrowBackend.write()` can corrupt or truncate the Parquet file.

**Recommended fix**: Phase 5 add a d-document sub-note to the ArrowBackend entry: "Write is NOT atomic — direct `pq.write_table()` call. ZarrBackend and FilesystemBackend both use temp+rename patterns; ArrowBackend does not. Downstream crash resilience for DataFrame outputs relies on re-running the block (ADR-018 checkpoint semantics), not atomic write guarantees."

**Severity**: P3 — behavioral omission that could mislead spec readers about durability guarantees.
