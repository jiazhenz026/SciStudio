### §3.1 `data-types.DataObject` — base class

- SSOT Source: `src/scieasy/core/types/dataobject.py:L1-L1`
- SSOT Status: `a`
- Actual: file is `src/scieasy/core/types/base.py` (lines 141–553). There is **no `dataobject.py`** in the repo (see `Glob src/scieasy/core/types/*.py`).

**Findings:**
- **P1 (Source path broken).** SSOT references a non-existent file. The class lives in `base.py`. The whole §3 section that points to `dataobject.py` (3.1, 3.2) inherits this break.
- **P1 (Docstring lies about the slot count).** SSOT entry's docstring says *"5-slot model (ADR-027 D5):"* and lists `_framework / _meta / _user / _storage_ref / _transient_data` as the 5 slots. ADR-027 D5 itself is the **three-slot** stratification (framework / meta / user). The actual class docstring at `base.py:141-161` correctly calls it the three-slot model and treats `_storage_ref` + `_transient_data` as separate concerns. SSOT's "5-slot" framing is a hallucination layered on top of ADR-027.
- **P1 (`to_memory` signature drift).** SSOT shows `to_memory(self) -> Any` with the *exact* docstring "Emits a 2GB size warning...". Actual `base.py:401-428` matches the behaviour but emits `ResourceWarning`, raises `ValueError` when `storage_ref is None`, and is documented to do so. SSOT does not mention the `ValueError`. Minor signature shape drift, **P2**.
- **P2 (`save` signature drift).** SSOT shows `save(self, path) -> None`. Actual `base.py:475-492` is `save(self, path: str | Path) -> StorageReference`. Both the type annotation and the return type are wrong in SSOT.
- **P2 (`with_meta` raises on `meta is None`).** SSOT docstring does not mention this; actual `base.py:312-319` raises `ValueError` when `self._meta is None`. Material contract omission.
- **P3 (`iter_chunks` docstring).** SSOT promises "Subclasses (Array) override with axis-aware chunking" — but `Array` does NOT override `iter_chunks` (only `to_memory` and `with_meta`). SSOT description is aspirational, not what the code does.

### §3.2 `data-types.DataObject._transient_data` (and `_data` / `_arrow_table` shadows)

- SSOT Source: `src/scieasy/core/types/dataobject.py:L1-L1`
- SSOT Status: `d` (document the deprecated forwarders)
- Actual: `base.py:213-217` declares `_transient_data` slot; `base.py:369-397` declares the `_data` / `_arrow_table` property shadows.

**Findings:**
- **P1 (Source path).** Same `dataobject.py` ghost; the slot lives in `base.py`.
- **P1 (Drift on shadow property behaviour).** SSOT docstring quotes:
  ```python
  @property
  def _data(self) -> Any | None:
      """DEPRECATED post-ADR-031. Read-only forwarder ...
      warnings.warn(DeprecationWarning(...))
      return self._transient_data
  ```
  Actual `base.py:369-382`:
  ```python
  @property
  def _data(self) -> Any:
      """Backward-compat bridge: reads ``_transient_data``."""
      return self._transient_data
  @_data.setter
  def _data(self, value: Any) -> None: ...
  @_data.deleter
  def _data(self) -> None: ...
  ```
  Three drifts: (1) **NOT read-only** — there is a setter and a deleter, used by legacy loaders that still write `obj._data = arr`; (2) **does not emit `DeprecationWarning`** at runtime; (3) return type is `Any` not `Any | None`. Same three drifts for `_arrow_table` (`base.py:384-397`).
- **P2 (issue link).** SSOT cites `Issue: #661` for the 32-doc-site cleanup. This is plausible (consistent with the `Phase 11` removal note in the docstring) but I cannot verify the GitHub issue from this audit scope; flagged as needs-confirmation in Phase 8.

### §3.3 `data-types.FrameworkMeta`

- SSOT Source: `src/scieasy/core/meta/framework.py`
- SSOT Status: `b`
- Actual: `src/scieasy/core/meta/framework.py:32-152`. Source path **correct**.

**Findings:**
- **P1 (Field type drift on `source`).** SSOT shows `source: str | None`. Actual `framework.py:63`:
  ```python
  source: str = ""
  ```
  Defaults to empty string, not `None`. The `b-code-wins` sub-label is defensible — but the SSOT description must show the actual field type.
- **P2 (Field type drift on `object_id` and `derived_from`).** SSOT documents "code uses `str`" — correct, code does:
  ```python
  object_id: str = Field(default_factory=lambda: uuid4().hex)
  derived_from: str | None = None
  ```
  But SSOT omits that `object_id` has a `default_factory=uuid4().hex` (i.e. NOT a required field, always populated). And `lineage_id` SSOT type matches. Net: medium drift on the *required vs default* nature, not on the type alone.
- **P2 (`derive` signature drift).** SSOT shows `derive(self) -> "FrameworkMeta"`. Actual `framework.py:67-107`:
  ```python
  def derive(self, **changes: Any) -> FrameworkMeta:
  ```
  Accepts `**changes` overrides for any field. The SSOT-quoted behaviour ("fresh object_id + created_at=now, derived_from=self.object_id") is correct as the default, but the override capability is hidden.
- **P2 (`with_lineage_id` parameter name drift).** SSOT shows `with_lineage_id(self, id: str)`. Actual `framework.py:109-151`:
  ```python
  def with_lineage_id(self, lineage_id: str) -> FrameworkMeta:
  ```
  Parameter name is `lineage_id`, not `id`. Cosmetic but a copy-paste of the SSOT will not type-check against the real signature.
- **P3 (model_config).** SSOT shows `class FrameworkMeta(BaseModel, frozen=True):`. Actual uses `model_config = ConfigDict(frozen=True)` (the modern Pydantic v2 idiom). Functionally identical, but the SSOT-quoted form is **invalid** Pydantic v2 syntax — `frozen=True` is not a class-keyword arg in v2. A reader who copied that line into a plugin would get a `TypeError`.

## Cross-cutting observations

1. **Systematic source-path drift in `core/storage` and `core/types`.** A reader using the SSOT as a navigation aid will hit 5 broken paths in §3+§4 alone: `dataobject.py` (does not exist, real file `base.py`), `filesystem_backend.py` (real `filesystem.py`), `router.py` (real `backend_router.py`), `TypeSignature in registry.py` (real `base.py`), `_get_backend in storage/__init__.py` (real `types/base.py`). All 5 are mechanical typos in the SSOT that a single grep would catch. Phase 8 fix: normalise file names via `glob src/scieasy/core/{types,storage}/*.py` and rewrite the Source: lines.

2. **The hallucinated "5-slot" framing on §3.1 is structurally important.** It conflicts with ADR-027 D5 (the canonical three-slot stratification) and would confuse anyone reading SSOT to learn the data model. The correct framing is *three logical metadata slots (framework / meta / user) plus two protocol slots (storage_ref / _transient_data)*. ADR-027 D5 should be re-read and §3.1 prose rewritten.

3. **§3.14 TypeSpec is the most distorted entry in §3.** Only 1 of 5 fields matches; the rest are renamed, retyped, or fictitious. ADR-009 explicitly forbids storing the class object itself (only `module_path` + `class_name` strings) — SSOT's `type_class: type` field would violate that invariant. Likely the entry was written from a remembered/proposed design rather than the implemented one.

4. **§3.16 `_serialise_one` two-level envelope vs SSOT flat shape is the single most dangerous drift in §3.** This is the wire-format contract between the engine and the worker subprocess. Any plugin author who writes serialiser/deserialiser code against the SSOT shape will fail to interoperate. Phase 8 fix: rewrite the return-shape block verbatim from `serialization.py:311-316`.

5. **§4.1 StorageReference field set is the single most dangerous drift in §4.** SSOT documents a design that was either never implemented or was rolled back to the flat `{backend, path, format, metadata}` model. Plugin storage backends will not find `type_chain`/`shape`/`dtype`/`axes`/`slot_refs`/`extras` on the actual class.

6. **`b`-class entries in §3 and §4 are well-framed.** The `b-code-wins` / `b-docs-wins` sub-labels (StorageReference immutability, ArrowBackend atomicity, CompositeStore iter_chunks/atomicity) all match the observed code reality. Whoever wrote those sub-label notes did look at the code; the §3 prose-only entries (TypeSpec, _get_backend, ChannelInfo, PhysicalQuantity) did not get the same care.

7. **Status labels are mostly defensible after fixes.** `a` is generally correct except for §3.10 (TypeSignature — `slot_schema` field name is wrong, may warrant `b`), §3.12 (ChannelInfo — class kind + field set drift, may warrant `b`), §3.13 (PhysicalQuantity — arithmetic drift, definitely `b`), §3.14 (TypeSpec — field set drift, `b`), and §4.7 (BackendRouter — `dispatch` non-existence, definitely `b`).

8. **No §3+§4 entry was completely unsalvageable.** Every drift listed is fixable inside the SSOT by re-reading code; no entries require code changes (this is correct — SSOT is the artefact under audit).
