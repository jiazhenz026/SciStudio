Highest-impact findings (all P1):
1. **AIBlock.auto_complete is a fiction** — the d-class entry claims "implemented but undocumented" but `auto_complete` does NOT exist anywhere in `ai_block.py` (no field, not in `config_schema`). Should be `c-impl`, not `d-document`. SSOT also embeds it in the `AIBlock.config_schema` snippet — a second drift.
2. **CompletionSource / CompletionEvent enum + dataclass shapes are wildly wrong** — all 3 enum value names differ from code, CompletionEvent has 5 SSOT-claimed fields vs 3 in code, missing `frozen=True`. Plugin authors reading the SSOT will write code that crashes at import.
3. **AppBlock._bin_outputs_by_extension** is marked `c-impl` with `Source: app_block.py:L1-L1` but is implemented at L182-L285 — full ~100-line implementation that even respects port `accepted_types`. SSOT is documenting fiction.
4. **FileExchangeBridge.prepare** is marked `c-impl` with `Source: bridge.py:L1-L1` but is implemented at L31-L74. Same fictional-status issue.
5. **SubWorkflowBlock._scheduler_factory** has literal `L?` placeholder in the `Source:` field — broken anchor.
6. **CompletionWatcher cancel() method documented but doesn't behave as described** — code has `cancel()` (L236) but its mechanism (setting `_cancel_event` flag) yields `WatcherCancelledError` not the SSOT-promised `CancelledError`. Documented type is wrong.

## Cross-cutting observations

1. **`Source: ...:L1-L1` is being used as a fictional placeholder.** It appears in:
   - `block-abc.AppBlock.bridge.prepare`
   - `block-abc.AppBlock._bin_outputs_by_extension`
   - `block-abc.AIBlock.auto_complete`
   - and as a literal `L?` in `block-abc.SubWorkflowBlock._scheduler_factory`.

   The SSOT grammar (line 24: `Source: \`path/to/file.py:Lnn-Lmm\``) does not provide a sanctioned "I have no anchor" sentinel. Two of these (`prepare`, `_bin_outputs_by_extension`) point at methods that DO exist — using `L1-L1` masks that reality. One (`auto_complete`) is mislabeled `d-document` when it is actually `c-impl` (code missing). The convention needs sharpening — propose: `Source: [CODE: not implemented — file path TBD]` for true c-class, and require real line ranges when the symbol exists.

2. **Multiple entries embed ASPIRATIONAL contracts and label them as `d-document` rather than `c-impl`.** This makes the SSOT internally inconsistent because the `d` taxonomy says "implemented but undocumented". Examples:
   - `AIBlock.auto_complete` — not implemented at all.
   - `port_accepts_signature` — Array/CompositeData richer checks not implemented.
   - `validate_connection` — `is_collection` check not implemented.

   A future audit pass should re-classify all such entries. For now, Phase 8 manager-fix should at minimum prefix the SSOT body with a clear "FUTURE / ASPIRATIONAL" marker on these.

3. **Module-1 preamble undercounts entries (says 30, actual is 40).** Module-2 preamble says 13, actual is 12. Phase 8 manager should reconcile counts in module preambles. Source: SSOT line 51 and 1558 (preamble fenced text).

4. **Line-range drift is systemic in block.py entries.** SSOT line ranges for Block.transition / Block.validate / Block.run / Block.get_effective_*_ports all point to wrong locations. This suggests either (a) the SSOT was authored from a pre-refactor version of block.py, or (b) the author was estimating line ranges from the SSOT body's own narrative length rather than measuring. Phase 8 should re-run a mechanical anchor check against current code for every block.py entry.

5. **CompletionEvent/CompletionSource entry is the riskiest in the audit** — it's not just stale, it documents shapes that would crash any consumer who copies them. Phase 8 should treat this as priority-zero.

6. **Detailedness invariant violations are common.** Per SSOT line 39, every entry must include "every field/parameter, every constraint/default/exception". Multiple entries (IOBlock, SubWorkflowBlock, AIBlock, ProcessBlock signature subset) drop ClassVars that are public surface in code.

7. **Several `d-private` recommend-rename entries are useful but a separate concern.** The SSOT correctly identifies `RunnerRegistry`, `FileExchangeBridge`, `FileWatcher`, `ProcessExitedWithoutOutputError`, `_PopenProcessAdapter`, `_sequential_execute`, `CompletionEvent`/`CompletionSource`/`WatcherCancelledError` as candidates for `_underscore` rename. Phase 8 can split: keep audit findings, defer rename PRs to follow-up issues.
