## Summary

**Verdict: pass-with-fixes — 4 P1, 7 P2, 5 P3 findings.**

The 5 skills + base SKILL.md + CLAUDE.md/AGENTS.md template are well-structured and pedagogically strong. Identity framing is clean, the rule hierarchy is consistent, and the worked examples (especially the thresholding block in `scieasy-write-block`) are detailed enough that a cold-read agent can succeed at most common tasks.

The **release-blocking issue** is **systematic naming drift** between every skill's tool-call documentation and the actual MCP tool signatures. The skills uniformly use `node_id` / `port_name` / `output_refs` / `block_path`, while the tools require `block_id` / `port` / `outputs` / `type_name`. The 3 Codex P1/P2 findings on PR #1059 catch only 3 of these instances; the same drift recurs in at least 6 additional places across all 5 skills. **A cold-read agent following any skill's worked example verbatim will fail with `pydantic.ValidationError` on the FIRST tool call after `list_blocks`.**

For e2e test (g) "write a thresholding block from zero hints": the agent will **succeed at scaffolding** (skill §7 example is good, port-type rules are crisp, list_blocks → list_types → scaffold_block sequence is clear), but **will fail at the final `run_block_tests` step** because the skill teaches the wrong parameter name (`block_path` not `type_name`). This is recoverable — agent will read the parameter-validation error and self-correct — but it breaks the documented "tool-call sequence works first try" promise of the skill.

## C. Skill body quality

### C.1 `scieasy-write-block` deep-dive (drives e2e test g)

**Setup**: I read the skill body cold as if I were a fresh agent given user prompt: "write a simple image thresholding block — input image, output binary mask, threshold parameter".

**Step-by-step simulation** of the agent following the skill:

1. **§1 Block-reuse rule** — Agent calls `list_blocks`. ✓ Clear instruction. ✓ Cited #875.
2. **§2 Block ABC contract** — Agent reads class-level metadata, understands `name`/`version`/`subcategory` are `ClassVar`, **understands NOT to set `base_category`** (explicit warning). ✓ Crisp.
3. **§3 Port-type selection** — Agent calls `list_types`, picks `Image` for input and `Mask` for output. ✓ Mandatory call to `list_types` is clear. ✓ Why-it-matters paragraph is the right teaching depth (preview, edge-time type checks, lineage navigation).
4. **§4 config_schema MRO merge** — Agent declares `method` enum + `threshold_value` number. ✓ Implications list is clear (don't duplicate parent fields, bump version on contract change).
5. **§5 Plugin entry-point pattern** — Agent reads, decides to use a project-local `blocks/*.py` (skill says "for most user blocks, drop a *.py file into blocks/" — matches the test scenario). ✓.
6. **§6 Block category taxonomy** — Agent maps `process` to `ProcessBlock`. ✓.
7. **§7 Worked example** — Agent copies the thresholding example, adapts the algorithm. ✓ Drop-in code. ✓ Realistic `process_item` body. ✓ Concrete `Image` + `Mask` types.
8. **§8 Tool-call sequence** — Agent follows: `list_blocks` → `list_types` → `scaffold_block(...)` → edit → `reload_blocks` → `list_blocks` (verify) → `run_block_tests`. ✓ Sequence is correct.

**HOWEVER**, the very last step is **broken**:

```
mcp__scieasy__run_block_tests block_path=blocks/threshold_simple.py
```

The actual MCP tool signature (per `tools_authoring.py:430-432`) is:

```python
async def run_block_tests(
    type_name: str = Field(description="Block type name. Test file is tests/blocks/test_<lower>.py."),
) -> RunBlockTestsResult:
```

The tool takes a `type_name` (e.g. `"imaging.threshold_simple"`), NOT a `block_path`. Agent following the skill verbatim will get a parameter-validation error.

**[P1 — F1]** (codex-confirmed): `scieasy-write-block:264` — replace `block_path=blocks/threshold_simple.py` with `type_name="imaging.threshold_simple"`.

Even if fixed, there's a second issue: `run_block_tests("imaging.threshold_simple")` resolves to `tests/blocks/test_imaging.threshold_simple.py` (per `tools_authoring.py:445`: `test_path = root / "tests" / "blocks" / f"test_{type_name.lower()}.py"`). The dot in the namespaced block type produces a malformed filename. The tool's docstring says "test file is `tests/blocks/test_<lower>.py`" — it doesn't say "we strip namespaces". An agent following this skill will end up with `run_block_tests` returning `found=False, returncode=-1, stderr="test file not found: …test_imaging.threshold_simple.py"`.

**[P2 — F2]**: `run_block_tests`'s `type_name` argument is ambiguous: does it want the registered name `imaging.threshold_simple` (with namespace) or the bare snake_case `threshold_simple`? Tool's behavior (per source) suggests bare snake_case is what produces a sensible test filename. Either:
- (a) Update the tool's docstring + Pydantic Field description to say "Block name in snake_case (matching the scaffold_block 'name' arg), NOT the namespaced type_name from list_blocks", OR
- (b) Update `run_block_tests` impl to strip namespaces from `type_name` before computing the test path, OR
- (c) Update the skill's worked example to also create `tests/blocks/test_threshold_simple.py` explicitly.

This is a tool-and-skill cross-cutting decision. Recommend (a) + clarify in skill §8.

**[P2 — F3]**: Skill §8 says the canonical sequence ends with `run_block_tests`. But the tool returns `found=False` when no test file exists, and `run_block_tests`'s docstring says "If found=False, create tests/blocks/test_<name>.py first" — this puts the agent in a chicken-and-egg loop on a fresh block. The skill should explicitly include a step "create tests/blocks/test_threshold_simple.py with at least one smoke test before calling run_block_tests" — perhaps even include a test-file template in §7 alongside the block code.

### C.4 `scieasy-debug-run` cold-read

**P1 — F6**: `scieasy-debug-run:39-44` — `get_run_status` envelope:

```
{
  state: "succeeded" | "failed" | "cancelled" | "running" | "pending",
  block_states: {node_id: "succeeded" | "failed" | ...},
  started_at: ISO-8601 timestamp,
  finished_at: ISO-8601 timestamp | null,
  error: str | null
}
```

Real envelope (per `tools_workflow.py:300-313`):

```python
class GetRunStatusResult(BaseModel):
    run_id: str
    state: str                # one of queued/running/succeeded/failed/cancelled/unknown
    progress: dict[str, Any]  # {'block_states': {block_id: STATE_NAME}}
    errors: list[BlockErrorEntry]   # NOT a single 'error' string
```

Three mismatches:
1. `block_states` is NESTED under `progress`, not at top level.
2. `error` (singular) doesn't exist; there is `errors` (plural list of `{block_id, error, summary}` envelopes).
3. State `"pending"` is wrong — real states are `queued / running / succeeded / failed / cancelled / unknown`.

Also `started_at` and `finished_at` are NOT in the real envelope. The skill's claim "Read every field. The top-level `error` may be the engine-level diagnostic" sends the agent looking for a field that doesn't exist.

Agent following this skill will write `result["state"]` (works) then `result["block_states"]` (`KeyError`).

**P1 — F7**: `scieasy-debug-run:133` — `finish_ai_block` signature:

```
`mcp__scieasy__finish_ai_block(run_id, output_refs)`
```

Real signature (per `tools_workflow.py:787-797`):

```python
async def finish_ai_block(
    outputs: dict[str, str] | None = None,
) -> FinishAIBlockOK | FinishAIBlockError:
```

Three mismatches:
1. **No `run_id` parameter** — the tool resolves the run dir from `MCPContext.ai_block_run_dir` OR `SCIEASY_AI_BLOCK_RUN_DIR` env var. Telling the agent to pass `run_id` will cause an MCP parameter-validation rejection.
2. Parameter name is `outputs`, NOT `output_refs`.
3. Type is `dict[str, str]` (port_name → file_path), NOT `dict[port_name → ref]`. The skill describes `output_refs — a dict {port_name: ref}` which is wrong — refs are NOT what `finish_ai_block` accepts; it accepts absolute or project-relative file paths.

**P2 — F8**: `scieasy-debug-run:136` — environment variable name:

```
`run_id` — the parent workflow's run id (read from `SCIEASY_AI_BLOCK_RUN_ID` or the environment).
```

Real env var (per `tools_workflow.py:779`) is `SCIEASY_AI_BLOCK_RUN_DIR` (suffix `_DIR`, value is a directory path), NOT `SCIEASY_AI_BLOCK_RUN_ID`. The skill teaches the agent to read the wrong env var, AND to do so for the wrong purpose (run_id isn't an argument to the tool).

**P2 — F9**: `scieasy-debug-run:25, 54, 167` — `get_block_logs(run_id, node_id)` should be `get_block_logs(run_id, block_id)` (real param name). Same pervasive `node_id`/`block_id` drift.

### C.5 Systemic `node_id` / `block_id` naming drift (P1 umbrella)

**[P1 — F10]** (umbrella): Across all 5 skills the parameter names are uniformly wrong:

| What skill teaches | Real tool param | Affected skills |
|---|---|---|
| `node_id` | `block_id` | build-workflow:43,73,74,85,250; debug-run:25,49,54,92,167; inspect-data:69,100 |
| `port_name` | `port` | build-workflow:73,74,85,250; debug-run:25; inspect-data:69,100 |

The skills use `node_id`/`port_name` consistently — that's actually the CANVAS-API convention (frontend uses node-id-and-port-name terminology). The MCP tools (`get_block_output`, `get_block_logs`) use `block_id`/`port`. The fix is a one-pass sweep across all 5 skill files replacing `node_id` → `block_id` and `port_name` → `port` in tool-call contexts (NOT in edge-string format documentation — `"<node_id>:<port_name>"` colon-string IS canonical workflow YAML, that's the edge wire shape, distinct from the MCP tool param names).

This is the most pervasive drift in the audit. A fresh agent following ANY of the 3 affected skills' tool-call examples will get parameter-validation errors on `get_block_output` / `get_block_logs` / `update_block_config` (which uses `block_id` per `tools_inspection.py:670`).

### C.6 `scieasy-inspect-data` cold-read

**P3 — F11**: `scieasy-inspect-data` doesn't mention the `ref` parameter shape. The real signature is `inspect_data(ref: dict[str, Any])` — the ref is a **wire-format dict** like `{"backend": "filesystem", "path": "...", "format": "...", "metadata": {...}}`. The skill consistently talks about refs as opaque IDs like `"rf-001"` (e.g. `inspect_data(ref="rf-099")` in the worked example §8). This is doubly misleading:
- The format isn't a string ID — it's a dict.
- `get_block_output` returns the ref dict; agent passes it back into `inspect_data` without unpacking.

A cold-read agent will try `inspect_data("rf-099")` and get an MCP validation error. The fix: rewrite §8 worked example to show:

```
result = get_block_output(run_id="r-abc123", block_id="thr", port="mask")
# result.ref → {backend: "filesystem", path: "...", metadata: {...}}
meta = inspect_data(ref=result.ref)
```

**P3 — F12**: `scieasy-inspect-data:79-82` `list_data` claim:

```
Enumerates data assets under the project's `data/` directory. Use
this when the user asks "what data is in this project?"
```

Real signature (per `tools_qa.py:243-247`):

```python
async def list_data(
    project_dir: str = Field(description="Absolute path to the project root."),
) -> ListDataResult:
```

The skill doesn't mention that `list_data` requires the **absolute project_dir** as an argument — which the agent typically knows from `_render_project_context` but never explicitly. Also `list_data` only walks `data/zarr`, `data/parquet`, `data/artifacts` subdirectories (per `tools_qa.py:266`) — NOT the bare `data/` tree the skill implies.

Recommendation: rewrite §6 to:

```
`list_data(project_dir=<absolute path from project_context>)` enumerates
data assets under `data/zarr/`, `data/parquet/`, `data/artifacts/`. For
other data, use `inspect_data` directly with a ref or use `search_docs`
to find file path references in project docs.
```

## D. Tool description ergonomics

I read every MCP tool's docstring as if reading the rendered `tools/list` `description` field. Per ADR §3.2 style guide each should have:
1. Imperative one-sentence first line
2. "Use when … / Do NOT use to …" anti-pattern paragraph
3. `next_step` on write-class tools

Sample-checked all 26 tools.

### D.1 Findings — overall (PASS)

- **GOOD**: All 26 tools have imperative one-liner first sentences. The first line becomes the catalog row's description — cold-read agents will see a useful summary.
- **GOOD**: 18 of 26 tools have explicit "Use when:" + "Do NOT use to:" sections. The remaining 8 are either trivially-named read tools (`list_blocks`, `list_types`, `list_data`, `list_block_examples`, `get_doc`) or have implicit anti-patterns embedded in the body.
- **GOOD**: All 7 write-class tools have `next_step` fields with concrete tool names + reasoning. Example: `scaffold_block.next_step` says "Edit the scaffolded file to implement the block's run() method, then call mcp__scieasy__reload_blocks to register it. If warnings list flagged generic DataObject ports, narrow them to concrete types from mcp__scieasy__list_types." — exact next tool + branching guidance.

### D.2 Findings — `scaffold_block` warnings (PASS)

The warning messages on `scaffold_block` (per `tools_authoring.py:367-379`) are actionable:

```
input port 'image' uses generic DataObject. Preview and edge-time type
checking will degrade. Confirm intentional (e.g. SubWorkflowBlock or
generic AppBlock); otherwise pick a concrete type from
mcp__scieasy__list_types().
```

Agent reading this will know: (a) WHY it matters (preview degrades), (b) WHEN to ignore (intentional generic), (c) WHAT to do (call list_types). ✓

### D.3 Findings — `finish_ai_block` error envelope (P2)

**P2 — F14**: `finish_ai_block` returns either `FinishAIBlockOK` or `FinishAIBlockError`. The error envelope has 4 codes documented (per `tools_workflow.py:332-337`): `not_in_ai_block_context | invalid_outputs | already_finished | io_error`. The tool's docstring covers all 4. ✓ However, the SKILL.md teaches `finish_ai_block` differently (see F7 above) — so the tool docstring is RIGHT and the skill is WRONG. Agent who reads the rendered tool catalog FIRST and the skill body SECOND will get a consistent message; agent who reads the skill FIRST and the tool description SECOND will be confused. Recommend rewriting `scieasy-debug-run` §7 to match the tool's docstring verbatim (single source of truth = tool docstring).

### D.5 Finding — error vs return shape inconsistency (P3)

**P3 — F15**: Some tools raise (`KeyError`, `FileNotFoundError`, `PermissionError`) while others return envelopes with error codes (e.g. `finish_ai_block` returns `FinishAIBlockError`). This is fine in principle but the agent has to handle both paradigms. The skills don't explicitly teach this asymmetry — recommend a one-paragraph note in the base SKILL.md identity section: "Most read tools raise exceptions on bad input (KeyError, FileNotFoundError); a few write tools return Result envelopes with `code`/`message` for soft errors (`finish_ai_block`)." This prevents the agent from trying to parse a raised exception as if it were a typed envelope.

## G. Codex vs Claude cross-check

ADR-040 §3.10 explicitly defers Codex hook governance ("apply_patch / MCP-call hooks: out of scope; Codex doesn't yet expose a stable hook protocol"). The provisioning code at `agent_provisioning/__init__.py` writes both `.claude/skills/scieasy/` AND `.agents/skills/scieasy/` mirrors and BOTH `CLAUDE.md` + `AGENTS.md`. But hooks live ONLY at `.claude/settings.json` (not `.agents/`).

### G.1 Findings — Codex agent reading CLAUDE.md/AGENTS.md (P2)

**P2 — F20**: AGENTS.md (Codex's view) has identical text to CLAUDE.md, including statements like:

- "A PreToolUse hook blocks such calls with exit code 2" (rule #1)
- "Hooks block direct edits" (rule #2)
- "A PostToolUse hook blocks `blocks/*.py` writes if list_blocks was not called earlier" (rule #3)
- "A PostToolUse hook stderr-warns when a block file declares a generic port type" (rule #4)

A Codex agent will read AGENTS.md and assume the hook backups exist — they don't. Codex agents are operating WITHOUT any hook safety net. Two failure modes:

1. **Codex agent makes a mistake the Claude agent would be blocked from making.** E.g. Codex agent runs `scieasy run-workflow` via Bash. With Claude, the `deny_scieasy_cli` hook would block and tell the agent to use MCP. With Codex, the command silently succeeds, bypassing lineage. The agent then assumes everything is fine — but the GUI didn't see the run and `ADR-038` lineage is wrong.

2. **Codex agent over-trusts the contract.** The agent reasons "rule #3 says hooks block bad writes, so I don't need to be careful about calling list_blocks first." Without the hook, the agent's misbehavior reaches disk and creates a junk block.

Fix options (both should land):

- (a) **At provisioning time**: write a slightly different `AGENTS.md` that says "On Codex, hook backup is NOT yet available (ADR-040 §3.10 deferred). Treat each rule as a self-discipline rule that you MUST follow on your own — there is no second layer to catch your mistake." OR
- (b) **At skill body level**: add to each skill's mandatory rules a sentence: "On Codex (no hooks): you are 100% responsible for these rules. On Claude Code: hooks provide a safety net but disciplined behavior is still expected."

Recommend (a) — single template branch at provisioning time. This is a real prod-env gap and should NOT ship without it. Tracking: matches what the planning doc anticipated as test (f) "Codex EXPECTED-GAP (documented)" — but the gap is currently UNDOCUMENTED in the prompt the Codex agent actually reads.
