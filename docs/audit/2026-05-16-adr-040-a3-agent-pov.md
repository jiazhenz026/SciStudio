# A3 Audit — ADR-040 Phase 3 Agent-POV Prompt + Skill Review

**Date:** 2026-05-16
**Auditor role:** A3 — fresh prod-env agent point-of-view
**Tracking branch:** `track/adr-040`
**Source under audit:** PR #1059 (I40b — skill content + CLAUDE.md/AGENTS.md template) on top of PR #1053 (I40a — FastMCP + project context) and PR #1047 (I40c — agent_provisioning orchestrator + hooks).
**Sibling agents:** A1 (ADR completeness), A2 (MCP wiring + cross-doc).

## Summary

**Verdict: pass-with-rewrites.**

The new 5-skill split is a real improvement over the monolithic
SKILL.md. Frontmatter `description:` fields trigger progressive
disclosure cleanly for ~80% of plausible user requests. The
`scistudio-build-workflow` and `scistudio-debug-run` skills are the
strongest: schema teaching is verbatim, pitfalls are concrete, worked
examples are runnable. The base `SKILL.md` is the right shape (~50 LOC
identity + index + project_context marker), and the CLAUDE.md/AGENTS.md
template is appropriately short.

**However, three classes of drift make me predict e2e (g) "thresholding
block from scratch" will FAIL on first try** unless the polish pass
in §G.5 lands first:

1. **API drift between `scaffold_block` output and skill teaching.**
   The scaffold template (`tools_authoring.py:225-258, 271, 278`) emits
   `InputPort(name=..., type=DataObject)` — the legacy `PortSpec` shape
   — while the `scistudio-write-block` skill correctly teaches
   `InputPort(name=..., accepted_types=[Image])` per the live
   dataclass at `src/scistudio/blocks/base/ports.py:23-29`. The
   scaffolded file will raise `TypeError: InputPort.__init__() got an
   unexpected keyword argument 'type'` on `reload_blocks`.
2. **`enforce_concrete_port_types` hook is dead code.** It AST-greps
   for `PortSpec(type=...)` (a class that no longer exists). It will
   never fire on real block code that uses `InputPort(accepted_types=[...])`.
   The skills + CLAUDE.md/AGENTS.md template both call out this hook
   as a safety net — that safety net does not function.
3. **`run()` arity disagreement.** Skill §2 teaches `run(self, inputs,
   config)`; scaffold template (`tools_authoring.py:255`) emits
   `run(self, inputs)` (1 arg).

The CLAUDE.md/AGENTS.md template is essentially correct but has one
P1 ambiguity (hook safety-net wording misleads Codex agents who have
no hooks). The Codex POV reveals the safety-net language is universally
mis-stated: Claude users have a false sense of safety because the
hook is dead; Codex users have no hook at all but aren't told.

**P1 / P2 / P3 counts:** 5 P1, 7 P2, 9 P3.
**e2e (g) acceptance prediction:** FAIL on first attempt; likely PASS
on second attempt after agent self-corrects from the TypeError stderr.

### A.2 What confused me

**A.2.1 (P2):** Line 17 says "not direct file edits to
`workflows/*.yaml`" but is silent on `blocks/*.py`. A fresh agent
will assume Edit/Write on `blocks/*.py` is permitted (correct) but

**Recommendation:** Add a fourth bullet to non-negotiable rules
(lines 45-60): "For `blocks/*.py`: prefer `mcp__scistudio__scaffold_block`

**A.2.2 (P3):** Line 24 doesn't say the task skills load on-demand.
Codex's skill discovery differs from Claude Code per ADR §3.5.
**Recommendation:** "## Skills available (loaded on-demand when the
user request matches a `description:` frontmatter)".

### A.3 Three-way conflict on DataObject scope

This is **P1** because it affects the agent's port-type decisions.

- Base SKILL.md line 57: "`DataObject` is reserved for generic blocks
  (see ADR-040 §3.2a)." (Vague.)
- `scistudio-write-block` §3 line 92-99: "`SubWorkflowBlock` inputs,
  generic `save_data` / `load_data` blocks". (`save_data` and
  `load_data` are `IOBlock` subclasses.)
- CLAUDE.md template line 24-27: "`SubWorkflowBlock` and generic
  `AppBlock` patterns". (Wrong family — `AppBlock` ≠ `IOBlock`.)

A fresh agent authoring an `IOBlock` will read the CLAUDE.md template
("AppBlock only → I'm not AppBlock → no DataObject allowed") and
declare overly narrow port types, then re-read the skill body and
flip — wasted cycles, possibly wrong final answer.

**Recommendation:** Pick one canonical phrasing and repeat verbatim
in all three places:

> `DataObject` is reserved for blocks that genuinely accept any type
> — `SubWorkflowBlock` inputs, the generic `load_data` / `save_data`
> `IOBlock`s, and certain `AppBlock` patterns.

### A.4 What's missing

**A.4.1 (P2):** No mention of the `<project>/.scistudio/` runtime state
directory rule in base SKILL.md. CLAUDE.md mentions it (line 70 "Do
not edit by hand") but skills don't.
**Recommendation:** Add to base SKILL.md rules: "Do NOT read or edit
anything under `<project>/.scistudio/`. That is runtime state
(lineage.db, session markers); use MCP lineage tools instead."

**A.4.2 (P3):** No explicit "what to do FIRST" instruction; minor.

## C. e2e (g) "thresholding block from scratch" simulation

The hardest e2e case. I read `scistudio-write-block` cold and walk
through what I'd actually do.

### C.1 Sequence the skill teaches

1. `mcp__scistudio__list_blocks` (#875 rule) ★
2. `mcp__scistudio__list_types` ★
3. `mcp__scistudio__scaffold_block(category="process",
   input_ports={"image": {"type": "Image", "required": true}},
   output_ports={"mask": {"type": "Mask"}})` ★
4. Read result envelope `warnings` field ★
5. Edit `blocks/threshold_simple.py` per the §7 worked example
6. `reload_blocks`
7. `list_blocks` verify
8. `run_block_tests`

Clean sequence. The skill body §7 has a runnable worked example using
the live `InputPort(name=..., accepted_types=[Image])` API. ★

### C.2 Where it will go wrong — the scaffold template

Step 3 returns a scaffolded file. The template (`tools_authoring.py:225-258`)
and `_render_port_block` (line 265-279, specifically line 278) emit:

```python
input_ports = [
    InputPort(name='image', type=Image, required=True),
]
output_ports = [
    OutputPort(name='mask', type=Mask),
]
```

But the live `InputPort` dataclass (`ports.py:23-29`) is:

```python
@dataclass(kw_only=True)
class InputPort(Port):
    name: str
    accepted_types: list[type]
    is_collection: bool = False
    description: str = ""
    required: bool = True
```

There is NO `type=` kwarg. Loading via `reload_blocks` raises
`TypeError: InputPort.__init__() got an unexpected keyword argument
'type'`. Additionally, `Image` / `Mask` are referenced by bare name
without import — the template emits no type imports. Likely
`NameError` follows.

### C.3 What the agent will do next

Read stderr from `reload_blocks`. Re-read skill §2 / §7. The skill
shows `accepted_types=[Image]`. The agent rewrites the file via
`Edit` and adds imports (skill §7 line 181: `from
scistudio_blocks_imaging.types import Image, Mask`). Retry
`reload_blocks` — likely passes.

### C.4 Prediction

- **First attempt:** FAIL (TypeError from `InputPort(type=...)`).
- **Self-correction:** YES — skill teaches the right API; failure is
  mechanical.
- **Second attempt:** likely PASS.
- **If Phase 4 bar is "first-try pass":** FAIL.

### C.5 Recommended fixes

Detail in §G.5 below. Summary: fix scaffold template (P1), fix dead
hook (P1), reconcile `run()` arity (P1), add "scaffold template emits
no imports" caveat to skill (P2).

Hooks fire only on Claude Code per ADR §3.10. I re-read every skill

### G.6 `run()` arity discrepancy

- `scistudio-write-block` §2 line 67: `def run(self, inputs, config)`.
- `_SCAFFOLD_TEMPLATE` line 255: `def run(self, inputs)` (1 arg).
- Live `Block.run` ABC (`block.py`) is `@abstractmethod` — signature
  is implementation-defined.
- §7 worked example overrides `process_item(item, config, state)` —
  3 args, ProcessBlock pattern.

A fresh agent reads §2 (2-arg), scaffolds (1-arg), and is confused.
**Recommendation:** Make §2 explicit:

> ```python
> @abstractmethod
> def run(self, inputs: dict[str, Any], config: BlockConfig) -> dict[str, Any]:
>     """Override for direct `Block`/`CodeBlock` subclasses. Return
>     {output_port_name: data_obj}. For `ProcessBlock` subclasses,
>     override `process_item(item, config, state)` INSTEAD — the parent
>     implements `run()` to iterate over Collections."""
> ```

And fix the scaffold template to emit either signature depending on
`category` (process → `process_item`; others → 2-arg `run`).

       for kw in node.keywords:

           elif kw.arg == "type":

       is_generic = False
       if accepted_types_arg is not None and isinstance(accepted_types_arg, ast.List):

       elif type_arg is not None:

       if is_generic:
