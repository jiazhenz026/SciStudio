### F1 — `Collection[T]` routes to a single-item previewer instead of the core collection fallback (P1)

- **Category:** routing / spec-conformance · **Confidence:** high
- **Locations:** `src/scistudio/previewers/router.py:69-80` (tiers 2 & 4 call
  `_pick(..., want_collection=False)` unconditionally for collection targets);
  `packages/scistudio-blocks-imaging/.../previewers/__init__.py:395-396,406-407`
  (Image/Label specs are `supports_collection=False`)
- **Evidence:** ADR-048 §3 / FR-003 precedence for a collection is: project exact
  `Collection[T]` → package exact `Collection[T]` → parents → **core collection
  fallback**. In `PreviewRouter.resolve`, for a collection target the code runs
  tier-1 (PROJECT, `want_collection=True`, guarded by `if is_collection`), then
  **unconditionally** tier-2 `_pick(PROJECT, …, want_collection=False)`, then
  tier-3 (PACKAGE, `want_collection=True`), then **unconditionally** tier-4
  `_pick(PACKAGE, …, want_collection=False)`. `_pick` matches specs where
  `bool(s.supports_collection) == want_collection`, so for `Collection[Image]`
  with the imaging package installed, tier-4 matches `imaging.image.viewer`
  (`target_type='Image'`, `supports_collection=False`) and **returns it** — a
  single-image previewer rendering a whole collection — before reaching the
  tier-7 core collection fallback. US4 scenario 2 requires that with no
  collection-specific previewer the target hits `core.collection.basic`.
- **Impact:** The canonical US4 ten-image scenario (ADR-048 §1.1 / SC-010) is
  misrouted to the single-image viewer instead of the gallery/collection
  fallback. The session API accepts client-supplied `kind: collection_ref`, so
  `PreviewHost` can request a collection target and trigger this. Breaks the
  FR-003 precedence contract.
- **Why CI didn't catch it:** the existing router tests cover (a) a
  collection-capable previewer present and (b) **no** Image previewer at all —
  not the realistic *imaging-installed + collection* case where the bug lives.
- **Recommendation:** Gate the item `_pick` calls (tiers 2/4, and the parent
  tiers 5/6 item branch) on `not is_collection`, so a collection target only
  matches collection-capable previewers before falling through to the core
  collection fallback. Add a router test for `Collection[Image]` **with** the
  imaging package present asserting it resolves to `core.collection.basic`, not
  `imaging.image.viewer`. **Fix in-PR before SPEC 1 merges.**

### F2 — `PreviewHost` is exported but mounted on no live UI surface; the live pane still renders the legacy core image path (P2)

- **Category:** scope / spec-conformance · **Confidence:** high
- **Locations:** `frontend/src/components/DataPreview.tsx` (renders
  `PreviewRenderer` + `useSlicePreview` + `useOmeMetadata`, legacy);
  `DataPreview.parts/{ImageViewer,luts,useOmeMetadata,useSlicePreview}` (legacy
  core image-domain code, unchanged)
- **Evidence:** The PR adds `PreviewHost`/`coreViewers`/`dynamicPreviewer` (2085
  insertions, 0 deletions to the existing parts). A repo-wide grep for
  `<PreviewHost` / `PreviewHost(` over `**/*.tsx` finds **no mount** outside
  `PreviewHost.tsx`/`PreviewHost.test.tsx`; `DataPreview.tsx` only re-exports it,
  with a comment "New preview surfaces should mount PreviewHost directly." The
  spec's Implementation Sequence steps 7-10 (introduce PreviewHost; move rich
  image viewer to the imaging package; remove core image-specific behavior) are
  therefore only half-realized: the backend migration is real and routed, but the
  **live** frontend still uses the legacy core renderer.
- **Impact:** The new routed previewer path (and the package-owned imaging
  viewer) is dead-wired on the live UI — users still get the old hardcoded core
  image renderer. US2 and SC-010 cannot be exercised by a real user through this
  PR, yet the CHANGELOG reads as if the rich Image/Label migration is complete
  end-to-end.
- **Recommendation:** Either (a) wire `PreviewHost` into the live `DataPreview`
  surface and retire the legacy `ImageViewer`/`luts`/`OME` core code per sequence
  step 10, or (b) explicitly scope this PR as "backend + `PreviewHost` component
  only; live cutover deferred" with a tracked `TODO(#NNN)` and correct the
  CHANGELOG wording.

### F3 — The "core" previewer subsystem imports **up** into the API layer (P2)

- **Category:** routing (layer inversion) · **Confidence:** medium
- **Locations:** `src/scistudio/previewers/data_access.py:175,465,558`
  (`from scistudio.api.runtime._preview_cache import _get_preview_table`;
  `from scistudio.api.runtime._preview_image import _image_data_uri_from_matrix,
  _downsample_matrix`); `tests/architecture/test_layer_deps.py` (LAYER_RULES
  guards only core/blocks/engine/ai, not `previewers`)
- **Evidence:** ADR-048 frames the previewer subsystem as **core** ("core owns
  routing, session lifecycle, safety limits, data access helpers"), yet
  `PreviewDataAccess` lazy-imports private helpers from the API layer
  (`scistudio.api.runtime._preview_cache`, `_preview_image`). The back-edge
  exists too (`api/runtime/_data` imports `scistudio.previewers`). The
  architecture layer-deps test only enforces rules for dirs named
  core/blocks/engine/ai, so `scistudio.previewers` importing `scistudio.api` is
  structurally **unguarded**; the cycle stays under the cycle/sentrux thresholds,
  so CI is green while the dependency direction is inverted.
- **Impact:** A supposedly-core subsystem is coupled to the API layer's private
  runtime helpers — the exact placement boundary AGENTS.md §3.5 calls out.
  Masked because `previewers` is a new top-level package outside the layer test's
  allowlist. This is the **same class of enforcement gap** the prevention sweep
  found (import-linter blind spots) from a different angle.
- **Recommendation:** Move the reusable bounded helpers (`_get_preview_table`,
  `_downsample_matrix`, `_image_data_uri_from_matrix`) into `scistudio.previewers`
  (or `scistudio.core`) and have `api/runtime` re-export them for compat, so the
  dependency points downward; and add `scistudio.previewers` to
  `test_layer_deps.py` LAYER_RULES (forbidding `scistudio.api`/`scistudio.engine`)
  to lock the direction. At minimum, leave a tracked TODO.

### F4 / F5 — P3

- **F4 (P3, bug):** single-page large-TIFF and shape-less array reads are still
  effectively unbounded in `PreviewDataAccess._open_array_handle` (multi-page
  `key=0` bound and the `.shape`-aware Zarr path are fine; a single huge page or
  a shape-less store still fully materializes). Partially undercuts SC-004 for
  those shapes; low likelihood.
- **F5 (P3):** duplicate-previewer-id semantics — `registry.register()` keeps
  first and returns False (implemented + tested), but `DuplicatePreviewerIdError`
  / `InvalidSpecError` are defined and exported yet **never raised**, advertising
  a raise-based path (FR-029) that does not exist.

## 4. PR #1580 — SPEC 2 (AI plot tools + preview-side plot jobs)

Faithfully and completely implements SPEC 2: all 35 functional requirements and
all 10 success criteria are satisfied with concrete code and meaningful tests.
Six `category:plot` MCP tools register via FastMCP (count 27→33, verified at unit
and transport level); the preview-side runtime reuses the CodeBlock subprocess
primitive by import only while enforcing timeout/byte/file caps and sanitized
errors. The critical isolation contract (FR-025/SC-005: no workflow YAML /
scheduler / lineage / downstream mutation) and the PlotPreviewer consumption
contract (FR-031/SC-010) are both directly asserted by tests. **No P1 or P2.**

- **F1 (P3, gate):** PR body `Closes #1575` does not register a GitHub closing
  link because the PR base is the stacked SPEC 1 branch, not `main` (matches the
  documented umbrella behavior). Re-confirm or close #1575 manually after the
  stack retargets to `main`.
- **F2 (P3, bug):** multi-artifact `current_N.<ext>` files are not cleared on
  rerun — `_clear_current_artifacts` globs `current.*`, which does **not** match
  `current_1.svg` (`_` not `.`). On a rerun producing fewer artifacts, stale
  indexed files persist and could show as PlotPreviewer tabs. Broaden the glob to
  `current_[0-9]*.*` and add a shrink-on-rerun test.
- **F3 (P3, bug):** the R-runner-unavailable early return writes `current.json`
  but leaves `result.metadata_path = None` (the main path sets it). The caller
  can't locate the failure metadata. Assign `metadata_path` in that branch.
- **F4 (P3, gate):** the committed ledger records `task_kind=bugfix` for net-new
  feature work (6 new tools, new skill, new runtime). All feature-grade
  obligations are in fact satisfied and CI passed; process nit only — use
  `feature` next time.

## 7. Cross-cutting note and recommendation

- **One enforcement theme spans both audit tracks.** The prevention sweep found
  import-linter blind to the `ai/agent` subtree (missing `__init__.py`); this
  review found the architecture layer-deps test blind to the new `previewers`
  package (#1577 F3). Both let a layer inversion pass CI. Worth a small, durable
  fix: make the layer/boundary guards **fail-closed** on un-allowlisted
  top-level packages so a new subsystem can't silently escape the contract.
- **Merge guidance.** SPEC 2 (#1580) and SPEC 3 (#1581) are merge-ready modulo
  P3 cleanups. SPEC 1 (#1577) should fix **F1 (P1)** in-PR with a regression test
  and resolve **F2** (wire `PreviewHost` or explicitly scope-defer + fix the
  CHANGELOG) before it lands, because the entire stack sits on it. F3 and the P3s
  can be in-PR or tracked follow-ups at the owner's discretion.
