### P3 (advisory, non-blocking)

- **P3-1 — `pyproject.toml` matplotlib `[dev]` add is justified and minimal.**
  `pyproject.toml:62` adds `"matplotlib>=3.8"` to the existing `[dev]` extra,
  beside the pandas/tifffile test-only data-format helpers, with an inline comment
  citing SC-006 (matplotlib SVG path must be exercised end to end in CI).
  matplotlib is a plot-job *runtime* helper loaded only inside the subprocess
  harness, never imported by core. It is **not** added to core/runtime deps. The
  change is one line, well-commented, and correctly declared `governance_touch:
  true` in the gate record (the mod_guard flags `pyproject.toml`). Verdict:
  appropriate and minimal — recorded as a finding per the dispatch instruction,
  no action required.

- **P3-2 — R `to_dataframe` `max_rows` shadowing is benign but confusing.**
  `_harness.py:211-213` (R harness) writes
  `cap <- if (is.null(max_rows)) max_rows else min(as.integer(max_rows), max_rows)`
  which compares the argument against itself, then falls back to the global
  `max_rows`. The net effect is correct (cap = arg when supplied, else the
  runtime cap), but the self-`min` is dead. Cosmetic only; R execution is
  `@requires_r`-gated so it is not exercised in default CI. Suggest a tracked
  cleanup if an R-focused follow-up touches the harness.

- **P3-3 — `models.PlotManifest.id` is not regex-validated at the model layer.**
  `scaffold_plot` and `resolve_manifest_path` validate the id via
  `validate_plot_id` before any filesystem use, so a hand-edited manifest with a
  hostile `id:` cannot traverse (the on-disk path is keyed by `plot_id`/`path`,
  not `manifest.id`). The model field itself is a free `str`. No security gap
  found; noting for defense-in-depth only.

- **Codex P2 — `s2-impl.md:49` "Use the keyword-only CodeBlock runner signature".**
  Codex notes the *dispatch prompt* recorded `run_codeblock_process(argv, cwd,
  env_delta, timeout_seconds)` as a positional call, whereas the real helper is
  keyword-only (`def run_codeblock_process(*, argv, cwd, env_delta,
  timeout_seconds)` — confirmed at `_backends_registry.py:160-166`). The warning
  is that a literal positional call would `TypeError` at runtime.

  **Reconciliation: ACCEPT-as-correct-in-code / the comment targets a doc, not
  shipped behavior.** The *implementation* already uses the keyword form:
  `runtime.py:279` calls `run_codeblock_process(argv=argv, cwd=work_dir,
  env_delta={}, timeout_seconds=timeout)`. The timeout branch correctly reads
  `getattr(exc, "stdout"/"stderr")`, which exist on `CodeBlockTimeoutError`. So
  the shipped code is correct and the integration works (verified by the passing
  Python SVG + timeout tests). The only artifact with the misleading positional
  shape is the `s2-impl.md` dispatch prompt under `docs/planning/`, which is an
  informational planning doc, not a contract or runtime path. **No code change is
  required before owner-merge.** If desired, the prompt line can be corrected in a
  later docs touch; it is not gating.
