## 3. Restore on dirty tree auto-commits (the user-reported bug)

```
$ curl /api/git/status   # dirty tree from user editing dummy.txt
{"dirty": true, "modified": ["dummy.txt"], ...}

$ curl -X POST /api/git/restore -d '{"commit_sha": "c5129ee...", "files": ["dummy.txt"]}'
{
    "status": "ok",
    "auto_commit_sha": "8c68d20d5db3627be85d1c3863e4a5b940bbc1d0"
}

$ curl /api/git/log?limit=1
{
    "sha": "8c68d20d5db3627be85d1c3863e4a5b940bbc1d0",
    "subject": "auto: pre-restore @ 2026-05-21T19:46:28+00:00 (target=c5129ee)",
    ...
}

$ cat dummy.txt   # restored to v1; the dirty content is one commit back
v1
```

Wire shape matches:

- Response: `{status: "ok", auto_commit_sha: "<sha>"}` (no `status:
  "stashed"` variant)
- Auto-commit subject: `auto: pre-restore @ <iso-ts> (target=<short_sha>)`
  matches the prefix pattern the frontend `RestoreWorkflowButton` parses
  to render the inline hint copy: "Your unsaved changes were committed
  as <short_sha> before the restore — see History tab to revert if
  unintended."

## 4. Pre-vs-post user experience

Before this PR: Restore on a dirty tree → backend silently stashed → UI
showed amber "Your unsaved changes were stashed as stash@{0} — recover
via Git tab → Stashes." The user does not know what a stash is; the
Stash drawer is gone.

After this PR: Restore on a dirty tree → backend auto-commits with
prefix `auto:` → UI shows neutral "Your unsaved changes were committed
as <sha> before the restore — see History tab to revert if unintended."
History is a tab the user already knows. `git checkout HEAD^ -- ...` is
inside the four-op user model.
