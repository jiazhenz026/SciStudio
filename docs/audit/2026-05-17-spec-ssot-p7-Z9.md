### F-Z9-003 [P1] — `data-types.FrameworkMeta` type drift on `source` field
**Category**: consistency-drift
**Module**: data-types
**SSOT entry** (`docs/specs/INTERFACE_SPEC.md:L1977-L1988`):
```python
class FrameworkMeta(BaseModel, frozen=True):
    ...
    created_at: datetime
    object_id: str                 # ARCH said UUID; code uses str
    source: str | None             # origin description (loader name, manual creation, etc.)
    lineage_id: str | None         # block_execution_id from lineage.db.block_executions per ADR-038

```
**Code reality** (`src/scieasy/core/meta/framework.py:L32, L59-L65`):
```python
class FrameworkMeta(BaseModel):
    ...
    model_config = ConfigDict(frozen=True)

    created_at: datetime = Field(default_factory=lambda: datetime.now(UTC))
    object_id: str = Field(default_factory=lambda: uuid4().hex)

(c) SSOT omits `default_factory` for `created_at`/`object_id` and omits the `= ""` / `= None` defaults on other fields. (d) SSOT entry has `Source: src/scieasy/core/meta/framework.py` with no line range.
**Recommended action**: fix-in-SSOT — replace with verbatim lift, including the correct `model_config` line and field defaults.

## Closing assessment

The SSOT, as committed at `c4b94ce`, is **not merge-ready** as-is and should NOT have CI audit gates wired up against it in Phase 8 without first fixing the `a`-class fidelity issues.

The biggest systemic risk is the `a`-class drift rate (F-Z9-039): roughly 80% of sampled `a`-class entries deviate from code in ways the SSOT preamble itself forbids ("Lifted verbatim from both sides (they agree)"). This includes:
- Cited source files that do not exist (F-Z9-001, F-Z9-005, F-Z9-006, F-Z9-007).
- Phantom symbol names that aren't anywhere in the codebase (F-Z9-009: ten phantom LineageStore methods; F-Z9-014: two phantom event constants; F-Z9-016: four phantom WS outbound messages).
- Synthesized docstrings (F-Z9-002, F-Z9-025, F-Z9-026, F-Z9-034) that misrepresent the contract.
- Wrong types and parameter names on widely-used public surfaces (F-Z9-018 MCP tools, F-Z9-008 lineage dataclasses, F-Z9-024 ProvisionResult).

The two Codex c-class cross-cuttings (rest-api.cross-layer-coupling and workflow-yaml.validator-strictness-variability) DID carry through from draft v3 — the architectural finding is present and correctly classified — but the second one (F-Z9-029) cites a hallucinated function signature for `validate_workflow`, which undermines the recommendation.

Pass 2 of the SSOT-write (acknowledged as pending in the preamble at L11) is therefore **not just** about adding modules 8-13 with full signatures: it should be re-doing Pass 1 to fix `a`-class fidelity AND adding the workflow module section (F-Z9-030) AND replacing every `:L1-L1` placeholder (F-Z9-032) AND fleshing out the content-free storage backend entries (F-Z9-033). The aggregate-count appendix (F-Z9-038) is also internally inconsistent and should be regenerated mechanically.

Recommended Phase 8 manager-fix scope:
1. (highest priority) Re-lift every `Status: a` entry verbatim from current code.
2. Add a 14th module (workflow-yaml) or extend an existing module to cover `WorkflowDefinition` and the workflow schema.
3. Replace every `:L1-L1` placeholder with a real range.
4. Fill in the content-free storage backend signature blocks.
5. Regenerate the count appendix mechanically.
6. Only then wire up the CI audit gate.
