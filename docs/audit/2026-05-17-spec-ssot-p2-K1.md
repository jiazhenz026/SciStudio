## Notes on judgment calls

**Row 4 — Block.allowed_input_types (b not a):** The code comment says `# An empty list means "accept any DataObject subclass"` which is semantically the same as `[DataObject]` at runtime. However, xcheck explicitly flags this as a both-differ signal because the *declaration form* differs from ARCH. Tools and the frontend may interpret the empty list differently from `[DataObject]`. Keeping as b.

**Row 67 — StorageBackend Protocol (a not b):** D1 documents 4 methods, code has 6. I classify a rather than b because the *Protocol concept* and 4 core methods match, and the two extra methods (`get_metadata`, `write_from_memory`) are implementation extensions not contradictions. The gap is at the ARCH level (docs missing 2 methods) which is a d-class sub-issue for the extra methods — but the class itself is documented and aligned. Flagging: the extra methods are individually d-class within the interface.

**Row 102-103 — ResourceSnapshot duplication:** I first wrote "d" then immediately reclassified as "a" when I checked D1 again. The row 102 entry should be discarded; row 103 is the authoritative classification. Final count excludes the duplicate.

**Row 83 — BlockTestHarness (a despite example method name mismatch):** D1 uses `harness.run()` in a code example but the actual method in code is `harness.smoke_test()`. The method name mismatch is not a signature-level contract difference (D1 describes `smoke_test()` in the bullet list even though the example uses `run()`). Classified a since the listed API matches. The example code in D1 is stale — a documentation bug, not a contract divergence.

**Row 141 — REST workflow routes (b not d):** The 10 routes in ARCH §8.2 ARE present in code (a-class subset), but code has 3 extra routes the ARCH omits. Per the decision tree "Yes/Yes but disagree" = b. If ARCH had zero entries for workflow routes it would be d. Since ARCH has a partial (10/13) listing with the missing 3, it is both-differ-b.

**Row 165 — MCP workflow tools (b for the group):** 9 of 10 tools align; `finish_ai_block` is in code but not in ARCH §7.2 category table. I classify the group b because the group entry in D1 exists but the count and membership disagree (the ARCH table is incorrect). Individual: `finish_ai_block` would be d by itself.
