## Executive summary

The draft v1 §7 entry is **substantially correct** — the 10 C4 d-hints are all
correctly captured and classified. However, this audit identifies **6 P1 errors**
and **5 P2 observations** that must be resolved before Phase 6 lifts verbatim
signatures.

The biggest P1 issues are:
1. The subscription matrix in `events.py` comments is **wrong** in two places,
   and draft v1 + C4 both copied the comment rather than the actual `subscribe()`
   call sites — the SSOT must use the code, not the comment.
2. The `#887 ResourceManager.acquire()` classification conflict between I1
   (b-code-wins) and I2 (b-docs-wins / code-fix-needed) must be resolved;
   draft v1 picked I1's label but the I2 rationale is more architecturally correct.
3. `deserialize_intermediate_refs` is classified `d-private` in M2 but has
   explicit **deprecation** semantics that make it `d-document` (must NOT be
   called), not just private.
4. Two module-level helpers in `checkpoint.py` (`save_checkpoint` /
   `load_checkpoint`) are entirely absent from C4's 47-iface inventory and
   from draft v1. They are load-bearing (called by `CheckpointManager.save`)
   and need an entry.

**P1 FINDING — subscription matrix comment is stale in two places**:

The subscription matrix at `events.py:L67-L79` (a COMMENT, not code) claims:
```
# DAGScheduler | BLOCK_READY, BLOCK_DONE, BLOCK_ERROR, BLOCK_CANCELLED,
#              | BLOCK_SKIPPED, CANCEL_BLOCK_REQUEST, CANCEL_WORKFLOW_REQUEST
```

The actual `subscribe()` calls in code are:

The comment lists `BLOCK_READY`, `BLOCK_CANCELLED`, `BLOCK_SKIPPED` as

C4's subscription matrix (`C4:L99-L107`) correctly reads from code (the C4

**Action required (P1)**: Phase 6 SSOT must use the verified call-site list

### 1.7 `ResourceManager` — P1 CLASSIFICATION CONFLICT on #887

Draft v1 says: `b-code-wins` (memory_high_watermark docstring says 0.80, code
default is 0.90).

**I1 says**: `b-code-wins` for `acquire()` zero callers (document reality,

**I2 says**: `b-code-wins` for `acquire()` zero callers — but with opposite fix
direction: "resolution is code-side (wire the dispatch path) not doc-side".

These two I-agents agree on the `b` label but disagree on the sub-label
semantics for the same finding. I1 says docs should be updated to match code
(b-code-wins = code is truth = update docs to say L1 is not wired). I2 says
code should be fixed to match docs (b-docs-wins equivalent = ARCH is truth =

**A4 verdict (P1)**: The sub-label must be resolved. A4 reads the situation as:

1. `memory_high_watermark` docstring vs code default: **b-code-wins** (clear,

   scheduler uses `block.resource_request` in dispatch — ARCH is the primary
   authority per the hierarchy; code is incomplete, not docs. The fix is

These are two distinct b-findings on the same class, each with a different
sub-label. Draft v1 conflates them. Phase 6 must separate them:
- `ResourceManager.__init__` watermark default: `b-code-wins` (trivial docfix)
- `ResourceManager.acquire()` / L1 dispatch wiring: `b-docs-wins` (code must

**Note**: Draft v1 says "Manager confirms I-agents' upgrade from K's `a` to b.
Add open issue." This is correct: the #887 entry is in draft v1. But the
sub-label is wrong — draft v1 says `b-code-wins` for acquire() when it should
