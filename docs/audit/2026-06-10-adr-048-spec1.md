## 2. Findings (by severity, with file:line)

### P1 — blocks owner-merge

**None.** No contract break or unimplemented FR.

### P2 — should-fix before merge (both mitigated today)

**P2-1 — SVG plot sanitizer regex is bypassable; the "sanitized" claim is
inaccurate.** `src/scistudio/previewers/fallbacks.py:46-51, 431-440`
(`sanitize_svg`).

The regex sanitizer is bypassable in at least two ways, confirmed by direct
probe:

- The `<script>` end-tag pattern `</script\s*>` does not match a closing tag
  with non-whitespace after `script`, e.g. `</script\t\nbar>`. Payload
  `<svg><script>alert(1)</script\t\nbar>...` survives **completely unstripped**
  (`removed=False`, `alert(1)` retained). This is exactly the CodeQL
  "Bad HTML filtering regexp" alert (`security/code-scanning/183`,
  `fallbacks.py:46`).
- The external-href stripper only matches `http(s):` / `//`-prefixed URLs, so a
  `javascript:` URI in an `href`/`xlink:href` survives
  (`<a href="javascript:alert(1)">` is untouched).
- `<foreignObject>` (which can embed XHTML) is not removed; its `on*` handlers
  happen to be caught by the handler regex, but the element-level vector is not
  closed.

**Why this is P2 and not P1:** the spec (FR-019) requires SVG be
"sanitized **or** sandboxed," and the frontend renders the SVG in an
`<iframe sandbox="" srcDoc={svg}>`
(`frontend/src/components/DataPreview.parts/coreViewers.tsx:602-608`). An empty
`sandbox=""` value grants **no** `allow-scripts` and **no** `allow-same-origin`,
so the browser blocks script execution and origin access even when the regex
misses something. The sandbox is the real boundary and it holds; the regex is
defense-in-depth. **Fix:** either (a) replace the regex with a proper SVG/XML
sanitizer (or at minimum fix the end-tag regex to `</script[^>]*>` and reject
non-`http(s)` script-capable URI schemes incl. `javascript:`), or (b) downgrade
the docstring/`payload["sanitized"]`-style claim to "best-effort strip; the
sandbox is authoritative." The CodeQL alert should be resolved or dismissed with
this rationale so the security tab stays clean.

**P2-2 — `PreviewDataAccess` TIFF path is not bounded; it materializes the full
TIFF.** `src/scistudio/previewers/data_access.py:478-482` (`_open_array_handle`).

The TIFF branch calls `tifffile.imread(str(path))` with no `key=`/page bound,
which reads **all pages** of a multi-page/large TIFF into RAM. The docstring
(line 224-226) claims "the first page is read… a single IFD," but `imread`
without `key` returns the full stack. The SC-004 bounded-read test
(`tests/previewers/test_preview_data_access.py:76`) only exercises the **Zarr**
path (which is genuinely bounded — it asserts the selector is not `Ellipsis`).
The spec verification plan explicitly lists "array access for large Zarr/**TIFF**
fixtures reads bounded slices only," and SC-004 names "large Zarr or TIFF
payloads." So this is a real, if narrow, FR-010/SC-004 gap for TIFF.

**Why this is P2 and not P1:** for the common single-plane 2-D TIFF preview the
behavior is correct and bounded; the gap only bites large multi-page TIFF
stacks, which are the less-common preview input (Zarr is the primary large-array
store). Notably, the *retained* MCP `preview_data` adapter already guards this
case (`_MAX_PREVIEW_BYTES` + `tiff_page_exceeds_cap_and_not_memmappable`,
`tests/ai/test_mcp_tools_inspection.py:143`), which both proves FR-027 and shows
the equivalent guard is missing in the new `PreviewDataAccess`. **Fix:** use
`tifffile.imread(path, key=0)` / a `TiffFile` page handle (or `memmap`) and a
byte-cap guard mirroring the MCP adapter, plus a bounded-TIFF test.

### P3 — improvement / follow-up

**P3-1 — The frontend-manifest delivery seam is an informal, provider-burden
contract rather than a framework-provided field.** (Recorded per dispatch
instruction; see §4 for the full analysis.) Manifest is delivered to the host via
`envelope.metadata.extra["frontend_manifest"]`
(`packages/.../previewers/__init__.py:189-198`,
`frontend/.../PreviewHost.tsx:64-71`). The session manager has the resolved
`spec` in `_render` (`src/scistudio/previewers/session.py:214-253`) but does not
inject `spec.frontend_manifest` into the envelope; each previewer author must
manually re-embed it on **every** envelope or the host never loads their UI.

**P3-2 — `_composite_raster_legacy` bypasses bounded access.**
`src/scistudio/api/runtime/_data.py:392-418` calls `_load_preview_matrix(slot_ref)`
and `_downsample_matrix`, materializing the raster matrix outside
`PreviewDataAccess`. This is intentional and acceptable per FR-008 (legacy compat
"preserved bit-for-bit"), but it is a second un-bounded read path; worth a tracked
follow-up to route it through `composite_raster_slot` once the legacy frontend
fully migrates off the compat route.

**P3-3 — Docstring drift.** `src/scistudio/previewers/models.py:189` documents
`module_url` as `/api/previews/assets/<asset_id>`, but the actual route +
manifest is `/api/previews/assets/{previewer_id}/{file}`
(`api/routes/data.py:208`, imaging `_module_url` line 86). Cosmetic; align the
docstring.

**P3-4 — Empty `previewer_id` registration returns `False` but does not record a
typed `InvalidSpecError`/diagnostic code.** `registry.py:61-63` appends a plain
string diagnostic. `InvalidSpecError` exists (`models.py:592`) but is unused.
Non-blocking; the diagnostic string is sufficient for FR-029.

## 4. Dispatch-Mandated Deep Dives

**Manifest-delivery seam — sound contract, or should it be first-class?**
The manifest rides in `envelope.metadata.extra["frontend_manifest"]`
(provider embeds it; host reads `envelope.metadata.frontend_manifest`). It
**works** end-to-end and is same-origin-validated, so it is not a defect. But it
is the weakest contract in SPEC 1: (1) `PreviewerSpec.frontend_manifest` is
first-class on the *spec* but is dropped from the `PreviewEnvelope` wire model
(`schemas.py:272-283` `PreviewEnvelopeModel` has no manifest field); (2) the
session manager holds the resolved `spec` in `_render` but never auto-injects its
manifest — so every package/project author must remember to re-embed the manifest
into *every* envelope (`_embed_manifest`), including error envelopes, or the host
silently never loads their UI. **Recommendation (record either way, per
dispatch): make the manifest a first-class, framework-populated field.** The
session manager should stamp `spec.frontend_manifest.to_dict()` onto the envelope
(e.g. a real `frontend_manifest` field on `PreviewEnvelope`/`PreviewEnvelopeModel`)
so authors cannot forget it and the contract is explicit on the wire. This is a
P3 today (it functions), but it is the right hardening for SPEC 2/3 package
authors and should be tracked.

**Bounded `PreviewDataAccess`:** Zarr is genuinely bounded (explicit slice,
proven by SC-004 test). TIFF is **not** bounded (P2-2). The `shape_attr is None`
Zarr branch (`data_access.py:497-503`) materializes via `handle[...]`, but only
for exotic/test-double handles without `.shape`; real Zarr always takes the
bounded path. Acceptable.

**SVG sanitization actually sanitized?** No — the regex is bypassable (P2-1), but
the sandboxed iframe makes the actual risk benign. Fix the claim and/or the
regex.

**Router precedence + ambiguity vs ADR-048 §3:** Exact match — project-exact
(collection then item) → package-exact (collection then item) → project-parent →
package-parent → core-collection → core-base → error, with priority tie-break and
`RoutingAmbiguityError` on unresolved ties, project default override. Verified
line-by-line against ADR-048 §3.

**Core ArrayPreviewer generic-only?** Yes — no LUT/OME/channel/label semantics in
`array_previewer`; all image-domain controls live in the imaging package
(`image_provider`, `viewer.js`). FR-013/FR-014 clean.

**Imaging falls back to core Array when absent (FR-026)?** Yes — imaging `Image`
spec is package-tier priority 100 (wins exact `Image`), but its envelope uses
`kind=ARRAY` with the exact payload the core Array viewer consumes, and the host
falls back to the core viewer on any dynamic-load failure. When the package is
absent the router resolves `Array` → `core.array.basic`.

**Semantic-dup baseline re-ratchet (#1578):** Justified. `duplicate_pct`
**improved** 14.06→13.17 (codebase grew); only absolute count/LOC ratchets moved
(88→94 clusters, 5038→5427 LOC) and `max_cluster_size` *tightened* 10→8. The
rationale is documented in the baseline file with a tracked follow-up (#1578) to
DRY the 9 structurally-similar envelope-building providers and re-ratchet DOWN.
The bump is the correct call: the new providers are legitimately similar (each
assembles a `PreviewEnvelope` + 6 metadata flags), and DRYing them now would add
indirection before SPEC 2/3 reveal the right abstraction. Defer the DRY to #1578
as recorded.

previewer half. All FR-001..FR-030 are implemented; SC-001..SC-009 are proven by
tests; SC-010 is the manager's browser-round obligation. No scope reduction, no

1. **P2-1** Fix/replace the SVG sanitizer regex (end-tag + `javascript:` URI) or

   (page handle / byte cap) + add a bounded-TIFF test, to close the FR-010/SC-004

legacy composite-raster compat through bounded access; docstring/checklist nits)
are good fast-follows and should be tracked, not blocking.
