The ¬ß7 SSOT has substantial drift from code. Two themes dominate:

1. **Source-path errors**: SSOT places `spawn_block_process`, `register_async_process`, `build_worker_payload`, `BlockTerminalStateReportedError`, and `RunHandle` at the wrong file paths. `runners/__init__.py` re-exports several from `process_handle.py` and `terminal_state.py`, which is presumably how SSOT was generated ‚Äî but the canonical `Source:` lines are wrong.
2. **Signature/contract drift**: every helper that has a documented signature in SSOT has actually drifted. `spawn_block_process` is the worst (no overlap between SSOT-listed params and real params). `ProcessHandle` methods are documented `async` but are all sync in code.

In addition, **the event-type-constants entry is materially incorrect**: SSOT lists 17 constants including two (`WORKFLOW_PAUSED`, `WORKFLOW_RESUMED`) that **do not exist anywhere** in code, and omits three (`INTERACTIVE_PROMPT`, `WORKFLOW_CHANGED`, `GIT_HEAD_CHANGED`) that **do**. The "Source: events.py:L67-L79" line range points to the subscription-matrix comment, not the constants block (which is L23-L54).

## Cross-cutting issues

### CC-1 ‚Äî Source path generation policy
Five entries (RunHandle, spawn_block_process, register_async_process, build_worker_payload, BlockTerminalStateReportedError) point to the WRONG `Source:` file. The common cause appears to be that whoever generated SSOT used `runners/__init__.py` re-exports as the canonical location. SSOT should use the file where the `class` or `def` keyword actually appears, not the re-export shim. Recommend adding a CI check that greps the SSOT `Source:` line range for the symbol name to catch these.

### CC-2 ‚Äî Async vs sync drift
SSOT shows several methods as `async` (ProcessHandle.{is_alive, exit_info, terminate, kill}, ResourceManager.acquire) where the code is sync (or in acquire's case the reverse ‚Äî code IS async, SSOT shows sync). This kind of drift breaks every caller. Recommend an SSOT linter that introspects the `inspect.iscoroutinefunction()` of each documented method.

### CC-3 ‚Äî ARCHITECTURE.md ¬ß6.1 carries the same wrong subscription matrix
ARCH ¬ß6.1 (line 1929-1938) lists ProcessRegistry as an EventBus subscriber. It is not. Both SSOT (¬ß7 entry 2 docstring and entry 18 docstring) and ARCH agree on this fiction ‚Äî likely the original design intent that was never coded. The fix is to either implement the subscription (ADR-018 update) or correct both ARCH and SSOT. ADR-018 reading suggests the design intent IS for ProcessRegistry to subscribe; the missing implementation is the bug, not the docs.

### CC-4 ‚Äî Subscription-matrix comment in events.py:L67-79 is stale
The comment block claims (a) ProcessRegistry subscribes to PROCESS_SPAWNED/EXITED (false ‚Äî see CC-3) and (b) CheckpointManager subscribes to CHECKPOINT_SAVED (false ‚Äî CheckpointManager subscribes only to 4 terminal events). The comment lies in the canonical declaration file, which makes downstream confusion likely. Fix as a single code-side cleanup PR alongside CHECKPOINT_SAVED + BLOCK_READY emit-or-remove decision.

### CC-5 ‚Äî Spec frozenness of `engine/runners/` claim
SSOT entry 24 footnote says "to avoid touching engine/runners/ which is frozen". This frozenness assumption affects engineering choices (pty_control.py is placed outside runners/ because runners/ is frozen). Auditors should verify this freeze rule is real (search for an ADR or hard-scope rule). If it's stale, simplifications could move pty_control logic into runners/.

### CC-6 ‚Äî `terminal_state.py` exists as a separate module
SSOT classifies BlockTerminalStateReportedError as living in `local.py`. The reality is this exception got hoisted into its own module (`terminal_state.py`) ‚Äî presumably to avoid an import cycle (scheduler.py imports from terminal_state.py, local.py also imports from terminal_state.py). SSOT should reflect the module split.
