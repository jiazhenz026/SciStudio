# ADR-045: Workflow And File State Version-Vector Contract

## 1. Decision Summary

SciStudio adopts a **server-authoritative monotonic version-vector contract** for
shared mutable state on disk. Each governed entity class (workflow YAML, file
tab content) gets a per-entity monotonic `version` integer bumped at every
write. Write sites (canvas save, agent MCP write, `gitRestore`, etc.) emit
`workflow.changed` / `file.changed` WS events directly with `{version, source,
source_id}` payload. The filesystem watcher is demoted from the primary
notification channel to a **fallback** that detects only changes the
application did not author (external editors, external git CLI). The frontend
tracks `(baseVersion, pendingVersion)` per opened entity and reconciles WS
events against them: server-side echoes of its own writes are dropped, true
remote changes refresh the canvas, conflicts with in-flight edits surface a
conflict UI.

This decision eliminates the family of races tracked in #1401 — autosave,
lineage restore, agent write, external editor, multi-session — and unlocks
deletion of three heuristic suppression layers currently scattered across
[workflow_watcher.py](../../src/scistudio/api/routes/workflow_watcher.py) and
[git.py](../../src/scistudio/api/routes/git.py).

## 2. Background — Why Current Architecture Loses

### 2.1 Two-Layer Abstraction Gap (Syscall vs Semantic)

Filesystem watchers (`watchdog` on the backend, `chokidar`/`fsevents`/`inotify`
in general) report **physical operations on inodes and paths**, not semantic
state changes. The application code, the user, and the AI agent reason in
terms of **logical entities** ("the workflow at `workflows/demo.yaml` is now
version 3"). These two abstraction levels do not align cleanly.

Specifically, a single logical "content update" can manifest as any of these
syscall sequences depending on the writer:

| Writer | Typical syscall sequence | Semantic |
|---|---|---|
| `f.write()` direct overwrite | `MODIFIED` | content update |
| `write tmp + os.replace(tmp, final)` | `CREATED(tmp)`, `MODIFIED(tmp)`, `DELETED(final)`, `MOVED(tmp → final)` | content update |
| vim / VSCode atomic save | `DELETED(final.swp)`, `CREATED(final)`, sometimes `MOVED` | content update |
| `git checkout` | `DELETED(final)`, `CREATED(final)` | content replacement |
| `rm` | `DELETED(final)` | true deletion |
| `mv across-dir` | `DELETED(src)`, `CREATED(dst)` | move |

A watcher seeing `DELETED(workflows/demo.yaml)` cannot tell whether the user
deleted the file, an atomic write replaced it, or `git checkout` swapped the
HEAD. The current implementation in
[workflow_watcher.py:241-246](../../src/scistudio/api/routes/workflow_watcher.py#L241)
uses `if path.exists(): suppress` as a heuristic, but this races with git's
unlink-then-rename sequence on Windows.

The mapping is fundamentally lossy. No amount of debouncing or coalescing at
the watcher layer can recover the semantic intent — that information must come
from the write site.

### 2.2 Even Perfect Semantic Events Are Insufficient — Identity And Source

Suppose the watcher were replaced by a perfect oracle that emits exactly one
semantic event per logical write, with the correct `kind` field. The autosave
race **still occurs** because the client cannot identify its own echo:

```
T+0     user drags block (workflowNodes mutated in-memory, dirty=true)
T+800   autosave debounce fires → client PUT /api/workflows/X
T+810   server writes YAML; perfect oracle emits semantic "modified"
T+820   user drags another block (workflowNodes mutated; dirty=true)
T+850   client receives WS modified event
T+855   client refetches; server returns content from T+810
T+860   client setWorkflow(T+810 content) → silently overwrites T+820 drag
```

The fix is to tag the notification with identity. The server's PUT response
carries a new `version`; the WS event carries the same `version` and a
`source` field naming the write origin (`canvas`, `agent`, `gitRestore`,
`external`). The client tracks the highest version it has seen and the
version it is about to write; on receipt of a WS event it can determine
"this is my own echo" (drop), "this is a clean remote change" (refresh), or
"this conflicts with my unsaved edits" (conflict UI).

This is the minimum information needed. Without `version`, the client must
guess via heuristics like timing windows or `(path, mtime, size)` tuples.
Without `source`, the contract cannot reason about the four classes of write
origin (user-canvas, agent, gitRestore, external) which have different UX
implications (e.g., agent writes deserve a notification banner; external
writes are silent merges).

## 3. Decision

### 3.1 Per-Entity-Class Version-Vector Contract

The contract is parameterized by **entity class** (a kind of versioned thing).
Two classes are in scope (§3.2). Each class has:

- A monotonic integer **version** per entity instance, owned by the server.
- A **WS event channel** that broadcasts state changes with version + source.
- A **GET endpoint** that returns content + current version.
- A **write endpoint family** (multiple write sources allowed) where each
  write bumps version atomically with the disk write.
- **Client-side per-entity tracking**: `baseVersion` (last version fetched
  cleanly) and `pendingVersion` (incremented when local edits are made
  but not yet sent to server).

The contract shape is identical across entity classes. Adding a new class
in the future (e.g., project metadata, lineage runs if mutability is ever
introduced) is purely additive.

### 3.2 In-Scope Entity Classes

**Class A: workflow YAML**

- Entity ID: workflow ID (filesystem stem of `workflows/<id>.yaml`).
- WS event: `workflow.changed`.
- Write sources: canvas save (PUT `/api/workflows/<id>`), import, create, agent
  MCP `write_workflow`, `gitRestore`, external (watcher fallback).
- GET: `/api/workflows/<id>` returns `{content, version, mtime_ns}`.

**Class B: file tab content**

- Entity ID: filesystem path relative to project root.
- WS event: `file.changed` (new — does not exist today).
- Write sources: file tab save (PUT `/api/projects/<id>/file`), agent MCP file
  write, external (watcher fallback).
- GET: existing file content endpoint returns `{content, state_version, mtime, size, encoding}`.
- Migrates the existing `FileTab.contentLoadedAt` field
  ([types.ts:286](../../frontend/src/store/types.ts#L286)) into `baseVersion`
  semantics, closing the TODO at [types.ts:274-277](../../frontend/src/store/types.ts#L274).

### 3.3 Write-Site Emit + Watcher Fallback

For both entity classes, the **write site** (HTTP endpoint or service-layer
function) is responsible for emitting the WS event after the disk write
completes. The flow:

```
write_site() {
    write_to_disk(content)                         # actual fs write
    new_version = bump_version_atomic(entity_id)   # write +1 to in-memory map
    mark_self_write_on_watcher(path, new_version)  # swallow only this exact echo
    emit_ws_event({
        kind: "modified" | "created" | "deleted",
        entity_class: "workflow" | "file",
        entity_id: <id>,
        version: new_version,
        source: <write source enum>,
        source_id: <optional UUID for client-correlation>,
    })
}
```

The watcher remains active but its emission criteria become:

```
watcher_on_event() {
    if not _is_governed_path(path): return       # outside scope
    if matches_recent_self_write_signature(path): return  # write site already emitted
    # External write (vim, git CLI, agent bypassing MCP, etc.)
    inferred_version = read_version_from_disk(path)
    if inferred_version > last_known_version[path]:
        emit_ws_event({source: "external", version: inferred_version, ...})
}
```

This replaces watcher's role as primary notifier with that of **drift
detector** for sources outside the application's write path. During the staged
rollout, watcher self-write suppression is limited to exact originating write
signatures. It must not suppress every event for a workflow ID during a time
window, because an external write to the same path immediately after a
first-party write is a real remote change.

### 3.4 WS Event Payload Schema

```typescript
interface VersionedChangeEvent {
  type: "workflow.changed" | "file.changed";
  entity_class: "workflow" | "file";
  entity_id: string;
  version: number;          // monotonic per entity
  source: "canvas" | "agent" | "gitRestore" | "import" | "external";
  source_id?: string;       // optional UUID for client-side correlation
  kind: "modified" | "created" | "deleted";
  timestamp: string;
  changed_by?: string;      // free-form, audit-only
  data?: Record<string, unknown>;
}
```

Workflow HTTP responses keep the existing YAML/schema `version: string`
contract for backward compatibility. The ADR-045 server counter is exposed as
`state_version: number` on workflow GET/write responses. `workflow.changed`
events continue to use `version: number` for the ADR-045 event contract.
Editable file HTTP responses follow the same non-overloading rule: file
GET/write responses expose the ADR-045 state counter as `state_version`, while
`file.changed` events use `version: number` in the shared event payload.

`source` lets the client choose UX:
- `canvas` → likely own echo, drop if version matches outstanding PUT
- `agent` → show notification banner ("agent updated this workflow")
- `gitRestore` → trigger explicit canvas refresh
- `external` → conflict UI if dirty, refresh if clean

`source_id` is optional but allows the client to match WS echoes to specific
client-issued PUTs (UUID generated client-side, included in PUT request,
echoed in WS event). This is the strongest dedup primitive when needed for
high-frequency writes.

### 3.5 Client Reconcile Algorithm

For each opened entity, the client maintains:

```typescript
interface EntityVersionState {
  entity_id: string;
  baseVersion: number;       // last version fetched cleanly
  pendingVersion: number;    // baseVersion + N where N = unsent local mutations
                             // (always >= baseVersion; equals baseVersion when clean)
  pendingSourceId: string | null;  // UUID of in-flight PUT, if any
}
```

On WS event for entity X:

```
if event.version <= baseVersion[X]:
    # Stale event (e.g., out-of-order delivery, or our own echo).
    drop()

elif event.version == pendingVersion[X] and event.source_id == pendingSourceId[X]:
    # Our own write just confirmed — advance baseVersion.
    baseVersion[X] = event.version
    pendingSourceId[X] = null

elif event.version > baseVersion[X] and pendingVersion[X] == baseVersion[X]:
    # Clean state, real remote change — refresh.
    fresh = await api.get(X)
    setEntity(X, fresh.content)
    baseVersion[X] = fresh.version
    pendingVersion[X] = fresh.version

elif event.version > baseVersion[X] and pendingVersion[X] > baseVersion[X]:
    # Conflict: we have pending edits AND remote moved forward.
    showConflictUI({local: X.content, remote: await api.get(X), source: event.source})
```

The conflict UI in the fourth branch surfaces a three-way prompt: **Keep
mine** (next PUT becomes a new version on top of the remote), **Use remote**
(discard local pending), **Merge** (textual diff for file tabs;
not applicable for workflow YAML structurally — falls back to keep-mine or
use-remote).

### 3.6 Version Namespace + Storage

**Namespace**: `(entity_class, entity_id)` — each entity has its own
monotonic counter independent of all others.

**Storage**: in-memory map in `ApiRuntime`, initialised at startup from
filesystem `mtime_ns`. No persistent storage layer:

- On server start: scan governed paths, populate `version_map[entity_id] =
  mtime_ns`. Future writes bump by `+1` rather than re-reading mtime.
- On server restart: client's `baseVersion` may be stale (lower than the new
  in-memory init). Standard behaviour: client receives a stale-event message,
  fetches the entity, learns the new baseline. No data loss because disk
  state is canonical.

This avoids:
- YAML pollution (no `__version: N` field in user-visible file)
- Sidecar files (no `.scistudio/versions.json` to keep in sync)
- Schema migrations (no DB / no existing files to upgrade)

Trade-off: server restart resets the counter floor. Acceptable because
client-side baseVersion is always re-established via GET after a reconnection,
and version comparison is per-entity-instance not cross-temporal.

**Growth**: per-entity counter (~16 bytes including Python overhead), typical
project has ≤ 100 workflows and ≤ 20-50 file tabs, total per-process state
< 2 KB. No growth concern.

### 3.7 Conflict Handling

The 4th branch in §3.5 triggers a conflict UI. The contract specifies
**detection**; the UI surface is an implementation concern but the
following minimum semantics are required:

- The client MUST NOT silently overwrite remote changes when local edits
  are pending.
- The conflict prompt MUST identify the `source` (e.g., "Agent claude
  modified this workflow", "Restored from commit abc1234", "Edited
  externally").
- For workflow YAML, the prompt offers: Keep mine / Use remote.
- For file tabs, the prompt additionally offers: Show diff (Monaco's
  built-in diff editor against the remote content).
- The chosen branch is persisted by issuing a new PUT (which bumps version
  again) or by reloading the remote (which resets pendingVersion to the
  remote's version).

#### 3.7.1 Implementation note — workflow YAML conflict UI (#1891)

The initial ADR-045 rollout implemented only the **detection** half of the 4th
branch for workflow YAML: the reconcile handler set `workflowConflict` in the
store, but nothing consumed it. The autosave debounce guarded on
`workflowDirty` alone, so it still fired ~800ms later and PUT the stale local
state — silently overwriting the remote (typically agent) write and violating
the "MUST NOT silently overwrite" requirement above.

#1891 completes the workflow-YAML conflict UI:

- The autosave debounce now also skips while `workflowConflict` is set
  ([useAppLifecycleEffects.ts](../../frontend/src/App.parts/useAppLifecycleEffects.ts)),
  freezing the save until the user resolves the conflict. Clearing the
  conflict re-runs the effect, so autosave resumes automatically.
- A `WorkflowConflictDialog`
  ([WorkflowConflictDialog.tsx](../../frontend/src/components/WorkflowConflictDialog.tsx))
  surfaces the prompt, identifies the `source`, and offers **Keep my version**
  (`keepLocal`) and **Use remote** (`loadRemote`) per §3.7.
- `resolveWorkflowConflict(resolution)` in the workflow slice applies the
  choice: `keepLocal` drops the conflict marker so the still-dirty edits
  resume autosaving (a now user-chosen last-write-wins); `loadRemote` adopts
  the remote workflow as the new clean base via `setWorkflow`.

The file-tab "Show diff" option (§3.7) remains unimplemented and out of scope
for #1891; the narrow in-flight-PUT-vs-remote-write timing window is likewise
not addressed (it would require server-side optimistic concurrency, which
ADR-039 §5.2 deliberately removed).

## 4. Scope

### 4.1 In-Scope

- Workflow YAML entity class — full contract per §3.
- File tab entity class — full contract per §3, including new
  `file.changed` WS event channel.
- Migration of existing heuristic suppression layers per §5.1.

### 4.2 Out-Of-Scope

- **Git refs (`HEAD`, `refs/heads/*`)**: already versioned by git SHA; the
  `git.head_changed` event already carries SHA + ref name implicitly. No
  additional version vector needed. The `_GitHeadHandler` debounce
  ([workflow_watcher.py:304+](../../src/scistudio/api/routes/workflow_watcher.py#L304))
  remains as load-reduction only.
- **Lineage runs**: append-only and immutable per ADR-038 / ADR-039. No
  in-place edit means no race surface to address.
- **Block execution events**: streaming with internal `EngineEvent` sequence
  numbers; different shape (per-run-instance, not per-entity-state).
- **Project tree state**: bulk fs change notifications (`bumpProjectTreeRefresh`)
  use a different coarse-invalidation pattern; race scope is small enough that
  the existing scheme is adequate.
- **Real-time collaborative editing** (OT / CRDT): out of scope; SciStudio
  expects a single primary editor per workflow at a time. Conflict UI handles
  the rare collision case.
- **Cross-workflow ordering**: each entity versions independently; no global
  sequence.

### 4.3 Future Extensions

The contract shape is intentionally generic. Future entity classes can adopt
the contract without changing this ADR — only `governs.modules` and the WS
event channel name change. Candidates for future adoption:

- Block configuration store (if mutable beyond workflow YAML)
- Plugin / type registry runtime state (if user-mutable)

## 5. Consequences

### 5.1 Heuristic Suppression Unlock Matrix

After this contract lands, the heuristic inventory from §2.3 resolves as:

| # | Layer | Verdict | Rationale |
|---|---|---|---|
| 1 | Path normalization (NFC/NFD/symlink) | **KEEP** | Filesystem-level concern, orthogonal to version semantics. |
| 2 | Transient-delete `path.exists()` probe | **KEEP DURING MIGRATION** | A1 preserves the existing guard until every relevant write path has precise versioned emission and regression coverage. |
| 3 | Self-write deque `(path, mtime, size)` | **NARROW** | The watcher may suppress only an exact originating write signature. It must not drop all events for a workflow ID within a time window. |
| 4 | 200ms per-path debounce | **SIMPLIFY** | Becomes pure load-reduction. Drop the race-related justification; keep as optimization if measurements show benefit. |
| 5 | Hash snapshot diff in git.py | **DELETE** | `gitRestore` becomes a write-site that bumps version per affected workflow; diff snapshots become unnecessary. |
| 6 | Frontend WS probe (#1400 hotfix) | **REPLACED-BY** | The version-comparison logic in §3.5 supersedes the probe. The probe was a symptom-layer patch for case 2. |

Plus:

- **Autosave debounce 800ms**: KEEP (UX, unrelated).
- **`FileTab.contentLoadedAt` TODO**: CLOSED (becomes `baseVersion`).

Net: 3 deletions + 1 simplification + 1 replacement = 5 hacks retired.

### 5.2 Affected Modules

| Module | Change |
|---|---|
| `scistudio.engine.events` | Existing `EngineEvent.data` carries version/source/entity fields; no event bus class change is required for A1. |
| `scistudio.api.runtime` | Owns the in-memory `version_map[entity_class][entity_id]`. Bumps on write. |
| `scistudio.api.routes.workflows` | `_emit_workflow_changed` adds version + source. `update`, `create`, `import` callsites pass `source`. |
| `scistudio.api.routes.projects` (file save endpoint) | New WS emit on save. Pass version + source. |
| `scistudio.api.routes.git` | `_emit_workflow_diff` becomes a write-site emit per affected workflow, with `source="gitRestore"`. Hash snapshot diff retired (#5). |
| `scistudio.api.routes.workflow_watcher._WorkflowFileHandler` | Becomes drift detector for external writes. Suppression layer 3 is narrowed to exact first-party signatures; layer 2 stays during migration. Emit carries `source="external"`. |
| MCP `write_workflow` tool | Wire through to runtime's write-site flow so emits carry `source="agent"`. |
| `frontend/src/hooks/useWebSocket.ts` | Handler implements the reconcile algorithm in §3.5. Probe (#6) deleted. |
| `frontend/src/store/workflowSlice.ts` | Add `baseVersion`, `pendingVersion`, `pendingSourceId` fields. |
| `frontend/src/store/tabSlice.ts` | Add same fields to `FileTab`. Migrate `contentLoadedAt` semantics into `baseVersion`. |
| `frontend/src/api/index.ts` | `getWorkflow` / file-GET return types include `state_version`; changed-event payloads include `version`. PUT functions accept and return `source_id`. |
| Conflict UI component (new) | Three-way prompt for §3.7. |

Estimated LOC across implementation: 450-550 (split roughly 60/40
backend/frontend), plus comprehensive race-matrix test suite (~200 LOC).

### 5.3 Migration Path

Implementation phases (each shippable independently behind a feature flag
`scistudio.versioned_state`):

1. **Phase 1 — Schema additions**: extend WS event payload, GET endpoint
   responses, and PUT request bodies to carry `version` / `source_id`.
   Backwards compatible: client tolerates missing version (treats as legacy
   path).
2. **Phase 2 — Write-site emit for workflows**: `_emit_workflow_changed` and
   `git.py` become source-tagged write-site emitters. Watcher continues to
   emit redundantly (idempotent).
3. **Phase 3 — Frontend reconcile algorithm**: implement §3.5 logic; ignore
   events lacking version (legacy fallback to existing behaviour).
4. **Phase 4 — Watcher demotion**: layers 2, 3 deleted; layer 4 simplified.
   Layer 5 (git.py hash diff) deleted.
5. **Phase 5 — File tab class**: add `file.changed` WS, file-save write-site
   emit, frontend reconcile for `FileTab`. Closes ADR-036 §3.10 TODO.
6. **Phase 6 — Frontend probe deletion**: remove the #1400 hotfix probe from
   `useWebSocket.ts` once Phase 3+4 are stable.

Each phase ships its own PR, gate record, and changelog entry. Phase 1 is
purely additive (no behavioural change); Phases 2-6 are behavioural and need
regression tests for the race matrix.

### 5.4 Risks

- **Server restart resets version counter**: client must re-fetch baseline on
  WS reconnect. Handled by §3.6 design but requires reconnect path
  hardening.
- **Phase 4 deletion races**: removing watcher layers 2/3 before all write
  sites (especially git operations) are properly write-site-emitting could
  drop legitimate external-change events. Mitigation: maintain layer 3
  (self-write deque) until Phase 5 completes, then delete in Phase 6 cleanup.
- **Conflict UI complexity**: the three-way prompt is a new UI surface. Risk
  of poor UX if not designed carefully. Mitigation: prototype with a single
  user story (agent overwrite) first; expand to git/external in subsequent
  phases.
- **`source_id` collisions**: if not properly UUID'd, client-side correlation
  fails. Mitigation: use `crypto.randomUUID()` consistently; require server
  to echo verbatim.

## 6. Alternatives Considered

### 6.1 Pure Frontend Solutions (Rejected)

| Approach | Why rejected |
|---|---|
| Dirty-guard: ignore WS events when local dirty | Cannot handle agent writes (user has no clue agent changed file); loses external-edit notifications |
| Track outstanding PUTs by timestamp, ignore WS within X ms after save | Heuristic with same failure modes as backend `(path, mtime, size)` deque; tuneable window is fragile |
| Client-side mutation queue (Redux-style optimistic UI) | Single-threaded JS already serializes mutations; queue doesn't solve client/server identity |

Pure-frontend approaches can solve race 1 (autosave) with heuristics but
cannot solve races 3/4/5 because those require server-side participation in
identifying source / sequencing remote changes.

### 6.2 Pure Backend Solutions (Rejected)

| Approach | Why rejected |
|---|---|
| Watcher event coalescence with longer windows (e.g., 500ms after `DELETE`, treat follow-up `CREATE` as merge) | Adds latency to legitimate delete; doesn't solve echo identification |
| Single-writer worker queue at HTTP layer | FastAPI already single-process serializes writes; this is a no-op |
| Stronger self-write suppression heuristics (more tuple fields) | Diminishing returns; tuples are still heuristic and don't cover git CLI |

### 6.3 CRDT / OT (Rejected as Out-of-Scope)

Operational transform or conflict-free replicated data types solve concurrent
collaborative editing rigorously, but the engineering surface (~5000 LOC of
CRDT plumbing) is disproportionate to SciStudio's single-primary-editor norm.
The version-vector contract handles the rare collision case via conflict UI;
real concurrent editing is out of scope.

### 6.4 Replace All HTTP With WS Mutations (Rejected as Re-architecture)

A pure-WS bidirectional mutation channel would eliminate the "client PUT then
WS echo" pattern by collapsing both into a single broadcast event. Conceptually
clean but requires rewriting all HTTP endpoints, breaking REST API
compatibility, and re-tooling the entire client/server transport. The
version-vector contract achieves the same race-elimination property with a
fraction of the cost.

## 7. Open Questions

1. **`source_id` cardinality**: should `source_id` be a UUID per write, a
   monotonic per-client counter, or a session-scoped sequence? UUID is
   simpler; counter is more compact in logs. Implementation can choose;
   contract requires only "unique within a session".

2. **Server restart version reset**: after restart, in-memory `version_map`
   re-initializes from `mtime_ns`. If a client's `baseVersion` is from
   pre-restart and happens to equal the new init value by coincidence, the
   echo-identification logic may misfire. Probability is low (mtime collision
   across restart) but non-zero. Mitigation candidates: include server-start
   epoch in event payload; or version baseline is `(epoch, counter)` tuple.
   Resolve during Phase 1 design.

3. **`_GitHeadHandler` lack of self-write deque**: the audit found this and
   asked whether HEAD ref monitoring needs similar suppression. Hypothesis:
   HEAD is rarely self-written by SciStudio (we don't directly modify it
   outside of explicit git operations which are already source-tagged), so
   no suppression needed. ADR-045 confirms this hypothesis as part of §4.2
   carve-out; if a regression surfaces, file a follow-up.

4. **EventBus backpressure**: the audit raised "no backpressure or dropping
   observed; if frontend lags, does the deque grow unbounded on heavy
   churn?". Not strictly a version-vector concern but worth investigating
   during Phase 2 — the additional payload size (3 new fields × ~50 bytes
   per event) is marginal but should be measured under load.

5. **Conflict UI for workflow YAML**: structural diff is harder than text
   diff. Phase 5 must decide whether to surface a graph-level diff
   (lossy but visual) or fall back to YAML text diff (precise but
   intimidating). Defer to a separate UX spec under the implementation
   tracking issue.
