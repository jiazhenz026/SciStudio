# ADR-041: CodeBlock v2 Script-as-AppBlock

## 1. Decision Summary

ADR-041 defines CodeBlock v2 as an AppBlock-shaped script integration block.
The goal is to let scientists drop existing scripts into a workflow without
rewriting them into a SciEasy-specific `run(inputs, config) -> dict` function.

This rewrite updates the original ADR-041 design to the ADR-042 document
schema and replaces its obsolete ADR-028 D8 format-routing assumptions with
ADR-043's IO format capability registry.

### 1.1 Problems Addressed

| Problem | Risk | ADR-041 response | Detailed section |
|---|---|---|---|
| Current CodeBlock requires framework-shaped Python functions | Existing scripts need invasive rewrites | Treat CodeBlock as a script runner over AppBlock file exchange | Section 3 |
| Inline execution and function mode blur responsibilities | UI, security, and documentation surface stays too broad | Remove inline and function modes from CodeBlock v2 | Section 3 |
| External apps, AI agents, and scripts all need boundary IO | Three parallel implementations would drift | Reuse the AppBlock boundary pattern for scripts | Section 3 |
| Old ADR-041 depended on `supported_extensions` as final format routing | Format dispatch would remain implicit and ambiguous | Route all typed boundary IO through ADR-043 capabilities | Section 6 |
| Users must know which exchange files scripts should read/write | Silent path conventions make script adaptation error-prone | Keep fixed exchange paths and provide UI guidance/snippets | Section 5 |
| Scripts need reproducible source and environment provenance | Runs may not be replayable or explainable | Require project-local scripts and capture environment snapshots | Section 8 |
| Notebook-like scripts produce executed artifacts | Reproducibility evidence can be lost | Auto-capture executed notebooks as Artifact outputs | Section 7 |

### 1.2 Accepted Design Direction

CodeBlock v2 will become an AppBlock variant where the external application is
a script interpreter and the user-supplied script is the macro, notebook, or
program being run.

The user-facing model is one palette block:

- select a script file inside the project;
- choose or accept the detected interpreter;
- choose an environment mode;
- define typed input and output ports;
- exchange data through files only.

Typed file exchange at the script boundary is governed by ADR-043:

- input materialisation resolves saver capabilities;
- output reconstruction resolves loader capabilities;
- ambiguous format choices require `capability_id`;
- metadata preservation is described by `MetadataFidelity`;
- CodeBlock does not add extension guessing or format-specific type logic.

### 1.3 Scope

In scope:

- CodeBlock as an AppBlock-shaped external execution block;
- files-mode delivery through an exchange directory;
- project-local script provenance;
- interpreter and environment selection;
- notebook output capture;
- ADR-043 capability-aware boundary IO.

Out of scope:

- moving external format support onto DataObject types;
- registering every script as a separate palette item;
- parameters-mode delivery into script variables, notebook cells, or document
  frontmatter;
- nested workflow-language semantics for Snakemake, Nextflow, or CWL;
- container orchestration for Docker, Apptainer, or Singularity;
- arbitrary local package management beyond selected environments;
- runtime implementation in this doc-only ADR rewrite.

## 2. Context

### 2.1 Current CodeBlock

Current `src/scieasy/blocks/code/code_block.py` is a framework-aware block with
inline and script modes. It expects code to be shaped around the SciEasy block
contract and dispatches to language runners that call an entry function.

That is useful for small framework-native snippets, but it does not solve the
inclusive integration case: a scientist already has a `.py`, `.R`, `.ipynb`,
`.qmd`, or MATLAB script that reads files, writes files, and was not designed
around SciEasy's `run()` API.

### 2.2 AppBlock Is The Boundary Pattern

AppBlock already expresses the correct structure for external tools:

1. materialise typed inputs into files;
2. launch an external process or GUI;
3. wait for completion;
4. collect output files;
5. reconstruct typed DataObjects.

CodeBlock v2 uses the same shape. The external process is not Fiji, Napari, or
another GUI application. It is a script interpreter such as Python, R, Quarto,
Jupyter, Bash, MATLAB, or Octave.

ADR-035 makes the same architectural move for AIBlock: an AI agent is an
AppBlock-like external thing with a special user interface. CodeBlock is the
script counterpart.

### 2.3 ADR-043 Replaces The Old Format Assumption

The original ADR-041 treated ADR-028 D8's `supported_extensions`,
`find_loader`, and `find_saver` model as the final format architecture. That
is no longer accurate.

ADR-043 supersedes that assumption with structured IO format capabilities:

- `FormatCapability` is the unit of load/save support;
- `capability_id` is the stable workflow replay key when ambiguity exists;
- `supported_extensions` is migration scaffolding, not the final package
  authoring model;
- the engine must not guess extension aliases or semantic conversions;
- type and extension combinations are validated before AppBlock or CodeBlock
  execution.

ADR-041 now consumes ADR-043 rather than defining a competing format model.

## 3. CodeBlock As An AppBlock Variant

### 3.1 Structural Decision

CodeBlock v2 SHOULD be implemented as an AppBlock variant. The precise Python
inheritance may be `class CodeBlock(AppBlock)` or a composition that shares the
same AppBlock bridge and lifecycle, but the runtime contract is AppBlock-like.

CodeBlock inherits or reuses:

- external execution mode;
- process lifecycle and cancellation;
- exchange directory layout;
- input materialisation;
- output file watching;
- typed output reconstruction;
- user-controlled completion where appropriate.

### 3.2 Single Palette Item

The palette exposes one `Code Block` item. Scripts are not registered as
individual block types and do not create one palette entry per file.

This keeps the UX consistent with ADR-043 aggregate IOBlocks: many concrete
handlers can sit behind one user-facing block when the user intent is one
conceptual operation.

### 3.3 Removed Modes

CodeBlock v2 removes:

- inline code execution from block config;
- function mode requiring a SciEasy-shaped entry function;
- language-specific palette items.

Users who want framework-native code should write a ProcessBlock or a custom
block. Users who want to reuse existing scripts should use CodeBlock v2.

## 4. Script Source, Interpreter, And Environment

### 4.1 Script Must Be Project-Local

The configured script path MUST resolve inside the SciEasy project directory.

If a user selects a file outside the project, the UI should offer:

- copy into project;
- symlink into project where supported;
- cancel.

This preserves ADR-038 and ADR-039 lineage expectations: the run can point to a
workflow git commit that includes the script source or an explicit project-local
link.

### 4.2 Supported Script Formats

The first implementation target SHOULD support:

| Extension | Interpreter family | Notes |
|---|---|---|
| `.py` | Python or `uv run` | Plain script |
| `.ipynb` | Jupyter nbconvert | Executed notebook output |
| `.qmd` | Quarto | May involve Python and R |
| `.R` | Rscript | Plain R script |
| `.Rmd` | R Markdown | Rendered document output |
| `.sh` | Bash | POSIX shell script |
| `.m` | MATLAB or Octave | Plain script |
| `.mlx` | MATLAB | Live script support depends on MATLAB |

Additional languages, containers, and workflow languages require follow-up
issues and are not part of this ADR's v1 scope.

### 4.3 Environment Selection

CodeBlock config SHOULD expose an environment selector:

- `auto`: infer from script metadata, project files, or known conventions;
- `existing`: user selects a discovered interpreter or environment path.

Auto mode may use conventions such as PEP 723, `requirements.txt`, project
`pyproject.toml`, `renv.lock`, Quarto configuration, or notebook metadata.

Existing mode may discover conda, venv, uv, system R, renv, MATLAB, or Octave
installations. Discovery is a user convenience; execution must record the
resolved interpreter and environment snapshot.

## 5. AppBlock File Delivery Mode

File delivery is the only CodeBlock v2 delivery mode. CodeBlock reuses the
AppBlock file-exchange contract instead of defining a separate script-only
protocol. This keeps external GUI tools, CLI tools, and user scripts on the
same path model.

Every AppBlock or CodeBlock run owns a block-scoped exchange directory:

```text
<exchange_root>/<block_slug>-<block_id>/
```

For example:

```text
exchange/fiji-7f3a2c/inputs/
exchange/fiji-7f3a2c/outputs/
```

The block-scoped directory prevents two blocks from sharing an `inputs/` or
`outputs/` folder. The default `<exchange_root>` is the project exchange area.
When a user configures an exchange or output root, the block still creates its
own child directory under that root.

Within the block exchange directory, every port gets its own folder:

```text
inputs/<port_folder>/
outputs/<port_folder>/
```

`<port_folder>` defaults to the port name. If that folder already exists and is
not owned by the current run's exchange layout, the runtime creates a
distinguished folder such as `<port_name>__scieasy`; if needed it appends a
numeric suffix. The manifest records the actual folder path used for each port.

### 5.1 Input Port Folders

The engine materialises every input port into its dedicated input folder before
launching the external process:

```text
<block_exchange>/inputs/<port_folder>/
```

Single-object inputs and Collection inputs use the same folder contract. For
Collection inputs, each item becomes one file in the folder. For single-object
inputs, the folder contains one materialised file. File names SHOULD preserve
the upstream source basename when the object has a source path. Generated
objects or objects without a source basename fall back to deterministic names
such as `item_0001.<extension>`.

The runtime cannot know at workflow-edit time whether an upstream Collection
will contain exactly one item. The UI therefore exposes the folder path as the
stable contract. If the user knows their workflow will provide exactly one file
for that port, they may append a concrete file name with the declared extension
inside that folder. The engine contract remains the port folder.

### 5.2 Output Port Folders

The engine creates one output folder per declared output port before launching
the external process:

```text
<block_exchange>/outputs/<port_folder>/
```

Users save files into the folder for the port they want to produce. Every file
inside a port folder whose extension matches the port's declared extension is
collected into that port's output Collection. This means a port may naturally
produce one file or many files:

```text
outputs/image/cell_0001.tif
outputs/image/cell_0002.tif
outputs/image/cell_0003.tif
```

Multiple output ports may declare the same extension because folder placement,
not file extension alone, determines the port:

```text
outputs/image/*.tif
outputs/mask/*.tif
outputs/label/*.tif
```

Files inside a port folder with an extension that does not match that port emit
a warning and are ignored. Required output ports fail if their folder contains
no matching files. Files placed directly under the output root, or under
unknown folders, emit warnings and are ignored.

### 5.3 Script Path Guidance UX

Using CodeBlock may require the user to change path strings inside their
script. This is acceptable: scripts that previously read arbitrary local paths
must be pointed at the exchange folders SciEasy provides.

The UI MUST make that adaptation obvious and low-friction. For each port, the
config panel should show:

- the direction;
- the declared type;
- the selected extension and capability;
- where the user should read from or save into;
- a copy button for the folder path.

Example guidance table:

| Port | Direction | Read from / Save at |
|---|---|---|
| `image` | input | `inputs/image/` |
| `mask` | output | `outputs/mask/` |
| `table` | output | `outputs/table/` |

For a script that expects a single file and a workflow that is known to provide
one file, the user may extend the copied folder path with a file name such as
`inputs/image/image.tif` or `outputs/table/table.csv`.

The UI SHOULD provide copy buttons and language-aware snippets. For Python:

```python
from pathlib import Path

image_files = sorted(Path("inputs/image").glob("*.tif"))
mask_dir = Path("outputs/mask")
mask_dir.mkdir(parents=True, exist_ok=True)
```

The UI MAY detect likely file path string literals in the script and offer
best-effort suggestions such as "replace `raw/image.tif` with
`inputs/image/`". This detection is guidance only; it must not become an engine
contract and must not rewrite the script without explicit user action.

The engine MUST create the block exchange directory, `inputs/`, `outputs/`, and
all declared port folders before launching the process. It MUST materialise
declared inputs before process start. It MUST NOT require input files to exist
when the user is editing or saving the workflow.

Absolute paths inside the user's script may still work if the user's machine
has those files, but they are not portable and bypass workflow dataflow. The UI
SHOULD warn when likely absolute input/output paths are detected and suggest
the exchange-relative folder path instead.

Parameters mode is intentionally excluded. Adapting every host language's
variable binding, notebook cell convention, document frontmatter, and type
coercion behavior would create a second integration framework inside CodeBlock.
Users who need framework-native parameter passing should write a ProcessBlock
or a custom block.

### 5.4 Port Config Shape

CodeBlock ports SHOULD use the same capability-aware shape as AppBlock ports:

```yaml
inputs:
  image:
    type: Image
    extension: .tif
    capability_id: imaging.image.tiff.save
    exchange_folder: inputs/image/
outputs:
  table:
    type: DataFrame
    extension: .csv
    capability_id: core.dataframe.csv.load
    exchange_folder: outputs/table/
```

The UI may omit `capability_id` only when the registry has exactly one
deterministic match for the declared direction, type, and extension.

`exchange_folder` is derived from direction, port name, folder conflict
resolution, and extension. It may be shown or persisted for explicitness, but
users should not edit it independently from the port name and extension in v1.

## 6. Format Capability Contract

### 6.1 CodeBlock Does Not Own Format Semantics

CodeBlock coordinates file exchange. It does not decide that `.tif` means
`Image`, that `Label` can fall back to `Image`, or that `.tif` and `.tiff` are
aliases. Those decisions belong to package-declared ADR-043 capabilities.

### 6.2 Input Materialisation

For every input port, validation resolves a saver capability:

```python
registry.find_saver_capability(
    data_type=declared_input_type,
    extension=extension,
    capability_id=capability_id,
)
```

Execution then uses:

```python
materialise_to_file(
    obj,
    port_input_dir,
    extension,
    filename_stem=source_basename_or_item_id,
    capability_id=capability_id,
    registry=registry,
)
```

Missing or ambiguous saver capabilities fail before the script starts.

### 6.3 Output Reconstruction

For every output port, validation resolves a loader capability:

```python
registry.find_loader_capability(
    data_type=declared_output_type,
    extension=extension,
    capability_id=capability_id,
)
```

Execution then uses:

```python
for path in sorted(port_output_dir.glob(f"*.{extension.lstrip('.')}")):
    reconstruct_from_file(
        path,
        target_type=declared_output_type,
        extension=extension,
        capability_id=capability_id,
        registry=registry,
    )
```

Missing or ambiguous loader capabilities fail before the script starts when the
port contract is known. If a required output port folder contains no matching
files, the run fails as an execution error rather than a capability error.

### 6.4 Metadata Fidelity

CodeBlock surfaces metadata fidelity from the selected loader or saver
capability. It does not create a separate metadata system.

The UI may warn when a selected saver is `pixel_only` and the input object has
typed `meta` fields that the saver does not declare in `typed_meta_writes`.

Lineage records the selected capability, resolved interpreter, environment
snapshot, script path, and produced files. It does not replace typed `meta`.

## 7. Notebook And Rendered Artifacts

Notebook-like formats SHOULD automatically produce an executed artifact output:

- `.ipynb`: executed notebook;
- `.qmd`: rendered document or executed notebook artifact;
- `.Rmd`: rendered document;
- `.mlx`: MATLAB live script output where supported.

The framework-managed output port should be named `_executed_notebook` or a
clear successor name and should have type `Artifact`.

This output is reproducibility evidence. It should be captured by lineage even
when the user's typed data outputs are separate files.

## 8. Provenance And Lineage

CodeBlock runs MUST record:

- script path relative to the project;
- script file content hash or git commit evidence;
- interpreter command;
- environment mode;
- resolved environment snapshot;
- file delivery layout;
- selected input and output capabilities;
- exchange directory or retained artifact references;
- executed notebook artifacts where applicable.

Environment snapshots may initially live in resolved block config or another
lineage-owned field. They are runtime information, not typed `meta`.

## 9. Implementation Expectations

Implementation should proceed in phases:

1. Add CodeBlock v2 skeleton using the AppBlock boundary lifecycle for `.py`
   files in files mode.
2. Add ADR-043 capability-aware input and output validation.
3. Add environment selector and environment snapshot capture.
4. Add `.ipynb` support through Jupyter nbconvert and capture the executed
   notebook artifact.
5. Add R, Quarto, Bash, MATLAB, and Octave interpreter support.
6. Update frontend config panels and port editors.
7. Update block-development docs.

Each phase should open its own implementation issue and tests. This ADR rewrite
does not itself implement CodeBlock v2.

## 11. Alternatives Considered

### 11.1 Keep Current CodeBlock Modes

Rejected.

The current inline and entry-function modes serve framework-native snippets,
not existing scripts. Keeping them inside CodeBlock v2 would preserve the
confusing UI that this ADR is trying to remove.

### 11.2 Make Scripts Registered Blocks

Rejected for v1.

Registering each script as a palette item creates the same UX problem ADR-043
rejects for one-loader-per-format designs. A script library can be large; the
user-facing palette should stay conceptual.

### 11.3 Put Format Logic In CodeBlock

Rejected.

CodeBlock should not contain TIFF, CSV, OME, or table-specific behavior.
Format support belongs to IO capabilities and package handlers.

### 11.4 Add Parameters Mode

Rejected.

Parameters mode looks attractive for notebooks and scripts that already expose
named parameters, but each host environment has a different binding model:
Python globals, notebook parameter cells, Quarto or R Markdown frontmatter, shell
environment variables, MATLAB workspaces, and language-specific type coercion.

Supporting all of those would make CodeBlock a cross-language parameter
adapter rather than a file-exchange integration block. File mode is the stable
inclusive contract. Users who want parameter-native behavior should write a
ProcessBlock or a custom block.

### 11.5 Require All Script Integration To Be ProcessBlocks

Rejected.

ProcessBlocks are correct for framework-native code. They are not inclusive for
existing file-oriented scripts that scientists already use outside SciEasy.

## 12. Consequences

Positive consequences:

- existing scripts become workflow nodes with minimal adaptation;
- CodeBlock shares the AppBlock boundary architecture;
- file format dispatch aligns with ADR-043 instead of competing with it;
- ambiguous type and extension combinations fail before execution;
- metadata loss can be made visible through selected capabilities;
- notebooks become lineage artifacts by default.

Negative consequences:

- CodeBlock v2 implementation depends on ADR-043 capability work;
- old inline/function CodeBlock workflows need migration;
- environment discovery adds platform-specific complexity;
- frontend port configuration needs capability selection controls;
- large file materialisation can be slow and needs user warnings.
- scripts that prefer parameter injection must adapt by reading files or become
  ProcessBlocks/custom blocks.

Open follow-up:

- Write an implementation spec for CodeBlock v2 before code changes.
- Decide whether CodeBlock should literally subclass AppBlock or share an
  external-boundary base class.
- Decide the exact lineage storage location for environment snapshots.
- Open implementation issues for language families beyond Python.
- Add tracked TODOs in implementation PRs for any v1 deferrals.

## Addendum 1: ADR-043 Materialisation Ownership Correction (2026-05-19)

ADR-043 implementation moved the canonical file materialisation helpers from
`scieasy.engine.materialisation` to `scieasy.blocks.io.materialisation` so
`scieasy.blocks` does not depend on `scieasy.engine`. The engine module remains
a compatibility re-export for existing callers. This addendum amends only the
ADR-042 governed-surface metadata for audit purposes; it does not change the
CodeBlock v2 architectural decisions above.

```yaml adr042-governance-amendment
governs:
  modules:

      - scieasy.blocks.io.materialisation

      - scieasy.engine.materialisation.materialise_to_file
      - scieasy.engine.materialisation.reconstruct_from_file

    add:
      - src/scieasy/blocks/io/materialisation.py
```
