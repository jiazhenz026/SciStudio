**Status**: **proposed** — drafted 2026-05-16 from a design session that began as "fix CodeBlock" and converged on "CodeBlock is structurally an AppBlock subclass". Promotes to **accepted** once the §D8 implementation cluster (#1073-#1080) lands and at least Phase 1 below ships green.

### 1. Purpose

Refactor `CodeBlock` so a user can drop one of their **existing automation scripts** (`.py` / `.ipynb` / `.qmd` / `.R` / `.Rmd` / `.sh` / `.m` / `.mlx`) into the workflow as a single block, **without restructuring the script** to fit a SciEasy-specific `run(inputs, config) → dict` function signature.

CodeBlock v2 is structurally an **`AppBlock` subclass** where the "app" is a script interpreter dispatched by file extension. This reuses ADR-006's file-exchange bridge, ADR-019's process lifecycle, and (once the §D8 implementation cluster lands) ADR-028's type↔extension dispatch.

The design is the strongest expression of the **inclusivity principle** (CLAUDE.md §2.4) yet: a researcher's existing `.py` / `.R` / `.ipynb` becomes a workflow node with **zero code modification** (files mode) or a 4-line header (parameters mode).

---

### 2. Context

#### 2.1 What's wrong with CodeBlock today (ADR-005 era)

- `src/scieasy/blocks/code/code_block.py` requires a `run(inputs, config) → dict` callable. Users must restructure their existing scripts to wrap them in this function.
- Inline mode allows arbitrary `exec()` from the config panel — a security surface and a UX dead end. Custom blocks via Monaco (ADR-036) + agent scaffolding (ADR-040) is the better path for "write me a small block".
- Per-port `InputDelivery` machinery (ADR-016) added complexity for marginal gain; ADR-020 already removed PROXY and CHUNKED, leaving MEMORY as the only mode.
- Three execution modes (`inline`, `script`-function, function) confuse the UI and inflate documentation surface.

The 858-line `src/scieasy/blocks/code/` directory cannot serve the "drop in my existing script" use case without significant per-user adaptation work.

#### 2.2 What FijiBlock proves works (the AppBlock pattern)

`FijiBlock` (in `scieasy-blocks-imaging/interactive/`) overrides `Block.run()` and calls `_prepare_image_exchange()` which:

1. Materialises each input `Image` via `tifffile.imwrite` to `<exchange_dir>/inputs/image_<i>.tif`.
2. Launches Fiji with `argv = [<staged_tiff_paths>]`.
3. Watches `<exchange_dir>/outputs/` for `*.tif` / `*.zip` / `*.roi`.
4. Returns Artifact (or, once #1079 lands, typed) outputs.

This file-exchange pattern works end-to-end. CodeBlock v2 generalises it: the **interpreter** is the app, the **script** is the macro, and the file-exchange bridge stays the same. AIBlock (per ADR-035) is the sibling pattern where the "app" is `claude` / `codex` in a PTY.

The structural collapse becoming visible: `AppBlock` is the universal "wrap an external thing" pattern. CodeBlock = AppBlock + script interpreter; AIBlock = AppBlock + PTY agent. Long-term, block taxonomy may converge on `ProcessBlock` (framework-internal, typed contract) and `AppBlock` (external wrapping). This ADR does not formalize that convergence — but it does take the second concrete step toward it.

#### 2.3 What ADR-028 §D8 delivers (the dependency)

ADR-028 §D8 (`docs/adr/ADR.md:5704-5721`) specifies a bidirectional type↔extension contract:

- Each `IOBlock` subclass declares `supported_extensions: ClassVar[dict[str, str]]` mapping extension → format key.
- The framework provides an `IOBlock._detect_format(path)` helper.
- Engine queries via `BlockRegistry.find_loader(type, ext)` / `find_saver(type, ext)`.
- Engine helpers `materialise_to_file` / `reconstruct_from_file` handle the boundary materialisation.

Audit on 2026-05-16 confirmed §D8 is **0% implemented**. The §D8 implementation cluster (#1073-#1080) closes this gap. CodeBlock v2 cannot land until that cluster merges, because:

- **Files-mode input mounting** needs `materialise_to_file(image, dir, ".tif")` to write the user's Image as a TIFF the script can read.
- **Files-mode output collection** needs `reconstruct_from_file(path, target_type=DataFrame, ".csv")` to lift the script's output CSV back into a typed `DataFrame` for downstream blocks.
- Without §D8, both directions degrade to the broken JSON-dump fallback (`bridge.py:31-74`) and Artifact-downgrade-with-warning (`app_block.py:240-258`).

This is the only architectural prerequisite. There is no new dispatch dimension to introduce; the format-handling story is fully covered by ADR-028 + ADR-031 once §D8 is real in code. See §6 below and `docs/architecture/ARCHITECTURE.md` "Format handling: canonical zone vs boundary" for the documented model.

#### 2.4 What this ADR is NOT

- **Not a new architectural decision about format**. Format handling lives in ADR-028 §D8 (block-centric) + ADR-031 (canonical Zarr/Arrow inside the graph). This ADR consumes those; it does not reverse or extend them.
- **Not a re-numbering of ADR-005**. ADR-005 is marked superseded by this ADR; its old `inline` / `function` modes are removed without backward-compat shim (SciEasy is pre-1.0).
- **Not a redesign of the block category taxonomy**. CodeBlock-as-AppBlock-subclass is the second case study (after AIBlock per ADR-035); a formal taxonomy refresh remains deferred.

### 3. Decisions

#### D1. CodeBlock extends AppBlock

`class CodeBlock(AppBlock)` — structural subclass. Inherits file-exchange bridge, ProcessHandle/cancellation (ADR-019), watcher, output binner, exit-code handling.

One small AppBlock parent extension is folded into the §D8 cluster (specifically #1080): the watcher's "completion = output files appear OR user clicks Mark done" model also accepts **"process exits with code 0"** as a clean completion signal. Currently the watcher raises `ProcessExitedWithoutOutputError` if files don't appear before exit; for CodeBlock, exit code is the authoritative completion signal and the absence of output files at exit-0 simply means "the script declared no outputs".

#### D2. Inline mode and function mode are removed

CodeBlock v2 has **no `mode` config field**. The block has exactly one input: a script file path. The script is treated as opaque — SciEasy launches the appropriate interpreter on it.

Migration policy: existing CodeBlock instances using inline or function mode break. Acceptable because SciEasy has not yet released; documentation guides affected users to either (a) extract inline code into a `.py` file under `<project>/scripts/`, or (b) write a real ProcessBlock via the Monaco editor (ADR-036) + agent scaffolding (ADR-040). No backward-compat shim — see ADR-028 D14 (a precedent for "pre-1.0 → break cleanly").

#### D3. Interpreter dispatch by file extension

The CodeBlock framework holds a small map of extension → interpreter command. Users do not select interpreter; it is determined by extension. UI shows the detected interpreter as read-only confirmation ("Detected: papermill via /opt/conda/envs/myenv/bin/python").

| Extension | Interpreter | Notebook? |
|---|---|---|
| `.py` | `uv run` (auto mode) or `<env_python>` (existing mode) | no |
| `.ipynb` | `papermill` via chosen Python env | yes |
| `.qmd` | `quarto render` | yes |
| `.R` | `Rscript` | no |
| `.Rmd` | `Rscript -e 'rmarkdown::render(...)'` | yes |
| `.sh` | `bash` | no |
| `.m` | `matlab -batch` (fallback `octave` if no MATLAB) | no |
| `.mlx` | `matlab -batch` (octave does not support `.mlx`) | yes |

Engine does not "know" what these formats mean semantically; the map says "this extension → invoke this command". `.jl` is deferred to v2 (interpreter map slot kept; no implementation until a real user need surfaces).

#### D4. Script must live inside the project

Per ADR-039 (git auto-commit on Run), source files outside the project tree cannot be pinned to the run's `workflow_git_commit`. To preserve lineage integrity:

- Configuring a `script_path` outside `<project_dir>` triggers a UI dialog: **Copy into project / Symlink into project / Cancel**.
- Default location convention: `<project>/scripts/` (soft — files elsewhere in the project tree are accepted; UI shows a non-blocking "consider moving to scripts/" suggestion).

This guarantees ADR-038 lineage (`workflow_yaml_snapshot` + `workflow_git_commit`) and ADR-039 (pre-run auto-commit captures the exact script state) work for CodeBlock runs.

#### D5. Environment selector — auto / existing

Block config exposes an `environment` selector with two modes:

**Auto** (default for `.py`):
- For `.py`: PEP 723 inline metadata (`# /// script ... # ///`) → `uv run` creates and caches a venv at `<project>/.scieasy/script-envs/<script-hash>/`.
- For `.R`: sibling `renv.lock` → `renv::restore` into per-script library.
- For `.ipynb`: sibling `requirements.txt` or `%pip install` cell detection.
- **Fallback when no metadata**: use project-level `pyproject.toml` / `requirements.txt`; final fallback is bundled Python (will likely raise `ModuleNotFoundError` with a UI suggestion to add dependencies).

**Existing**: user picks from a discovered list. v1 discovery scope:
- conda: parse `conda env list` (covers miniconda / miniforge / mambaforge / anaconda).
- venv: scan `~/.virtualenvs/`, `<project>/.venv/`, `<project>/venvs/*`.
- uv: `uv venv --list`.
- **Browse...** button: manual interpreter path for power users / not-discovered envs.
- R: detect system R + `renv` profiles.
- MATLAB: detect `matlab` binary + version, fallback `octave`.

Out of v1 (TODO + tracking issue at implementation time): pyenv, rye, hatch env discovery.

**Lineage capture**: regardless of mode, the engine captures `pip freeze` / `sessionInfo()` per run. This requires a small extension to ADR-038 — a `block_executions.environment_snapshot` TEXT column (or reuse of `block_config_resolved`'s JSON to nest it). Decided in implementation.

#### D6. User's own packages — not supported in v1

Sibling `from utils import foo` works because the script's directory is added to `sys.path` (D9). Editable installs (`pip install -e .`) and arbitrary local package paths are NOT supported in v1. Users with custom packages should pick **Existing** mode and select an env that has those packages already installed.

#### D7. Supported script formats in v1

`.py`, `.ipynb`, `.qmd`, `.R`, `.Rmd`, `.sh`, `.m`, `.mlx`.

Explicitly deferred (each gets a tracking issue at implementation time):

- `.jl` (Julia)
- Container scripts (`docker run`, `apptainer`, `singularity`) — properly an `AppBlock` track, not CodeBlock.
- Workflow languages (Nextflow, Snakemake, CWL) — would need nested-DAG semantics; separate ADR.

#### D8. Single palette item, fixed UI

Palette shows one **"Code Block"** item. Drag onto canvas. Block config panel:

```
Environment:    ( ) Auto   ( ) Existing → [dropdown of discovered envs]
Script path:    [browse... ]   detected: <interpreter>
Delivery:       ( ) files   ( ) parameters
Input ports:    [port editor — name / type / extension]
Output ports:   [port editor — name / type / extension]
```

No per-script palette registration. No "scripts as registered blocks" mechanism. Simplicity over feature creep — adopting the AppBlock pattern keeps the palette uncluttered as the script library grows.

#### D9. Files mode delivery (default)

The engine mounts upstream port data into `<exchange_dir>/inputs/<port_name>.<ext>` and watches `<exchange_dir>/outputs/<port_name>.<ext>`. Working dir for the script is `<exchange_dir>`; the script's own directory is added to `sys.path` / `.libPaths()` so relative imports work.

Per-port extension selection in the port editor:

- **Default**: the first extension in the matching saver's `supported_extensions` for the declared port type (resolved at workflow-load time via `BlockRegistry.find_saver(type, ext)` — provided by #1077).
- **User override**: user picks a different extension from the saver's supported list.
- **Native (zero-copy)**: when source `storage_ref.format` (always Zarr/Arrow inside the canonical zone per §6) maps onto the target extension via a registered native-pass-through path, the engine uses `mount_pathlike` (symlink on POSIX, junction on Windows for dirs, hardlink for same-volume files, copy fallback). This is engine-mechanical; user does not configure it.

Memory-cost warnings (UI):
- Materialised input > 2 GB → yellow "this materialisation will be slow".
- Materialised input > 20 GB → red "OOM risk; consider a smaller input or rework via ProcessBlock".

Implementation: §D9 leans entirely on `materialise_to_file` (#1078). CodeBlock contributes the per-port dispatch loop and the file-watcher pattern; the actual format conversion lives in IOBlock savers/loaders per ADR-028 §D8.

#### D10. Parameters mode delivery

Header / cell-tag declares which inputs become script variables. The format-specific syntax follows existing community conventions where possible (no SciEasy invention required for `.ipynb` / `.qmd` / `.Rmd`):

| Script format | Header convention |
|---|---|
| `.py` / `.R` / `.sh` / `.m` | PEP 723-style comment block: `# /// scieasy\n# inputs: {...}\n# outputs: {...}\n# ///` |
| `.ipynb` | papermill standard: cell tagged `parameters` |
| `.qmd` / `.Rmd` | knitr `params:` YAML frontmatter |
| `.mlx` | TBD (v1.1) — fallback to `.m` header rules for now |

For files-mode users, the header block is optional (port definitions come from the block config). For parameters-mode users, the header is required and the framework parses it at workflow-load time to surface mismatches between header and config in the port editor.

#### D11. Notebook-type scripts auto-add an executed-notebook output port

When script extension is `.ipynb` / `.qmd` / `.Rmd` / `.mlx`, the framework automatically appends a variadic output port `_executed_notebook: Artifact` to the block (reusing ADR-029 variadic-port infrastructure). The executed notebook (with cells run, outputs embedded) lands here. The user does not configure this port; it is framework-managed.

Rationale: executed notebooks are reproducibility evidence (cells + outputs + timing in one artefact). Capturing them automatically into lineage (ADR-038) is the right default; opting out would surprise scientists who expect their `.ipynb` runs to produce a viewable result.

#### D12. Multi-language `.qmd` supports separate Python and R envs

The Environment selector for `.qmd` (and `.Rmd` if it embeds Python via `reticulate`) presents **two columns**: Python env + R env. Quarto orchestrates the cross-language calls; SciEasy passes both interpreters' paths via standard quarto config. Pure-Python or pure-R `.qmd` documents need only one selector (UI hides the other).

This supports the user requirement "both — `.qmd` must work for mixed-language documents" without forcing a single conda env that has everything.

#### D13. Scripts folder convention

`<project>/scripts/` is the default location for CodeBlock-driven scripts. UI file-browser defaults there. Scripts placed elsewhere in the project tree are accepted; UI shows a non-blocking suggestion "consider moving to scripts/". Scripts outside the project tree trigger D4's copy/symlink/cancel dialog.

Aligns with the existing `<project>/{workflows, blocks, notes}/` conventions documented in `docs/architecture/PROJECT_TREE.md`.

#### D14. AppBlock `prepare()` and `_bin_outputs_by_extension` upgrade (delivered by #1079 + #1080)

Both methods are upgraded to dispatch through `supported_extensions` ClassVar + `materialise_to_file` / `reconstruct_from_file` helpers from the §D8 implementation cluster.

**prepare()** (output direction — for AppBlock/CodeBlock inputs being written to disk): for each input port, resolves the target extension from port config or saver default; if source `storage_ref.format` allows native pass-through, uses `mount_pathlike`; otherwise materialises via `materialise_to_file`. The current JSON-dump dead path (`bridge.py:31-74`) is removed.

**_bin_outputs_by_extension** (input direction — for output files being lifted back to typed DataObjects): uses `reconstruct_from_file(path, target_type, ext)` to build typed instances; falls back to `Artifact` only when no matching loader exists. The silent Artifact-downgrade-with-warning at `app_block.py:240-258` is removed.

CodeBlock inherits these upgraded methods directly. Concrete AppBlock subclasses (`FijiBlock`, `NapariBlock`) may continue to override but no longer need to — the framework default handles typed I/O correctly once #1079 + #1080 land.

#### D15. Function mode is removed (final)

CodeBlock v2 has no `function` delivery mode. Users wanting framework-aware, no-IO-overhead processing write a real `ProcessBlock` (in the Monaco editor per ADR-036, or via agent scaffolding per ADR-040). Removed because:

- Function mode requires script restructuring → directly violates v2's "inclusive, no-modification" goal.
- Custom ProcessBlocks are the proper path for power users.
- Reduces UI mode-count from 3 (inline / function / script) to 2 (files / parameters).

### 6. Architectural framing (canonical zone vs boundary)

This section restates the format-handling model so ADR-041 is self-contained for future readers. The model is also documented in `docs/architecture/ARCHITECTURE.md` (closes #1081) as the canonical reference.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    canonical zone (Zarr / Arrow only)                │
│                                                                      │
│   ProcessBlock ──► ProcessBlock ──► ProcessBlock ──► CustomBlock     │
│                                                                      │
│   storage_ref.format == "zarr" (or "arrow") everywhere               │
│   format is not a dispatch concern within this zone                  │
└─────────────────────────────────────────────────────────────────────┘
                  ▲                                       ▼
            ┌─────┴─────┐                          ┌──────┴──────┐
            │  Load     │                          │  Save       │
            │ (boundary)│                          │ (boundary)  │
            └───────────┘                          └─────────────┘
                  ▲                                       ▼
                  │            ┌─────────────────────┐
                  └───────────►│ AppBlock / CodeBlock │
                               │  / AIBlock           │
                               │ (boundary)           │
                               └─────────────────────┘
```

**Five format-aware boundary points**:

1. **Load**: read user's arbitrary-format file → write canonical Zarr/Arrow → return reference-only DataObject (per ADR-031 D4).
2. **Save**: read canonical store → write user's chosen output format.
3. **AppBlock invocation**: prepare canonical data as file(s) in the external tool's expected format; collect output files back into typed DataObjects.
4. **CodeBlock invocation** (this ADR): same pattern; "external tool" is a script interpreter.
5. **AIBlock invocation** (per ADR-035): same pattern; "external tool" is `claude` / `codex`.

**Inside the canonical zone**:
- `storage_ref.format` is always `"zarr"` or `"arrow"` (or `"file"` for Artifact).
- Process / Code / Custom blocks operate on canonical data via the `to_memory()` / `sel()` / `iter_chunks()` interfaces.
- Format is not part of edge compatibility checks; only type compatibility matters.

**Format conversion** is naturally `Load(format_A) → Save(format_B)`. There is no "format converter block" concept; the engine supports conversion as a side effect of the standard Load/Save path. For users who need a specialised conversion tool (e.g. `bfconvert` for OIR → OME-TIFF), that's an `AppBlock` whose input and output ports both accept `Artifact` (file-path level), operating outside the canonical zone.

**Provenance vs dispatch**:
- `framework.source` (every DataObject) and `Image.Meta.source_file` (imaging plugin) record the original input path / format as provenance strings. These do **not** participate in dispatch.
- `storage_ref.format` is the current canonical format-key. Inside the zone it is invariably `"zarr"` / `"arrow"`; only at boundaries does it transiently take other values (during materialisation to a tool's expected format).

This framing was the implicit assumption of ADR-028 + ADR-031 + the AppBlock pattern but was never stated as a coherent picture. The ARCHITECTURE.md addition (closing #1081) makes it the canonical reference for plugin authors and future ADRs.

### 8. Architectural impacts and review notes

- **Inclusivity principle (CLAUDE.md §2.4)**: this ADR is the strongest expression of "everything connectable, not everything native" yet — the user's existing `.py` / `.R` / `.qmd` / `.m` script becomes a workflow node with zero modification (files mode) or 4 lines of header (parameters mode).
- **Block category collapse signal**: CodeBlock-as-AppBlock-variant + AIBlock-as-AppBlock-variant (ADR-035) suggests the long-term block taxonomy converges on `ProcessBlock` (framework-internal, typed contract) and `AppBlock` (external wrapping). Worth surfacing in a future architecture refresh; not in scope here.
- **ADR-031 reference-only contract**: native pass-through (D9) is the canonical case-study for ref-only at the AppBlock boundary; the existing `_load_zarr` + `_win_junction` precedents (already in code) are the implementation seeds.
- **Documentation impact** (substantial):
  - `docs/block-development/external-apps.md` — add "your script as a CodeBlock" section.
  - `docs/specs/` — possibly a `code-block-v2-spec.md` for header conventions (D10).
  - `docs/architecture/ARCHITECTURE.md` — section added in this PR (closes #1081).
  - `docs/architecture/PROJECT_TREE.md` — update `src/scieasy/blocks/code/` annotations once Phase 1 lands.

### 9. Open questions for review

1. **D11 executed-notebook port toggle**: should the auto-added port be required (always present) or optional (config toggle to disable)? Current decision: always present, no toggle (notebook output is reproducibility evidence, not optional). Open for review.
2. **D5 third env mode "Project-level"**: should the env selector expose a third explicit option for `<project>/.venv/` as a peer to Auto/Existing? Currently subsumed under Auto's fallback. Promoting it would surface a clear "project python" affordance in the UI.
3. **D14 binner disambiguation**: when multiple loaders claim the same extension (e.g., generic `LoadData` and plugin `LoadImage` both register `.tif` in some configuration), the prefer-most-specific-type rule needs precise definition. Resolution: rank by `len(accepted_types[0].__mro__)` descending; ties → first-registered. Documented at implementation time in #1077.

### 10. What this ADR is NOT deciding

- Block category taxonomy reshape (deferred future ADR).
- WorkflowBlock category for nested Nextflow / Snakemake / CWL (separate future ADR).
- Multi-script-per-block (Snakemake-style main + helper scripts). v1 = one script per block.
- stdin/stdout streaming I/O for CodeBlock. v1 = files only.
- Registering scripts as palette items (`scieasy.scripts` entry-point group). v1 = file-path-driven instances under the generic CodeBlock palette item.
