# ADR-023: Frontend Layout Redesign With Inline Config, Data Preview, And Start-From-Here Execution

## 1. Decision Summary

SciStudio's workflow editor uses a richer three-column canvas layout with an
inline block configuration surface, a dedicated data preview panel, a bottom
panel for secondary workspaces, and a backend-supported "start from here"
execution mode. Project management is also part of the GUI workflow: users can
create, open, save, switch, and close projects from the frontend.

This ADR preserves the 2026-04-04 frontend redesign and folds in its project
management addendum. The frontend may evolve in implementation detail, but the
architectural boundary is stable: the backend remains the source of truth for
workflow definition, project state, execution state, cached outputs, previews,
and selective re-run validation.

### 1.1 Problems Addressed

| Problem | Risk | ADR-023 response | Detailed section |
|---|---|---|---|
| The original three-panel layout had no durable data preview surface | Users would leave the workflow editor to inspect outputs | Add a right-side Data Preview panel with lazy, type-specific preview loading | Section 3 |
| Config and preview competed for the same right panel | Editing and execution workflows would fight for screen space | Put top parameters inline on nodes and full config in the bottom panel | Section 4 |
| Users need execution logs, AI chat, lineage, and problems without modal churn | Important runtime context would be hidden or detached | Add a bottom panel with browser-style tabs | Section 5 |
| Iterative science workflows need partial re-execution | Users would re-run expensive upstream work unnecessarily | Add `execute-from` API and scheduler support for checkpoint-backed selective re-run | Section 6 |
| Project lifecycle was missing from the GUI flow | Non-code users could not naturally create or switch SciStudio projects | Add project UI and API response contracts | Section 7 |
| Frontend contracts need backend-auditable anchors | UI ADRs can drift from executable API behavior | Govern request/response schemas, API routes, and scheduler method contracts | Section 10 |

## 2. Context

ADR-014 established ReactFlow and FastAPI as the frontend-backend stack. The
initial visual shape was a simple block palette, canvas, and config panel.
That sketch was enough to prove the stack, but not enough for day-to-day
scientific workflow editing.

Real users need to inspect outputs, tune parameters, watch logs, understand
lineage, ask AI-assisted authoring questions, and re-run only the downstream
portion affected by a change. They also need project operations exposed as GUI
actions instead of only CLI commands.

## 3. Layout Decision

The editor has:

- a fixed toolbar across the top;
- a left block palette;
- a central ReactFlow canvas;
- a right Data Preview panel;
- a bottom panel below the canvas.

The block palette and preview panel run from the toolbar to the bottom of the
viewport. The bottom panel belongs to the center canvas area. Palette, preview,
and bottom panel dimensions are resizable and collapsible.

The right panel is preview-first, not config-first. Preview is the stronger
need during execution and review.

## 4. Block Node And Port Design

Canvas nodes are custom ReactFlow nodes. They show:

- category icon and block name;
- run controls, including a "start from here" affordance;
- the highest-priority config parameters inline;
- left input ports and right output ports;
- state badge and compact runtime summary.

The full config form lives in the bottom panel. Inline fields are for the top
parameters only; all parameters, descriptions, validation hints, and advanced
controls remain accessible in the complete config tab.

Ports use type-driven visual styling. Base data types receive stable colors,
collection ports use an additional visual distinction, and subtype-specific
ring color may be exposed through backend schema metadata.

## 5. Data Preview And Bottom Panel

The Data Preview panel renders the selected block's outputs lazily. It supports
type-specific preview payloads such as:

- tables for `DataFrame` and table-like data;
- images for array/image data;
- charts for series/spectrum data;
- text for textual outputs;
- metadata and links for artifacts;
- expandable slot previews for composite data.

Preview fetches are on demand through the data preview API and cached on the
frontend. The frontend must not eagerly load every item in a large collection.

The bottom panel contains secondary workspaces such as Config, Logs, AI Chat,
Lineage, Jobs, and Problems. The MVP surface may ship a subset, but the panel
is the reserved architectural home for those views.

## 6. Start-From-Here Execution

"Start from here" re-runs a selected block and its downstream descendants while
reusing cached upstream outputs. The frontend sends:

```text
POST /api/workflows/{workflow_id}/execute-from
```

The backend validates that every predecessor of the selected block has cached
output in the checkpoint. On success, the scheduler marks predecessors done
with cached outputs, resets the selected block and descendants to idle, clears
their cached outputs, and dispatches ready work. If required upstream output is
missing, the API returns a conflict or validation error instead of silently
running from an invalid state.

The frontend never computes this state transition locally. It sends the
request and renders the event stream and returned response.

## 7. Project Management Addendum

The toolbar includes project operations before ordinary file workflow actions:

- create a project;
- open a known or browsed project;
- save current project and workflow state;
- switch recent projects;
- close the current project.

When no project is open, the canvas shows a welcome screen rather than an
empty ReactFlow graph. Project API responses include identity, name,
description, path, last-opened metadata, and workflow count.

## 8. Backend Source Of Truth

The frontend store is a mirror, not the authority. The backend owns:

- workflow definitions and validation;
- project creation and loading;
- block registry schemas and port metadata;
- execution state and terminal transitions;
- checkpointed intermediate references;
- data preview materialization;
- lineage and run history.

Frontend state may cache, select, resize, and present. It must not become the
runtime source of truth for execution or persistence.

## 9. Scope

In scope:

- frontend layout and panel semantics;
- block node display model;
- type-driven port rendering inputs;
- preview API consumption;
- project menu and welcome screen behavior;
- `execute-from` API and scheduler behavior.

Out of scope:

- implementing a full scientific renderer for every plugin data type in this
  ADR alone;
- replacing checkpoint or lineage persistence;
- making frontend state authoritative;
- automatic stale-graph invalidation on every config edit.

## 10. Verification And Tooling Impact

Tests should verify project CRUD responses, workflow selective re-run API
behavior, scheduler `execute_from()` precondition checks, data preview payloads,
frontend lazy preview loading, project welcome/open flows, and that API and
WebSocket routes are not shadowed by SPA routing from ADR-024.

### 10.1 Signature-Level Contracts

```text

    block_id: str


class ExecuteFromResponse:
    workflow_id: str

async def execute_from_workflow(
    workflow_id: str,

    runtime: RuntimeDep,

class DAGScheduler:
    async def execute_from(self, block_id: str) -> None: ...
```

## 11. Consequences

- The right panel becomes Data Preview rather than Config.
- Block nodes become denser and more self-describing.
- The bottom panel becomes a major extension surface.
- Selective re-run depends on checkpoint quality and cached output survival.
- The backend must expose enough schema and preview metadata for the frontend
  to render typed ports and previews without hardcoded scientific assumptions.

## 12. Alternatives Considered

Keeping config in the right panel was rejected because preview is more valuable
during execution.

Floating config modals were rejected because they block canvas work.

No inline config was rejected because users need to scan important parameters
directly on the graph.

Automatic stale invalidation was considered but deferred in favor of an
explicit "start from here" action controlled by the user.
