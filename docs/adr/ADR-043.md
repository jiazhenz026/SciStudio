# ADR-043: IO Format Capability Registry

## 1. Decision Summary

ADR-043 defines a package-level IO format capability model for SciStudio. It
replaces the current loose `supported_extensions`-only dispatch model with a
structured capability registry built from IOBlock declarations.

The decision preserves ADR-041's canonical-zone model: data types do not own
file formats, and workflow edges inside the canonical zone remain type-only.
File formats are boundary concerns owned by IO loaders, IO savers, AppBlocks,
and CodeBlocks.

### 1.1 Problems Addressed

| Problem | Risk | ADR-043 response | Detailed section |
|---|---|---|---|
| One package may need many file-format handlers | Exposing one block per handler creates unusable palettes | Introduce user-facing aggregate IOBlocks backed by internal format capabilities | Section 3 |
| Multiple packages may claim the same extension and type | Current first-registered behavior is implicit and environment-dependent | Add capability IDs, defaults, and ambiguity errors | Section 6 |
| AppBlock/CodeBlock ports can declare type + extension combinations with no loader/saver | Failures occur after external tools run, producing late confusing errors | Validate boundary IO capability at workflow-load/run time | Section 7 |
| Data types and file formats are conceptually conflated | Type hierarchy may explode into format-specific subtypes | Keep formats on IO capabilities, not DataObject classes | Section 2 |
| Capability metadata can increase development burden for scientist users | Simple custom loaders become too hard to write | Add simple IO base classes and decorators that synthesize capabilities | Section 4 |
| Package authors need a clear contract for format support | Incomplete loader/saver declarations cause runtime surprises | Require structured package declarations and scan-time validation | Section 9 |
| External formats carry domain metadata | Pixel data may round-trip while scale/channel metadata silently disappears | Make typed `meta` fidelity an explicit capability field | Section 8 |

### 1.2 Accepted Decision

SciStudio will introduce IO format capabilities as structured declarations owned
by IOBlock classes or their package-level metadata.

A capability describes a single supported boundary conversion:

- direction: `load` or `save`;
- target data type;
- format identifier;
- extensions;
- user-visible label;
- internal handler;
- capability ID;
- default/priority metadata;
- round-trip status;
- typed `meta` fidelity.

User-facing IOBlocks may aggregate many capabilities behind one block. For
example, `Load Image` may offer TIFF, PNG, OIR, Zarr, and OME-TIFF in a config
dropdown while remaining a single palette item.

Simple local IOBlocks may use ergonomic base classes that synthesize
capabilities from minimal declarations. Full capability declarations are
required for published packages and ambiguous format cases, but they are not
required for a scientist writing a one-off local loader.

### 1.3 Scope

In scope:

- structured capability declarations for loader/saver IOBlocks;
- aggregate IOBlocks with dynamic format dropdowns;
- simple local IOBlock authoring paths;
- registry indexing and deterministic lookup;
- package scan-time validation;
- workflow validation for AppBlock/CodeBlock boundary ports;
- typed `meta` fidelity declarations for IO boundaries;
- replacement of silent first-registered dispatch where ambiguous.

Out of scope:

- moving extension declarations onto DataObject types;
- adding format-specific DataObject subclasses such as `TiffImage`;
- engine-level semantic guessing or extension alias prediction;
- automatic conversion inside canonical workflow edges;
- requiring ProcessBlock authors to understand external file formats;
- requiring every real-world format to be round-trip capable.

## 2. Type, Format, And Boundary Ownership

DataObject types describe semantic shape: `Image`, `Label`, `PeakTable`,
`MSRawFile`, `Artifact`, and similar concepts.

File formats describe boundary representation: TIFF, PNG, OIR, CSV, mzML,
Parquet, Zarr, HDF5, and similar encodings.

Formats MUST NOT become DataObject class attributes. The same extension may map
to different semantic types, and the same type may be supported by multiple
packages with different implementation quality, metadata support, or external
dependencies.

Therefore:

- DataObject classes remain format-independent.
- IOBlock capabilities declare format support.
- The registry answers capability questions.
- The workflow records the selected capability when ambiguity exists.

Internal runtime information, provenance rows, execution parameters,
environment snapshots, and block input/output edges belong to Lineage. They are
not format capability metadata.

Domain metadata is represented by the DataObject subtype's typed `meta` model.
ADR-043 uses `meta` as the only metadata surface relevant to IO format
fidelity. Free-form user metadata is out of scope and is expected to be handled
by a future metadata package.

## 3. Aggregate IOBlocks

Package authors SHOULD expose a small number of user-facing IOBlocks. They
SHOULD NOT expose one palette block per format when the block semantics are the
same.

Example:

```python
class LoadImage(IOBlock):
    direction = "input"
    output_type = Image

    format_capabilities = [
        FormatCapability(
            id="imaging.image.tiff.load",
            direction="load",
            data_type=Image,
            format_id="tiff",
            extensions=[".tif", ".tiff"],
            label="TIFF",
            handler="_load_tiff",
            is_default=True,
            roundtrip_group="imaging.image.tiff",
            metadata_fidelity=MetadataFidelity(
                level="typed_meta",
                typed_meta_reads=["axes", "pixel_size", "channels"],
            ),
        ),
        FormatCapability(
            id="imaging.image.png.load",
            direction="load",
            data_type=Image,
            format_id="png",
            extensions=[".png"],
            label="PNG",
            handler="_load_png",
            metadata_fidelity=MetadataFidelity(level="pixel_only"),
        ),
    ]
```

The UI renders one `Load Image` block. Its config panel exposes a format
dropdown populated from `format_capabilities`.

Internal handlers remain small and explicit. The aggregate block dispatches to
the selected handler. This keeps developer implementation precise while keeping
the user palette compact.

## 4. Developer Ergonomics For Custom IOBlocks

ADR-043 must not make simple local development feel like package-author
ceremony. The target local custom-block author is a scientist who can write
some Python, not a framework maintainer.

The model therefore has three authoring levels.

| Level | Intended author | Capability requirement |
|---|---|---|
| Ordinary custom ProcessBlock | Scientist/user | No format capability declaration |
| Simple custom IOBlock | Scientist/user | Minimal declaration; framework synthesizes capability |
| Published IO package | Package author | Explicit `FormatCapability` records and scan-time validation |

### 4.1 Ordinary Custom Blocks

Custom ProcessBlocks that operate on internal DataObjects are unaffected. They
consume objects that have already crossed the boundary and must not inspect or
depend on original external file extensions.

### 4.2 Simple Local Loaders

Simple local loaders may subclass `SimpleLoader` and implement one method:

```python
from pathlib import Path
from typing import Any, ClassVar

from scistudio.blocks.io.simple_io import SimpleLoader
from scistudio_blocks_imaging.types import Image


class LoadMyTiff(SimpleLoader):
    output_type: ClassVar[type] = Image
    extensions: ClassVar[list[str]] = [".tif", ".tiff"]
    format_id: ClassVar[str] = "tiff"

    def load_file(self, path: Path, config: dict[str, Any]) -> Image:
        ...
```

The framework synthesizes a capability equivalent to:

```python
FormatCapability(
    id="<local-module>.LoadMyTiff.load.tiff",
    direction="load",
    data_type=Image,
    format_id="tiff",
    extensions=[".tif", ".tiff"],
    label="TIFF",
    block_type="LoadMyTiff",
    handler="load_file",
    metadata_fidelity=MetadataFidelity(level="pixel_only"),
)
```

The default metadata fidelity is `pixel_only`; users do not need to understand
metadata fidelity until they want to make stronger promises.

### 4.3 Simple Local Savers

Simple local savers mirror loaders:

```python
from pathlib import Path
from typing import Any, ClassVar

from scistudio.blocks.io.simple_io import SimpleSaver
from scistudio_blocks_imaging.types import Image


class SaveMyTiff(SimpleSaver):
    input_type: ClassVar[type] = Image
    extensions: ClassVar[list[str]] = [".tif", ".tiff"]
    format_id: ClassVar[str] = "tiff"

    def save_file(self, obj: Image, path: Path, config: dict[str, Any]) -> None:
        ...
```

### 4.4 Published Package IOBlocks

Published packages SHOULD use explicit `FormatCapability` records when:

- one user-facing block supports multiple formats;
- multiple semantic types share the same extension;
- metadata fidelity is stronger than `pixel_only`;
- the package wants stable capability IDs for workflow replay;
- the package marks a load/save pair as round-trip capable.

This keeps local experimentation lightweight while giving published packages a
strong contract.

## 5. Internal Blocks Are Format-Independent

ProcessBlocks, ordinary custom blocks, and framework-internal blocks operate in
the canonical zone. They consume typed DataObjects and their typed `meta`, not
external files.

Therefore:

- a ProcessBlock must not require that an `Image` originally came from TIFF;
- PNG, TIFF, OIR, CZI, and Zarr loaders must all produce valid internal
  `Image` objects if they claim to load `Image`;
- format-specific assumptions must be normalized at the load boundary or
  represented as typed `meta`;
- blocks that truly need file-level TIFF structure are AppBlocks or
  Artifact-level tools, not ProcessBlocks.

If an algorithm needs scale, channel labels, acquisition timing, or similar
domain information, the input type's `meta` contract should express those
requirements. The original external format name is provenance, not a runtime
dispatch key for internal blocks.

## 6. Capability Selection And Ambiguity

The registry resolves capabilities by:

1. direction;
2. target type compatibility;
3. extension or format ID;
4. explicit capability ID, when supplied;
5. default/priority rules.

If exactly one capability matches, the registry returns it.

If multiple capabilities match:

- an explicit capability ID wins;
- otherwise an explicitly declared default wins;
- otherwise the most-specific type wins;
- if still tied, the registry MUST report an ambiguity error.

The previous final fallback, `first-registered wins`, is not acceptable for
runtime dispatch once capability validation is implemented. Registration order
may still be used for deterministic display ordering, but not as silent
semantic selection.

## 7. Workflow Boundary Validation

The workflow validator MUST validate IO boundaries before execution.

For AppBlock and CodeBlock input materialisation:

- each input port with declared `type + extension` MUST resolve to a saver
  capability;
- if no saver exists, validation fails;
- if multiple savers exist and no capability is selected, validation fails as
  ambiguous.

For AppBlock and CodeBlock output reconstruction:

- each output port with declared `type + extension` MUST resolve to a loader
  capability;
- if no loader exists, validation fails;
- if multiple loaders exist and no capability is selected, validation fails as
  ambiguous.

This turns errors such as `Label + .tif has no loader` into pre-run contract
validation failures rather than post-external-tool crashes.

## 8. Typed `meta` Fidelity At IO Boundaries

External formats often carry domain metadata. Examples include image physical
scale, z spacing, channel names, OME XML, acquisition time, instrument
identity, LC-MS run metadata, and sample annotations.

ADR-043 does not treat internal runtime metadata as IO metadata. Run IDs,
block execution records, resolved config, environment snapshots, and
input/output object edges belong to Lineage. Free-form user metadata is also
out of scope for this ADR and is expected to be handled by a future metadata
package.

The IO fidelity contract concerns the typed `meta` model on the DataObject
subtype.

### 8.1 Metadata Fidelity Levels

Capabilities declare exactly one fidelity level. The level is a promise about
what the loader or saver preserves at the IO boundary.

| Level | Definition | Use when |
|---|---|---|
| `pixel_only` | The capability promises only the primary data payload and the minimum structural fields required to construct the target DataObject. No domain metadata is promised. | A simple local loader/saver or a format with no reliable metadata mapping |
| `typed_meta` | The capability reads or writes a declared subset of the target type's typed `meta` fields. Metadata outside those declared fields is not promised. | The package maps external metadata into stable SciStudio fields such as `pixel_size`, `axes`, `channels`, or sample labels |
| `format_specific` | The capability preserves declared format-native metadata in addition to typed `meta`, but that metadata is still represented explicitly as typed fields, a typed sidecar, or a package-defined metadata object. | The package understands format-native metadata that has no stable cross-format SciStudio field yet |
| `lossless` | The capability can reconstruct the original boundary representation without semantic loss for the declared round-trip group. This includes primary data, declared typed `meta`, declared format-specific metadata, and any required raw representation. | The package provides tested load/save round-trip behavior for the same format family |

Most simple user-authored IOBlocks default to `pixel_only`. Published packages
SHOULD declare stronger fidelity only when tests cover it.

The levels are ordered by strength:

```text
pixel_only < typed_meta < format_specific < lossless
```

A stronger level includes the obligations of weaker levels. For example,
`format_specific` still needs to declare which typed `meta` fields are read or
written if it touches typed `meta`.

### 8.2 Level Usage Rules

#### 8.2.1 `pixel_only`

`pixel_only` is the conservative default. A loader at this level may populate
only structural metadata required to make the object usable, such as shape,
dtype, axes defaults, or table columns. It MUST NOT imply preservation of
scale, acquisition parameters, instrument settings, channel names, or other
domain metadata.

Example:

```python
MetadataFidelity(level="pixel_only")
```

UI behavior:

- no metadata preservation should be advertised;
- saving an object with meaningful typed `meta` through a `pixel_only` saver may
  show a lossy-save warning.

Validation behavior:

- no typed `meta` field validation is required;
- package tests need only prove that the primary payload is readable or
  writable.

#### 8.2.2 `typed_meta`

`typed_meta` means the capability maps external metadata into fields defined by
the target DataObject's typed `Meta` model. The package MUST list each field it
reads or writes.

Example:

```python
MetadataFidelity(
    level="typed_meta",
    typed_meta_reads=("pixel_size", "z_spacing", "channels"),
    typed_meta_writes=("pixel_size", "z_spacing", "channels"),
)
```

UI behavior:

- the UI may display which stable metadata fields are preserved;
- the UI may warn if a saver omits typed `meta` fields currently present on the
  object.

Validation behavior:

- every field in `typed_meta_reads` and `typed_meta_writes` MUST exist on the
  declared type's `Meta` model;
- package tests SHOULD cover representative reads and writes for each declared
  field.

#### 8.2.3 `format_specific`

`format_specific` means the capability preserves metadata that is meaningful
for the external format but is not yet part of the cross-format typed `Meta`
contract. This level is not a license for hidden blobs. The package MUST
declare how the format-specific metadata is represented inside SciStudio.

Allowed representations include:

- typed fields on the DataObject's `Meta` model;
- a typed package-specific metadata object referenced from `meta`;
- an explicit Artifact sidecar with a declared schema;
- a format-native metadata payload retained only when the round-trip group is
  selected.

Example:

```python
MetadataFidelity(
    level="format_specific",
    typed_meta_reads=("pixel_size", "channels"),
    format_metadata_reads=("ome_xml",),
    format_metadata_writes=("ome_xml",),
)
```

UI behavior:

- the UI may label the capability as preserving format metadata;
- the UI should distinguish stable typed `meta` from format-specific metadata.

Validation behavior:

- `format_metadata_reads` and `format_metadata_writes` MUST name declared
  package metadata fields or sidecar schemas;
- package tests SHOULD prove that declared format metadata survives at least a
  load-observe-save or load-save-reload cycle, depending on direction.

#### 8.2.4 `lossless`

`lossless` is the strongest claim. It means the package can preserve the full
declared boundary representation for a round-trip group. A `lossless`
capability MUST be paired with a compatible loader or saver through
`roundtrip_group`.

Example:

```python
MetadataFidelity(
    level="lossless",
    typed_meta_reads=("pixel_size", "z_spacing", "channels"),
    typed_meta_writes=("pixel_size", "z_spacing", "channels"),
    format_metadata_reads=("ome_xml",),
    format_metadata_writes=("ome_xml",),
)
```

UI behavior:

- the UI may advertise the capability as round-trip preserving only inside the
  declared round-trip group;
- the UI must not imply that `lossless` conversion holds across unrelated
  formats.

Validation behavior:

- `roundtrip_group` MUST be present;
- both load and save sides of the group MUST exist unless the capability is
  explicitly one-way and therefore cannot claim `lossless`;
- package tests MUST cover round-trip preservation for the declared primary
  data, typed `meta`, and format-specific metadata.

### 8.3 Capability Metadata Fields

```python
from dataclasses import dataclass
from typing import Literal


MetadataFidelityLevel = Literal[
    "pixel_only",
    "typed_meta",
    "format_specific",
    "lossless",
]


@dataclass(frozen=True)
class MetadataFidelity:
    level: MetadataFidelityLevel = "pixel_only"
    typed_meta_reads: tuple[str, ...] = ()
    typed_meta_writes: tuple[str, ...] = ()
    format_metadata_reads: tuple[str, ...] = ()
    format_metadata_writes: tuple[str, ...] = ()
    notes: str | None = None
```

For example, an OME-TIFF Image loader may declare that it reads
`pixel_size`, `z_spacing`, and `channels` into `Image.Meta`. A PNG saver may
declare `pixel_only`, making metadata loss visible rather than surprising.

### 8.4 Boundary Save Behavior

Saver capabilities MUST NOT imply that all source metadata is preserved. A
saver writes only the typed `meta` fields it declares in
`typed_meta_writes`, plus any format-specific metadata declared in
`format_metadata_writes`.

The UI may use fidelity declarations to warn users before lossy saves. For
example, saving an `Image` with `pixel_size` metadata through a `pixel_only`
PNG saver can display a warning that scale metadata will not be written.

## 9. Package Validity Rules

A package that declares IO format support MUST provide internally consistent
capability metadata.

Scan-time validation MUST check:

- referenced IOBlocks exist;
- referenced handlers exist;
- declared type is compatible with the block's effective port type;
- extensions are normalized lowercase strings with leading dots;
- capability IDs are globally stable and package-qualified;
- defaults do not conflict within the same `(direction, type, extension)`
  group;
- round-trip groups marked as round-trip have both load and save capabilities.
- metadata fidelity declarations reference fields that exist on the declared
  type's `Meta` model when `level` is `typed_meta` or stronger.

Packages MAY declare one-way formats. Examples:

- raw instrument formats that are load-only;
- report or notebook formats that are save-only;
- opaque Artifact formats that are copied rather than semantically parsed.

A package is invalid only when it claims round-trip support without both sides,
or when its declared capability metadata does not match the actual block
contract.

## 10. AppBlock And CodeBlock Port Config

AppBlock and CodeBlock port config SHOULD evolve from extension-only routing to
capability-aware routing.

A port may contain:

```yaml
name: label
type: Label
extension: .tif
capability_id: imaging.label.tiff.load
```

For user ergonomics, the UI may infer and prefill `capability_id` when there is
a unique match. If there is no unique match, the UI must ask the user to
choose.

The engine must treat `capability_id` as the stable replay key. Extension is
still useful for filenames and human readability, but capability ID is the
semantic IO selection.

## 11. API And Function Signatures

This section defines the expected public shape. Exact module names may change
during implementation, but any change must preserve the same responsibilities.

| `packages/scistudio-blocks-imaging/src/**/io/**` | Migrate LoadImage/SaveImage and future Label loaders to capabilities |
| `packages/scistudio-blocks-lcms/src/**/io/**` | Migrate LCMS loaders/savers, including one-way raw-file declarations |
| `docs/architecture/ARCHITECTURE.md` | Update format-handling and metadata sections to align with ADR-043 |
| `docs/block-development/**` | Document simple IO authoring and package capability authoring |

## 13. Implementation Expectations

Implementation should proceed in phases:

1. Add capability dataclasses/schemas and scan-time normalization.
2. Add `SimpleLoader` and `SimpleSaver` for local custom IOBlocks.
3. Extend `BlockRegistry` with capability indexing and query APIs.
4. Migrate existing `supported_extensions` declarations into compatibility
   capability records.
5. Add explicit ambiguity errors.
6. Update AppBlock/CodeBlock validation to require matching capabilities.
7. Update UI format dropdowns to read registry capabilities and persist
   `capability_id`.
8. Update package author docs and simple custom loader docs.

Backward compatibility may temporarily synthesize capabilities from existing
`supported_extensions`, but synthesized capabilities must be treated as
migration scaffolding, not the final package authoring model.

## 14. Alternatives Considered

### 14.1 Put Extensions On DataObject Types

Rejected.

This conflates semantic data shape with file representation, forces plugin
format knowledge into type definitions, and cannot represent multiple packages
with different loaders for the same type and extension.

### 14.2 One Loader Block Per Format

Rejected as the user-facing model.

It is reasonable internally for each format to have its own handler, but
exposing thousands of palette blocks creates poor UX and hides the conceptual
fact that the user wants to "load image data", not choose from an unbounded
block catalog.

### 14.3 Require Full Capability Records For Every Local User IOBlock

Rejected.

It gives the framework the cleanest metadata, but it asks too much of the
scientist-user profile. Local users should be able to write a short loader or
saver and let the framework synthesize conservative capability metadata.

### 14.4 Keep First-Registered Wins

Rejected for semantic dispatch.

It is deterministic within one registry instance but not robust across package
installation order, plugin availability, or future ecosystem growth.

### 14.5 Engine Guessing And Alias Rules

Rejected.

The engine must not infer `.tif` means `.tiff`, or that an `Image` loader should
construct a `Label`. Such behavior belongs in package-declared capabilities or
explicit user choices.

## 15. Consequences

Positive consequences:

- UI remains compact even when packages support many formats.
- Package IO contracts become explicit and testable.
- AppBlock and CodeBlock failures move from late runtime errors to validation.
- Ambiguous loader/saver situations become visible and user-resolvable.
- DataObject types stay stable and format-independent.
- Simple local custom IOBlocks remain approachable for scientist users.
- Metadata loss becomes visible through capability fidelity declarations.

Negative consequences:

- Package authors must maintain more structured metadata.
- Registry implementation becomes more complex.
- Existing `supported_extensions` users need migration support.
- Workflow YAML may need to record capability IDs for reproducibility.
- The UI needs to surface capability ambiguity and metadata-loss warnings
  without overwhelming users.

Open follow-up:

- Decide exact schema name and location for `FormatCapability`.
- Decide whether capability declarations live only on IOBlock classes or may
  also be package-level manifests.
- Decide UI ordering and grouping rules for large capability lists.
- Decide whether metadata fidelity field names are strings or validated paths
  against the target type's `Meta` model.
