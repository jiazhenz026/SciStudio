## ADR-039: Git-Backed Source Version Control for SciEasy Projects

**Status**: **proposed** ‚Äî design accepted in session 2026-05-15; implementation tracked in phased post-merge issues. Promotes to "accepted" after Phase 1 wires `pygit2` into project lifecycle with a green commit/restore smoke test.
**Date**: 2026-05-15
**Related**: ADR-014 (FastAPI + React stack), ADR-036 (embedded code editor), ADR-037 (desktop packaging ‚Äî pygit2 bundles here), ADR-038 (run lineage ‚Äî sibling: `runs.workflow_git_commit` is the join key)

---

### 1. Purpose

Every SciEasy project becomes a git repository. Workflow YAML, custom block Python files, and project notes are version-controlled with standard git semantics: history, diff, restore, branch. Commits are **manual** ‚Äî the user decides what and when, like every other use of git.

This ADR is the source-version half of the history model. Its sibling ADR-038 covers run-level history (the recorded executions of those source files). Together they replace the prior implicit and confusing single-namespace "metadata" model.

---

### 2. Context

#### 2.1 Current state ‚Äî no version control

- Workflow YAML files are saved to `<project>/workflows/<workflow_id>.yaml`; each save **overwrites** the previous version. No history.
- Custom block files at `<project>/blocks/*.py` (per ADR-036) are similarly overwritten on save.
- Notes (`<project>/notes/*.md` per ADR-036 ¬ß3.7) are overwritten.
- `ApiRuntime.bump_revision` exists as an in-memory `revision: int` for optimistic-concurrency ETag checks (`If-Match` header), but **resets on server restart** and tracks only the active session ‚Äî not history.

The result: a researcher who wrote a workflow 4 months ago, modified it 2 months ago, and now wants the original cannot recover it from SciEasy. They can hope they have a backup or `git init` it themselves outside SciEasy. Neither is acceptable for the non-technical-user audience.

#### 2.2 What the user needs

From the requirements discussion 2026-05-15:

- Editorial history of **workflows / blocks / notes**: who changed what, when, why.
- **Restore** any prior version at any time.
- **Branching** for trying variants (e.g. "try a different segmentation pipeline without losing the current one").
- **Diff** view to see what changed between versions.
- Operations must feel familiar to anyone who knows git, and must be approachable to users who do not.

#### 2.3 Why git, not a custom solution

Git is 40 years old, format-stable, exhaustively tooled, well-understood, and there is no scientific upside to writing a competing source-version-control system. The architectural decision is "embed git" + "build SciEasy-shaped UI affordances on top". Per CLAUDE.md ¬ß2.4 "everything connectable, not everything native", git is the canonical "everything connectable" example.

#### 2.4 Relation to ADR-038

ADR-038 records each workflow run, including the field `runs.workflow_git_commit` ‚Äî the SHA of the commit the run executed against. The two ADRs share that single join key. Otherwise they are independent:

- ADR-039 (this) operates on text source files when the user clicks "Commit".
- ADR-038 operates on engine events when the user clicks "Run".

A run can reference a git commit; a git commit can be checked-out to see what produced any past run.

### 3. Decisions

#### 3.1 Git engine = `pygit2` (libgit2 binding)

`pygit2` is the Python binding to libgit2. Selected because:

- Mature, actively maintained, used in production by many tools.
- Native bindings ‚Üí fast: history walks, diffs, commits all ~10√ó faster than `dulwich`.
- Bundles into `python-build-standalone` cleanly via ADR-037 desktop packaging (libgit2 ships as a shared library, included in the resources tree).
- Does **not** require system `git` binary ‚Äî important for the desktop-app audience who do not have a terminal.

Alternatives rejected:

- **`dulwich`** (pure Python): slower (no native code), but more portable. Acceptable as a fallback if `pygit2` fails to build on a future platform; v1 mandates `pygit2`.
- **Shell out to system `git`**: requires git installation, breaks the desktop double-click-and-run promise.
- **`gitpython`** (also wraps system git): same problem.

#### 3.2 Auto-init on project open

When a SciEasy project is opened:

1. If `<project>/.git/` exists ‚Üí use it as-is.
2. If `.git/` does not exist ‚Üí `pygit2.init_repository(<project>)`, write the default `.gitignore` (¬ß3.3), make an **initial commit** containing whatever source files already exist (`workflows/*.yaml`, `blocks/*.py`, `notes/*.md`, `project.yaml`, `README.md`). Commit message: `Initial commit (auto-generated by SciEasy)`.

**No user prompt.** The user wants git versioning per the requirement set; defaulting to "yes, do it" is the right behavior. Users who want to disable can `rm -rf .git` and SciEasy will warn but proceed without versioning (degraded mode ‚Äî re-init available via Settings).

#### 3.3 Default `.gitignore` template

Written on `git init`; user may edit:

```gitignore
# SciEasy auto-generated .gitignore.
# Edit freely ‚Äî SciEasy will not overwrite this file.

# Data files (not versioned ‚Äî see ADR-038 for run lineage)
data/

# SciEasy runtime state (per-project, per-machine; not portable)
.scieasy/

# Python caches
__pycache__/
*.py[cod]
*$py.class

# OS noise
.DS_Store
Thumbs.db

# Plugin venvs (per ADR-037)
*-venv/
*.venv/

# Editor caches
.idea/
.vscode/
*.swp
```

**Note on `.scieasy/lineage.db`**: it is git-ignored by default. Rationale evaluated:

- SQLite binary, no useful diff/merge.
- Conceptually local: "runs that happened on this machine" travels with the machine, not the source. Two researchers cloning the same project each generate their own local lineage.
- Source still reproduces the runs: `workflow_yaml_snapshot` is also stored inside `lineage.db`, so the recipe survives independently when needed.

Users who explicitly want lineage shared with collaborators can delete the `.scieasy/` line from `.gitignore`. The ADR documents the trade-off so it is a conscious choice.

#### 3.4 Zero auto-commit ‚Äî user commits manually

Git was not designed for auto-commit. Every save-triggered auto-commit pollutes history and forces the user to navigate around noise. SciEasy does **not** auto-commit on:

- File save (workflow YAML, .py, .md)
- Project open
- Workflow run start or completion

The user clicks **Commit** explicitly. A `CommitDialog` opens with a pre-filled template (see ¬ß3.5).

Implication for ADR-038: `runs.workflow_git_commit` records the HEAD SHA at run start. The `workflow_dirty` field records whether the working tree had uncommitted changes ‚Äî usually true during active editing. The full YAML snapshot in `workflow_yaml_snapshot` still preserves reproducibility regardless of dirty state.

#### 3.5 v1 feature set ‚Äî MUST-haves

The following are non-negotiable for v1 ship:

| Feature | UI element | API endpoint |
|---|---|---|
| **Commit** with author + message | `CommitDialog.tsx` with multi-line message textbox + Commit button | `POST /api/git/commit` `{message, author?, files?}` |
| **History (`git log`)** | `GitHistoryList.tsx` left-pane reverse-chrono list: SHA / time / message / author | `GET /api/git/log?branch=...&limit=...` |
| **Diff** between two commits or commit-vs-working | `GitDiffModal.tsx` using `react-diff-viewer-continued` for text files | `GET /api/git/diff?from=<sha>&to=<sha\|HEAD\|WORKING>&file=<path>` |
| **Restore** any prior version (`git checkout`) | "Restore this version" button on each history row | `POST /api/git/restore` `{commit_sha, files?}` (files=[] means whole worktree) |
| **Branch list + switch** | `BranchPicker.tsx` toolbar dropdown | `GET /api/git/branches`, `POST /api/git/branch/switch` `{name}` |
| **Branch create** | "New branch" item inside `BranchPicker` | `POST /api/git/branch/create` `{name, from_sha?}` |
| **Working-tree status** | Toolbar badge: dirty / clean | `GET /api/git/status` |

Commit dialog pre-filled template:

```
<one-line subject>

# What changed:
# - workflows/<id>.yaml: ‚Ä¶
# - blocks/<file>.py:    ‚Ä¶

# Auto-detected modified files:
#   M  workflows/image_pipeline.yaml
#   M  blocks/cellpose_wrap.py
#   A  notes/2026-05-15-thoughts.md
```

The `# ` lines are git-style comments (stripped at commit time). The auto-detected file list is generated from `pygit2.Repository.status()`.

#### 3.6 Restore semantics

"Restore this version" defaults to **`git checkout <sha> -- <files>`** restoring listed files to the working tree without changing HEAD. This is the safer mode: user can keep editing, see what the old version looked like, decide whether to commit.

Optional "Hard restore (move HEAD)" advanced action does `git reset --hard <sha>` ‚Äî gated behind a confirmation dialog explaining the consequences. v1 surfaces the soft restore prominently; hard restore is deferred or hidden until user feedback shows demand.

If the working tree is dirty during restore, SciEasy auto-stashes (`pygit2 stash`) first, restores, and prompts: "Your unsaved changes are stashed. Apply them? [Apply now / Keep stashed / Discard]".

#### 3.7 Branch UI scope

Branch operations covered:
- List existing branches
- Switch to an existing branch (only if working tree is clean; otherwise stash dialog)
- Create new branch from current HEAD or from any historical commit
- Delete a branch (with confirmation)

Branch operations **NOT covered in v1**:
- Merge between branches (deferred; users with merge needs use git CLI externally)
- Conflict resolution UI (same)
- Branch graph visualization (defer to v2; `@gitgraph/react` evaluated as future option)

When the working tree is on a branch other than the one used during a run, the run record's `workflow_git_commit` SHA still resolves correctly ‚Äî branches are pointer abstractions over commits.

#### 3.8 External git changes are respected

If a user runs `git commit -a` from a terminal outside SciEasy:

1. SciEasy's existing file watcher (ADR-034 Phase 2 `workflow_watcher`) detects YAML changes ‚Üí refreshes canvas.
2. A new git-aware watcher detects HEAD changes (poll `.git/HEAD` and `.git/refs/heads/` mtime, or `pygit2.Repository.head` revision check on file activity) ‚Üí refreshes `BranchPicker` + `GitHistoryList`.
3. No locks on `.git/` are taken. Users can run arbitrary git CLI alongside SciEasy.

#### 3.9 Pre-existing projects without `.git`

Per ¬ß3.2, projects opened that lack `.git/` get auto-init. The initial commit captures whatever already exists in the project (workflow YAML, .py, .md files). No prompt; this is the right default given the user-stated requirement.

If a user wants to opt out, they can:
- Delete `.git/` from the filesystem (SciEasy will not aggressively re-init ‚Äî only on project open if absent and no `.scieasy/no_git` marker)
- Add a `.scieasy/no_git` marker file to suppress re-init on subsequent opens

The opt-out path is documented but does not have a Settings checkbox ‚Äî it is intentionally inconvenient because the use case is rare.

#### 3.10 Out of scope (defer to future ADRs)

- **Remote push / pull (GitHub, GitLab, etc.)** ‚Äî v1 is local-only. Users who want to push their project elsewhere can use git CLI manually with `git remote add` + `git push`. v2 may add a UI affordance.
- **Git LFS** ‚Äî `data/` is git-ignored; large binary files do not enter git. Users who explicitly want versioned data files can configure LFS manually.
- **Conflict resolution UI** ‚Äî merge conflicts are sufficiently complex that even mature tools (VS Code, SourceTree) struggle. SciEasy v1 defers to git CLI.
- **Submodules** ‚Äî not used by any SciEasy workflow; out of scope.
- **Hooks (`pre-commit`, etc.)** ‚Äî users can configure manually in `.git/hooks/`; SciEasy does not write hooks.
- **Tag management** ‚Äî git tags work via CLI; UI element deferred.

### 4. Alternatives considered

1. **No versioning** ‚Äî current state. Rejected per ¬ß2.1: 4-month-rollback is a user-stated requirement.
2. **Custom history mechanism (snapshot files in `.scieasy/`)** ‚Äî Rejected: reinvents git poorly. Forty years of tooling, conventions, and user familiarity argue for embedding the real thing.
3. **`dulwich` (pure Python)** ‚Äî Rejected for v1 per ¬ß3.1 performance; available as a fallback if `pygit2` build issues arise on an OS.
4. **Shell out to system `git`** ‚Äî Rejected: requires system installation; breaks ADR-037 desktop double-click promise.
5. **Auto-commit on file save** ‚Äî Rejected per ¬ß3.4: pollutes history beyond usefulness; git was not designed for it.
6. **Auto-commit on workflow run** (pre-run squash) ‚Äî Considered and rejected. The user-control philosophy is consistent: commit is a user act. Run records pin the YAML snapshot regardless of commit state (ADR-038 ¬ß3.1 `workflow_yaml_snapshot`), so reproducibility does not require pre-run auto-commit.
7. **Force `.scieasy/lineage.db` into git by default** ‚Äî Rejected per ¬ß3.3: binary, machine-local, no merge story. User can opt-in by editing `.gitignore`.
8. **Pre-built React git UI library** ‚Äî Surveyed: no all-in-one option exists. `react-diff-viewer-continued` covers the diff piece; the rest (commit dialog, history list, branch picker) is ~500 LOC of straightforward React.

### 6. Phased implementation

**Phase 1 ‚Äî Engine + auto-init + REST API (~700 LOC)**

- File watcher extension to detect `.git/HEAD` changes.
- Tests: round-trip commit / restore / branch ops via REST; auto-init idempotence.

**Phase 2 ‚Äî Frontend UI (~700 LOC)**

- New `frontend/src/components/Git/` with CommitDialog / GitHistoryList / GitDiffModal / BranchPicker / GitStatusBadge / StashApplyDialog.
- New `frontend/src/store/gitSlice.ts`.
- Wire toolbar (`Toolbar.tsx`) with BranchPicker + GitStatusBadge + Commit button.
- Wire WebSocket handler to refresh on `git.head_changed`.
- Tests (vitest): Commit dialog populates message template; HistoryList renders commits; Diff modal renders text diff; BranchPicker switch handler.

**Phase 3 ‚Äî Polish + integration with ADR-038**

- ADR-038 `runs.workflow_git_commit` field populated by lineage recorder from `git_engine.head_state()`.
- "Restore this run's workflow" affordance in Lineage tab (per ADR-038 ¬ß3.8) wires to `gitRestore({commit_sha: run.workflow_git_commit, files: [workflow_yaml_path]})`.
- Stash-on-restore prompt.
- Status flips to `accepted`.

Phases are linear: 1 ‚Üí 2 ‚Üí 3.

### 7. Consequences

#### 7.1 Wins

- Forty years of git semantics, file format, and tooling come for free.
- The user can run external `git` (CLI, GUI tools, GitHub Desktop) alongside SciEasy without conflict.
- The 4-month-rollback user requirement is satisfied with a one-click restore.
- Branching enables variant exploration without copy-paste filename hacks.
- Reproducibility chain locks in: every run records the git commit it ran against (ADR-038), and git history preserves the source exactly.
- Custom block development gets normal source control automatically.

#### 7.2 Losses

- **pygit2 + libgit2 add ~3MB to the desktop bundle** (ADR-037).
- Users who do not understand git at all are exposed to "branch" and "commit" terminology. Mitigation: tooltips + minimal onboarding tooltip on first commit. Worst case: they ignore the version-control UI entirely and SciEasy still works (saves + runs proceed).
- Merge conflicts, if they occur, route the user to the git CLI. Acceptable v1 stance.
- The optimistic-concurrency ETag flow goes away. Concurrent multi-tab edits of the same workflow YAML can now race silently (last-write-wins on file save). The existing workflow_watcher emits `workflow.changed` on FS change ‚Üí other tabs invalidate cache. This is the same model as VS Code / most editors: detect external change, prompt to reload. Not worse than the current bump_revision (which only protects within one server process).

#### 7.3 Risks

- **pygit2 wheel availability across platforms** (especially Windows ARM, fresh Linux distros). Mitigation: bundle libgit2 with the desktop app (ADR-037); fall back to `dulwich` if pygit2 fails to import at runtime (not in v1; tracked as a future fallback path).
- **`.git/` corruption** if user or another tool wedges the repo. Mitigation: SciEasy operations on `.git/` are read-mostly + write-via-pygit2 (which is robust). If `.git/` is corrupt, SciEasy degrades to "no version control available" state with a Settings ‚Üí "Reinitialize git" recovery button.
- **Big binary mistakenly committed** (e.g. user removes `data/` from `.gitignore`). Mitigation: pre-commit size warning if any file > 10MB is staged. v1 implements the warning; cleanup (BFG / git-filter-repo) is user's responsibility.
- **Branch checkout while working tree is dirty**: SciEasy auto-stashes per ¬ß3.6; documented but a real failure mode if user does not understand stash. Mitigation: clear modal explaining what stash does and offering "Cancel" prominently.

### 8. Open questions

1. **OQ-1**: Initial commit author when no git config is set. Default to `SciEasy User <noreply@scieasy.local>`? Or prompt user once for name/email? Tentatively: prompt-once on first commit; cache in `<project>/.scieasy/git_author.json`.

2. **OQ-2**: Should `git tag` operations be exposed in v1? Tentative: no ‚Äî tags are rare for the target audience. Reconsider in Phase 3 polish.

3. **OQ-3**: Large file warning threshold. 10MB is a starting guess; some scientific output files (CSVs of analysis results) can legitimately be > 10MB. Soft warning + override is the right shape; tune the threshold based on user reports.

4. **OQ-4**: When ADR-038 Lineage tab shows a run with `workflow_dirty=true`, what message do we show? "This run executed against uncommitted changes. The workflow YAML snapshot was saved with the run for full reproducibility." Tentatively yes.

5. **OQ-5**: Multi-tab editing ‚Äî if the user has the same workflow open in two browser tabs and saves from both: the second save wins on disk; the file watcher invalidates the first tab's cache; user gets a "Reload? Discard local edits?" prompt. Same as VS Code. Decide UX in Phase 2.

### 10. Decision summary

**Every SciEasy project is a git repository. `pygit2` is the engine. Auto-init on project open. Manual commit, never auto-commit. Default `.gitignore` excludes `data/` and `.scieasy/`. v1 ships the four MUST features (commit, history, diff, restore) plus branch list/switch/create. Merge UI, remote sync, and LFS are explicitly v2.** The reproducibility chain joins via `runs.workflow_git_commit` (ADR-038): a run record points to the exact commit that produced it.

The architectural invariant: **history is two layers ‚Äî source via git (this ADR) and runs via lineage.db (ADR-038). The conflation of these two concerns ends here.**
