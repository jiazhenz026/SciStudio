## ADR-039: Git-Backed Source Version Control for SciEasy Projects

**Status**: **accepted** — promoted 2026-05-15 by D39-2.5 (issue #954). The bundled-git engine, REST surface, frontend UI, conflict resolution, branch-graph visualization, and ADR-038 lineage join (`runs.workflow_git_commit` populated by `start_workflow` per §3.4 + §6 Phase 4) are all wired. Phase 3.5 + Phase 4 e2e remain.
**Date**: 2026-05-15
**Related**: ADR-014 (FastAPI + React stack), ADR-036 (embedded code editor), ADR-037 (desktop packaging — bundled git binary ships here), ADR-038 (run lineage — sibling: `runs.workflow_git_commit` is the join key)

---

### 1. Purpose

Every SciEasy project becomes a git repository. Workflow YAML, custom block Python files, and project notes are version-controlled with standard git semantics: history, diff, restore, branch. Commits are **manual** — the user decides what and when, like every other use of git.

This ADR is the source-version half of the history model. Its sibling ADR-038 covers run-level history (the recorded executions of those source files). Together they replace the prior implicit and confusing single-namespace "metadata" model.

---

### 2. Context

#### 2.1 Current state — no version control

- Workflow YAML files are saved to `<project>/workflows/<workflow_id>.yaml`; each save **overwrites** the previous version. No history.
- Custom block files at `<project>/blocks/*.py` (per ADR-036) are similarly overwritten on save.
- Notes (`<project>/notes/*.md` per ADR-036 §3.7) are overwritten.
- `ApiRuntime.bump_revision` exists as an in-memory `revision: int` for optimistic-concurrency ETag checks (`If-Match` header), but **resets on server restart** and tracks only the active session — not history.

The result: a researcher who wrote a workflow 4 months ago, modified it 2 months ago, and now wants the original cannot recover it from SciEasy. They can hope they have a backup or `git init` it themselves outside SciEasy. Neither is acceptable for the non-technical-user audience.

#### 2.2 What the user needs

From the requirements discussion 2026-05-15:

- Editorial history of **workflows / blocks / notes**: who changed what, when, why.
- **Restore** any prior version at any time.
- **Branching** for trying variants (e.g. "try a different segmentation pipeline without losing the current one").
- **Diff** view to see what changed between versions.
- Operations must feel familiar to anyone who knows git, and must be approachable to users who do not.

#### 2.3 Why git, not a custom solution

Git is 40 years old, format-stable, exhaustively tooled, well-understood, and there is no scientific upside to writing a competing source-version-control system. The architectural decision is "embed git" + "build SciEasy-shaped UI affordances on top". Per CLAUDE.md §2.4 "everything connectable, not everything native", git is the canonical "everything connectable" example.

#### 2.4 Relation to ADR-038

ADR-038 records each workflow run, including the field `runs.workflow_git_commit` — the SHA of the commit the run executed against. The two ADRs share that single join key. Otherwise they are independent:

- ADR-039 (this) operates on text source files when the user clicks "Commit".
- ADR-038 operates on engine events when the user clicks "Run".

A run can reference a git commit; a git commit can be checked-out to see what produced any past run.

### 3. Decisions

#### 3.1 Git engine = bundled `git` CLI (subprocess)

SciEasy invokes a **bundled portable `git` binary** as a subprocess. The binary ships inside the ADR-037 desktop bundle (`<bundle>/resources/git/`) alongside the bundled Python runtime — the user does not need `git` installed on their system. Per-platform packaging:

| Platform | Binary | Size |
|---|---|---|
| Windows | **MinGit** (the official trimmed portable distribution shipped by Git for Windows) | ~30 MB |
| macOS | Static-built `git` from Git's source release (ARM + Intel universal or per-arch slices via the ADR-037 universal2 build) | ~25 MB |
| Linux | Static-built `git` (musl libc, no system dependencies) | ~25 MB |

Selected because:

- **Maximum cross-platform reliability.** Git binary distributions exist for every platform and architecture and have been maintained for 25+ years. No wheel build issues on new OS / Windows ARM / fresh Linux distros. Bundling git binary is a well-trodden path (VS Code's git extension, GitHub Desktop, SourceTree, GitKraken all do this).
- **100% bug-for-bug compatibility with external git tools.** When a user opens the same project in VS Code or runs `git status` in a terminal, the behavior is identical because both invoke the same git binary semantics.
- **Universal debugging.** Bug reports become "paste this command in a terminal"; we can copy/paste the exact subprocess invocation we ran.
- **License-clean.** Git is GPLv2. Shelling out and bundling the binary do not subject SciEasy itself to GPL terms. (VS Code, GitHub Desktop, etc. all ship under their own licenses while bundling git.)
- **Stable wire format.** Plumbing commands (`git log --format=...`, `git diff --raw`, `git status --porcelain=v2`) have stable machine-readable output across versions; we parse those, not human-readable porcelain.

Subprocess overhead (10–100 ms per call vs <1 ms native) is acceptable for SciEasy's interactive UI use case (< 100 git ops per day per user).

Alternatives rejected:

- **`pygit2`** (libgit2 binding): initially preferred, reversed 2026-05-15. Trade-offs: ~3 MB bundle vs ~30 MB, but wheel availability on new platforms is unreliable (especially Windows ARM and pre-release macOS / Linux), libgit2 diverges from git in subtle edge cases (hooks not run by default, some merge strategies, submodule handling), and pygit2 has had documented maintenance lag relative to libgit2 upstream. The bundle-size delta is irrelevant against ADR-037's ~250 MB total. The cross-platform reliability gap is decisive.
- **`dulwich`** (pure Python): slower than both pygit2 and git CLI; even more divergence from git's reference behavior. Rejected.
- **`gitpython`** (wraps system git, does not bundle): requires system git installation; same problem we are avoiding for non-technical-user desktop audience.
- **Embed git via libgit2 directly (C library, no Python wrapper)**: extra implementation cost for a problem subprocess already solves.

Sourcing strategy:

- Windows: `MinGit-<version>-busybox-64-bit.zip` from Git for Windows releases.
- macOS: build from `git-<version>.tar.gz` source release with universal2 flags.
- Linux: build static binary from same source release.
- Refresh cadence: every quarter, tracking git's stable releases and CVE advisories. The bundled binary version is pinned per SciEasy release.
- Build pipeline: `desktop/scripts/fetch-git-portable.{ps1,sh}` automates download / build / SHA-verify / extraction into `desktop/resources/git/`.

#### 3.2 Auto-init on project open

When a SciEasy project is opened:

1. If `<project>/.git/` exists → use it as-is.
2. If `.git/` does not exist → `<bundled_git> init <project>`, write the default `.gitignore` (§3.3), make an **initial commit** containing whatever source files already exist (`workflows/*.yaml`, `blocks/*.py`, `notes/*.md`, `project.yaml`, `README.md`). Commit message: `Initial commit (auto-generated by SciEasy)`.

**No user prompt.** The user wants git versioning per the requirement set; defaulting to "yes, do it" is the right behavior. Users who want to disable can `rm -rf .git` and SciEasy will warn but proceed without versioning (degraded mode — re-init available via Settings).

#### 3.3 Default `.gitignore` template

Written on `git init`; user may edit:

```gitignore
# SciEasy auto-generated .gitignore.
# Edit freely — SciEasy will not overwrite this file.

# Data files (not versioned — see ADR-038 for run lineage)
data/

# SciEasy runtime state (per-project, per-machine; not portable)
.scieasy/

# Python caches
__pycache__/
*.py[cod]
*$py.class

# OS noise
.DS_Store
Thumbs.db

# Plugin venvs (per ADR-037)
*-venv/
*.venv/

# Editor caches
.idea/
.vscode/
*.swp
```

**Note on `.scieasy/lineage.db`**: it is git-ignored by default. Rationale evaluated:

- SQLite binary, no useful diff/merge.
- Conceptually local: "runs that happened on this machine" travels with the machine, not the source. Two researchers cloning the same project each generate their own local lineage.
- Source still reproduces the runs: `workflow_yaml_snapshot` is also stored inside `lineage.db`, so the recipe survives independently when needed.

Users who explicitly want lineage shared with collaborators can delete the `.scieasy/` line from `.gitignore`. The ADR documents the trade-off so it is a conscious choice.

#### 3.4 Pre-run auto-commit + GUI filter (not zero auto-commit)

SciEasy's primary user is a scientist who does not know git. If commits are user-driven only, the typical scientist will never commit — and a year later their `git log` is empty, useless for reproducibility. Auto-commit on every workflow run gives them a complete history "for free", at the cost of log noise that the GUI filters out.

**Pre-run auto-commit semantics**:

```
User clicks Run
   ├─ Working tree clean → use HEAD SHA, no commit
   └─ Working tree dirty → squash-commit all modified files
                            Message: "auto: pre-run @ <iso-ts> (run=<short_id>, workflow=<id>)"
   ↓
Insert lineage.db runs row with workflow_git_commit = HEAD SHA
```

The `workflow_dirty` field on `lineage.db.runs` (ADR-038 §3.1) is **always 0** when auto-commit succeeds. The field exists as an escape hatch for users who explicitly disable auto-commit (see below) or for runs that occurred before this feature existed.

**Settings toggle**:

```
Settings → Git
  ☑ Auto-commit before each workflow run (recommended)
     When OFF: lineage records workflow_dirty=1 + the full YAML snapshot is
     stored inline (workflow_yaml_snapshot) so reproducibility is preserved
     even without a commit.
```

Default is ON. Power users wanting strict editorial control can disable.

**GUI filter** (the noise problem):

`git log` ends up with two commit categories:
- **Manual commits**: user-authored milestones via the Commit button (`feat:`, `fix:`, custom messages)
- **Auto commits**: pre-run auto-commits with `auto: pre-run @ ...` prefix

The History panel filters them visually:

```
History panel toolbar:
  [Show: Manual milestones ▾]  [⚙ Filter]
         Manual milestones        ← default
         All (incl. auto)
         Auto only (debug)

  Filter rules:
  - "manual milestones": hide commits whose message starts with "auto:" or "agent:"
  - "all": show everything
  - "auto only": inverse, for debugging
```

In the branch-graph view (§3.5b), filtered commits are rendered as small grey dots (◯) to preserve graph topology, while visible commits are large coloured dots (●) with labels. This prevents the graph from looking topologically broken when filtering is on.

#### 3.4a Agent commit prefix

When commits are generated by an agent (ADR-034 embedded coding agent, ADR-035 AI Block, or future programmatic flows) rather than by interactive user action, the commit message uses the `agent:` prefix:

```
agent: <short summary> (session=<chat_or_block_run_id>)
```

UI default: `agent:` commits are **shown** (not filtered like `auto:`), but rendered with a distinct icon (🤖) to make the source obvious. Users want to see what agents did to their code; they don't want to see every pre-run squashed dirty state.

Prefix legend:
- `auto:` — pre-run squash; hidden by default filter
- `agent:` — agent-authored milestone; visible with 🤖 icon
- *(no prefix)* — user-authored milestone; visible with 👤 icon

The prefix convention is documented in `docs/cli-integration.md` so external tools and CI can apply the same convention.

Implication for ADR-038: `runs.workflow_git_commit` always points at a real, clean commit when auto-commit is enabled (the default). The `workflow_yaml_snapshot` field remains the safety net for the auto-commit-disabled case.

#### 3.5 v1 feature set — MUST-haves (full branch ops for agent collaboration)

Originally v1 was scoped to commit / history / diff / restore / branch-list-switch-create. The full branch-ops set is now in v1 because of agent collaboration (ADR-034 embedded coding agent / ADR-035 AI Block) — agents may work on their own branches, and users need merge / conflict resolution without dropping to git CLI.

| Feature | UI element | API endpoint |
|---|---|---|
| **Commit** with author + message | `CommitDialog.tsx` with pre-filled template + Commit button | `POST /api/git/commit` `{message, author?, files?}` |
| **History (`git log`)** | `GitHistoryList.tsx` reverse-chrono list with filter dropdown (per §3.4) | `GET /api/git/log?branch=...&limit=...` |
| **Diff** between two commits or commit-vs-working | `GitDiffModal.tsx` using `react-diff-viewer-continued` for text files | `GET /api/git/diff?from=<sha>&to=<sha\|HEAD\|WORKING>&file=<path>` |
| **Restore** any prior version | "Restore this version" button on each history row | `POST /api/git/restore` `{commit_sha, files?}` (files=[] means whole worktree) |
| **Branch list / switch / create / delete** | `BranchPicker.tsx` toolbar dropdown | `GET /api/git/branches`, `POST /api/git/branch/switch`, `POST /api/git/branch/create`, `DELETE /api/git/branches/{name}` |
| **Working-tree status** | `GitStatusBadge.tsx` toolbar dirty/clean indicator | `GET /api/git/status` |
| **Merge** (FF + non-conflict + conflict path) | "Merge into current" in `BranchPicker` + `MergeFlow.tsx` orchestrator + `ConflictResolveView.tsx` for conflicts | `POST /api/git/merge` `{source_branch}` (returns `{result: "fast-forward" \| "clean" \| "conflict"}` + conflict file list) |
| **Cherry-pick** | "Cherry-pick onto current" on history rows | `POST /api/git/cherry-pick` `{commit_sha}` (may also produce conflicts → same resolution flow) |
| **Stash list + apply** | `StashListPanel.tsx` in a Stash drawer | `GET /api/git/stash`, `POST /api/git/stash/apply` `{stash_id}`, `POST /api/git/stash/save` `{message?}`, `DELETE /api/git/stash/{stash_id}` |
| **Branch graph viz** | `GitGraphView.tsx` (clean-room — see §3.5b) | (consumes existing `/api/git/log` data) |
| **Filter UI** for History view + Graph view | Filter dropdown in panel header per §3.4 | client-side filter on cached log |

Commit dialog pre-filled template:

```
<one-line subject>

# What changed:
# - workflows/<id>.yaml: …
# - blocks/<file>.py:    …

# Auto-detected modified files:
#   M  workflows/image_pipeline.yaml
#   M  blocks/cellpose_wrap.py
#   A  notes/2026-05-15-thoughts.md
```

The `# ` lines are git-style comments (stripped at commit time). The auto-detected file list is generated from `git status --porcelain=v2` on the bundled git binary.

#### 3.5a Conflict resolution UI — reuse ADR-036 Monaco editor

Merge / cherry-pick conflicts are resolved using the existing Monaco editor from ADR-036 with a thin merge-conflict integration layer. **No new editor; no third-party 3-way merge component.**

Flow:

```
1. User triggers Merge (or Cherry-pick).
2. SciEasy runs `git merge <branch>` (or `git cherry-pick <sha>`) via
   subprocess. Result classified from exit code + stdout/stderr:
   - fast-forward       → git already advanced HEAD, done
   - clean three-way    → git already created merge commit, done
   - conflict (exit 1)  → enter conflict-resolution state
3. In conflict state: git has written the conflicted files in-place with
   <<<<<<, ======, >>>>>> markers (standard git behavior). The
   MergeFlow.tsx component:
   a. Lists all conflicted files from `git status --porcelain=v2`
   b. Click a file → opens it in the existing Monaco editor (ADR-036
      file tab system) with conflict-marker decorations applied:
        - Custom Monaco decorations highlight each <<<<<<…>>>>>> region
        - Inline action buttons appear above each region:
            [Accept Current] [Accept Incoming] [Accept Both] [Manual edit]
        - Click resolves that region by rewriting the file content +
          removing markers
   c. Once all regions in a file are resolved → "Mark resolved" button
      runs `git add <file>` to stage it
4. When all conflicted files staged → "Complete merge" button runs
   `git commit --no-edit` to finalize the merge commit (git itself
   carries the merge state and parent list). Or "Abort merge" runs
   `git merge --abort` (or `git cherry-pick --abort`) to revert the
   working tree.
```

Implementation cost: Monaco decoration integration ~150 LOC + MergeFlow + ConflictResolveView wrappers ~400 LOC + REST endpoints ~150 LOC ≈ **~700 LOC** added on top of the ADR-036 editor infrastructure.

Why this works without a true 3-way merge editor:
- Monaco supports rich text decorations and inline widgets; the conflict-marker pattern is a standard pre-merge-editor UX (VS Code itself shipped with this style for years before adding the dedicated merge editor in 2022)
- For users wanting a full 3-way side-by-side view, "Compare Changes" button opens Monaco's `DiffEditor` showing source vs target separately — also already shipping for ADR-036 diff use

#### 3.5b Branch graph visualization — clean-room implementation

A VSCode-style branch graph rendering, implemented from scratch. **No third-party React git-graph library** (CommitGraph was evaluated and rejected on supply-chain grounds; `@gitgraph/react` is archived; `mhutchie/vscode-git-graph` uses a custom license forbidding derivative redistribution).

Layout algorithm is the standard DAG lane-assignment problem — public knowledge, no copying required. Visual style references VS Code / GitLens / JetBrains as design inspiration (functional UI conventions are not copyright-protected).

Implementation lives at `frontend/src/components/Git/GitGraph/`:

```
GitGraph/
├── laneAssign.ts          # commit → lane assignment via topo iteration + lane recycling (~500 LOC)
├── edgeRouter.ts          # connector lines between commit and parents; bezier curves for lane jumps (~250 LOC)
├── GraphSVG.tsx           # SVG rendering: dots, edges, labels, filter dimming (~300 LOC)
├── colorPalette.ts        # branch color rotation (~80 LOC)
├── interactions.ts        # hover preview, click → diff/checkout, virtualized scrolling for >1k commits (~200 LOC)
└── integration.ts         # gitSlice consumption, filter-state integration, theme (~150 LOC)
```

Algorithm sketch:

```typescript
function assignLanes(commits: GitCommit[]): GitCommit[] {
  // active_lanes[i] = the SHA the lane is "waiting for" (commit's child already drawn)
  const active_lanes: (string | null)[] = []
  for (const commit of commits) {  // commits already in topological order
    // 1. Reuse a lane if some already-drawn child expects this commit as parent
    let lane = active_lanes.findIndex(sha => sha === commit.sha)
    if (lane === -1) {
      // 2. Fresh lane: first null slot, or extend
      lane = active_lanes.findIndex(s => s === null)
      if (lane === -1) lane = active_lanes.length, active_lanes.push(null)
    }
    commit.lane = lane
    // 3. This lane now expects commit's first parent
    active_lanes[lane] = commit.parents[0] ?? null
    // 4. For merge parents (beyond first), allocate new lanes
    for (const extra of commit.parents.slice(1)) {
      let new_lane = active_lanes.findIndex(s => s === null)
      if (new_lane === -1) new_lane = active_lanes.length, active_lanes.push(null)
      active_lanes[new_lane] = extra
      commit.merge_lanes ??= []
      commit.merge_lanes.push(new_lane)
    }
  }
  return commits
}
```

This is textbook material; mhutchie's project, GitKraken, JetBrains, and many others all converge on the same approach. Our implementation references only the published algorithmic literature.

Visual details (commit dot size 8px, edge stroke 2px, ~10 branch colors with deterministic assignment, filtered commits as 4px grey dots) are design choices, copied from common conventions across git GUI tools.

Performance target: smooth scroll on a project with 1000 commits without virtualization; with virtualization (`@tanstack/react-virtual` or `react-window`), handle 10000+ commits.

#### 3.5c Filter and commit-prefix integration

The History panel filter dropdown (§3.4) drives both `GitHistoryList` and `GitGraphView`:

- **Manual milestones** (default): hide commits whose message starts with `auto:` or `agent:` (from §3.4a). In `GitGraphView` they render as small grey dots to preserve topology; the connecting edges still flow through them.
- **All**: show every commit fully.
- **Auto only / Agent only**: inverse filters for debugging.

Filter state lives in `gitSlice` so the dropdown choice persists across panel re-renders and across the History view <-> Graph view tab switch.

#### 3.6 Restore semantics

"Restore this version" defaults to **`git checkout <sha> -- <files>`** restoring listed files to the working tree without changing HEAD. This is the safer mode: user can keep editing, see what the old version looked like, decide whether to commit.

Optional "Hard restore (move HEAD)" advanced action does `git reset --hard <sha>` — gated behind a confirmation dialog explaining the consequences. v1 surfaces the soft restore prominently; hard restore is deferred or hidden until user feedback shows demand.

If the working tree is dirty during restore, SciEasy auto-stashes (`git stash push -m "auto-stash before restore"`) first, restores, and prompts: "Your unsaved changes are stashed. Apply them? [Apply now / Keep stashed / Discard]".

#### 3.7 Branch UI scope — full operations in v1

Branch operations all in v1 (expanded from earlier draft because of agent collaboration needs):

- List existing branches
- Switch to an existing branch (working-tree-dirty → stash dialog per §3.6)
- Create new branch from current HEAD or from any historical commit
- Delete a branch (with confirmation)
- **Merge** between branches: fast-forward / clean three-way / conflict path with Monaco-based resolution UI (§3.5a)
- **Cherry-pick** a commit onto current branch (same conflict path as merge)
- **Stash list + save + apply + drop** via Stash drawer
- **Branch graph visualization** (§3.5b) rendering all branches in a single lane-assigned view

When the working tree is on a branch other than the one used during a run, the run record's `workflow_git_commit` SHA still resolves correctly — branches are pointer abstractions over commits.

#### 3.8 External git changes are respected

If a user runs `git commit -a` from a terminal outside SciEasy:

1. SciEasy's existing file watcher (ADR-034 Phase 2 `workflow_watcher`) detects YAML changes → refreshes canvas.
2. A new git-aware watcher detects HEAD changes (poll `.git/HEAD` and `.git/refs/heads/` mtime on file activity) → refreshes `BranchPicker` + `GitHistoryList`.
3. No locks on `.git/` are taken. Users can run arbitrary git CLI alongside SciEasy.

#### 3.9 Pre-existing projects without `.git`

Per §3.2, projects opened that lack `.git/` get auto-init. The initial commit captures whatever already exists in the project (workflow YAML, .py, .md files). No prompt; this is the right default given the user-stated requirement.

If a user wants to opt out, they can:
- Delete `.git/` from the filesystem (SciEasy will not aggressively re-init — only on project open if absent and no `.scieasy/no_git` marker)
- Add a `.scieasy/no_git` marker file to suppress re-init on subsequent opens

The opt-out path is documented but does not have a Settings checkbox — it is intentionally inconvenient because the use case is rare.

#### 3.10 Out of scope (defer to future ADRs)

- **Remote push / pull (GitHub, GitLab, etc.)** — v1 is local-only. Users who want to push their project elsewhere can use git CLI manually with `git remote add` + `git push`. v2 may add a UI affordance.
- **Git LFS** — `data/` is git-ignored; large binary files do not enter git. Users who explicitly want versioned data files can configure LFS manually.
- **Submodules** — not used by any SciEasy workflow; out of scope.
- **Hooks (`pre-commit`, etc.)** — users can configure manually in `.git/hooks/`; SciEasy does not write hooks.
- **Tag management** — git tags work via CLI; UI element deferred.
- **Rebase / Bisect** — power-user operations; v1 surfaces via git CLI only.

### 4. Alternatives considered

1. **No versioning** — current state. Rejected per §2.1: 4-month-rollback is a user-stated requirement.
2. **Custom history mechanism (snapshot files in `.scieasy/`)** — Rejected: reinvents git poorly. Forty years of tooling, conventions, and user familiarity argue for embedding the real thing.
3. **`pygit2` (libgit2 binding)** — Initial draft choice (~3 MB bundle). Reversed 2026-05-15: cross-platform wheel availability lags on new OS / Windows ARM / fresh Linux distros, libgit2 has documented edge-case divergence from git's reference behavior, and the bundle-size delta is irrelevant against ADR-037's ~250 MB total. Bundled git CLI (§3.1) chosen instead.
4. **`dulwich` (pure Python)** — Slower than both pygit2 and git CLI; greater divergence from git's reference. Rejected.
5. **Shell out to system `git`** without bundling — Requires user-side git installation; conflicts with ADR-037 desktop double-click promise. Bundling git binary (§3.1) is the actual decision.
5. **Auto-commit on every file save** — Rejected: pollutes history beyond usefulness (one commit per Ctrl+S = 100s of commits per session). User explored this as one of five options 2026-05-15 and selected per-run instead.
6. **Zero auto-commit (user manually commits everything)** — Initial draft chose this. Rejected after user feedback: target audience is non-technical scientists who will not manually commit, leaving them with an empty `git log` and no usable history. Per-run auto-commit + GUI filter (§3.4) is the chosen path.
7. **Force `.scieasy/lineage.db` into git by default** — Rejected per §3.3: binary, machine-local, no merge story. User can opt-in by editing `.gitignore`.
8. **Vendor a third-party React git-graph library (CommitGraph by liuliu-dev)** — Evaluated 2026-05-15; rejected on supply-chain grounds (single maintainer, 28 stars, no published npm releases). Clean-room implementation (§3.5b) chosen instead.
9. **Vendor mhutchie/vscode-git-graph** — Evaluated 2026-05-15; rejected because its custom license forbids redistribution of derivative works. Clean-room implementation (§3.5b) chosen instead.
10. **Defer branch graph visualization to v2** — Rejected: user-stated UX requirement for v1; clean-room cost (~1500 LOC) is manageable.
11. **`@gitgraph/react`** — Archived (no longer maintained as of 2025). Rejected.
12. **Embed VS Code's git module** (built-in or via Theia) — Investigated 2026-05-15; not architecturally separable. VS Code's git extension depends on the VS Code extension host APIs; embedding the full host (via Eclipse Theia, ~50 MB) is disproportionate to the value. Rejected. Monaco editor (which IS embeddable, already used by ADR-036) covers the conflict-resolution UI use case.
13. **Defer merge / conflict resolution UI to v2** — Initial draft chose this. Rejected after user feedback: agent collaboration (ADR-034 / ADR-035) requires merge UI in v1 so users can integrate agent-authored branches without dropping to git CLI.

### 7. Consequences

#### 7.1 Wins

- Forty years of git semantics, file format, and tooling come for free.
- The user can run external `git` (CLI, GUI tools, GitHub Desktop) alongside SciEasy without conflict.
- The 4-month-rollback user requirement is satisfied with a one-click restore.
- Branching enables variant exploration without copy-paste filename hacks.
- Reproducibility chain locks in: every run records the git commit it ran against (ADR-038), and git history preserves the source exactly.
- Custom block development gets normal source control automatically.

#### 7.2 Losses

- **Bundled git binary adds ~25-30 MB to the desktop bundle** (ADR-037). Acceptable against the ~250 MB total. Net trade-off vs the pygit2 alternative (~3 MB): we accept the size for cross-platform reliability + 100% bug-for-bug git compatibility.
- Users who do not understand git at all are exposed to "branch" / "commit" / "merge" terminology in the UI. Mitigation: pre-run auto-commit (§3.4) means most users never need to type a commit message; tooltips + minimal onboarding cover the rest. Worst case: they ignore the version-control UI entirely and SciEasy still works.
- The optimistic-concurrency ETag flow goes away. Concurrent multi-tab edits of the same workflow YAML can now race silently (last-write-wins on file save). The existing workflow_watcher emits `workflow.changed` on FS change → other tabs invalidate cache. This is the same model as VS Code / most editors: detect external change, prompt to reload. Not worse than the current bump_revision (which only protects within one server process).
- Pre-run auto-commit (§3.4) is ON by default; users who want strict editorial control must explicitly disable it in Settings.
- Graph visualization is clean-room (~1500 LOC) — visual polish takes iteration to match VS Code / JetBrains; v1 ship may be functional-but-rough until Phase 4 polish.

#### 7.3 Risks

- **Bundled git binary maintenance**: SciEasy must track upstream git releases for CVE advisories (e.g. CVE-2024-32002 et al.). Mitigation: pin git version per SciEasy release; refresh quarterly; subscribe to git's announce list. Bumping the bundled version is a one-line change in `fetch-git-portable.{ps1,sh}` + a CI rebuild.
- **Subprocess overhead** (10-100 ms per call). Mitigation: not a concern for interactive UI use case (< 100 git ops per day per user). Batch operations where possible (e.g. one `git log` with `--all --format=...` rather than per-branch calls).
- **`.git/` corruption** if user or another tool wedges the repo. Mitigation: SciEasy invokes git CLI which is the reference implementation; if git itself cannot recover, neither can we. Surface as "no version control available" state with a Settings → "Reinitialize git" recovery button (move `.git/` to `.git.broken/` + re-init).
- **Output parsing fragility** if git changes its `--porcelain` or `--format=` output. Mitigation: pin git version per SciEasy release so upstream output changes are observed in our CI before users see them; use plumbing commands (stable contract) not porcelain (human-readable).
- **Big binary mistakenly committed** (e.g. user removes `data/` from `.gitignore`). Mitigation: pre-commit size warning if any file > 10MB is staged. v1 implements the warning; cleanup (BFG / git-filter-repo) is user's responsibility.
- **Branch checkout while working tree is dirty**: SciEasy auto-stashes per §3.6; documented but a real failure mode if user does not understand stash. Mitigation: clear modal explaining what stash does and offering "Cancel" prominently.

### 8. Open questions

1. **OQ-1**: Initial commit author when no git config is set. Default to `SciEasy User <noreply@scieasy.local>`? Or prompt user once for name/email? Tentatively: prompt-once on first commit; cache in `<project>/.scieasy/git_author.json`.

2. **OQ-2**: Should `git tag` operations be exposed in v1? Tentative: no — tags are rare for the target audience. Reconsider in Phase 3 polish.

3. **OQ-3**: Large file warning threshold. 10MB is a starting guess; some scientific output files (CSVs of analysis results) can legitimately be > 10MB. Soft warning + override is the right shape; tune the threshold based on user reports.

4. **OQ-4**: When ADR-038 Lineage tab shows a run with `workflow_dirty=true`, what message do we show? "This run executed against uncommitted changes. The workflow YAML snapshot was saved with the run for full reproducibility." Tentatively yes.

5. **OQ-5**: Multi-tab editing — if the user has the same workflow open in two browser tabs and saves from both: the second save wins on disk; the file watcher invalidates the first tab's cache; user gets a "Reload? Discard local edits?" prompt. Same as VS Code. Decide UX in Phase 2.

### 10. Decision summary

**Every SciEasy project is a git repository. The engine is a **bundled portable `git` CLI binary** invoked via subprocess (MinGit on Windows ~30 MB, static-built `git` on macOS/Linux ~25 MB) shipped as part of the ADR-037 desktop bundle — pygit2 was the initial draft choice and was reversed 2026-05-15 per §3.1. Auto-init on project open. Pre-run auto-commit ON by default (`auto:` prefix, hidden from `git log` by GUI default filter); user manual commit always available. Agent commits use `agent:` prefix and remain visible. Default `.gitignore` excludes `data/` and `.scieasy/`. v1 ships the full local feature set: commit, history with filter, diff, restore, branch list/switch/create/delete, merge (fast-forward / clean / conflict via Monaco-based resolution UI reusing ADR-036), cherry-pick, stash list+apply, and clean-room branch graph visualization. Remote sync and LFS are explicitly v2.** The reproducibility chain joins via `runs.workflow_git_commit` (ADR-038): a run record points to the exact commit that produced it, and that commit is guaranteed to exist (not a dirty snapshot) when auto-commit is enabled.

The architectural invariant: **history is two layers — source via git (this ADR) and runs via lineage.db (ADR-038). The conflation of these two concerns ends here.**
