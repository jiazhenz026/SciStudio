# ADR-048: Extensible Type Previewers And Preview-Side Plot Jobs

## 1. Decision Summary

SciStudio adopts a routed, extensible preview subsystem. Core owns the preview
host, routing, safety limits, base-type fallback viewers, plot-job execution
boundary, preview cache, artifact export behavior, and AI-facing authoring
tools. Packages and projects may provide full interactive previewers for their
own data types, including backend data-access providers and frontend React
components. User-authored plot scripts are modeled separately as preview-side
plot jobs: they consume a selected block output collection, run through isolated
execution paths, write display-only plot artifacts, and never enter the workflow
DAG or downstream dataflow.

This replaces hardcoded preview dispatch with a first-class extension surface
matching SciStudio's block, type, and IO capability model. Preview behavior is
data-type-aware, collection-aware, bounded for large data, and extensible
without moving domain-specific UI into core.

### 1.1 Problems Addressed

| Problem | Risk | ADR response | Detailed section |
|---|---|---|---|
| Preview dispatch is hardcoded in core API and frontend renderer branches | New scientific data types require core edits and domain UI accumulates in the main app | Introduce routed previewers registered by core, packages, and projects | Section 2 and Section 3 |
| Rich image behavior belongs to imaging-domain packages, not core | Core becomes a dumping ground for LUTs, z-stack controls, labels, and metadata browsers | Core keeps only base `ArrayPreviewer`; rich image previewers move to imaging packages | Section 6 |
| Project users need quick scientific plots from block outputs | One-off plots become notebooks outside SciStudio or fake workflow blocks inside the DAG | Add preview-side plot jobs that consume block output collections and produce display-only artifacts | Section 5 |
| Plot binding by block name is ambiguous and fragile | Users may not know block IDs, and repeated block labels can collide | Store plot bindings in project-local plot manifests generated by UI/CLI/MCP, using workflow path, stable node ID, and output port | Section 5 |
| Package and project previewers need real interactive UI | A standard JSON payload cannot express LUT controls, slice navigation, brushing, or domain tools | Define previewers as backend provider plus frontend component plus manifest | Section 4 |
| Large scientific data must remain reference-based | Preview can accidentally materialize huge arrays or collections in the API process | Add bounded `PreviewDataAccess` and sampling contracts | Section 7 |
| Multiple previewers may match the same data type | Registration order can produce unstable or surprising UI | Require deterministic type-specific routing and explicit ambiguity errors | Section 3 |
| Plot outputs need publication-friendly formats | Raster-only plots lose vector quality required for figures and reports | Add core `PlotPreviewer` for PNG, JPEG, SVG, and PDF artifacts | Section 6 |
| AI assistants need a safe plot-authoring workflow | Agents may guess block names, write invalid scripts, or confuse plot jobs with workflow blocks | Add MCP tools and a packaged skill for plot target discovery, scaffolding, validation, and test runs | Section 9 |
| Package and block developer docs are stale | Extension authors will not know where previewers, plot templates, and package-owned viewers belong | Include developer documentation delete-and-rewrite in this ADR scope; exact inventory belongs to the companion spec | Section 10 |

### 1.2 Accepted Decision

SciStudio separates two concepts:

1. **Type previewers** decide how a `DataObject` or `Collection` should be
   inspected interactively.
2. **Preview-side plot jobs** run user-authored Python or R plotting scripts
   against a selected block output collection and produce plot artifacts for the
   core `PlotPreviewer`.

A previewer may be supplied by core, an installed package, or a project-local
extension. A previewer may include both backend code and a frontend React
component. The frontend component is loaded through a backend-validated manifest
and interacts only through the preview session API.

A plot job is not a workflow block, not a graph node, and not a downstream data
producer. It may reuse existing CodeBlock runner mechanics for subprocess
execution, language support, logs, cancellation, timeout, and file exchange, but
its outputs are written to preview cache and rendered only in the selected
block's preview panel.

## 2. Preview Subsystem Model

The preview subsystem has five architectural roles.

| Role | Responsibility |
|---|---|
| `PreviewHost` | Frontend shell that owns selected block/output UI, session mounting, fallback/error display, and export actions. |
| `PreviewRouter` | Backend resolver that selects the applicable previewer for a data reference or collection. |
| `PreviewerRegistry` | Registry of core, package, and project previewer specs. |
| `PreviewSession` | Short-lived state for one mounted preview, including selected provider, query parameters, and safe data endpoints. |
| `PreviewDataAccess` | Backend helper surface for bounded, reference-only reads, sampling, slicing, and metadata summaries. |

The backend remains authoritative for data references, provider resolution,
preview sessions, cache paths, and plot-job execution. The frontend may keep UI
state such as zoom, selected slice, selected tab, or LUT control values, but
frontend state is not workflow truth.

## 3. Previewer Resolution Contract

Preview routing is type-based and collection-aware. Providers may declare support
for a single item type, a collection of a type, or both.

Resolution order is:

1. Project previewer for exact `Collection[T]`.
2. Project previewer for exact `T`.
3. Package previewer for exact `Collection[T]`.
4. Package previewer for exact `T`.
5. Project previewer for the nearest parent type.
6. Package previewer for the nearest parent type.
7. Core collection fallback.
8. Core base-type fallback.
9. Core unknown/error fallback.

Within the same tier and type specificity, higher `priority` wins. If multiple
previewers tie on tier, type specificity, and priority, the router must return
an ambiguity error instead of choosing by registration order. A project may
declare an explicit default previewer for a type to resolve ambiguity.

Previewer specs include:

| Field | Meaning |
|---|---|
| `previewer_id` | Stable globally unique ID, such as `core.array.basic` or `imaging.image.viewer`. |
| `owner_kind` | `core`, `package`, or `project`. |
| `target_type` | Fully qualified `DataObject` type or base type. |
| `supports_collection` | Whether the previewer can inspect `Collection[target_type]`. |
| `backend_provider` | Optional Python provider for metadata, sampling, tile, slice, or session APIs. |
| `frontend_manifest` | Optional frontend component manifest. |
| `priority` | Deterministic tie-breaker within one tier and specificity. |
| `capabilities` | Declared features such as `slice`, `tile`, `lut`, `export`, `table`, `plot`, `metadata`. |

## 4. Interactive Previewer Contract

A full previewer has two halves:

- a backend provider that reads data through `PreviewDataAccess`, creates
  preview sessions, and exposes bounded session endpoints;
- a frontend component that renders the interactive UI.

Package and project frontend modules are loaded only through backend-validated,
same-origin manifests. Remote CDN imports and arbitrary external URLs are not
part of the contract. The manifest must name the previewer ID, ESM module URL,
export name, optional CSS assets, version, and compatible preview API version.

The session manager stamps the resolved previewer's `frontend_manifest` onto the
`PreviewEnvelope` it returns, so providers declare the manifest once on their
`PreviewerSpec` and never re-embed it per envelope; the host reads it from
`envelope.frontend_manifest`.

The frontend component receives a constrained `PreviewHost` API:

- `previewSessionId`;
- provider metadata;
- current block/output identity;
- data/session request helpers;
- export/save helpers;
- error-reporting helpers.

It must not directly mutate workflow definitions, runtime state, lineage, or
project files. Any mutation outside preview cache must go through explicit
backend APIs.

## 5. Preview-Side Plot Job Contract

Project users may define plot jobs under a dedicated project directory such as:

```text
plots/
  cell_scatter/
    plot.yaml
    render.py
  qc_summary/
    plot.yaml
    render.R
```

Plot bindings are project-local source artifacts stored in per-plot manifests.
The UI, CLI, or MCP tooling creates the manifest from the selected workflow node
and output port. Users and agents do not need to hand-type block labels or guess
node IDs.

Example manifest:

```yaml
id: cell_scatter
title: Cell Scatter
target:
  workflow: workflows/main.yaml
  node_id: node_8f3a2c
  output_port: measurements
script:
  language: python
  entrypoint: render.py:render
outputs:
  preferred_format: svg
```

The target is not a workflow edge and does not become part of the executable DAG.
The script receives the selected collection and does not need to know the block
name.

Python template:

```python
def render(collection, context):
    df = context.to_dataframe(collection, max_rows=10000)
    fig, ax = context.plt.subplots()
    ax.scatter(df["x"], df["y"], s=4)
    return context.save_figure(fig, "figure.svg")
```

R template:

```r
render <- function(collection, context) {
  df <- context$to_dataframe(collection, max_rows = 10000)
  p <- ggplot2::ggplot(df, ggplot2::aes(x = x, y = y)) +
    ggplot2::geom_point()
  context$save_plot(p, "figure.pdf")
}
```

Plot jobs reuse the existing block/code execution route where possible:
subprocess execution, language runner selection, cancellation, timeout, logs,
sanitized errors, and declared input exchange. They differ from workflow blocks
in these ways:

| Concern | Plot job rule |
|---|---|
| DAG membership | Not a workflow node and not visible to the scheduler as executable graph work. |
| Input | Receives one selected block output collection. |
| Output | Writes display-only artifacts to preview cache. |
| Downstream flow | Does not produce downstream collections or edges. |
| Lineage | Does not claim scientific output lineage; may record lightweight preview metadata. |
| Failure | Fails only the preview panel, not the workflow run. |

Preview cache paths follow a current-overwrite model:

```text
.scistudio/previews/<workflow_id>/<node_id>/<output_port>/<plot_id>/current.*
.scistudio/previews/<workflow_id>/<node_id>/<output_port>/<plot_id>/current.json
```

`current.json` records script path, script hash, target workflow/node/output,
input collection object IDs, created time, selected runner, output formats, and
error state. Rerunning the plot overwrites `current.*`. Export/save actions copy
the artifact to a user-selected destination or project artifact location.

## 6. Core Fallback Viewers

Core fallback previewers are inspection-grade, not domain-grade. They guarantee
that base data remains visible without package-specific assumptions.

| Previewer | Core responsibility | Explicitly not core |
|---|---|---|
| `DataFramePreviewer` | Paginated table, column summary, schema, row count, sorting/filter basics. | Domain table semantics or specialized QC dashboards. |
| `ArrayPreviewer` | Bounded N-D numeric inspection, shape/dtype/axes metadata, scalar/1D/2D display, generic slicing, generic heatmap. | Image semantics, RGB/channel merge, label overlays, OME metadata, domain LUT tools. |
| `SeriesPreviewer` | Chart and index/value table for ordered values, time-index handling when metadata exists, decimated large charts. | Domain-specific spectrum, chromatogram, electrophysiology, or time-series analysis tools. |
| `TextPreviewer` | Bounded text preview, copy, syntax highlighting for safe plain formats, editor handoff for long content. | Active HTML execution or external resource loading. |
| `ArtifactPreviewer` | Metadata, MIME, size, safe inline display for supported static formats, open/export actions. | Interpreting domain-specific binary files. |
| `CompositePreviewer` | Slot inventory, slot names, slot data types, lightweight metadata, child preview routing. | Recursive eager materialization of every slot. |
| `PlotPreviewer` | Display and export plot artifacts from preview-side plot jobs. Supports PNG, JPEG, SVG, and PDF. | Running plot code or modifying workflow data. |

`ArrayPreviewer` behavior:

- `ndim == 0`: scalar value and metadata.
- `ndim == 1`: simple line chart plus index/value table.
- `ndim == 2`: bounded matrix or heatmap view plus numeric summary.
- `ndim >= 3`: choose two display axes and expose selectors for remaining axes.
- Prefer `y`/`x` axes when present; otherwise use the last two dimensions or
  the largest two dimensions.
- Use generic colormap/range controls only as numeric display controls.

`SeriesPreviewer` behavior:

- shows length, dtype, index metadata, missing count, and numeric summary when
  applicable;
- offers both chart and table views;
- uses line chart for ordered numeric/time indices and point/scatter display
  when appropriate;
- decimates or bins chart data under a preview budget while table access remains
  paginated.

`TextPreviewer` behavior:

- previews a bounded number of bytes, characters, or lines;
- marks truncation explicitly;
- uses safe plain-text rendering;
- offers an editor handoff for long text or file-backed text.

`PlotPreviewer` behavior:

- supports `image/png`, `image/jpeg`, `image/svg+xml`, and `application/pdf`;
- preserves vector outputs for SVG and PDF;
- supports fit, zoom, pan where possible, and export/save for every format;
- sanitizes or sandboxes SVG display so scripts do not execute;
- displays multi-artifact plot outputs as tabs.

## 7. Data Access, Sampling, And Cache

Previewers receive references and context, not eager payloads. `PreviewDataAccess`
must provide bounded helpers for common data shapes:

| Type | Bounded access |
|---|---|
| `DataFrame` | Column projection, row limit, pagination, optional sampled rows. |
| `Array` | Shape/dtype/axes metadata, Zarr-aware slice reads, downsampled 2D planes, tile reads when supported. |
| `Series` | Head/tail, index/value pages, chart decimation such as min/max bins or LTTB. |
| `Text` | Byte/line chunks with encoding and truncation metadata. |
| `Artifact` | Metadata and safe static-file serving. |
| `CompositeData` | Slot metadata first; child payloads only when selected. |
| `Collection` | Item count when available, sampled item references, bounded iteration. |

Every preview response must indicate whether it is sampled, truncated, cached,
derived, or complete. Previewers must not use `to_memory()` on large objects
unless the access helper explicitly determines the object is within the preview
budget.

## 8. Security And Failure Semantics

Previewer and plot execution is a local single-user reliability boundary, not a
multi-tenant hostile-code sandbox. Even so, the runtime must isolate failures:

- backend preview providers and plot jobs must not run arbitrary long work in
  the API event loop;
- plot jobs must enforce timeout, output-size caps, cancellation, and sanitized
  errors;
- frontend modules must load only from validated same-origin manifests;
- package/project previewer load failures must degrade to diagnostics or a core
  fallback;
- preview failures must not mutate workflow definitions, data objects, lineage,
  or downstream outputs;
- preview cache is runtime state and is not a scientific result unless the user
  explicitly saves or exports it.

## 9. AI Plot Authoring Tools

SciStudio will provide AI-facing tooling for plot jobs, separate from block
authoring. The pattern mirrors `scistudio-write-block`, but targets project
plot manifests and scripts rather than `blocks/*.py` or workflow nodes.

The required skill is:

| Skill | Purpose |
|---|---|
| `scistudio-write-plot` | Guides agents through target discovery, plot scaffolding, Python/R script authoring, validation, and test execution. |

The required MCP tools are:

| Tool | Purpose |
|---|---|
| `list_plot_targets` | Lists previewable workflow outputs with workflow path, stable node ID, node label, block type, output port, output type, and latest output availability. |
| `scaffold_plot` | Creates `plots/<plot_id>/plot.yaml` plus `render.py` or `render.R` from a selected target and language. |
| `list_plot_examples` | Returns curated matplotlib, seaborn, and ggplot2 examples. |
| `read_plot_source` | Reads an existing plot manifest and script. |
| `validate_plot` | Validates manifest schema, entrypoint, language, target existence, and output format declarations. |
| `run_plot_job` | Executes a plot against the latest available target collection and returns artifact paths, logs, and sanitized errors. |

Agent rules:

- agents must call `list_plot_targets` before creating a plot;
- agents must not guess node IDs or bind by human-facing block labels alone;
- agents should use `scaffold_plot` rather than hand-creating manifests;
- agents must run `validate_plot` and `run_plot_job` before declaring a plot
  ready;
- plot jobs must remain preview-only and must not edit workflow DAGs.

## 10. Documentation Scope

This ADR includes a delete-and-rewrite pass for active block and package
developer documentation. The rewritten documentation must explain the new
previewer registration contract, project-local previewers, plot templates,
package-owned domain previewers, and the current block/package extension
posture after recent ADRs and package/block changes.

The ADR does not enumerate every document to update. The companion spec owns the
exact documentation inventory, affected paths, and acceptance criteria.

## 11. Verification And Tooling Impact

Implementation must add tests for:

- previewer registry discovery for core, package, and project previewers;
- resolution order, exact type matches, parent fallback, collection routing,
  priority, and ambiguity errors;
- core fallback viewers for all six base types plus plot artifacts;
- `ArrayPreviewer` axis selection, slice clamping, downsample metadata, and
  large-array bounded reads;
- `SeriesPreviewer` chart/table modes and decimation;
- `TextPreviewer` truncation and editor handoff metadata;
- Python matplotlib/seaborn plot jobs;
- R ggplot2 plot jobs;
- plot manifest creation, validation, broken target detection, and cache
  overwrite behavior;
- dynamic frontend manifest validation and load failure diagnostics;
- AI skill and MCP tool packaging for `scistudio-write-plot`;
- imaging package registration for rich `Image` and `Label` previewers;
- regression behavior for existing DataFrame, Array, Series, Text, Artifact,
  and CompositeData previews.

Manual/browser verification should include:

- previewing a block output collection containing multiple images;
- selecting an image package previewer and interacting with slice/LUT controls;
- falling back to core `ArrayPreviewer` when the imaging previewer is absent;
- creating a plot from the UI/CLI/MCP target picker rather than hand-typing a
  block name;
- running a Python plot job and viewing SVG/PDF output;
- running an R plot job and viewing PDF/PNG output;
- confirming plot failure does not alter workflow execution state.

## 12. Consequences And Risks

Positive consequences:

- preview behavior becomes extensible in the same architectural style as blocks,
  types, and IO capabilities;
- domain packages own their domain UI instead of pushing it into core;
- project-specific plotting becomes first-class without polluting the DAG;
- core remains useful in minimal installs through base-type fallback viewers;
- vector plot outputs become first-class preview artifacts;
- AI assistants get a safe, discoverable path for writing plots.

Costs and risks:

- dynamic frontend extension loading adds packaging and security complexity;
- project-local React previewers are an advanced authoring surface and require
  build/manifest validation;
- plot jobs introduce Python and R runtime availability concerns;
- preview cache and saved artifacts must be clearly distinguished from workflow
  outputs;
- moving rich image preview out of core requires careful compatibility tests;
- stale developer documentation can mislead package authors unless refreshed
  with the implementation.

## 13. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Keep adding `preview.kind` branches in core | Continues coupling every domain preview to core and blocks package-owned UI. |
| Make data types own preview rendering directly | Conflates semantic type contracts with UI/frontend dependencies. |
| Treat user plots as ordinary workflow blocks | Pollutes the DAG with exploratory display-only work and creates false downstream data semantics. |
| Treat user plots as previewers | Mixes two concepts: type inspection and block-output plotting. Plot jobs should output artifacts rendered by `PlotPreviewer`. |
| Store plot target only inside the script | Requires users or agents to guess block labels or node IDs and creates fragile bindings. |
| Restrict external previewers to JSON payloads | Cannot support interactive image, spectrum, or domain-specific controls. |
| Load previewer JavaScript from arbitrary URLs | Too hard to audit and inconsistent with local package/project ownership. |
| Move all image behavior to imaging with no core fallback | Generic arrays would become unpreviewable in minimal installs. |
