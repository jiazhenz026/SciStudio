# ADR-011: Workflow Definition As Declarative YAML, Decoupled From Frontend

## 1. Decision Summary

SciStudio workflows are declarative graph documents. The frontend renders and
edits them, but frontend state is not the source of workflow truth.

### 1.1 Problems Addressed

| Problem | Risk | ADR-011 response | Detailed section |
|---|---|---|---|
| Workflows must be portable and versionable | UI-only state cannot be shared, diffed, or run headlessly | Use declarative workflow files | Section 3 |
| Frontend layout should not own runtime semantics | A ReactFlow model would leak into CLI and AI tooling | Keep layout optional metadata | Section 4 |

## 2. Decision

Workflow files define nodes, edges, config, and metadata. Canvas position is
optional layout metadata. CLI, AI tools, tests, and the GUI all read the same
workflow definition contract.

## 3. Scope

In scope: graph definition, validation, optional layout, and frontend/backend
decoupling. Out of scope: making ReactFlow JSON the canonical workflow format.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- Workflow serializer and validator tests cover graph portability.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

ReactFlow JSON was rejected as frontend-coupled. A Python DSL was rejected as
too developer-only. Visual-only workflows were rejected because they cannot be
versioned or executed headlessly.
