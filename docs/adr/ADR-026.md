# ADR-026: Block SDK Scaffolding, Test Harness, And Developer Documentation

## 1. Decision Summary

SciStudio provides a small Block SDK so external developers can create,
validate, and package block libraries without reading internal runtime ADRs.
The SDK consists of a CLI scaffold command, package templates, and a public
`BlockTestHarness` for contract validation and smoke tests.

This ADR preserves the accepted 2026-04-05 SDK decision. Its original custom
IO adapter guidance is historical only: ADR-028 removes the central adapter
registry and current block packages should expose IO behavior as `IOBlock`
subclasses through `scistudio.blocks`.

### 1.1 Problems Addressed

| Problem | Risk | ADR-026 response | Detailed section |
|---|---|---|---|
| External developers need a repeatable package shape | Block packages drift in layout, metadata, and entry-point wiring | Provide `scistudio init-block-package` scaffolding | Section 3 |
| Block authors need contract checks before distribution | Invalid ports or entry-point returns reach users late | Provide `BlockTestHarness` validation and smoke tests | Section 4 |
| SDK docs must map architecture concepts to author tasks | Authors would need to read ADR-017, ADR-020, and ADR-025 directly | Document execution, Collection transport, package metadata, and testing patterns | Section 5 |
| Historical IO guidance changed | New packages might implement deleted `scistudio.adapters` APIs | Point IO authors at ADR-028 plugin-owned IO blocks | Section 6 |
| SDK APIs must be auditable | Docs could describe helper signatures that no longer exist | Govern CLI and harness symbols explicitly | Section 8 |

## 2. Context

ADR-025 defines entry-point based block distribution. That protocol is useful
only if package authors can produce the expected project layout, metadata,
tests, and examples quickly. ADR-026 adds the developer-facing layer on top of
the distribution protocol.

The SDK is intentionally small. It does not add a separate framework for block
authoring; it packages the existing `Block`, `PackageInfo`, ports, `Collection`,
and entry-point contracts into a usable starter project.

## 3. Scaffolding

The CLI command `scistudio init-block-package` creates a package directory with:

- `pyproject.toml` entry-point wiring for `scistudio.blocks`;
- a source package with `PackageInfo` and example block exports;
- a minimal example block;
- pytest tests using `BlockTestHarness`;
- a README for local development and publication.

Template rendering is owned by `scistudio.cli._scaffold`. The scaffolder must be
deterministic, avoid overwriting an existing target directory, and return the
created root plus the relative files it wrote so the CLI can show a concise
creation summary.

## 4. Block Test Harness

`BlockTestHarness` is the public SDK test surface. It validates:

- the class inherits from `Block`;
- the class is concrete;
- `input_ports` and `output_ports` contain `InputPort` and `OutputPort`
  instances;
- `run()` exists;
- the block has a descriptive name;
- entry-point callables return either `(PackageInfo, list[type[Block]])` or
  the backward-compatible plain `list[type[Block]]`.

The smoke-test path instantiates the block, builds a `BlockConfig` through the
normal constructor path, calls `run(inputs, config)`, and returns the output
mapping. Exceptions raised by the block are intentionally propagated so pytest
failures show the real block error.

## 5. Developer Documentation Contract

SDK documentation should explain block development in task terms:

- the subprocess execution model from ADR-017;
- the cancellation model from ADR-018;
- `Collection` transport and block-local iteration from ADR-020;
- entry-point packaging and `PackageInfo` from ADR-025;
- type registration and domain package boundaries from ADR-027;
- current IO authoring through `IOBlock` subclasses from ADR-028.

Docs may reference ADRs for traceability, but human-facing block-development
guides should not require authors to read the ADR corpus before writing a
block.

## 6. Historical IO Guidance

The original ADR-026 source described a custom `FormatAdapter` path and
`scistudio.adapters` entry-points. That guidance is no longer active. ADR-028
deletes the central adapter registry and makes plugin-owned loaders and savers
normal blocks. Current SDK examples should teach `IOBlock`, `LoadData`,
`SaveData`, and plugin loader/saver subclasses instead.

## 7. Scope

In scope:

- CLI package scaffolding;
- scaffold templates;
- `scistudio.testing` and `BlockTestHarness`;
- SDK-facing documentation expectations;
- correction of historical adapter guidance.

Out of scope:

- changing the runtime block contract;
- resurrecting a central adapter registry;
- plugin dependency installation UX beyond generated packaging metadata;
- broad generated reference documentation.

## 8. Verification And Tooling Impact

Tests should verify template rendering, directory creation, refusal to
overwrite an existing target, generated entry-point metadata, harness block
validation, package-info validation, entry-point callable validation, and
smoke-test execution.

## 9. Consequences

- External developers get a consistent starting point for block packages.
- The test harness catches common contract mistakes before publication.
- The SDK creates a public `scistudio.testing` surface that must remain stable.
- Historical adapter examples must be kept out of current SDK guidance.

## 10. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Documentation only | Leaves too much boilerplate and entry-point detail to each author |
| Cookiecutter template | Adds another dependency and command surface for a small scaffold |
| Raw pytest examples only | Repeats block validation setup in every package |
| ADR-only developer guidance | Internal ADRs are not an ergonomic authoring manual |
