# ADR-046: DAGScheduler Sub-Package Decomposition

## 1. Decision Summary

`src/scistudio/engine/scheduler.py` (1744 LOC, single file) becomes the
sub-package `src/scistudio/engine/scheduler/`. The `DAGScheduler` class and
`RunHandle` stay defined in `__init__.py`; `DAGScheduler`'s 33 methods split
across 5 responsibility-grouped private sibling modules. Method bodies live
in the siblings as `_underscore` private module-level functions and bind
into the class via class-body assignment — the **Path D pattern**
established by Phase 2 PR #1460 (per umbrella #1427). No public surface
change; no behavior change; #1449 scheduler state-machine contract test
passes byte-identically.

This is a pure structural refactor. The single-file form has crossed every
reasonable god-file threshold; the decomposition relocates code without
touching semantics.

### 1.1 Problems Addressed

| Problem | Current pain | This ADR's response |
|---|---|---|
| `scheduler.py` is 1744 LOC | hardest file to review in the codebase; single-file change-impact analysis is impractical | Split by responsibility (5 sibling modules) |
| 6 distinct concerns interleave (dispatch / events / lineage / state / rerun / lifecycle) | a change to lineage emit touches the same file as cancellation logic | Each concern owns its own module |
| God-file advisory (#1427 Phase 1) tracks this as Bucket D | Bucket D is the final phase to reach the threshold-clean state | Closes the last large protected-core god-file |

## 2. Decomposition layout

```
src/scistudio/engine/scheduler/
├── __init__.py     # DAGScheduler class body + bindings, RunHandle, module helpers (~280 LOC)
├── _dispatch.py    # block dispatch loop                                          (~500 LOC)
├── _events.py      # EventBus subscriber handlers (_on_*)                         (~300 LOC)
├── _lineage.py     # output + lineage persistence                                 (~300 LOC)
├── _state.py       # state machine helpers — only moves, semantics unchanged      (~50 LOC)
└── _rerun.py       # rerun / reset / graph-traversal                              (~150 LOC)
```

Method assignment per group (full inventory in [docs/audit/2026-05-22-umbrella-1427-phase1-with-context.md] is the Phase 1 audit's data; the table below is the canonical Phase 3 mapping):

| Group | Methods | Sibling |
|---|---|---|
| Public lifecycle (stays in `__init__.py`) | `__init__`, `execute`, `pause`, `resume`, `cancel_workflow`, `cancel_block`, `execute_from`, `rerun_block`, `reset_block`, `save_checkpoint`, `block_states`, `set_state` | — |
| Block dispatch | `_emit_block_ready`, `_dispatch`, `_run_and_finalize`, `_run_interactive`, `_dispatch_newly_ready`, `_instantiate_block`, `_gather_inputs` | `_dispatch.py` |
| Event handlers | `_on_block_done`, `_on_block_error`, `_on_cancel_block`, `_on_cancel_workflow`, `_on_process_exited`, `_on_interactive_complete` | `_events.py` |
| Lineage + output | `_persist_output_metadata`, `_resolve_lineage_store`, `_persist_single_output`, `_upsert_wire_row`, `_sync_checkpoint_to_store`, `_build_block_done_data`, `_build_block_terminal_data`, `_resolve_block_type`, `_resolve_block_version` | `_lineage.py` |
| State machine | `_check_readiness`, `_check_completion`, `_propagate_skip` | `_state.py` |
| Rerun / graph | `_cancel_if_active`, `_ancestors_of`, `_reset_upstream`, `_reset_downstream_skipped`, `_drain_active_tasks`, `_cancel_active_tasks_on_shutdown` | `_rerun.py` |

`RunHandle` (small dataclass-like, ~10 LOC) stays in `__init__.py` next to `DAGScheduler` for read locality.
The 3 module-level helpers `_extract_error_summary`, `_collect_object_ids`, `_object_ids_for_value` also stay in `__init__.py`.

## 4. Path D binding sketch

```python
# src/scistudio/engine/scheduler/__init__.py (excerpt)
from scistudio.engine.scheduler import _dispatch, _events, _lineage, _state, _rerun

class DAGScheduler:
    def __init__(self, ...): ...
    async def execute(self) -> None: ...

    # bindings — method bodies live in private siblings
    _dispatch = _dispatch._dispatch
    _emit_block_ready = _dispatch._emit_block_ready
    _on_block_done = _events._on_block_done
    _check_readiness = _state._check_readiness
    # … 30 more bindings
```

This is identical to Phase 1 A1's `ApiRuntime` pattern (PR #1445) and Phase 2's `SaveData` / `LoadData` (PR #1460). §C9-style compliance: no helper classes; siblings contain only private functions.

## 5. State machine: physical move only

`_check_readiness`, `_check_completion`, `_propagate_skip` are governed
by **#1449 scheduler state-machine contract test** (`tests/engine/test_scheduler_state_machine_contract.py`, merged 2026-05-22). This ADR explicitly **does not refactor these three functions**; the implementer agent must relocate them to `_state.py` with byte-identical semantics. The contract test runs against `DAGScheduler` via the public surface so a Path D move passes without modification.

If any future implementer believes a semantic improvement is warranted, that change requires a separate ADR and a contract-test update. Path D is for code organization; state-machine cleanup is a different decision.

## 6. Scope

In scope:

- The 6-way sub-package split exactly as Section 3 specifies.
- `tests/engine/test_scheduler*.py` import-surface verification (must continue to pass without test edits).
- Frontmatter `governs.files` glob expansion to cover `scheduler/**` (this ADR plus ADR-018, ADR-020, ADR-038, ADR-039, ADR-042 if they reference `scheduler.py`).
- `scripts/check_god_files.py` waiver removal once the new files all measure <750 LOC.
- One new test file under `tests/engine/test_scheduler_package_layout.py` asserting (via AST walk) that `_dispatch.py` / `_events.py` / `_lineage.py` / `_state.py` / `_rerun.py` contain zero `class` definitions — protects the Path D contract from drift.

Out of scope:

- Any behavior change in any of the 33 methods.
- State-machine semantic edits (see Section 5).
- Engine-wide architectural changes (event ordering, subscription wiring, lineage schema).
- Resource accounting (#887) — orthogonal future work.
- ADR-018 subprocess-isolation contract (preserved verbatim).

## 8. Consequences

Positive:

- Each sibling module < 750 LOC, individually reviewable.
- Reviewers can navigate by concern (event subscription / lineage emit / dispatch loop) without re-loading the same 1744-line file.
- New tests pinning sub-package shape catch future drift early.

Negative / accepted:

- The `DAGScheduler` class API surface is now defined by ~30 binding lines in `__init__.py` rather than 1744 inline lines. Phase 1 audit accepted this pattern; same trade-off applies here.
- The state-machine module (`_state.py`) is only ~50 LOC. The justification for its own file is the `#1449` contract semantic locality, not size.

## 9. Alternatives considered

| Alternative | Reason rejected |
|---|---|
| Leave `scheduler.py` as 1744 LOC, no decomposition | Defeats the entire purpose of umbrella #1427 |
| Decompose into multiple `Block`-style classes (composition) | Breaks `#1449` contract (its test imports `DAGScheduler.<method>` paths directly); architectural change masquerading as refactor |
| Single `_methods.py` for all 30 bindings | Trades one god file for a slightly smaller god file; defeats the per-concern split |
| Use `mixin`s to group methods | §C9-style compliance forbids helper classes; mixins are helper classes by another name |
