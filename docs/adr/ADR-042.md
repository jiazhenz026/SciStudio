# ADR-042: Repository Quality Governance

## 1. Decision Summary

ADR-042 defines the repository quality governance system for SciEasy. It
establishes a non-recursive governance ADR for code, documentation, and
AI-assisted development. The purpose is not to create a self-iterating
process. The purpose is to define a stricter, machine-checked repository
management contract.

### 1.1 Problems Addressed

| Problem | Risk | ADR-042 response | Detailed section |
|---|---|---|---|
| Code implementation and documentation disagree | Contributors and users trust stale or false contracts | Define facts, drift classes, and code/documentation cross-check tools | Section 5 |
| Documentation is inaccurate or unstructured | Docs become long, duplicated, unowned, and hard for humans or agents to follow | Define document categories, frontmatter, length rules, and doc lint | Section 3 |
| Agent harness engineering has limits | AI agents may skip process, broaden scope, or weaken rules | Define AI developer workflow, persona, skill, and rule-protection constraints | Section 7 |
| Documentation is not automatically built | API, CLI, schema, and entry-point references drift when maintained by hand | Define Sphinx and generator-based auto documentation | Section 6 |
| Code quality rules are incomplete or inconsistent | Style, type, import, test, security, and frontend quality are enforced unevenly | Define the code normalization tool stack and CI/pre-commit topology | Section 4 |

## 2. Scope

### 2.1 In Scope

ADR-042 defines repository governance rules for:

- code normalization;
- documentation normalization;
- code/documentation consistency checking;
- documentation auto-generation;
- AI-assisted development constraints;
- human developer exemption principles;
- the common and custom tools required to enforce these rules.

ADR-042 may require new schemas, audit tools, CI jobs, pre-commit hooks,
documentation structure, and contributor workflow documentation.

### 2.2 Enforcement Boundary

ADR-042 defines the target governance model. A rule becomes blocking only
when:

1. the corresponding tool exists;
2. the tool has focused tests;
3. the current repository baseline is known;
4. the owner approves the transition from report-only to hard-fail.

Before that transition, tools may run in report-only mode. A later section may
declare a stricter rule for a specific problem class. Section 5 does so for
code/documentation consistency: once a consistency check is wired, findings
hard-fail immediately so existing technical debt is forced into the cleanup
queue instead of being silently grandfathered.

### 2.3 Migration Boundary

Accepting ADR-042 does not mean all existing repository files are immediately
compliant. Existing drift, missing docs, weak configs, incomplete generated
references, or missing governance metadata are cleanup work tracked
separately.

New work after ADR-042 acceptance should follow the new rules unless the owner
explicitly grants an exemption.

### 2.4 Out of Scope

ADR-042 does not define:

- a self-iterating ADR system;
- temporary release-readiness gates;
- automatic governance-rule modification by AI agents;
- automatic acceptance of implementation work without owner review;
- replacement of the existing project workflow state machine;
- product features unrelated to repository governance;
- immediate cleanup of every historical violation.

## 3. Documentation Normalization

### 3.1 What And Why

SciEasy needs documentation that is short, structured, owned, and verifiable.
Long prose files are difficult for humans to keep current and difficult for
agents to follow. Unstructured docs also make drift detection ambiguous:
tools cannot determine whether a file is an ADR, a contract, a workflow, a
user guide, a generated reference, or an audit record.

Documentation normalization gives every document a category, owner, expected
shape, and validation path. The purpose is not to make every document short.
The purpose is to make each document type have a fixed responsibility, fixed
reader model, fixed structure, and fixed automated check.

### 3.2 Documentation Categories

| Document type | Path | Required standard |
|---|---|---|
| ADR | `docs/adr/ADR-NNN.md` | Records why an architectural decision was made, its status, consequences, and governed artifacts |
| Spec | `docs/specs/**` | Defines contracts and expected behavior; must identify affected APIs/schemas/runtime behavior |
| Architecture docs | `docs/architecture/**` | Describes stable system structure; longer form allowed |
| Contributor docs | `docs/contributing/**` | Short procedural docs for humans and agents contributing to this repo |
| User docs | `docs/user/**` | End-user installation, quickstart, guides, tutorials, generated reference |
| Production-agent docs | `docs/prod-agent/**` | Maintainer/operator docs for the production embedded-agent stack |
| Doc-guide docs | `docs/doc-guide/**` | Rules for writing and maintaining documentation |
| Audit reports | `docs/audit/**` | Machine-readable or human-readable verification output |
| Generated docs | `docs/user/reference/**`, `docs/user/llms.txt` | Generated from code or facts; hand edits rejected |

### 3.3 ADR Schema

Every ADR MUST have YAML frontmatter that validates against this schema.
Implementation may use Pydantic, JSON Schema, or an equivalent typed schema,
but the accepted field set and validation rules are normative.

| Field | Type | Required | Validation |
|---|---|---:|---|
| `adr` | integer | yes | Positive integer; filename MUST be `ADR-{adr:03d}.md` |
| `title` | string | yes | 4-160 characters |
| `status` | enum | yes | `Proposed`, `Accepted`, `Deprecated`, `Superseded` |
| `date_created` | date | yes | ISO date |
| `date_accepted` | date or null | yes | Required when `status=Accepted`; null when `status=Proposed` |
| `date_superseded` | date or null | yes | Required when `status=Superseded`; otherwise null |
| `supersedes` | list[int] | yes | Existing ADR numbers or empty list |
| `superseded_by` | int or null | yes | Required when `status=Superseded`; otherwise null |
| `related` | list[int] | yes | Existing ADR numbers or empty list |
| `closes_issues` | list[int] | yes | GitHub issue numbers or empty list |
| `tracking_issue` | int or null | yes | Required when implementation work is expected |
| `is_code_implementation` | bool | yes | True when ADR requires code/tooling changes |
| `governs.modules` | list[dotted-path] | yes | Importable module prefixes or empty list |
| `governs.contracts` | list[dotted-path] | yes | Importable symbols or empty list |
| `governs.entry_points` | list[string] | yes | Entry-point names/groups or empty list |
| `governs.files` | list[repo-path-or-glob] | yes | Repo-relative paths or globs |
| `governs.excludes` | list[repo-path-or-glob] | yes | Exclusions from governed files |
| `tests` | list[repo-path] | yes | Required when `is_code_implementation=true`; otherwise may be empty |
| `agent_editable` | bool or enum | yes | `false` for governance ADRs unless owner explicitly allows edits |
| `assisted_by` | list[string] | yes | AI/runtime assistance lines or empty list |
| `phase` | enum | yes | `planning`, `implementation`, `complete`, `maintenance` |
| `tags` | list[string] | yes | Lowercase slugs |
| `owner` | string | yes | GitHub handle or owner alias |
| `co_authors` | list[string] | yes | Handles or empty list |
| `language_source` | enum | yes | `en` by default |
| `translations` | list[Translation] | yes | Translation records or empty list |

`Translation` has this schema:

| Field | Type | Required | Validation |
|---|---|---:|---|
| `locale` | string | yes | BCP-47-like locale, e.g. `zh-CN` |
| `path` | repo path | yes | Must live under `docs/<locale>/` |
| `auto_generated` | bool | yes | True if generated by translation workflow |
| `source_sha` | string or null | yes | Source commit SHA when generated |

ADR body structure is also normative. Every ADR MUST follow this outline:

1. `# ADR-NNN: <title>`
2. `## 1. Decision Summary`
3. `### 1.1 Problems Addressed`
4. One table whose rows list every problem the ADR addresses.
5. Each problem row MUST include a `Detailed section` column pointing to a
   later section in the same ADR.
6. Later sections MUST explain every problem listed in `1.1`.
7. The ADR MUST include scope, decision details, verification/tooling impact,
   consequences, and alternatives considered.

ADR-042 itself is the template for governance ADRs. Ordinary ADRs may be
shorter, but the first section and `1.1 Problems Addressed` traceability rule
still applies when the ADR addresses more than one problem.

### 3.4 Spec Schema

Specs MUST be compatible with the repository's SpecKit workflow. A SpecKit
feature spec lives at `specs/[###-feature-name]/spec.md`; legacy project specs
under `docs/specs/` may remain during migration, but new specs MUST follow the
same structural contract.

Every normalized spec MUST have YAML frontmatter with this schema:

| Field | Type | Required | Validation |
|---|---|---:|---|
| `spec_id` | string | yes | Lowercase slug or SpecKit branch id |
| `title` | string | yes | 4-160 characters |
| `status` | enum | yes | `Draft`, `Clarifying`, `Planned`, `Implemented`, `Deprecated` |
| `feature_branch` | string | yes | SpecKit branch name, e.g. `012-feature-name` |
| `created` | date | yes | ISO date |
| `input` | string | yes | Original user/request summary |
| `owners` | list[string] | yes | At least one owner |
| `related_adrs` | list[int] | yes | Governing ADRs or empty list |
| `related_specs` | list[string] | yes | Related spec ids or empty list |
| `scope.in` | list[string] | yes | Explicitly in-scope items |
| `scope.out` | list[string] | yes | Explicitly out-of-scope items |
| `governs.modules` | list[dotted-path] | yes | Affected modules or empty list |
| `governs.contracts` | list[dotted-path] | yes | Affected public contracts or empty list |
| `governs.files` | list[repo-path-or-glob] | yes | Affected files or globs |
| `tests` | list[repo-path] | yes | Expected test files or empty list when unknown |
| `acceptance_source` | enum | yes | `speckit`, `issue`, `adr`, or `manual` |
| `language_source` | enum | yes | `en` by default |

The spec body MUST follow the SpecKit section contract:

| Section | Required | Validation |
|---|---:|---|
| `## 1. Change Summary` | yes | First H2; explains what changes and why |
| `## 2. User Scenarios & Testing` | yes | Contains prioritized independently testable user stories |
| `### User Story N - <title> (Priority: Pn)` | yes | Priorities start at P1 and are unique within the spec |
| `Why this priority` | yes | Required for each user story |
| `Independent Test` | yes | Required for each user story |
| `Acceptance Scenarios` | yes | At least one Given/When/Then scenario per user story |
| `### Edge Cases` | yes | May be empty only with explicit `None identified` |
| `## 3. Requirements` | yes | Contains functional requirements and key entities when data is involved |
| `### Functional Requirements` | yes | IDs MUST be `FR-001`, `FR-002`, etc. without gaps |
| `### Key Entities` | conditional | Required when feature introduces or changes data entities |
| `## 4. Implementation Plan` | yes | Defines technical approach, affected files, sequencing, and verification plan |
| `## 5. Success Criteria` | yes | Contains measurable outcomes |
| `### Measurable Outcomes` | yes | IDs MUST be `SC-001`, `SC-002`, etc. without gaps |
| `## 6. Assumptions` | yes | Lists assumptions or explicitly states none |

Spec body entities use this normalized schema:

| Entity | Required fields | Validation |
|---|---|---|
| `UserStory` | `id`, `title`, `priority`, `description`, `why_priority`, `independent_test`, `acceptance_scenarios` | `priority` values are `P1`, `P2`, etc.; each story is independently testable |
| `AcceptanceScenario` | `given`, `when`, `then` | Must be written in Given/When/Then form |
| `FunctionalRequirement` | `id`, `text` | IDs are contiguous `FR-NNN`; unclear requirements must include `NEEDS CLARIFICATION` |
| `KeyEntity` | `name`, `description`, `attributes`, `relationships` | Required when the feature changes data shape, persistence, or public schema |
| `ImplementationPlan` | `technical_approach`, `affected_files`, `sequence`, `verification`, `risks` | Required for every normalized spec; may link to SpecKit `plan.md` for expanded detail |
| `ImplementationTask` | `id`, `title`, `story`, `files`, `depends_on`, `verification` | IDs are contiguous `T-NNN`; each task maps to a user story, foundation task, or cross-cutting concern |
| `SuccessCriterion` | `id`, `metric`, `target`, `measurement_method` | IDs are contiguous `SC-NNN`; criteria are technology-agnostic and measurable |
| `Assumption` | `text`, `source` | `source` is `owner`, `spec`, `adr`, `existing-system`, or `inferred` |

The `## 4. Implementation Plan` section MUST include these subsections:

| Subsection | Required content |
|---|---|
| `### 4.1 Technical Approach` | Proposed architecture, data flow, dependencies, and major design choices |
| `### 4.2 Affected Files` | Table of files or globs, action (`create`, `modify`, `delete`, `generate`), and rationale |
| `### 4.3 Implementation Sequence` | Ordered task list with dependencies; may reference SpecKit `tasks.md` task ids |
| `### 4.4 Verification Plan` | Tests, lint/type checks, docs checks, and manual verification expected before merge |
| `### 4.5 Risks And Rollback` | Known risks, migration concerns, and rollback or mitigation plan |

SpecKit plan artifacts are normalized as follows:

- `plan.md` starts with `# Implementation Plan: <feature>` and expands the
  spec's `## 4. Implementation Plan`; it MUST include technical context,
  constitution check, project structure, and any complexity tracking.
- `tasks.md` starts with `# Tasks: <feature>` and its first H2 is
  `## 1. Change Summary`; it then follows the SpecKit phased task format.
  Every implementation task MUST map back to a user story, a foundational
  prerequisite, or a cross-cutting concern from the spec's implementation
  plan.
- `research.md`, `data-model.md`, `quickstart.md`, and `contracts/` are
  created when required by `/speckit.plan`.

### 3.5 Document Structure Standard

All governed Markdown documents MUST use one of two first sections:

- `## 1. Decision Summary` for ADRs, architecture decisions, policy docs, and
  governance docs.
- `## 1. Change Summary` for specs, implementation plans, audit reports,
  changelogs, migration notes, and task/result reports.

The first H2 after the H1 MUST be one of those exact headings. The rule is
machine-checkable by scanning Markdown headings after stripping YAML
frontmatter.

For ADRs, `## 1. Decision Summary` MUST contain `### 1.1 Problems Addressed`
when the ADR addresses one or more concrete problems. The `1.1` table MUST
have these columns:

| Column | Meaning |
|---|---|
| `Problem` | The problem being solved |
| `Risk` | What happens if the problem is not solved |
| `ADR response` | The decision-level response |
| `Detailed section` | Later section that explains the response |

The structure checker MUST verify that every `Detailed section` value points
to an existing later section. This makes the summary table an index into the
ADR rather than a decorative overview.

For specs, `## 1. Change Summary` MUST summarize the requested behavior
change and identify whether the spec came from `/speckit.specify`, a GitHub
issue, an ADR, or a manual owner request.

### 3.6 Short-Form Documentation Standard

Hand-authored documents under `docs/contributing/`, `docs/user/`,
`docs/prod-agent/`, and `docs/doc-guide/` must be concise. The default limit
is:

- at most 120 non-empty source lines, excluding YAML frontmatter;
- at most 600 prose words, excluding YAML frontmatter and fenced code blocks.

Generated reference docs are exempt from this length rule because their source
of truth is code.

## 4. Code Normalization

### 4.1 What And Why

Code normalization gives the repository a stable, automatically enforceable
baseline for style, types, tests, import boundaries, security, complexity, and
frontend quality. It reduces review noise and makes higher-level consistency
checks more reliable because code shape becomes predictable.

### 4.2 Code Quality Categories

| Category | Requirement | Primary tools |
|---|---|---|
| Formatting | Python, TypeScript, and config formatting are deterministic | `ruff format`, `prettier`, `pyproject-fmt` |
| Lint | Static findings are visible, stable, and ratcheted down | `ruff check`, `eslint`, `sphinx-lint`, `markdownlint-cli2` for code-adjacent docs |
| Typing | Python and TypeScript public surfaces type-check | `mypy`, `pyright`, `tsc --noEmit` |
| Tests | Tests are deterministic, timeout-bounded, parallel-safe, and meaningful | `pytest`, `pytest-xdist`, `pytest-timeout`, `pytest-randomly`, `vitest` |
| Coverage | Coverage is measured and ratcheted; new QA tooling has stronger floors | `pytest-cov`, `coverage` |
| Import boundaries | Architecture layers do not drift through convenience imports | `import-linter` |
| API surface | Public API additions and breaking changes are visible | `griffe`, `griffe-pydantic`, `typedoc` |
| Dead code | Unused or unreachable code is detected after baseline | `vulture` |
| Complexity | Excessively complex functions are detected after baseline | `xenon` |
| Security | Source and dependency risks are detected | `ruff` `S` rules, optional `bandit`, `pip-audit`, `zizmor` |
| Workflow and config | CI/workflow/config files are syntactically valid and secure | `actionlint`, `yamllint`, `pyproject-fmt`, `pre-commit` |
| Spelling | Common typos in code-facing text are detected | `codespell` |

### 4.3 Enforcement Levels

Each code normalization tool has an explicit enforcement level:

| Level | Meaning |
|---|---|
| `required-now` | Existing hard-fail check; a violation blocks merge |
| `pre-commit-required` | Runs before commit; a local F-grade or configured violation blocks the commit |
| `report-only` | Runs and reports findings, but does not block |
| `baseline-required` | Must collect and review current findings before hard-fail |
| `future-hard-fail` | Intended to become blocking after implementation, focused tests, and owner approval |
| `manual-only` | Used on demand or scheduled because it is too slow or context-heavy for every PR |

No tool becomes hard-fail merely because it is listed here. The transition
rules in Section 2.2 apply, except for `code_score` F grades: once the scoring

### 4.7 Code Scoring

ADR-042 introduces a custom `code_score` tool. It assigns A/B/C/D/F grades to
code changes and modules without letting historical module debt block small
safe edits.

| Grade | Meaning | Merge behavior |
|---|---|---|
| A | Clean module: no hard findings; strong type/test/coverage/complexity signals | Allowed |
| B | Minor findings only; no blocking risk | Allowed |
| C | Noticeable debt but no merge blocker | Allowed with visible report |
| D | High debt or missing quality signals; owner review required | Allowed only while baseline permits it |
| F | Failing module: hard tool failure, critical security issue, invalid import boundary, test failure, syntax/import error, or score below floor | Blocks pre-commit and cannot enter `main` |

`code_score` has four score layers:

| Layer | Timing | Purpose | Blocking? | Storage |
|---|---|---|---:|---|
| Change score | Pre-commit and PR CI | Score only changed files, changed hunks, and changed symbols | Yes; F blocks | `.git/scieasy/code-score/latest-change.json` locally; `docs/audit/code-score/changes/<sha>.json` in CI when committed or uploaded |
| Module health score | Weekly scheduled audit | Score entire modules/packages, including historical debt | No direct block for small edits | `docs/audit/code-score/module-health/<YYYY-MM-DD>.json` |
| AI advisory score | Pre-commit or CI when a headless CLI is available | Ask Claude Code/Codex CLI to review changed modules and deterministic reports | No; skipped if CLI unavailable | `docs/audit/code-score/ai/<sha>.json` when available |
| Final gate score | Pre-commit and PR CI | Combine change score, current module health, and AI advisory status | Yes; only deterministic F blocks | Same report as change score |

The change score is the gate score for ordinary commits. It evaluates whether
the diff introduces or worsens objective problems. It MUST NOT fail a small
bugfix merely because the touched module has pre-existing historical debt.

The module health score is a background signal produced by a weekly scheduled
audit. It helps reviewers understand whether a change touches a weak module,
but it does not block a safe change by itself. A change that improves an
F-health module may merge even if the full module remains F.

The final gate rule is:

- `Change score = F` blocks pre-commit and PR merge.
- `Module health score = F` does not block if `Change score` is C or better
  and the diff does not worsen the module's known findings.
- Touching the exact function/class that owns an existing F finding without
  improving it may lower the change score.
- AI advisory scores never block unless the owner later explicitly changes
  this ADR.

`code_score` deterministic scoring consumes outputs from:

- `ruff`;
- `mypy`;
- `pyright`;
- `pytest`;
- `pytest-cov` / `coverage`;
- `import-linter`;
- `griffe`;
- `vulture`;
- `xenon`;
- `pip-audit` / security lint;
- `test_quality`;
- targeted mutation results when available.

AI advisory scoring is optional and best-effort:

- The tool may call a headless CLI such as `claude` or `codex`.
- Input is the changed diff, touched file excerpts, and deterministic tool
  report.
- Output MUST be machine-readable JSON with `score`, `summary`, `findings`,
  and `confidence`.
- If the CLI is missing, times out, exits non-zero, or returns invalid JSON,
  the AI advisory score is recorded as `skipped`.
- A skipped AI score never fails the commit or PR.

Reports use this shape:

```yaml
scores:
  deterministic:
    change_score: B
    module_health_score: C
    final_gate_score: B
  ai:
    provider: codex-cli
    score: C
    confidence: medium
    status: completed
  final:
    blocks_merge: false
    reason: deterministic final gate score is not F
```

Storage rules:

- Local pre-commit reports go under `.git/scieasy/code-score/` and are not
  committed.
- CI reports are uploaded as workflow artifacts and may also be committed
  under `docs/audit/code-score/changes/` when the owner wants persistent
  history.
- Weekly module-health reports are committed or published under
  `docs/audit/code-score/module-health/`.
- AI advisory reports are stored separately from deterministic reports under
  `docs/audit/code-score/ai/` so reviewers can compare AI and non-AI scores.

### 4.8 Custom Code-Quality Tools

Custom code-quality tools do not replace `pytest`, type checkers, or human
review. They only check mechanical risks that are otherwise easy to miss.

| Need | Custom tool | Enforcement |
|---|---|---|
| Score modules A/B/C/D/F | `code_score` | Pre-commit F hard-fail; CI F hard-fail |
| Detect weak tests | `test_quality.ast_lint` | Report-only, then hard-fail for hard anti-patterns |
| Detect mutation gaps | `test_quality.mutation_runner` / `mutmut` wrapper | Manual/scheduled, then targeted CI |

| Verify completion artifacts for governed changes | `complete_artifacts` | Future hard-fail for governed changes |
| Verify codemod metadata and idempotence | `codemod_lint` | Future hard-fail when codemods exist |

| Placement | Tools |

## 5. Code And Documentation Consistency

### 5.1 What And Why

SciEasy's architecture is documented through ADRs, specs, architecture docs,
and generated reference. If those documents claim code exists, behaves a
certain way, or exposes a public contract, the repository must be able to
verify that claim.

The goal is not to make prose impossible to write. The goal is to make claims
about modules, symbols, files, commands, workflows, schemas, entry points, and
ownership machine-checkable.

### 5.2 Fact Model

After ADR-042, a fact is a machine-verifiable statement about the repository.
Facts are represented with this normalized model:

| Field | Type | Meaning |
|---|---|---|
| `id` | string | Stable fact id, e.g. `python.symbol.scieasy.foo.Bar` |
| `kind` | enum | `adr`, `spec`, `file`, `symbol`, `entry-point`, `cli`, `openapi`, `schema`, `workflow`, `maintainer`, `skill`, `tool-output`, `generated-doc`, `expected-signature`, `expected-model-field`, `expected-cli-command` |
| `source` | repo path or tool id | Where the fact came from |
| `subject` | string | Path, symbol, command, workflow, or document being described |
| `value` | scalar/object/list | Machine-readable fact payload |
| `owner` | string or null | Owning ADR/spec/maintainer where known |
| `generated_at` | datetime | Extraction time |
| `source_sha` | string | Git commit used for extraction |
| `confidence` | enum | `normative`, `generated`, or `observed` |
| `stability` | enum | `stable`, `experimental`, `deprecated`, or `unknown` |

The facts registry is the generated YAML snapshot of these facts. It is an
input to drift tools and documentation substitution. Hand-authored prose may
refer to facts, but the facts registry itself is generated.

### 5.3 Fact Sources

Fact sources have three confidence levels:

| Source | Confidence | Extracted facts |
|---|---|---|
| ADR frontmatter and body structure | `normative` | Governed modules/files/contracts, decisions, problem-to-section mapping |
| Spec frontmatter and SpecKit body structure | `normative` | Requirements, implementation plan, affected files, tests, success criteria |
| Spec signature-level contracts | `normative` | Expected function/class signatures, Pydantic model fields, CLI commands, and exit-code behavior |
| `MAINTAINERS` | `normative` | Ownership and review routing |
| Workflow files | `normative` | Required CI jobs, triggers, protected paths, scheduled audits |
| Tool config files | `normative` | Lint/type/test/security/doc tool settings |
| Code symbols via `griffe` | `generated` | Public modules, classes, functions, signatures, docstrings |
| Package entry points | `generated` | Plugin/block/runner extension surfaces |
| CLI definitions | `generated` | Commands, options, arguments |
| OpenAPI schemas | `generated` | HTTP endpoints and request/response contracts |
| Pydantic schemas | `generated` | Structured data and settings contracts |
| Generated docs | `generated` | API/schema/CLI/reference pages and `llms.txt` |
| CI/tool outputs | `observed` | Current test, lint, type, coverage, security, and score results |
| `code_score` reports | `observed` | Change scores, module health scores, AI advisory scores |
| Audit reports | `observed` | Aggregated findings and historical trends |

### 5.4 Drift Classes

ADR-042 uses four drift classes. All non-`match` classes hard-fail once their
checker is wired. There is no report-only or grandfathered baseline for
code/documentation consistency: existing drift is technical debt that must be
fixed, not hidden.

| Class | Meaning | Detection source | Owner action | Severity |
|---|---|---|---|---|
| `match` | Documentation and facts agree | Any cross-check tool | None | None |
| `behavior-drift` | Documentation describes behavior that code no longer implements | `doc_drift`, tests, specs, generated facts | Fix code or update the governing document | Error |
| `phantom-reference` | Documentation references missing files, symbols, commands, paths, jobs, or generated artifacts | `doc_drift`, `closure`, Sphinx, `griffe` | Create the artifact, correct the reference, or mark future work with a tracked issue | Error |
| `missing-documentation` | Public code surface exists without required ADR/spec/user/generated documentation | `closure`, `griffe`, entry-point extraction, generated docs checks | Add the required documentation or intentionally make the surface private | Error |
| `signature-drift` | A spec-defined function, class, model field, CLI command, or exit-code contract differs from implementation | `signature_drift`, `griffe`, `inspect`, CLI dry-run fixtures | Update the implementation or revise the governing spec/ADR | Error |

### 5.5 Cross-Check Tools

Each consistency tool has one primary responsibility:

| Tool | Inputs | Output | Responsibility | Failure rule |
|---|---|---|---|---|
| `generate_facts` | Code, docs, workflows, maintainers, skills, tool configs | `docs/facts/generated.yaml` | Produce fact snapshot | Stale generated facts fail |
| `fact_drift` | Facts and prose substitutions | `AuditReport` findings | Detect stale or invalid fact substitutions in prose | Any finding fails |
| `doc_drift` | Facts, ADRs, specs, architecture docs, curated docs | `AuditReport` findings | Detect prose/fact contradiction and phantom references | Any finding fails |
| `closure` | ADR governs, specs, `MAINTAINERS`, public surfaces, generated docs | `AuditReport` findings | Verify ownership and required documentation coverage in both directions | Any finding fails |
| `signature_drift` | Signature facts, griffe symbols, Pydantic models, CLI dry-run output | `AuditReport` findings | Verify spec-level implementation signatures match code and CLI entry points | Any signature mismatch fails once wired |
| `full_audit` | All cross-check tools | Aggregate `AuditReport` | Run the consistency suite and preserve a single report envelope | Any child error fails |
| `griffe` | Python source | Symbol/API graph | Feed facts, API docs, and closure checks | Extraction failure fails |
| Sphinx build | Docs source and generated reference | HTML and warnings | Validate links, references, generated docs, and docs build health | Warning-as-error fails |

### 5.6 Ownership And Coverage Closure

Ownership closure has two directions:

- Every governed code path or public surface must have an owning ADR/spec or
  maintainer entry.
- Every ADR/spec governed file or symbol claim must resolve to a real path or
  symbol unless explicitly marked future work.

Future work must be tracked in an issue. Silent placeholder claims are drift.

### 5.7 Enforcement

Code/documentation consistency is a zero-baseline hard-fail area. When a
checker from Section 5.5 is wired into pre-commit or CI, every detected error
fails immediately. The cleanup process is therefore:

1. wire the checker;
2. let it fail on existing debt;
3. fix the debt or add a tracked future-work record where future work is
   legitimate;
4. keep the checker hard-failing thereafter.

Allowed future-work records must be explicit and machine-readable enough for
the checker to distinguish "not implemented yet, tracked" from "hallucinated
or stale claim." A verbal promise or untracked TODO is not sufficient.

## 6. Documentation Auto-Generation

### 6.1 Why

Generated reference docs prevent hand-written references from drifting. API,
CLI, schema, OpenAPI, and entry-point information should be derived from code
whenever possible. Hand-authored docs should explain concepts and workflows;
generated docs should list exact surfaces.

### 6.2 Boundary: Generation Backend Versus Governance Tooling

Sphinx and griffe are generation and extraction backends. They are required,
but they are not the complete governance system.

| Tool | Handles | Does not handle |
|---|---|---|
| `griffe` | Python module/symbol/signature/docstring extraction; API surface input for generated docs and facts | ADR/spec schema validation, ownership closure, behavior drift classification, AI governance |
| `sphinx` | Documentation build, references, nitpicky warnings, generated reference integration, link checking | Whether prose claims match code behavior, whether ADR `governs` claims are complete, whether generated docs were hand-edited |

Repository-specific governance remains in Section 5 and Section 8 custom
tools. Generation backends provide facts and generated pages; custom tools
decide whether those facts satisfy SciEasy policy.

### 6.3 Generated Documents

| Generated document | Target path | Source |
|---|---|---|
| Python API reference | `docs/user/reference/api/` | `sphinx-autoapi`, `griffe` |
| Schema reference | `docs/user/reference/schemas/` | Pydantic models, `autodoc-pydantic` |
| CLI reference | `docs/user/reference/cli.md` | Typer/click command tree |
| Server API reference | `docs/user/reference/server-api.md` | FastAPI OpenAPI |
| Entry-point catalog | `docs/user/reference/entry-points.md` | `importlib.metadata.entry_points()` |
| Block catalog | `docs/user/reference/blocks/` | SciEasy block registry / entry points |
| Runner catalog | `docs/user/reference/runners/` | Runner registry / entry points |
| Tutorial gallery | `docs/user/tutorials/`, `docs/user/examples-gallery/` | `sphinx-gallery` examples |
| LLM context file | `docs/user/llms.txt` | Docs toctrees and selected generated references |
| Facts registry | `docs/facts/generated.yaml` | `generate_facts` |

### 6.4 Required Tools

Documentation generation uses these tool groups:

| Group | Tools |
|---|---|
| Sphinx core | `sphinx`, `myst-parser`, `sphinx-needs`, `pydata-sphinx-theme` |
| API extraction/reference | `griffe`, `sphinx-autoapi`, `sphinx-autosummary`, `numpydoc`, `sphinx.ext.linkcode`, `sphinx.ext.intersphinx` |
| Schema/CLI/server generators | `autodoc-pydantic`, `sphinx-click` or equivalent Typer integration, `sphinxcontrib-openapi` |
| Examples/tutorials | `sphinx-gallery`, `pytest-examples` |
| Site UX | `sphinx-design`, `sphinx-copybutton`, `sphinx-issues` |
| SciEasy generators | `llms_txt`, `entry_point_catalog`, `cli_reference`, `openapi_reference`, `schema_reference`, block/runner catalog generators |
| Freshness/governance checks | `auto_generated_lint`, `generate_facts`, `fact_drift`, `doc_drift`, `closure` |

### 6.5 Generation Pipeline

The documentation build pipeline runs in this order:

1. Run `generate_facts` to refresh `docs/facts/generated.yaml`.
2. Generate API, schema, CLI, OpenAPI, entry-point, block, and runner
   reference docs.
3. Generate `docs/user/llms.txt`.
4. Run Sphinx with warnings treated as errors and nitpicky references enabled.
5. Run `auto_generated_lint` to verify committed generated files match the
   generator output.
6. Run `fact_drift`, `doc_drift`, and `closure` to validate generated and
   hand-authored docs against the fact model.

This order is mandatory. Consistency checks run after generation so they see
the same facts and generated pages that readers will see.

### 6.6 Hand-Edit Policy

Files marked as generated must not be hand-edited. `auto_generated_lint`
compares generated outputs with committed files and fails if the committed
tree is stale or manually modified.

If generated documentation is wrong, the contributor must change the source:
code, schema, CLI definition, OpenAPI source, entry-point metadata, facts
extractor, or generator. Editing generated Markdown is a policy violation.

### 6.7 Failure Semantics

Documentation auto-generation is a hard-fail area once wired. The following
conditions fail pre-commit or CI immediately:

- generated file is stale;
- generated file was manually edited;
- Sphinx emits a warning;
- nitpicky reference resolution fails;
- linkcheck fails for a non-baselined link;
- required generated target is missing;
- `docs/facts/generated.yaml` differs from generator output;
- post-generation `fact_drift`, `doc_drift`, or `closure` finds an error.

## 7. AI Developer Governance

### 7.1 Why

AI developers are useful but constrained by context limits, instruction
loading behavior, tool availability, and a tendency to overfit local prompts.
Repository governance must therefore be enforced by deterministic tools, not
only by instruction text.

AI-specific policy must also avoid treating AI agents as magical: AI agents
follow the same repository quality checks as humans, plus additional harness
constraints where their failure modes are different.

### 7.2 AI Local Gate Workflow

AI work is not always a new code feature. The gate workflow therefore models a
reviewable task lifecycle, not a fixed "code implementation" pipeline.

Supported task kinds are:

| Kind | Examples | Notes |
|---|---|---|
| `hotfix` | Live-debugging and a batch of related bug fixes | May start from owner directive; formal issues must be linked or created before commit |
| `bugfix` | Fix one known issue | Prefer existing issue; create one only if none exists |
| `feature` | Implement new behavior | Usually requires spec and/or ADR links |
| `docs` | Write or update documentation | Still requires scope, checks, and issue linkage |
| `maintenance` | Cleanup, dependency/config hygiene, repository housekeeping | Must declare why it is safe and scoped |
| `manager` | Agent-manager coordination, checklist updates, merge preparation | May touch planning docs and reports without code changes |

Gate state is local temporary state stored under `.git/scieasy/gates/`. It is
not committed. The local gate exists to make AI behavior auditable before a
commit is created. Remote CI remains the final enforcement boundary.

The gate does not replace the repository's branch or PR rules. It protects the
local commit boundary for AI-authored work. The branch, PR, reviewer, and
remote CI requirements still apply after the commit exists.

The AI local gate has six stages:

| Stage | Required record | Blocking rule |
|---|---|---|
| 1. Scope | Task kind, what will be done, owner directive, in-scope paths/globs, out-of-scope paths/globs, governance-touch flag, expected docs/tests/checks | No scope means no commit |
| 2. Issue | Existing issue to close, or newly created issue if none exists; hotfix batches may link multiple issues | No issue linkage means no commit |
| 3. Implement | Work happens within the declared scope; scope changes require an explicit gate amendment | Staged files outside scope fail pre-commit |
| 4. Test And Checks | Required tests plus ADR-042 normative checks: code score, formatting/lint/type/test checks, docs checks, consistency checks where wired | Failing checks fail pre-commit |
| 5. Documentation Landing | Required docs/spec/ADR/changelog/checklist updates are completed or explicitly marked not applicable with rationale | Missing required docs fail pre-commit |
| 6. Commit | Commit records gate session, issue, task kind, and AI assistance metadata | Missing trailers fail commit-msg hook |

The gate session schema is:

| Field | Meaning |
|---|---|
| `session_id` | Local unique identifier, stored under `.git/scieasy/gates/` |
| `task_kind` | One of `hotfix`, `bugfix`, `feature`, `docs`, `maintenance`, `manager` |
| `branch` | Branch the session applies to |
| `owner_directive` | Human instruction or issue that authorized the task |
| `scope.include` | Allowed path globs for staged files |
| `scope.exclude` | Denied path globs for staged files |
| `governance_touch` | Whether governance files may be touched |
| `issues` | Existing or newly created GitHub issues to close or reference |
| `required_checks` | Tests and normative checks required before commit |
| `check_results` | Recorded command, exit code, timestamp, and output path |
| `docs_landing` | Docs/spec/ADR/changelog/checklist updates or N/A rationale |
| `persona` | One of the four allowed AI personas in Section 7.3 |
| `runtime` | Runtime identifier, for example Codex or Claude Code |
| `admin_labels` | Required administrator-applied labels for protected operations |
| `amendments` | Explicit scope/check/doc changes made after session start |

Hotfix work may begin from a human owner directive before every bug is fully
triaged. Before commit, however, the issue stage must be satisfied: each fixed
bug is linked to an existing issue or a newly created one, and a batch hotfix
records the list of issues in the same gate session. This permits live
debugging without allowing untracked bug fixes to enter history.

Manager tasks also require issue linkage. A coordination-only manager task may
use a lightweight tracking issue, but it must still be visible in the issue
tracker before commit.

The local pre-commit hook is an error gate, not a warning. It MUST fail when:

- no active local gate session exists for an AI-authored commit;
- the current branch does not match the gate session;
- staged files are outside the declared scope;
- staged files match denied paths;
- governance surfaces are touched without explicit owner-approved scope;
- required tests or normative checks have not been recorded;
- `code_score --changed --fast` returns F;
- a consistency checker wired in Section 5 reports an error;
- required documentation landing is missing;
- untracked deferrals or TODOs lack a tracking issue.

The hook must run from `.pre-commit-config.yaml`, not only from an optional
manual `.git/hooks` copy. It checks only staged content and local session
state, so it remains fast enough for ordinary commits. Slower checks run in CI
or in the recorded test stage.

The commit-msg hook MUST require machine-readable trailers for AI-authored
commits:

```text
Gate-Session: <local-session-id>
Task-Kind: hotfix|bugfix|feature|docs|maintenance|manager
Issue: #<number>
Assisted-by: <runtime>:<model-or-agent-id>
```

These trailers are the required AI-authorship marker. ADR-042 does not require
source-code comments or inline markers for AI-authored code.

Remote CI cannot read `.git/scieasy/gates/`, so it validates the durable
evidence that reaches the PR: issue links, trailers, changed files,
governance-file review requirements, docs/tests presence, and all required
quality checks.

AI agents are prohibited from using `--no-verify`. Git cannot fully prevent an
agent from attempting that bypass locally, so the durable enforcement is the
combination of missing-trailer CI failure, issue-link CI failure, and reviewer
rejection. Human exemptions are governed by Section 9.

### 7.3 Persona Rules

Personas are routing labels. They help decide which kind of agent should do a
task, but they do not grant permissions and they do not change quality gates.
There are exactly four repository personas:

| Persona | Use | Required skill | Root policy |
|---|---|---|---|
| `manager` | Coordinate agents, maintain checklists, prepare merges, summarize status | `agent-manager` | Must obey root `AGENTS.md`, gate scope, branch isolation, and owner scheduling decisions |
| `implementer` | Implement code, tests, tool wiring, docs generated from implementation | `implementation-worker` | Must obey root `AGENTS.md`, local gate, scope allowlist, tests, documentation landing, and code quality checks |
| `adr_author` | Draft or revise ADR/spec governance text and schemas | `adr-author` | Must obey root `AGENTS.md`, ADR schema, document structure rules, and governance-file review requirements |
| `audit_reviewer` | Inspect diffs, audit findings, CI failures, and conformance gaps | `audit-reviewer` | Must obey root `AGENTS.md`, review-only scope unless explicitly assigned a fix, and finding-first reporting |

Every AI task must declare one persona in the local gate session. A task may
change persona only by amending the gate session with a new scope rationale.
Persona declarations are AI-agnostic: Codex, Claude Code, Gemini, local CLI
agents, and future agent runtimes use the same persona names and the same
policy checks.

### 7.4 Runtime Configuration, Constitution, Skills, And Memory

AI runtimes commonly load project-local configuration from runtime-specific
directories, such as `.claude/`. ADR-042 adopts that pattern, but keeps it
AI-agnostic: no repository rule may exist only for one AI runtime when the same
concept applies to others.

Root policy, constitution, skills, memory, and workflow docs have distinct
roles:

| Layer | Location | Role |
|---|---|---|
| Root policy | `AGENTS.md` | Hard repository rules every agent must obey |
| AI runtime config roots | `.agents/**`, `.claude/**`, `.codex/**`, `.gemini/**`, or successor runtime roots | Project-specific AI assets loaded by each runtime |
| AI constitution | Runtime config root | Stable project principles and non-negotiable engineering values for AI runtimes |
| Persona skills | Runtime config root skill directories for the four personas | Short operational entry points that route work and link to canonical procedures |
| Runtime memory | Runtime config root memory files | Project-specific recall and reusable context; not a hard policy layer |
| Workflow docs | `docs/contributing/workflows/**` | Human-readable procedures used by skills and contributors |

Skills are pointers and workflow helpers. They must not duplicate or override
root policy. Runtime memory may summarize project facts, but it must not create
hidden requirements or bypass gates. The canonical rule is:

- root policy lives in `AGENTS.md`;
- AI constitution, skills, and memory live in project-local AI runtime config
  directories, with equivalent assets for every supported runtime;
- the AI constitution records stable project principles for repository AI work;
- SpecKit's `.specify/memory/constitution.md` is a separate feature-planning
  artifact and is not the main repository AI constitution;
- all supported AI runtimes must be able to read the same root policy,
  constitution, persona skills, and project memory at equivalent fidelity;
- workflow procedures live under `docs/contributing/workflows/`;
- skills link to those workflows and provide only runtime-specific command
  details;
- `persona_policy` verifies that the declared persona has a matching skill,
  AI constitution, and root-policy reference in that runtime's config root;
- `skill_pointer_sync` verifies that pointers remain valid across runtime
  config roots.

No tool, workflow, skill, or documentation may privilege one AI runtime over
another. A governance rule is valid only if it can be implemented for all
supported runtimes or cleanly degraded into the same report-only diagnostic for
all of them.

### 7.5 AI Restriction Tool Set

| Risk | Tool |
|---|---|
| Agent commits without a gate session | `local_gate.check_pre_commit` |
| Agent stages files outside declared scope | `local_gate.staged_scope_check` |
| Agent declares an unsupported persona or runtime-specific policy | `persona_policy.check` |

| Agent omits required tests or normative checks | `local_gate.check_evidence` |
| Agent omits required documentation landing | `docs_landing.check` |
| Agent modifies protected core components without administrator authorization | `core_change_guard` |
| Agent modifies governance rules without review | `governance_mod_guard` |
| Agent weakens CI, lint, tests, or thresholds | `weakened_ci_check` |
| Agent or PR claims human bypass without maintainer label provenance | `human_bypass_guard` |
| Agent claims rules were loaded when they were not | `instructions_loaded_audit` |
| Agent edits disallowed paths | path allowlist checks in hooks/CI |

These tools enforce AI-specific process. They do not decide whether code is

### 7.6 Repository Rule Protection

Repository governance files include:

- `AGENTS.md`;
- `.agents/**`;
- `.claude/**`;
- `.codex/**`;
- `.gemini/**`;
- `.github/workflows/**`;
- `.pre-commit-config.yaml`;
- `pyproject.toml` tool sections;
- `pyrightconfig.json`;
- `.codespellrc`;
- `.yamllint`;
- `.markdownlint.yaml`;
- `MAINTAINERS`;
- docs that define workflow or governance rules;
- scripts and modules under `scripts/audit/**` and `src/scieasy/qa/**`.

Changes to these files require focused scope, owner review, and hard-fail
checks for weakening patterns. A PR may strengthen governance rules as part of
its scope. It may not weaken them incidentally.

Protected core component paths require administrator authorization before an AI
agent may stage or commit changes:

- `src/scieasy/core/**`;
- `src/scieasy/engine/**`;
- `src/scieasy/blocks/**`;
- `src/scieasy/workflow/**`;
- `src/scieasy/utils/**`.

The authorization is represented by GitHub-native labels or approving reviews,
not by a free-form local record. Without that authorization signal,
`core_change_guard` fails for AI-authored protected-path changes in pre-commit
where the signal is available and in remote CI.

AI agents may not merge PRs. `pr_merge_guard` blocks AI-initiated merge
commands and CI must reject missing administrator authorization for any
automation path that attempts to merge. Repository administrators may merge
after review and CI according to normal branch protection.

Administrator authorization should use GitHub-native labels or reviews rather
than free-form records. Required labels include:

| Operation | Required label or review signal |
|---|---|
| Human AI-harness bypass | `human-authored` label |
| Protected core component change | `admin-approved:core-change` label or administrator approving review |
| AI merge automation | `admin-approved:merge` label |
| One-off AI harness override | `admin-approved:ai-override` label |

CI verifies the label or review actor has administrator or maintainer
permission. If label/review provenance cannot be verified, the authorization is
invalid. This avoids asking humans to fill a separate authorization record.

### 7.7 Test Quality Supervision

AI-generated tests are prone to verifying implementation shape rather than
behavior. ADR-042 therefore requires test-quality checks:

- AST checks for empty assertions, mocked-away behavior, snapshot-only tests,
  broad exception swallowing, and skipped tests without tracked rationale.
- Mutation testing for QA tools and selected critical logic.
- Test-first evidence for changes where owner policy requires it.
- Coverage ratchets with stronger thresholds for new QA tooling.

## 9. Human Developer Exemption Principle

### 9.1 Statement

Human developers may receive narrowly scoped exemptions from AI harness rules.
They do not receive exemptions from repository correctness, branch protection,
security rules, or CI hard-fail checks.

The purpose is practical: humans can make contextual judgment calls that AI
agents must not make silently. Human discretion is allowed only when it is
visible and reviewable.

### 9.2 Exemptible Rules

Humans may be exempted from AI harness checks. These checks exist to constrain
agent behavior, not to replace human judgment:

| Check | Human behavior |
|---|---|
| `local_gate` session requirement | Exempt. Human commits do not need `.git/scieasy/gates/` state |
| `persona_policy` | Exempt unless the human is editing project AI runtime config |
| AI-specific commit trailers | Exempt. `Assisted-by` is required only for AI-authored commits |
| `committer_enforce` for approved AI commit path | Exempt |
| `instructions_loaded_audit` | Exempt |
| `complete_artifacts` AI artifact checklist | Exempt when the PR has the `human-authored` label |
| `codemod_lint` AI codemod requirement | Exempt when the PR has the `human-authored` label |
| AI runtime skill parity for personal local tooling | Exempt unless committed as project runtime config |
| AI manager checklist discipline | Exempt unless the human is acting as an agent manager for delegated AI work |

### 9.3 Non-Exemptible Rules

Humans may not be exempted from repository correctness checks:

- protected branch rules;
- security checks;
- hard-fail quality CI checks;
- code owner review where configured;
- governance weakening checks;
- administrator authorization for protected core component changes;
- public API documentation requirements once hard-fail is enabled;
- tests required by the change scope;
- `code_score` deterministic F failure;
- Section 5 code/documentation consistency failures once wired.

Human developers should run the checks that match their change:

| Change type | Expected human checks |
|---|---|
| Python source | `ruff check`, `ruff format --check`, `mypy` or `pyright` where relevant, targeted `pytest`, `code_score --changed --full` before PR when available |
| Documentation | `frontmatter_lint`, `doc_length_lint` where applicable, `sphinx-build -b html -W --keep-going --nitpicky` for Sphinx docs |
| Governance files | `governance_mod_guard`, `weakened_ci_check`, `actionlint`/`zizmor` for workflow edits, owner review |
| Frontend | `eslint`, `prettier`, `tsc --noEmit`, `vitest`, `vite build` |
| Protected core paths | Targeted tests plus required administrator label or approving review |

### 9.4 Human-Authored Label

Human exemptions are manual and administrator-controlled. CI must not infer
that a PR is human-authored from PR text, branch name, commit author, or absence
of AI metadata.

The only normal bypass signal is a `human-authored` PR label applied by a
repository administrator or maintainer with the required permission. Without
that label, AI harness checks run and fail normally if their evidence is
missing.

The `human-authored` label means:

- the administrator has reviewed the PR source and treats it as human-authored
  for AI harness purposes;
- AI-only checks may report `skipped-human`;
- repository quality checks still run and fail normally.

If a PR has AI-authored commits, explicit `Assisted-by:` trailers, or an active
AI task session, the `human-authored` label is not sufficient for those commits.
The PR needs either normal AI harness evidence or a separate administrator
override recorded in PR review or an audit override log.

### 9.5 CI Bypass Semantics For Humans

CI must distinguish AI harness checks from repository quality checks. The
distinction is controlled by the `human-authored` label:

| CI item | Human-authored PR behavior |
|---|---|
| `local_gate` / gate-session existence | If `human-authored` label exists, mark `skipped-human`; otherwise run and fail normally |
| `persona_policy` | If `human-authored` label exists, mark `skipped-human` unless project AI runtime config changed; otherwise run and fail normally |
| `trailer_lint` for AI trailers | If `human-authored` label exists and no AI evidence is present, pass; otherwise require AI trailers |
| `committer_enforce` | If `human-authored` label exists, mark `skipped-human`; otherwise run and fail normally |
| `instructions_loaded_audit` | If `human-authored` label exists, mark `skipped-human`; otherwise run and fail normally |
| `complete_artifacts` AI artifact checklist | If `human-authored` label exists, mark AI-only artifact requirements `skipped-human`; otherwise run and fail normally |
| `codemod_lint` AI codemod requirement | If `human-authored` label exists, mark AI-only codemod requirements `skipped-human`; otherwise run and fail normally |
| `lint`, `typecheck`, `test`, `docs-build`, `audit`, `code-score`, `frontend`, `security`, `workflow-lint` | Run normally and fail normally |

CI must also verify that the actor applying or editing the `human-authored`
label has administrator or maintainer permission. AI agents may not apply this
label. If label provenance cannot be verified, CI must treat the PR as not
human-authored.

### 9.6 Local Bypass Semantics For Humans

Human developers may bypass AI-only local hooks, but should not bypass general
quality hooks.

Local hooks cannot be the authoritative human-bypass decision because they
cannot verify the PR's administrator-applied `human-authored` label before the
PR exists. Local bypass is therefore only a convenience path; CI remains the
source of truth.

The preferred local behavior is:

- general hooks such as `ruff`, `pytest` wrappers, docs checks, and
  `code_score` still run;
- if a hook framework requires manual skipping, humans may skip only AI-only
  hook ids such as `local_gate`, `persona_policy`, `committer_enforce`,
  `complete_artifacts`, `codemod_lint`, and `instructions_loaded_audit`;
- the PR must later receive the administrator-applied `human-authored` label
  for the corresponding CI bypass to be valid.

Humans should avoid `--no-verify` because it skips both AI-only checks and
quality checks. If `--no-verify` is unavoidable for local recovery, the PR must
receive the `human-authored` label and CI must still be green.

## 10. ADR Structure, Rollout, And Consequences

### 10.1 Future ADR Structure

Future ADRs MUST follow the Section 3 ADR schema. The required high-level
shape is:

1. Frontmatter.
2. H1 title.
3. `## 1. Decision Summary`.
4. `### 1.1 Problems Addressed`, with a table mapping every addressed
   problem to a later detailed section.
5. Decision summary subsections for the accepted decision, scope, and affected
   artifacts.
6. Later numbered sections that explain each problem, tradeoff, affected
   contract, implementation expectation, verification expectation, and
   consequence.
7. Alternatives considered.

ADR-042 is longer because it defines repository governance. Ordinary ADRs
should be much shorter.

### 10.2 Spec Structure

Future specs MUST follow the Section 3 spec schema and remain compatible with
the repository's SpecKit workflow. The required high-level shape is:

1. Frontmatter.
2. H1 title.
3. `## 1. Change Summary`.
4. User scenarios and acceptance tests.
5. Functional requirements and edge cases.
6. `## 4. Implementation Plan`, including technical approach, affected files,
   implementation sequence, verification plan, and risks/rollback.
7. Success criteria, assumptions, and links to governing ADRs.

### 10.3 Rollout

ADR-042 tools should roll out in four stages:

| Stage | Behavior |
|---|---|
| Scaffold | Tool exists, focused tests exist, no CI dependency |
| Report-only | Tool runs locally or in CI and uploads/report findings without blocking |
| Baseline | Existing findings are reviewed, classified, and either fixed or baselined |
| Hard-fail | New violations fail CI or pre-commit according to owner-approved policy |

This rollout is not self-iteration. It is normal staged adoption of stricter
repository rules.

Section 5 consistency checks are an explicit stricter case: once a consistency
checker is wired, it hard-fails immediately so existing technical debt is
cleaned rather than silently grandfathered.

### 10.4 CI Topology

The final target is a small set of clear CI jobs, optionally summarized by one
branch-protected aggregate check:

| CI job | Required tools/checks | Blocking rule |
|---|---|---|
| `lint` | `ruff check`, `ruff format --check`, `codespell`, `yamllint`, `pyproject-fmt` | Any configured violation fails |
| `typecheck` | `mypy`, `pyright` | Any configured error fails |
| `test` | `pytest`, `pytest-xdist`, `pytest-timeout`, `pytest-randomly`, `pytest-cov`, `coverage`, focused Pydantic schema tests | Test failure or configured coverage failure fails |
| `docs-build` | `sphinx-build -b html -W --keep-going --nitpicky`, `sphinx-lint`, Sphinx `linkcheck` where enabled | Warning, unresolved reference, stale generated doc, or non-baselined broken link fails |
| `audit` | Pydantic-backed schema validation, `frontmatter_lint`, `generate_facts`, `doc_drift`, `fact_drift`, `closure`, `signature_drift`, `full_audit`, `griffe`, `griffe-pydantic` | Any consistency or schema error fails once wired |
| `code-score` | `code_score --changed --full` | Deterministic F grade fails |
| `frontend` | `eslint`, `prettier`, `tsc --noEmit`, `vitest`, `vite build` | Any configured violation fails |
| `security` | `pip-audit`, `ruff` `S` rules, optional `bandit` | Configured critical/security violations fail |
| `workflow-lint` | `actionlint`, `zizmor`, governance weakening checks for workflow edits | Syntax, security, or weakening violation fails |

The `code-score` CI job runs `code_score --changed --full` on every PR. A
deterministic F grade fails the job and must be included in any branch-protected
aggregate check. Weekly module-health scoring may run separately on a scheduled
workflow, but PR change scoring is mandatory CI.

Pydantic is a schema engine rather than a standalone CI job. It enters CI
through schema tests and Pydantic-backed validators such as `frontmatter_lint`,
`ADRFrontmatter`, `SpecFrontmatter`, `AuditReport`, and `FactsRegistry`
validation.

The exact GitHub Actions layout may evolve as long as the required tools and
hard-fail semantics remain visible.

### 10.5 Explicitly Removed From Previous Drafts

The following concepts are explicitly not part of ADR-042:

- ADR-042 self-iteration.
- Meta-recursive proof requirements.
- Temporary release-readiness tooling.
- Temporary release milestone checks.
- Automatic governance-rule modification by agents.
- Requiring each new QA tool to validate ADR-042 itself before it can land.
- Treating ADR implementation status as an automated global completion gate.

### 10.6 Consequences

Positive consequences:

- Repository rules become deterministic and auditable.
- Documentation becomes shorter, categorized, and easier to maintain.
- Generated references reduce hand-maintained drift.
- AI agents are constrained by tools rather than only prompts.
- Human exemptions remain possible but visible.

Negative consequences:

- More tooling must be implemented and maintained.
- CI will become stricter over time.
- Some existing docs and code will need cleanup before hard-fail enforcement.
- Contributors must learn the document categories and tool expectations.

### 10.7 Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Use a self-iterating governance design | Too complex, too recursive, and not aligned with the owner's desired system |
| Only use common tools | Common tools cannot encode SciEasy-specific documentation, governance, and AI harness rules |
| Only write policy docs without tools | Prompt and prose rules are insufficient for AI-assisted development |
| Generate all docs | Conceptual and workflow docs require human judgment |
| Let human review catch drift manually | Review alone does not scale and misses repeated mechanical issues |
