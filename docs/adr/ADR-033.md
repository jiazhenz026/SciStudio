## ADR-033: Embedded Coding Agent via Claude Code / Codex Subprocess

**Status**: **superseded by ADR-034** (mostly). The stream-json IPC, custom agent UI, in-process agent runtime, and frontend renderer pieces (§4–§6) were rolled back in PR #808 and replaced by ADR-034's PTY + xterm.js terminal embed. The MCP server design from §3.D2 + the dev-CLI integration (`scieasy install`, `scieasy mcp-bridge`) survive and continue to ship.
**Date**: 2026-05-12 (superseded 2026-05-13)
**Related**: ADR-013 (AI as Layer 4), ADR-023 (frontend layout), ADR-025 (plugin entry-points), ADR-026 (Block SDK), **ADR-034** (the replacement)
**Supersedes**: parts of #452 (AIBlock MVP), parts of the AI service stack from Phase 9

---

### 1. Purpose

This ADR replaces SciEasy's existing single-call LLM architecture with an **embedded agent runtime** that wraps the user's locally installed Claude Code (or Codex) CLI as a subprocess. The decision covers:

1. **Agent runtime** — `AgentProvider` protocol, `ClaudeCodeProvider` first, `CodexProvider` as a parallel implementation; binary discovery; subprocess lifecycle; stream-json IPC; session management.
2. **SciEasy MCP server** — in-process tool exposure of `BlockRegistry`, `TypeRegistry`, workflow validation/execution, lineage, data inspection. ~22 tools spanning four user-need categories.
3. **System prompt** — three-tier composition (builtin / user-override / project-override), all stored under `{project}/.scieasy/`. Production-mode work discipline distinct from CLAUDE.md developer discipline.
4. **Permission model** — strict by default; Bypass mode opt-in; implemented via Claude Code's `PreToolUse` hook.
5. **Frontend rewrite** — stream-json renderer, permission UI, session sidebar, login-state banner.
6. **Project-local state** — sessions, transcripts, skills, memory all under `{project}/.scieasy/`.
7. **Deprecations** — delete `ai/generation/`, `ai/synthesis/`, `ai/optimization/`, single-call providers, `/api/ai/*` REST endpoints, and the keyword-routed `AIChat`. Keep `AIBlock` (workflow LLM node) unchanged for v1; revisit in a future ADR.
8. **Phased implementation** — four phases, each parallelizable across 2–4 agents.

The intent is to give SciEasy users a real, multi-turn, tool-using coding agent that understands SciEasy's domain (blocks, ports, types, workflow graphs, lineage), without SciEasy itself maintaining an agent loop, an OAuth flow, or a prompt-caching strategy. Those concerns are delegated to the upstream CLI (Claude Code / Codex).

### 2. Context

#### 2.1 Current state of AI in SciEasy

SciEasy ships three AI surfaces, all built on a single-call LLM pattern with no tool use and no agent loop:

| Surface | File(s) | Behavior |
|--------|---------|----------|
| `AIBlock` (workflow LLM node) | `src/scieasy/blocks/ai/{ai_block,providers}.py` | One-shot: serialize input → prompt template → call Anthropic or OpenAI Messages API → wrap response as `Text`. No tool use, no loop, no file access. |
| Generation / synthesis / optimization pipelines | `src/scieasy/ai/{generation,synthesis,optimization}/*.py` | Each pipeline = one LLM call (sometimes with a retry loop) and string parsing. `generate-block` validates output Python; `suggest-workflow` is a 501 stub; `optimize-params` produces parameter suggestions. |
| API routes | `src/scieasy/api/routes/ai.py` | Three REST endpoints (`/generate-block`, `/suggest-workflow`, `/optimize-params`). No streaming, no session, no conversation. |
| `AIChat` tab | `frontend/src/components/AIChat.tsx`, `App.tsx:580` | UI is a chat-style list, but `onSendChat` routes user input by keyword (`"generate block"` → `/generate-block`, `"workflow"` → `/suggest-workflow`, else canned help). One turn, one API call. |

Authentication is API-key-only:

- `AnthropicProvider` and `OpenAIProvider` read `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` from the environment (`src/scieasy/blocks/ai/providers.py`).
- `AIConfig.from_env()` (`src/scieasy/ai/config.py`) reads `SCIEASY_AI_PROVIDER`, `SCIEASY_AI_API_KEY`, `SCIEASY_AI_MODEL`, etc.
- The GUI has **no login interface** at all — a user must set environment variables before launching `scieasy gui` or AI features fail silently.

#### 2.2 What "agent" means and what we are missing

An **agent**, in the technical sense used here, is:

1. **Multi-turn** — keeps conversation state across user messages.
2. **Tool-using** — can invoke tools (read file, write file, run shell, query SciEasy registry, run workflow) between user turns. The tool results feed back into the model context.
3. **Self-directed** — the model decides which tools to call, in what order, and when to stop. The user does not script the steps.
4. **Streaming** — partial output streams to the UI as it is generated; tool calls and their results are visible in flight.
5. **Permissionable** — sensitive operations are gated by a policy and possibly a user prompt.

SciEasy's current AI surfaces have **none of these properties.** They are essentially "send a prompt, render the reply." That model is sufficient for one-shot template completion (e.g., "summarize this image"), but not for the four user-need categories driving this ADR (§2.4).

Building an agent from scratch on top of the Anthropic / OpenAI SDK is doable but substantial — at minimum: a tool-call loop, prompt caching, context-window management, retry on transient errors, file-edit tool implementations, shell sandboxing, MCP integration, and conversation persistence. This is several months of work for parity with existing tools.

#### 2.3 Reference precedent: ClaudePrism

[ClaudePrism](https://github.com/delibae/claude-prism) (Tauri + Rust desktop app for academic LaTeX writing) demonstrates a working pattern for **GUIs that wrap, rather than rebuild, an existing coding agent CLI**:

- **Discovers, not bundles, the `claude` binary.** Searches 8 fallback paths (`~/.local/bin/claude`, `NVM_BIN`, `PNPM_HOME`, `which`, login-shell `command -v claude`, Windows registry HKCU/HKLM Environment, NVM dirs, npm global). If absent, exposes a one-click `install_claude_cli` that runs Anthropic's official installer.
- **Spawns `claude` as a subprocess** with `--output-format stream-json --verbose --dangerously-skip-permissions --append-system-prompt "<prism prompt>"` (and `--model`, `-c` for continue, `--resume <session_id>` for resume). Prompts go in via stdin (avoiding argv-length issues); stdout is read line-by-line as NDJSON and re-emitted to the webview as Tauri events.
- **Delegates OAuth to the CLI.** `login_claude` simply spawns `claude auth login` and streams its output. Credentials live in `~/.claude/`; ClaudePrism never touches them. No third-party OAuth dance.
- **Sessions** are keyed by `window_label:tab_id`. The first `init` event in stream-json carries a `session_id`; ClaudePrism stores it and uses `--resume <id>` on subsequent turns. Child processes are tracked in a `HashMap` for cancellation (`child.kill()`).
- **Permissions in the GUI, not the CLI.** Because `--dangerously-skip-permissions` is passed, the CLI does not prompt. ClaudePrism implements a **diff-review panel** in the editor for any `Edit` tool call — users approve / reject / undo at the chunk level. The host owns the UX gate.
- **System prompt** is heavily customized via `--append-system-prompt`. ClaudePrism's prompt enforces LaTeX conventions, `TodoWrite`-first planning, `.venv/`-based Python, etc. The base CC prompt is preserved (append, not override).
- **No Node.js bundled.** Wheel stays small; first-run install is the only setup tax.

The same approach maps cleanly onto SciEasy's web architecture (FastAPI + React) with the subprocess management moving from Rust to Python asyncio. ClaudePrism validates the model end-to-end: it ships, it works, and the user OAuth experience is good.

#### 2.4 User requirements (from owner)

The agent must serve four user-need categories. MCP tool scope (§3 D2) maps tools to these:

(a) **Workflow design and execution** — "Build me a workflow that loads my mzML files, runs ElMAVEN, then a Raman pipeline, then exports CSV. And run it."

(b) **Custom block authoring** — "Write me a block that does Savitzky-Golay smoothing on Raman spectra, with configurable window size."

(c) **Result inspection and parameter tuning** — "The Cellpose step output looks bad. Show me a preview. What parameters should I adjust?"

(d) **Q&A about the project** — "What blocks are installed? What did this run produce? Where is the data stored?"

#### 2.5 Owner decisions captured in this ADR

The owner reviewed an earlier draft of this ADR and provided the following direction (paraphrased and recorded for traceability):

1. Replace the current single-call LLM API model entirely; do not maintain it in parallel.
2. Permission policy: **every** write operation needs approval, even for files inside the current project, to prevent the agent from silently corrupting workflows; provide a Bypass option for users who want it.
3. No in-GUI installer flow — show the install command and let the user run it; don't block other SciEasy features when the agent is unavailable.
4. MCP server is required from v1, not v2.
5. System prompt must (a) include SciEasy's basic concepts, (b) **not** include CLAUDE.md's developer-facing discipline (gates, branches, commit style — those are for project maintainers, not end users), (c) explicitly enumerate the available tools.
6. Session metadata, transcripts, skills, and the agent's memory all live under `{project}/.scieasy/`. The user-level system prompt was also relocated under the project to keep everything project-local.
7. AIBlock is retained as-is for v1; a future ADR will revisit.
8. Implementation is condensed into four phases, each parallelizable by 2–4 agents.
9. Permission enforcement uses Claude Code's hook protocol.
10. MCP server runs in the same process as the FastAPI engine.
11. Concurrent chat cap defaults to 5 (configurable).

These decisions are reflected in §3.

### 3. Decisions

#### D1. Agent runtime: local CLI subprocess via `AgentProvider`

##### D1.1 Provider protocol

A new module `src/scieasy/ai/agent/` defines:

```python
class AgentProvider(Protocol):
    """Embedded coding-agent runtime."""

    name: ClassVar[str]                 # "claude-code" | "codex"
    binary_name: ClassVar[str]          # "claude" | "codex"

    @classmethod
    def discover(cls) -> ProviderStatus:
        """Locate the binary; report availability, version, login state."""

    def start_session(
        self,
        *,
        project_dir: Path,
        system_prompt: str,
        mcp_config: dict,
        resume_session_id: str | None = None,
        permission_mode: PermissionMode,
    ) -> AgentSession: ...

class AgentSession(Protocol):
    """One active subprocess + its IPC channels."""

    session_id: str | None              # populated on first init event
    pid: int

    async def send_user_message(self, content: str) -> None: ...
    async def cancel(self) -> None: ...
    async def stream_events(self) -> AsyncIterator[AgentEvent]: ...
    async def close(self) -> None: ...
```

Two implementations in v1 are required by §2.5 #1; one ships in Phase 1, the other in Phase 4:

- `ClaudeCodeProvider` — wraps `@anthropic-ai/claude-code`.
- `CodexProvider` — wraps OpenAI's Codex CLI. Same protocol, same subprocess pattern, different `--output-format` quirks.

##### D1.2 Binary discovery

Ported from ClaudePrism's `find_claude_binary()` (`apps/desktop/src-tauri/src/claude.rs`). Order:

1. `$HOME/.local/bin/<binary>` (Anthropic's installer default)
2. `$NVM_BIN/<binary>`
3. `$PNPM_HOME/<binary>`
4. `shutil.which("<binary>")`
5. Login-shell resolution: `bash -lc "command -v <binary>"` (Unix). This catches shell-config-only PATH entries that GUIs miss.
6. Windows: `HKCU\Environment\Path` and `HKLM\System\CurrentControlSet\Control\Session Manager\Environment\Path` via `winreg`. (PATH discovery on Windows after a fresh install is the main reliability problem ClaudePrism solves; we copy the solution.)
7. NVM directories: `$HOME/.nvm/versions/node/*/bin/<binary>` (globbed; pick latest).
8. npm global: `$(npm root -g)/../bin/<binary>` (only if `npm` itself is on PATH).
9. Standard fallbacks: `/usr/local/bin`, `/usr/bin`.

Result: a `ProviderStatus`:

```python
@dataclass
class ProviderStatus:
    name: str                          # "claude-code"
    available: bool
    binary_path: Path | None
    version: str | None                # "claude --version" output
    logged_in: bool                    # heuristic; see §3 D1.4
    install_hint: str | None           # platform-specific install command if not available
```

##### D1.3 Subprocess invocation

```bash
<binary> \
  --output-format stream-json \
  --verbose \
  --append-system-prompt @<tmp_prompt_file> \
  --mcp-config @<tmp_mcp_config> \
  [--resume <session_id>] \
  [--model <model>]
```

Implementation notes:

- `asyncio.create_subprocess_exec` with `stdin=PIPE`, `stdout=PIPE`, `stderr=PIPE`.
- `cwd = project_dir` so the CLI's relative-path tooling resolves correctly.
- User prompt goes to stdin, **not argv**, to avoid argv length limits and shell-escaping bugs. Close stdin after writing.
- Stdout is read line-by-line; each line is a JSON event (NDJSON). A bounded parser tolerates partial / split / oversized lines.
- Stderr is captured and exposed for diagnostics; it should never reach the user-visible chat.
- On Windows, use `CREATE_NEW_PROCESS_GROUP` so we can group-kill child processes Claude Code itself spawns (e.g., for bash tools).
- Cross-platform process-tree termination uses the existing `ProcessHandle` abstraction from ADR-019, extended with an `AgentProcess` flavor if needed.

##### D1.4 Session management

- The first stream-json event after spawn is an `init` event containing `session_id`. The provider stores it on `AgentSession.session_id`.
- Subsequent turns within the same chat use `--resume <session_id>`.
- Login state is **probed**, not authoritative — there is no clean public API. Heuristic: spawn `claude config get -g installMethod` (or equivalent) with a short timeout; if it exits 0, treat as logged in. If the user actually isn't, the first user message will fail and the error surfaces through the stream-json `error` event.

##### D1.5 Concurrency cap

A `SessionManager` keeps a `dict[chat_id, AgentSession]` keyed per `(project_dir, chat_id)`. Cap is `5` concurrent sessions per project; configurable via `{project}/.scieasy/settings.json`. Excess spawn attempts return HTTP 429 from the chat endpoint with a hint.

#### D2. SciEasy MCP server (in-process, ~22 tools across 4 use-case categories)

##### D2.1 Process model

The MCP server runs **in the same process as FastAPI** (decision per §2.5 #10). Implementation:

- Mount a stdio MCP server inside the FastAPI app startup hook. When `start_session()` writes `mcp.json`, it points Claude Code at a Unix socket / named pipe (or stdio bridge subprocess) that proxies to the in-process server.
- Tools share the same dependency-injection container as REST routes: `get_block_registry()`, `get_type_registry()`, `get_engine()`, `get_lineage_store()` from `src/scieasy/api/deps.py`.
- A small adapter layer between MCP request format and the existing service functions, to avoid duplicating logic.

This keeps the database / registry single-truth and avoids cross-process state sync. Write-side tools that mutate the workflow YAML or run the scheduler use the same code paths as the REST API.

##### D2.2 Tool set

Each tool has `name`, `description`, `input_schema` (JSON Schema), and `output_schema`. Schemas are derived from existing Pydantic models where possible.

**(a) Workflow design and execution** (9 tools)

| Tool | Mutation? | Description |
|------|-----------|-------------|
| `list_blocks` | read | Returns all registered blocks (name, category, subcategory, port summary, package source). Mirrors `GET /api/blocks/`. |
| `get_block_schema(type_name)` | read | Returns the block's `input_ports`, `output_ports`, `config_schema` (with `ui_priority`), and port type hierarchy. |
| `list_types` | read | Returns the `TypeRegistry` hierarchy with base / domain types and accepted-type compatibility info. |
| `get_workflow(path)` | read | Loads `workflow.yaml` from `path` and returns the deserialized `WorkflowDefinition`. |
| `validate_workflow(yaml_or_path)` | read | Runs `workflow.validator.validate_workflow()`; returns the structured error list (port-type mismatches, cycles, missing connections, cardinality violations, duplicate-extension AppBlock outputs, etc.). |
| `write_workflow(path, yaml)` | **write** | Writes / overwrites `workflow.yaml`. Subject to permission policy (D4). |
| `run_workflow(path)` | **write** | Submits the workflow to the DAG scheduler; returns `run_id`. Subject to permission policy. |
| `cancel_run(run_id)` | **write** | Cancels via the existing CANCEL_WORKFLOW_REQUEST event. Subject to permission policy. |
| `get_run_status(run_id)` | read | Returns per-block state, progress, recent events, output port summary. |

**(b) Custom block authoring** (5 tools)

| Tool | Mutation? | Description |
|------|-----------|-------------|
| `read_block_source(type_name)` | read | Returns the `.py` source of a registered block (file path + content). For Tier-2 entry-point blocks, resolves via `importlib` to the installed package. |
| `list_block_examples(category)` | read | Returns 3–5 representative example block paths from `src/scieasy/blocks/{io,process,...}/` for the agent to pattern-match. |
| `scaffold_block(name, category)` | **write** | Generates a starter block file at `{project}/blocks/{name}.py` from `block-development/` templates. Subject to permission policy. |
| `reload_blocks` | **write** | Calls `BlockRegistry.hot_reload()` after the agent writes a new block. Subject to permission policy because it re-imports user code. |
| `run_block_tests(type_name)` | **write** | Runs `pytest -k <block_test_pattern>` for a single block; returns pass/fail summary and any failure traces. Subject to permission policy because tests run user code. |

Generic file IO (read/write/edit/glob/grep) is **not** exposed via MCP — Claude Code already has native `Read`/`Write`/`Edit`/`Glob`/`Grep` tools. MCP focuses on SciEasy-semantic operations.

**(c) Result inspection and parameter tuning** (7 tools)

| Tool | Mutation? | Description |
|------|-----------|-------------|
| `get_block_output(run_id, block_id, port_name)` | read | Returns the `StorageReference` and `TypeSignature` of an output. Does not load the payload. |
| `inspect_data(ref)` | read | Returns full metadata envelope (FrameworkMeta / typed meta / user) plus shape / axes / columns / row count / dtype. No payload. Backed by the unified `LineageStore` (ADR-038; supersedes ADR-032 MetadataStore — same on-disk DataObject identity catalog now lives in the `data_objects` table). |
| `preview_data(ref, fmt)` | read | Returns a small preview: DataFrame first-100 rows, Array thumbnail (PNG base64, capped at 256×256), Series first 200 points, Text first 4 KB, Artifact metadata + thumbnail. Bandwidth-bounded. |
| `get_lineage(ref)` | read | Recursive `derived_from` ancestors via the unified `LineageStore` (ADR-038) SQL query against `data_objects.derived_from`. |
| `get_block_config(workflow_path, block_id)` | read | Current parameter values from the workflow YAML. |
| `update_block_config(workflow_path, block_id, params)` | **write** | Patches a block's parameters in the YAML. Subject to permission policy. |
| `get_block_logs(run_id, block_id)` | read | Stdout/stderr captured by the block runner during the most recent execution. |

**(d) Q&A about the project** (4 tools)

| Tool | Mutation? | Description |
|------|-----------|-------------|
| `search_docs(query, scope)` | read | Full-text search over `docs/` filtered by `scope` ("architecture", "block-development", "specs", "all"). |
| `get_doc(path)` | read | Returns markdown text of a doc file. |
| `list_data(project_dir)` | read | Lists `data/zarr/`, `data/parquet/`, `data/artifacts/` contents with type / size / created-at. |
| `get_project_info` | read | Returns `project.yaml`, workflow file list, last-run summary. |

**Total: 25 tools, 16 read, 9 write.** All 9 write tools route through D4's permission policy.

##### D2.3 MCP config emission

When a session starts, the provider writes a temp `mcp.json` and passes its path with `--mcp-config`. Example contents:

```json
{
  "mcpServers": {
    "scieasy": {
      "command": "/abs/path/to/scieasy",
      "args": ["mcp-bridge", "--socket", "/tmp/scieasy-mcp.sock"],
      "env": {
        "SCIEASY_PROJECT_DIR": "/abs/path/to/project",
        "SCIEASY_CHAT_ID": "<chat_id>"
      }
    }
  }
}
```

The `scieasy mcp-bridge` subcommand opens a connection to the running FastAPI process's MCP socket and proxies frames in both directions. (If the user invokes `scieasy serve` and then opens a chat, the bridge subprocess connects to the existing server. If no server is running, the bridge spins up an in-process registry on demand for `mcp-bridge`'s lifetime.)

#### D3. Three-tier system prompt

##### D3.1 Composition order

The final `--append-system-prompt` is the concatenation, in order:

1. **Builtin** — from `src/scieasy/ai/agent/system_prompt.py` (string constant or assembled from sub-fragments).
2. **Project-level user prompt** — `{project}/.scieasy/system_prompt.md` if present.
3. **Per-project additional prompt** — `{project}/.scieasy/system_prompt.local.md` if present (gitignored, for per-machine tweaks).

§2.5 #6 relocates the previously-considered user-level prompt (`~/.scieasy/system_prompt.md`) into the project directory. The two slots above (`system_prompt.md` and `system_prompt.local.md`) replace the earlier `user` + `project` tiers; both now live inside the project.

##### D3.2 Builtin content

The builtin prompt is split into four sections, written for end-user-facing production behavior (NOT developer / maintainer behavior):

**Section A — Identity & scope**

> You are an AI assistant embedded inside SciEasy, an AI-native workflow runtime for multimodal scientific data. You help researchers design and run workflows, write custom blocks, inspect run results, tune parameters, and answer questions about their projects.

**Section B — SciEasy core concepts**

A 200–300 word digest of:

- The block model (six categories, typed input_ports / output_ports / config_schema).
- The six base data types (Array / Series / DataFrame / Text / Artifact / CompositeData) and domain subtypes.
- References are not payloads — use `inspect_data` / `preview_data`, never load full arrays.
- Workflows are YAML in `{project}/workflows/`; the runtime is the source of truth.
- Lineage chain via `derived_from`.

**Section C — Available tools**

An explicit enumeration of every MCP tool by category, with one-sentence descriptions. The model uses this list to discover capabilities; it should not have to read source code to know what tools exist. The Claude Code native tool set (Read, Write, Edit, Glob, Grep, Bash, WebFetch, WebSearch, Task) is also enumerated with a note that MCP tools are preferred when applicable.

**Section D — Working principles** (the production-mode discipline, see §4 for full text)

A condensed version of §4 inlined into the prompt: plan-before-act, verify-before-claim, cite-real-data, prefer-minimal-change, use-SciEasy-semantics, be-honest, respect-data-scale, never-silently-overwrite.

##### D3.3 What the builtin prompt deliberately omits

Explicitly **not** in the builtin prompt:

- Anything from `CLAUDE.md` (gate workflow, 6 stages, conventional commit format, branch naming, ADR / Spec requirements, CI nuances). Those are project-maintainer rules. End users running an agent in their own scientific workflow should not be told to file GitHub issues against the upstream SciEasy repo every time they edit a block.
- Sentences about `git`, `gh`, `pytest`, `mypy`, `ruff`. The agent can run these via `Bash` if needed, but it should not be coached to treat the user's project like a CI'd open-source repo.
- SpecKit / `/speckit.*` references.

Project-maintainer agent invocations (e.g., a developer using SciEasy itself in dev mode) can re-introduce these via the `{project}/.scieasy/system_prompt.md` override.

#### D4. Permission model

##### D4.1 Default policy: strict

| Operation | Default |
|-----------|---------|
| Claude Code native: `Read`, `Glob`, `Grep`, `WebSearch` | auto-approve |
| Claude Code native: `Edit`, `Write` (any path) | **approval required** |
| Claude Code native: `Bash`, `WebFetch` | **approval required** |
| MCP tool: any read tool (16 tools per D2.2) | auto-approve |
| MCP tool: any write tool (9 tools per D2.2) | **approval required** |

Per §2.5 #2, there is no project-internal carve-out. Every Edit, every Write, every Bash, every write-MCP-tool, gets a confirmation prompt. This is a deliberate choice favoring safety over flow — the agent is allowed to destroy a workflow file in principle, so we make it impossible to do so silently.

##### D4.2 Bypass mode

The user can opt into a Bypass mode that skips all approvals — equivalent to launching Claude Code with `--dangerously-skip-permissions`. Bypass:

- Is **per-session**, not per-project. Each new chat starts in strict mode unless the user explicitly toggles Bypass.
- Displays a persistent red banner at the top of the chat tab while active.
- Is logged to `{project}/.scieasy/sessions/<chat_id>.json` for audit.

##### D4.3 Enforcement: Claude Code `PreToolUse` hook

Claude Code's hooks system supports a `PreToolUse` hook: a user-defined command invoked before each tool call, with the tool name + arguments on stdin, returning approve/deny/ask via exit code or JSON output.

SciEasy registers a hook in `{project}/.scieasy/claude-hooks.json` that points at the local `scieasy hook-bridge` subcommand. Claude Code's actual hook config schema is nested under a top-level `hooks` key with per-matcher arrays:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "scieasy hook-bridge" }
        ]
      }
    ]
  }
}
```

(The `matcher` field empty-string matches all tools. `scieasy hook-bridge` reads the tool name + arguments from stdin and POSTs them to `http://127.0.0.1:8000/api/ai/permission-check`.)

The endpoint receives:

```json
{
  "chat_id": "...",
  "tool_name": "Edit",
  "tool_input": { "file_path": "/abs/path/foo.yaml", "old_string": "...", "new_string": "..." }
}
```

And returns:

- `{"action": "approve"}` — in Bypass mode, or for auto-approved tool classes.
- `{"action": "deny", "reason": "..."}` — if the user rejected.
- `{"action": "ask"}` — to signal "show the user a prompt and block until they decide." The frontend's permission UI handles the prompt; the endpoint returns approve/deny once the user clicks.

The blocking-ask path uses a server-side `asyncio.Event` per pending tool call, signaled by the frontend's `POST /api/ai/permission-decision`. WebSocket pushes the pending-tool-call event to the chat UI when the hook fires.

##### D4.4 Spec-phase technical spike

Before committing to the hook mechanism, Phase-1 includes a small spike (1–2 days) to verify:

- Claude Code's hook hooks fire for all MCP tool calls (not just native tools).
- The hook can block synchronously while waiting for a user decision without timing out the agent.
- The hook's exit-code / JSON output protocol works on Windows.

If any of these fail, the fallback is `--allowed-tools` + `--disallowed-tools` static lists plus front-end-side post-hoc diff review (ClaudePrism's pattern). The fallback is less precise but is known-good.

#### D5. Frontend chat rewrite

Existing `frontend/src/components/AIChat.tsx` and the `onSendChat` callback in `App.tsx` are replaced. New surface:

##### D5.1 Components

- `AIChat.tsx` — top-level chat container; manages session selection, message list, composer.
- `ChatMessageList.tsx` — virtualized list rendering stream-json event-derived messages.
- `EventRenderer.tsx` — dispatch over event kinds:
  - `assistant_text` — token-by-token streaming.
  - `tool_use` — collapsed card showing tool name + args summary; click to expand full args.
  - `tool_result` — nested under the corresponding `tool_use` card; success / error coloring.
  - `permission_request` — modal-style card with the tool call + Approve / Deny / Approve-and-don't-ask-again buttons.
  - `thinking` — italicized faded text.
  - `error` — red banner with full message + dismiss.
- `SessionSidebar.tsx` — list of chats in the current project, with create / rename / fork / delete; persisted to `{project}/.scieasy/sessions/`.
- `AgentStatusBanner.tsx` — shows one of: `not_installed` (with copy-able install command), `installed_not_logged_in` (with the `claude /login` reminder + `Open Terminal` button on macOS/Linux), `bypass_mode_active` (red), `ready` (hidden).
- `PermissionPrompt.tsx` — modal with tool name, args, diff preview (for Edit / Write / update_block_config / write_workflow), Approve / Deny / Always-allow-this-tool toggles.
- `SettingsPanel.tsx` (under bottom panel) — Provider selector (Claude Code / Codex once §3 D1 Phase 4 ships), permission mode (Strict / Bypass), concurrent-chat limit.

##### D5.2 Wire protocol (WebSocket)

The chat tab opens a WebSocket per session:

```
WS /api/ai/chat/{chat_id}
```

Client → server messages:

```json
{ "type": "user_message", "content": "..." }
{ "type": "cancel" }
{ "type": "permission_decision", "request_id": "...", "decision": "approve|deny" }
```

Server → client messages: pass-through of stream-json events, plus framing:

```json
{ "type": "agent_event", "event": { ... }}            // raw stream-json
{ "type": "permission_request", "request_id": "...", "tool": {...}}
{ "type": "session_ended", "reason": "..."}
{ "type": "error", "message": "..."}
```

##### D5.3 Removed UI

- The keyword-routed `onSendChat` is deleted.
- The "Apply to Canvas" button for JSON-detected workflows is replaced by the agent calling `write_workflow` directly (with approval) — the canvas refreshes via the workflow store's file watcher.

#### D6. No installer flow; non-blocking unavailability

Per §2.5 #3:

- On every chat-tab mount, the frontend calls `GET /api/ai/status` to retrieve `ProviderStatus`.
- If `available=false`, the chat tab body shows `AgentStatusBanner` in `not_installed` mode: one paragraph explaining what Claude Code is, the install command for the user's OS (`curl ... | bash` / `irm ... | iex`), and a "Refresh" button that re-runs the status check.
- All other SciEasy features (canvas, palette, blocks, run, preview, lineage) work unchanged.
- The bottom panel's AI Chat tab itself remains visible — it just renders the install-prompt instead of a chat composer.
- No modal interruption. No "Setup wizard." No telemetry / analytics check-ins.

If `available=true` but `logged_in=false`, the banner shows `installed_not_logged_in` mode: the `claude /login` reminder and a "I've logged in — refresh" button. We do **not** programmatically spawn `claude auth login` ourselves; the user runs it in their own terminal. (Future enhancement: bundle a "spawn login subprocess and stream OAuth URL" path, but Phase 1 keeps it manual.)

#### D7. Project-local state directory

All agent-related persistent state lives under the project workspace:

```
{project}/
└── .scieasy/
    ├── settings.json                # permission mode, concurrent-chat cap, model preference
    ├── system_prompt.md             # project-level system prompt overlay (gitignored opt-in)
    ├── system_prompt.local.md       # per-machine prompt overlay (gitignored by default)
    ├── mcp.json                     # generated; regenerated each session start
    ├── claude-hooks.json            # generated; PreToolUse hook config
    ├── sessions/
    │   ├── <chat_id>.json           # SciEasy chat metadata: title, created, last_active,
    │   │                            # provider, model, system_prompt_hash, session_id,
    │   │                            # bypass_mode, total_turns
    │   └── <chat_id>/
    │       └── transcript.jsonl     # write-through snapshot of stream-json events
    ├── memory/
    │   └── MEMORY.md                # project-local agent memory (cross-session notes)
    └── skills/
        └── *.md                     # project-local Claude Code skills
```

##### D7.1 Write-through transcript snapshot

Claude Code stores its own transcript under `~/.claude/projects/<sha>/...`. Per §2.5 #6 (everything project-local), SciEasy writes a **mirror** of each stream-json event into `{project}/.scieasy/sessions/<chat_id>/transcript.jsonl` as it streams to the frontend. This makes the project directory portable (zip it, share it, git-track it) and self-contained for reproducibility.

The mirror is best-effort: a write failure logs a warning but does not fail the chat. The CC home-directory copy remains canonical for `--resume` purposes.

##### D7.2 Memory and skills

Claude Code natively loads:

- Global memory from `~/.claude/projects/<sha>/memory/MEMORY.md`.
- Global skills from `~/.claude/skills/*.md`.
- Project skills from `<cwd>/.claude/skills/*.md` (relative to `cwd`).

SciEasy supplements with `{project}/.scieasy/memory/MEMORY.md` and `{project}/.scieasy/skills/*.md`. Two paths:

- For memory: SciEasy reads `{project}/.scieasy/memory/MEMORY.md` and appends it to the system prompt at session start (since CC's memory system is per-CC-project-dir, not configurable per-invocation).
- For skills: copy or symlink `{project}/.scieasy/skills/` into `{project}/.claude/skills/` at session start (CC discovers it).

Implementation detail (Phase 2 Spec): keep `.claude/skills/` populated automatically from `.scieasy/skills/`. Users edit only the `.scieasy/skills/` versions; the `.claude/skills/` copy is regenerated.

### 4. Working principles (production-mode agent discipline)

These principles go into the builtin system prompt (D3.2 Section D). They are deliberately **distinct from** the developer-facing `CLAUDE.md` discipline. The audience is an end user — a scientist — running an agent against their own project. They are not maintaining the SciEasy repo.

1. **Plan before acting.** For any non-trivial change (new workflow, new block, parameter sweep), describe the plan in plain language and wait for user confirmation before invoking write tools.
2. **Verify before claiming success.** After running a workflow, call `get_run_status` and check each block's final state. Don't say "done" on a workflow you haven't confirmed completed.
3. **Cite real data.** When discussing results, fetch them via `inspect_data` or `preview_data`. Never fabricate numbers, shapes, or column names. If a value is unknown, say "I don't know — let me check" and use a tool.
4. **Prefer minimal change.** Edit the specific block parameter; don't rewrite working blocks. Don't introduce abstractions the user didn't ask for.
5. **Use SciEasy semantics, not raw file ops.** `list_blocks` beats `grep -r 'class .*Block'`. `validate_workflow` beats reasoning about port types in your head. `inspect_data` beats reading a 50 GB Zarr.
6. **Be honest about limits.** If a tool call is denied, accept it and ask the user how to proceed. If you can't do something, say so — don't bluff. If a tool returned an error, report the error verbatim.
7. **Respect data scale.** Don't load large arrays into memory. `preview_data` returns a thumbnail or first-N rows; that's enough for most reasoning.
8. **Never silently overwrite.** Before `write_workflow` / `Write` / `update_block_config` on an existing artifact, briefly describe the diff or confirm it's the intended target.

Notably absent: anything about commit messages, branch names, PR descriptions, changelogs, gate workflows, issue numbers, ADRs. These are developer rules. If a developer happens to use the agent to maintain the SciEasy repo itself, they can re-introduce them via the project-level `system_prompt.md`.

### 5. Alternatives considered

##### Alt A. Build a native Python agent loop on Anthropic / OpenAI SDK

**Rejected.** Multiple-month effort to reach parity with what Claude Code already does (prompt caching, context-window management, tool dispatch, retry policy, sub-agent spawning, hook protocol, skills, slash commands, MCP). The supplier (Anthropic / OpenAI) iterates faster than we could keep up. We also can't get OAuth as a third party — no public partner-OAuth program for either supplier.

##### Alt B. API-key-only, no OAuth

**Rejected.** This is what we have today. Users with paid Claude / OpenAI subscriptions cannot reuse them; they must separately obtain an API key and pay metered. Friction is too high for non-developer scientists.

##### Alt C. Bundle Claude Code (and Node.js) into the SciEasy wheel

**Rejected.** Wheel size jumps from ~5 MB to ~200 MB. Anthropic's distribution terms for `@anthropic-ai/claude-code` are not friendly to repackaging. We would also lag the upstream CLI on bugfixes by however long our release cadence is.

##### Alt D. Use the Claude Agent SDK (Python library) instead of the CLI

**Rejected (for v1).** The Python SDK exposes agent primitives — tool calls, sub-agents — but we still need to write the OAuth dance, the tool-execution engine, and the streaming UI on our side. The SDK does NOT give us OAuth-for-free; it expects an API key. We lose the value proposition (free OAuth via the CLI). The SDK may be the right answer later for a server-side scripted agent (e.g., AIBlock's future replacement, D9), but not for the embedded chat.

##### Alt E. Use a third-party agent shell (Cursor, Cline, Continue)

**Rejected.** They are GUIs; we'd be embedding a GUI in a GUI. They don't expose programmatic interfaces. And they don't understand SciEasy's data model — we would still need to provide MCP tools, which we can do directly against any MCP-compatible client. The relevant clients for "embedded in another app" are Claude Code and Codex.

##### Alt F. Keep the existing single-call surfaces and incrementally add agentic features

**Rejected.** Per §2.5 #1, the owner has explicitly chosen a clean replacement over incremental migration. The current AI surfaces have low usage (keyword routing in the chat, three REST endpoints, one workflow-node block) and parallel maintenance is not justified.

##### Alt G. Permissions-by-policy (per-tool / per-path allowlists), no per-call prompts

**Rejected.** Per §2.5 #2, the owner explicitly chose a per-call prompt model with a Bypass escape hatch. The rationale: the agent should not be able to corrupt a workflow without the user being aware. Allowlists are too coarse and create false confidence.

### 6. Consequences

##### 6.1 New runtime requirements for users

- Local installation of `claude` (or `codex`) CLI. SciEasy detects but does not install.
- A one-time OAuth login (`claude /login`) per user.
- Node.js is an indirect prerequisite of `claude` (the npm package). SciEasy itself adds no Node.js dependency beyond the existing frontend dev requirement.

##### 6.2 New SciEasy dependencies

- Python `mcp` SDK (or equivalent JSON-RPC over stdio implementation) for the in-process MCP server.
- No new dependency on `anthropic` or `openai` SDKs — those are retired with D8. (AIBlock's local provider wrappers keep them, but as workflow-graph-local optional extras, not as a framework-wide AI layer.)

##### 6.3 Architectural impact

- `src/scieasy/ai/` is restructured around `ai/agent/` (provider, session manager, system prompt, MCP server, hook bridge) instead of `ai/generation`, `ai/synthesis`, `ai/optimization`.
- Layer rules unchanged: `ai/` (Layer 4) consumes `blocks/` (Layer 2), `engine/` (Layer 3), `core/` (Layer 1). The MCP server holds references to registries via dependency injection.
- `api/routes/ai.py` is rewritten: `WS /api/ai/chat/{chat_id}`, `GET /api/ai/status`, `POST /api/ai/permission-decision`, plus a `scieasy hook-bridge` and `scieasy mcp-bridge` CLI subcommand for CC integration. The three legacy REST endpoints are deleted.
- The frontend's chat surface is rewritten end-to-end. Routing in `App.tsx` is simplified (no more keyword dispatch).

##### 6.4 Documentation updates required

The following docs are updated as part of Phase 4:

- `docs/architecture/ARCHITECTURE.md` — Layer 4 description: agent runtime + MCP, not generation pipelines.
- `docs/roadmap/ROADMAP.md` — Phase 9 retitled to "Embedded Agent" with link to this ADR. Old `Phase 9: AI services` content marked superseded.
- `docs/guides/ai-chat.md` (new) — install, OAuth, basic usage, permission policy, customizing the system prompt, troubleshooting.
- `docs/adr/ADR.md` — append index entry on acceptance.
- `README.md` — "AI Integration" paragraph rewritten; new "Embedded Coding Agent" section.

##### 6.5 Breaking changes for users

- Env vars `SCIEASY_AI_*` are no longer read. Documented in CHANGELOG `[Unreleased] > Removed`.
- The keyword-routed `AIChat` is replaced; users typing "generate a block that..." get a real agent instead of a single-shot block generator. This is an upgrade, but the surface is different.
- The `/api/ai/generate-block` REST contract is removed. No known external consumers; the only call site was the in-app chat.

##### 6.6 What is preserved

- `AIBlock` (workflow LLM node) continues to function identically. Workflows containing AIBlocks run unchanged.
- All non-AI features (workflow editing, execution, palette, preview, lineage, project management) work normally even when the agent is unavailable.
- Existing workflow YAMLs are unaffected.
- The block / type / adapter registries are unchanged.

### 7. Open questions deferred to Spec

Items below are resolved at the decision level in this ADR but require detailed mechanism in the SpecKit-generated spec (`/speckit.specify`) before Phase 1 implementation:

1. **Hook protocol verification.** Confirm via spike that Claude Code `PreToolUse` hooks (a) fire for MCP tool calls, (b) support synchronous blocking on a user decision, (c) work on Windows. Fallback design if any fail (D4.4).
2. **MCP transport.** stdio bridge subprocess vs. Unix socket vs. named pipe (Windows). Performance and lifecycle implications.
3. **Concurrent-chat resource accounting.** How does the 5-chat cap interact with the existing `ResourceManager` GPU/CPU slot model? Are agent sessions counted as resource consumers? (Probably yes, but with a separate slot pool.)
4. **Login-state detection.** Heuristic in D1.4 may be brittle across CC versions; spec needs a recipe that survives upstream changes.
5. **Stream-json schema versioning.** CC's stream-json output is not strictly versioned; we need a defensive parser that tolerates new event kinds and discards unknown fields.
6. **Permission-decision UI latency.** What happens if the user does not respond to a permission prompt for a long time? Timeout? Default deny after N minutes?
7. **MCP write tool atomicity.** `write_workflow` and `update_block_config` should be atomic (read-modify-write with locking) to avoid mid-edit corruption when the user is also editing on the canvas.
8. **Codex provider parity.** Codex CLI's output format differs from `stream-json`; the `AgentProvider` abstraction must accommodate both without leaking provider specifics into the chat UI.
9. **Skill propagation.** Should `{project}/.scieasy/skills/` be symlinked into `{project}/.claude/skills/`, or rewritten on each session start? Symlinks have permissions issues on Windows.
10. **Project security boundary.** The agent has `cwd = project_dir` but `Bash` can `cd` elsewhere. Strict mode catches `Bash` calls, but should we additionally pass an env var to scope helpers? Probably out of scope — Strict mode covers it.

### 8. Phased implementation

Four phases, each parallelizable across 2–4 agents. Phase ordering is strict (Phase 2 needs Phase 1's provider abstraction, etc.) but tasks **within** a phase can fan out.

##### Phase 1 — Agent runtime backbone

Builds the subprocess infrastructure end-to-end with one tool (Claude Code's native `Read`) working through the chat WebSocket. No MCP yet, no permission UI yet, no frontend rewrite yet — but a CLI-level user can fire up a chat against a real CC instance from a Python REPL.

Tasks (parallelizable):

- T1.1 — `AgentProvider` protocol + `ProviderStatus` + binary discovery on all three OSes. Unit tests with mocked binaries.
- T1.2 — `ClaudeCodeProvider.start_session` + subprocess lifecycle + stream-json parser. Integration test against a stub CC executable.
- T1.3 — `SessionManager` (in-memory `dict[chat_id, AgentSession]` with cap=5) + session metadata write to `{project}/.scieasy/sessions/`.
- T1.4 — `WS /api/ai/chat/{chat_id}` route + `GET /api/ai/status` route. Tests via WebSocket test client.
- T1.5 — Permission spike per D4.4. Outcome: either confirm hook mechanism, or escalate to fallback design.

Exit criteria: a developer can `pytest tests/ai/test_agent_provider.py` and see a mocked CC session produce stream-json that flows out the chat WebSocket. Working tree, mypy, ruff, CI green.

##### Phase 2 — MCP server + tools

Builds the in-process MCP server and all 25 tools. With Phase 1 wiring in place, this gives the agent its SciEasy-specific capabilities.

Tasks (parallelizable):

- T2.1 — MCP server scaffold + stdio transport + `scieasy mcp-bridge` subcommand. Generator for `.scieasy/mcp.json`.
- T2.2 — Read-only tools for category (a) workflow: `list_blocks`, `get_block_schema`, `list_types`, `get_workflow`, `validate_workflow`, `get_run_status`. Tests against fixture workflows.
- T2.3 — Write tools for category (a): `write_workflow`, `run_workflow`, `cancel_run` (gated behind permission stub returning auto-approve for now).
- T2.4 — Category (b) authoring tools: `read_block_source`, `list_block_examples`, `scaffold_block`, `reload_blocks`, `run_block_tests`.
- T2.5 — Category (c) inspection tools: `get_block_output`, `inspect_data`, `preview_data` (with format-specific previewers), `get_lineage`, `get_block_config`, `update_block_config`, `get_block_logs`.
- T2.6 — Category (d) Q&A tools: `search_docs`, `get_doc`, `list_data`, `get_project_info`.
- T2.7 — `tests/ai/test_mcp_tools.py` covering each tool with at least one happy-path and one error-path case.

Exit criteria: a developer can spawn a CC session, type "list all blocks", and see Claude correctly invoke `list_blocks` and report results. mypy / ruff / 85% coverage gate green.

##### Phase 3 — Frontend rewrite + permission UI + system prompt

Rebuilds the chat UI on the Phase 1+2 backend; adds the permission prompts and the system-prompt composition.

Tasks (parallelizable):

- T3.1 — System prompt composition (`src/scieasy/ai/agent/system_prompt.py`): builtin content per D3.2, overlay loading per D3.1, hashing for cache-busting in session metadata.
- T3.2 — `PreToolUse` hook bridge + `scieasy hook-bridge` CLI + `POST /api/ai/permission-decision` route.
- T3.3 — Frontend `AIChat.tsx` rewrite per D5.1 components.
- T3.4 — Frontend `PermissionPrompt.tsx` modal with diff preview for Edit/Write/update_block_config/write_workflow.
- T3.5 — `SessionSidebar.tsx` + persistence to `.scieasy/sessions/`.
- T3.6 — `AgentStatusBanner.tsx` + status polling + manual refresh.
- T3.7 — `SettingsPanel.tsx` for permission mode + Bypass toggle + concurrent-chat cap.
- T3.8 — WebSocket event handling rewrite in `useWebSocket.ts`; new `aiChatSlice` in Zustand store replacing the existing chat slice.

Exit criteria: end-to-end user flow works in a browser: install CC → log in → open AIChat → ask "what blocks are installed?" → see streamed answer → ask "edit my workflow to add a Cellpose step" → see approval prompt → approve → workflow updated on canvas. Frontend tests via Vitest + RTL pass.

##### Phase 4 — Deprecations + Codex provider + docs

Cleanup phase; can be done in parallel sub-phases but conceptually one phase.

Tasks (parallelizable):

### Appendix A: Comparison with ClaudePrism

| Aspect | ClaudePrism | SciEasy (this ADR) |
|--------|-------------|--------------------|
| Frame | Tauri + Rust desktop app | FastAPI + React web app (existing) |
| Subprocess control | `tokio::process::Command` (Rust) | `asyncio.create_subprocess_exec` (Python) |
| Binary discovery | 8-fallback search incl. Windows registry | Same; ported to Python |
| Spawn flags | `--output-format stream-json --verbose --dangerously-skip-permissions --append-system-prompt ...` | Same minus `--dangerously-skip-permissions` (we use hooks); plus `--mcp-config` |
| OAuth | Delegated to `claude auth login` subprocess | Delegated; we do not even spawn `claude /login` automatically — user runs it themselves |
| System prompt | Single static `--append-system-prompt` per app | Three-tier: builtin + project + project-local |
| Permission UX | All-or-nothing `--dangerously-skip-permissions` + post-hoc diff review per Edit | Per-call approval via `PreToolUse` hook, Bypass mode opt-in |
| MCP | Not used | Required from v1, in-process FastAPI integration |
| Session persistence | Per-tab in `Mutex<Option<String>>` | Per-chat JSONL transcript + metadata JSON in `{project}/.scieasy/sessions/` |
| Multi-session | One per tab | 5-cap per project |
| Codex parity | Not in scope | Required (Phase 4) |

The two projects share the **core architectural insight**: spawning the user's installed coding-agent CLI is faster, more reliable, and more sustainable than rebuilding the agent stack inside the host application. SciEasy adapts the same pattern to a different host architecture (Python web service) and to a different domain (multimodal scientific workflows rather than LaTeX authoring), with deeper MCP integration appropriate to the domain.

### Appendix B: Glossary

- **Agent**: A model-driven loop that decides which tools to call and when to stop, given a user goal.
- **Stream-json**: Claude Code's NDJSON output format. Each line is a JSON event (`init`, `assistant_text_delta`, `tool_use`, `tool_result`, `permission_request`, `error`, etc.).
- **MCP (Model Context Protocol)**: Open protocol for exposing tools to LLM clients. Used by Claude Code, Cursor, Cline, and others.
- **PreToolUse hook**: Claude Code's pre-tool-call extension point; a shell command invoked before each tool with the tool's parameters on stdin.
- **Bypass mode**: SciEasy's name for running Claude Code with `--dangerously-skip-permissions`.
- **Session**: One Claude Code subprocess instance with a unique `session_id`. Sessions can be resumed via `--resume <id>`.
- **Project workspace**: SciEasy's project directory layout (`workflows/`, `blocks/`, `data/`, `.scieasy/`, etc.) defined by `scieasy init` and used as `cwd` for the agent.

The real Claude Code hook config schema is nested under a top-level `hooks` key with per-matcher arrays. T-ECA-108 (PR #704) implemented the correct shape; this addendum updates §3 D4.3 in-place to match. No code change was required because the implementation already used the correct format. Reviewers should consult the updated §3 D4.3 above for the canonical example.

The `scieasy hook-bridge` command line is also simpler than the original prose suggested: the bridge reads its target endpoint from `127.0.0.1:8000` (the local FastAPI server) without needing an `--endpoint` flag. The endpoint URL becomes a deployment concern only if SciEasy is ever served on a non-default host/port; in that case `hook-bridge` reads it from an env var.
