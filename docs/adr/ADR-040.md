## ADR-040: Production-environment agent reliability ‚Äî FastMCP, project context injection, multi-skill split, prod-env CLAUDE/AGENTS provisioning, project hooks, Codex parity

**Status**: **proposed** ‚Äî drafted 2026-05-15 from a discussion session about observed behavior drift in the production-environment agent. Promotes to "accepted" after Phase 1 ships FastMCP migration with a green parity test against the existing 26 tools.
**Date**: 2026-05-15
**Related**: ADR-033 (mostly superseded), ADR-034 (PTY architecture ‚Äî **accepted**), ADR-035 (AI Block PTY-tab variant ‚Äî proposed), ADR-037 (desktop packaging), ADR-039 (git source control ‚Äî proposed)
**Parent tracking issue**: **#903** (umbrella ‚Äî "optimize embedded agent system prompt + SKILL.md content"). This ADR is a sub-deliverable; #903 stays open after this ADR ships because its dimensions D (overlays) and E (empirical evaluation) are not addressed here.
**Closes**: **#824**, **#825**, **#832**, **#875** (see ¬ß11 for the mapping).
**Supersedes**: none. Refines ADR-034 ¬ß3.5 (MCP catalog re-render) and ADR-033 ¬ß3 D2.2 (mutation flag taxonomy).

### 1. Purpose

**This ADR is exclusively about the production-environment agent ‚Äî the claude/codex CLI spawned by the SciEasy GUI inside a user's SciEasy project, working on the user's workflows and data. It does not govern the development-environment agent (claude/codex used by SciEasy contributors against the SciEasy source repo).** The dev/prod boundary is defined explicitly in ¬ß2.1 and must be respected throughout.

Observed drift in production:

1. The embedded agent reads `skills/scieasy/SKILL.md` (372 lines, attached at PTY spawn via `--append-system-prompt @file`) and is told "use MCP tools, never call `scieasy` CLI via Bash", yet routinely falls back to `Bash(scieasy run ‚Ä¶)`. The CLI path bypasses live GUI refresh, lineage capture, and the schema-validated MCP write paths (#875).
2. The agent skips listing existing blocks before authoring new ones, bloating `blocks/` with near-duplicates (#875).
3. `compose_system_prompt(project_dir)` accepts a `project_dir` argument but does not use it; every project gets the byte-identical prompt, leaving the agent context-blind across projects (#825).
4. **Codex tabs are entirely broken for any user who has not manually run `scieasy install --target codex`** ‚Äî `spawn_codex` passes no `--mcp-config` and no `--append-system-prompt`, and the project tree has no `.codex/`, no `.agents/skills/scieasy/`, and no `AGENTS.md`. The user opens a codex tab and sees no SciEasy MCP tools and no behavioral guidance.

These are structural problems, not wording problems ‚Äî a one-layer prompt-only steering surface cannot enforce constraints once the context window dilutes, and the asymmetric provisioning of Claude Code vs Codex is a configuration gap not a prompt gap.

Six concrete engineering changes, all in production environment:

1. **MCP layer**: migrate the hand-rolled 251-LOC JSON-RPC server to **FastMCP 3.x**; rewrite all 26 tool descriptions; add `next_step` fields to write-class tool result models so the agent gets just-in-time guidance on the next correct MCP call.
2. **Per-project dynamic context injection** (closes **#825**): extend `compose_system_prompt` to splice a `<!-- project_context:begin/end -->` block with per-project facts (project name, workflow count, installed plugins, git branch/SHA, recently-modified workflows). Makes the previously-unused `project_dir` argument do real work.
3. **Skill split** (closes **#832**, **#824**): break the monolithic `skills/scieasy/SKILL.md` into 5 task-scoped skills (`scieasy-build-workflow`, `scieasy-write-block`, `scieasy-debug-run`, `scieasy-inspect-data`, `scieasy-project-qa`) using Claude Code Skills / Codex Skills progressive-disclosure semantics. Base SKILL.md shrinks to a thin identity + skill-pointer document. Skills directory relocates from repo-root `skills/` to packaged `src/scieasy/_skills/` and loads via `importlib.resources` so wheel installs work.
4. **Prod-env CLAUDE.md + AGENTS.md provisioning**: write a single source-of-truth content fragment (~50 lines) at both `<project>/CLAUDE.md` and `<project>/AGENTS.md` so both providers' base-instruction discovery picks it up. Content is purpose-written for end-user agents, distinct from the dev-env `CLAUDE.md` at the SciEasy source repo root.
5. **Project-scoped hooks** (closes **#875** for the CLI-vs-MCP enforcement half): provision `<project>/.claude/settings.json` + `<project>/.claude/hooks/` at project creation time, enforcing deterministic `PreToolUse` matchers that block known-bad patterns (Bash `scieasy *`, direct Edit on `workflows/*.yaml`) before the call reaches the runtime. The "block reuse" half of #875 lands via ¬ß3.4 CLAUDE.md content + ¬ß3.3 `scieasy-write-block` skill.
6. **Codex MCP provisioning + cross-skill install**: write `<project>/.codex/config.toml` (newly supported in Codex 2026 per [docs](https://developers.openai.com/codex/config-reference)) at project creation so Codex tabs see SciEasy MCP tools with zero manual setup. Extend `scieasy install --skill` to cross-install all 5 task skills to both `~/.claude/skills/` and `~/.agents/skills/`; update `scieasy install --target codex --scope project` to take advantage of Codex's new project-scope config support.

The ADR is incremental: no new subsystem, no contract change. It moves prod-env agent enforcement from prompt-only (Layer 1) to a four-layer stack (prompt + tool description + tool-result hints + deterministic hooks).

### 2. Context

#### 2.1 Dev environment vs production environment ‚Äî the binding boundary

These two contexts have always existed in the project but were not explicitly documented; conflating them when designing agent reliability is the single largest hazard of this work.

| | **Development environment** | **Production environment** |
|---|---|---|
| What is it | The SciEasy **source repository** (this codebase) | A user's SciEasy **project workspace** (created by `scieasy init` or via the GUI's Create Project) |
| Who is the agent | claude/codex used by SciEasy contributors developing SciEasy itself | claude/codex spawned by the SciEasy GUI's PTY tabs, working in the user's project root |
| Whose code is the agent touching | `src/scieasy/**`, `tests/**`, `docs/**` | `workflows/*.yaml`, `blocks/*.py`, `data/`, etc. |
| Governing instructions | `CLAUDE.md` at SciEasy repo root (‚âà800 lines: gate workflow, ADR process, etc.) | `skills/scieasy/SKILL.md` (372 lines, injected via `--append-system-prompt`) |
| Hooks today | None | None |
| MCP tools today | None (contributors don't have SciEasy MCP installed against the source repo) | 26 tools for Claude Code via `--mcp-config`; **Codex: zero tools unless user manually ran `scieasy install`** |
| Lifecycle | Git clone + dev install | `scieasy init` / `ApiRuntime.create_project` |
| **ADR-040 scope** | **OUT OF SCOPE.** Only cross-referenced where the prod artifact has a dev analogue (¬ß3.8). | **The entire ADR.** |

**Every "agent reliability" change in this ADR lands inside the user's SciEasy project directory.** The SciEasy source repo's `.claude/`, the source repo's `CLAUDE.md`, and contributors' personal `~/.claude/` configurations are deliberately not touched.

(Making the SciEasy source repo's `AGENTS.md` mirror its `CLAUDE.md` so Codex-using contributors get the same dev-env guidance is a worthwhile separate task. It is called out as out-of-scope in ¬ß3.8 and tracked as a follow-up issue. It is fundamentally a docs/content task, not an architecture decision, and does not warrant its own ADR.)

#### 2.2 Prod-env steering today ‚Äî one layer, statically frozen

`scieasy.ai.agent.system_prompt.compose_system_prompt()` reads `skills/scieasy/SKILL.md`, splices a re-rendered tool catalog between `<!-- tool_catalog:begin -->`/`<!-- tool_catalog:end -->`, writes the result to `<project>/.scieasy/.tmp/scieasy-prompt-*.md`, and `spawn_claude` passes it via `--append-system-prompt @<path>`. The string is **frozen for the lifetime of the claude subprocess**. The user's SciEasy project has no `CLAUDE.md`, no `AGENTS.md`, no `.claude/settings.json`, and no `.claude/skills/` directory. Steering surface = the single injected SKILL.md plus the 26 MCP tools' minimalist descriptions.

This works as long as:
- the conversation stays short enough that SKILL.md retains attention weight at the system-message slot;
- the agent has no equally-valid second path to the same outcome.

Neither holds in practice. SKILL.md says "Never use `Bash` to call the SciEasy CLI" ‚Äî but as soon as the user asks the agent to do something easier via shell than via the multi-step MCP path, the agent falls back to its trained-distribution default. **#875** documents the two specific drift patterns observed in production.

#### 2.3 `project_dir` is accepted but unused (#825)

`compose_system_prompt(project_dir: Path) -> str` takes `project_dir` purely "for forward compatibility". Today the same byte-for-byte prompt ships to claude/codex regardless of which project the user opened. The agent has no signal for project name, project root path, workflow count, installed block plugins, git branch/SHA, or recently-modified workflows. **#825** filed the proposed mechanism (a `<!-- project_context:begin/end -->` marker mirroring the existing `<!-- tool_catalog -->` pattern) but was queued behind ADR-034 Phase work.

#### 2.4 Prod-env MCP server is hand-rolled

`src/scieasy/ai/agent/mcp/server.py` (251 LOC) implements JSON-RPC 2.0 line-delimited framing, `initialize` / `tools/list` / `tools/call` dispatch, and error envelopes by hand. `_registry.py` (279 LOC) declares 26 `ToolEntry` dataclasses. Every `tools/list` response wraps every tool with `inputSchema: {"type": "object", "additionalProperties": true}` ‚Äî the **schema accepts anything**; validation happens (if at all) inside each handler. [FastMCP 3.x](https://gofastmcp.com/servers/tools) inverts the model via decorator + type hints ‚Üí `inputSchema` generation.

#### 2.5 SKILL.md is monolithic, statically loaded, and not wheel-installable (#832, #824)

`skills/scieasy/SKILL.md` is a single 372-line file covering identity, environment, project layout, YAML schema teaching, tool catalog, custom block authoring, 9 working principles, worked example. The agent reads all 372 lines per session regardless of task (#832).

Worse: `_load_skill_md` locates the file by walking up from `__file__` looking for a sibling `skills/` directory ‚Äî works for source checkout + editable install, **fails for wheel install** because `skills/` is sibling to `src/` and does not land in `site-packages` (#824). For ADR-034 production deployment via wheel this is a release blocker: `compose_system_prompt` raises `FileNotFoundError`, PTY spawns claude without `--append-system-prompt`, claude has no idea it's inside SciEasy.

Both [Claude Code Skills](https://code.claude.com/docs/en/skills) and [Codex Skills](https://developers.openai.com/codex/skills) support progressive-disclosure semantics (frontmatter `description` triggers JIT body load) ‚Äî the two providers use different paths (`~/.claude/skills/<name>/` vs `~/.agents/skills/<name>/`) but identical SKILL.md structure.

#### 2.6 Prod-env hooks scaffolding does not exist

`grep -r PreToolUse src/` finds one comment in `_registry.py` referencing the obsolete ADR-033 STRICT-mode plan. No hook scripts exist. No `scieasy install --hooks` flag exists. [Claude Code's hooks system](https://code.claude.com/docs/en/hooks) is mature: project-scoped hooks at `<project>/.claude/settings.json` reference scripts at `<project>/.claude/hooks/*` via `$CLAUDE_PROJECT_DIR`, exit code 2 blocks with stderr returned to the agent. The `command` field accepts any executable ‚Äî Python, Bash, PowerShell, Node, binaries.

#### 2.7 Codex prod-env is broken today (newly discovered)

Audit of `terminal.py::spawn_codex` (verified 2026-05-15):

```python
def spawn_codex(*, project_dir, dangerous, ...):
    argv = ["codex"]
    if dangerous:
        argv.append("--dangerously-bypass-approvals-and-sandbox")
    return PtyProcess(argv, cwd=project_dir, ...)
```

No `--mcp-config`, no `--append-system-prompt` ‚Äî codex CLI doesn't accept either flag. Codex discovers MCP servers and skills via filesystem-level scanning:

- **MCP**: `~/.codex/config.toml` (user-scope) and now `<project>/.codex/config.toml` (project-scope, [introduced in Codex 2026](https://developers.openai.com/codex/config-reference) ‚Äî Codex walks from project root to cwd loading every `.codex/config.toml`)
- **Skills**: `~/.agents/skills/<name>/` and `<repo>/.agents/skills/<name>/`
- **Instructions**: `~/.codex/AGENTS.md`, `<project>/AGENTS.md` (hierarchical concatenation)

Today's SciEasy installs **none** of these for the user's project, and `scieasy install --skill` writes only to `~/.claude/skills/` (not `~/.agents/skills/`). Result: a GUI user who never ran `scieasy install --target codex --scope user` manually opens a codex tab and gets **zero SciEasy MCP tools and zero SKILL.md guidance**. This is a current prod-env defect, not introduced by ADR-040 but unaddressed by ADR-040 v2.

Additionally, the existing `install.py::_install_codex` comment ("Codex has no project-scope config file; force user-scope for codex even when the user picked --scope project") is **outdated as of Codex 2026** ‚Äî project-scope `.codex/config.toml` is supported and this fallback logic should be removed.

#### 2.8 `scieasy install --skill` is Claude-only (#824 paths, plus Codex gap)

`install.py::_install_skill` copies `skills/scieasy/` to either `~/.claude/skills/scieasy/` (user scope) or `<cwd>/.claude/skills/scieasy/` (project scope). There is no equivalent for Codex's `~/.agents/skills/scieasy/`. With Codex Skills (¬ß2.5) using identical SKILL.md format, the gap is one extra `shutil.copytree` per scope.

#### 2.9 Project-creation lifecycle has two entry points (both prod-env)

Both prod-env entry points must learn to install hooks + skills + CLAUDE/AGENTS + codex config:

- `src/scieasy/cli/main.py::init` ‚Äî CLI path (`scieasy init my_project`).
- `src/scieasy/api/runtime.py::create_project` ‚Äî GUI path, called by `POST /api/projects`.

ADR-039 already documented that *git auto-init* must happen at both entry points. The same pattern applies here: all prod-env agent assets auto-install at both. Additionally, `open_project` should idempotently top-up any old project missing these assets ‚Äî necessary because alpha-stage projects on disk pre-date this ADR.

### 3. Decision

#### 3.1 Migrate prod-env MCP server to FastMCP 3.x

Adopt `fastmcp >=3.1, <4.0` as a runtime dependency. Replace the contents of `src/scieasy/ai/agent/mcp/server.py` with a thin FastMCP-backed implementation. Tool handlers move from plain functions in `tools_workflow.py` / `tools_authoring.py` / etc. to `@mcp.tool()`-decorated functions with explicit type hints.

```python
# Before (today)
def write_workflow(path: str, yaml: str) -> dict[str, Any]: ...

# After (FastMCP)
from pydantic import BaseModel, Field

class WriteWorkflowResult(BaseModel):
    path: str
    bytes_written: int
    diff_summary: str
    next_step: str = Field(
        description="Suggested next MCP call to maintain workflow integrity.",
    )

@mcp.tool(name="write_workflow")
async def write_workflow(
    path: str = Field(description="Project-relative path under workflows/."),
    yaml: str = Field(description="Full workflow YAML content."),
) -> WriteWorkflowResult:
    """Persist a workflow YAML to disk with a file lock and pre-write schema validation.

    Use when writing or replacing a workflow YAML. Do NOT use Bash or Edit on
    workflows/*.yaml ‚Äî this tool validates against the SciEasy schema; direct
    writes can produce malformed YAML the GUI cannot load. Returns a next_step
    field pointing at validate_workflow.
    """
    ...
```

Three deliverables fall out automatically:

1. **`inputSchema` is generated from type hints + Pydantic models**, replacing today's `additionalProperties: true` stub. The model layer (Claude / Codex) rejects malformed calls before reaching the runtime.
2. **Tool descriptions become the function docstring** ‚Äî one source of truth.
3. **Tool results are typed Pydantic models with `next_step` and `warnings` fields**:
   - `next_step: str` on every write-class tool ‚Äî names the canonical follow-up MCP call.
   - `warnings: list[str]` on tools that perform soft validation ‚Äî populated when the call succeeds but with caveats the agent should review (canonical example: `scaffold_block` flagging generic-`DataObject` ports or unregistered type names ‚Äî see ¬ß3.2a).

   FastMCP serializes the result Pydantic model into the MCP `content` text block returned to the agent; both fields land in the agent's next reasoning turn inline with the result. **These fields are unrelated to engine block payload, ADR-031 reference-only contract, or ADR-038 lineage ‚Äî they are purely string fields inside the MCP response envelope.**

`TOOL_REGISTRY` (`_registry.py`) is deleted; FastMCP discovers tools by decorator. `compose_system_prompt._render_tool_catalog` reads the FastMCP server's `list_tools()` instead.

`finish_ai_block` (ADR-035 ¬ß3.5) ports over with the same shape; its error envelopes become Pydantic `Union` returns.

#### 3.2 Tool description + `next_step` content rewrite

A one-time pass across all 26 tool functions rewrites docstrings and adds `next_step` fields. Style guide:

- **First line**: imperative one-sentence summary (becomes MCP `description`).
- **Second paragraph**: "Use when: ‚Ä¶" + "Do NOT use to: ‚Ä¶" anti-patterns.
- **`next_step`** on write-class tools: name the exact MCP tool the agent should call next + the reason.

The rewrite happens once, lives in code (docstrings), and reflects automatically in both MCP `tools/list` and the rendered system-prompt catalog.

#### 3.2a Soft validation via tool-result `warnings` (port-type narrowness, etc.)

For situations where a hard schema enum would be too strict but the agent's chosen argument deserves a flagged review, tools return a `warnings: list[str]` field on the result. The warning text reaches the agent as part of the MCP `content` text block; the agent's next turn reads it and typically corrects course without user intervention.

**Canonical use case ‚Äî port type narrowness on `scaffold_block`** (closes part of the agent-drift pattern documented in this section):

Observed: agents authoring new blocks routinely set port `expected_type` to the abstract `DataObject` root because it's the safest default ‚Äî registration accepts anything. The cost is silent: preview rendering breaks (data viewers can't dispatch on type), and edge-connection type checking degrades to "anything to anything". The user only notices when they try to inspect the block's output and get no view.

A hard schema enum (`expected_type ‚àà TypeRegistry.all() ‚àí {"DataObject"}`) was considered and rejected because legitimate cases exist:

- a plugin author may declare a new `DataObject` subtype in the same Python module as the block that produces it;
- generic blocks (`SubWorkflowBlock`, certain `AppBlock`s, generic save/load) legitimately accept any `DataObject` subtype.

Soft validation in the tool result is the right shape:

```python
class ScaffoldBlockResult(BaseModel):
    path: str
    bytes_written: int
    next_step: str
    warnings: list[str] = Field(
        default_factory=list,
        description="Soft advisory notes the agent should review in its next turn.",
    )

@mcp.tool(name="scaffold_block")
async def scaffold_block(name, category, input_ports, output_ports, ...) -> ScaffoldBlockResult:
    """..."""
    warnings: list[str] = []
    for port_name, spec in {**input_ports, **output_ports}.items():
        if spec.type == "DataObject":
            warnings.append(
                f"Port '{port_name}' uses generic DataObject. Preview and "
                "edge-time type checking will degrade. Confirm intentional "
                "(e.g. SubWorkflowBlock or generic AppBlock); otherwise pick "
                "a concrete type from mcp__scieasy__list_types()."
            )
        elif not type_registry.has(spec.type):
            warnings.append(
                f"Port '{port_name}' references unregistered type "
                f"'{spec.type}'. Either pick from list_types() or register "
                "the new type via the scieasy.types entry-point in this plugin."
            )
    # write the block file
    return ScaffoldBlockResult(path=..., warnings=warnings, next_step=...)
```

The same pattern applies to other tools where the agent's argument choice has hidden downstream cost (e.g. choosing the wrong `category` for a block, picking a `subcategory` outside the established taxonomy). Tool authors populate `warnings` opportunistically as those patterns surface.

Hook-level enforcement (¬ß3.6 `enforce_concrete_port_types.py`) provides a second layer: PostToolUse on `Edit|Write|mcp__scieasy__scaffold_block` matching `blocks/*.py` AST-parses the written file and re-flags the same issues ‚Äî catches the case where the agent bypasses `scaffold_block` entirely and writes a block via Edit/Write/Bash. Both layers are advisory (no hard block); BlockRegistry runtime validation of port types is a stronger enforcement worth considering but is **out of scope of this ADR** (cross-cutting policy decision affecting human-authored blocks too ‚Äî deferred to ADR-041 if pursued).

#### 3.3 Per-project dynamic context injection (closes #825)

Extend `compose_system_prompt(project_dir)` so the previously-unused `project_dir` argument actually shapes the prompt. Add a `<!-- project_context:begin/end -->` marker block to the base SKILL.md (mirroring the `tool_catalog` pattern); the composer splices a rendered project-state section between the markers:

```markdown
## Project context

You are operating inside the **{project_name}** project at `{project_dir}`.

- Workflows directory: `{project_dir}/workflows/` ({workflow_count} workflows present)
- Installed block plugins: scieasy_blocks_imaging, scieasy_blocks_lcms, scieasy_blocks_srs
- Git: branch `{branch}` at commit `{sha[:7]}` (omit section if not a git repo)
- Recently-modified workflows (top 3 by mtime):
  - `microplastics.yaml` (modified 2h ago)
  - `calibration.yaml` (modified 3d ago)

Always reference these instead of inventing paths.
```

Implementation lives in a new helper `_render_project_context(project_dir: Path) -> str`. Each field source:

| Field | Source |
|---|---|
| `project_name` | `project.yaml::project.name` (or `project_dir.name` fallback) |
| `workflow_count` | `len(list((project_dir / 'workflows').glob('*.yaml')))` |
| `installed_plugins` | enumerate via the live `BlockRegistry` |
| `branch`, `sha` | best-effort `git -C <project_dir> rev-parse`; omit section if not a git repo |
| `recent_workflows` | top 3 by `Path.stat().st_mtime`, formatted as "N{h,d,w} ago" |

Performance budget: <100ms even at 1000 workflows (use `os.scandir`). Composer is called once per PTY spawn; no caching needed.

**Note**: the prompt is still frozen-at-spawn (PTY architecture inherits this; per-turn re-composition would require owning the LLM API call). This addition makes the prompt **per-project dynamic** rather than **per-turn dynamic** ‚Äî sufficient for the user-visible drift cases observed (#825 acceptance criteria).

#### 3.4 Split monolithic SKILL.md into 5 task-scoped skills + relocate for wheel packaging (closes #832 + #824)

Replace `skills/scieasy/SKILL.md` with a directory of 5 task skills + a thin base index file, and **relocate the entire `skills/` tree from repo-root to `src/scieasy/_skills/` so wheel installs find it**:

```
src/scieasy/_skills/                       ‚Üê relocated from repo-root skills/
‚îî‚îÄ‚îÄ scieasy/
    ‚îú‚îÄ‚îÄ SKILL.md                           ‚Üê base: identity + skill index + project_context marker (~50 LOC)
    ‚îú‚îÄ‚îÄ scieasy-build-workflow/SKILL.md    ‚Üê writing new workflows
    ‚îú‚îÄ‚îÄ scieasy-write-block/SKILL.md       ‚Üê authoring custom blocks (covers #875 block-reuse rule)
    ‚îú‚îÄ‚îÄ scieasy-debug-run/SKILL.md         ‚Üê diagnosing failed runs
    ‚îú‚îÄ‚îÄ scieasy-inspect-data/SKILL.md      ‚Üê exploring data refs / lineage
    ‚îî‚îÄ‚îÄ scieasy-project-qa/SKILL.md        ‚Üê project structure / docs / data Q&A
```

Each task skill has frontmatter:

```yaml
---
name: scieasy-build-workflow
description: |
  Use when the user wants to design a new workflow, choose block types,
  wire edges between blocks, or convert a verbal pipeline description
  into a valid workflow YAML. NOT for debugging existing runs (use
  scieasy-debug-run) or for writing custom blocks (use scieasy-write-block).
---

[body: YAML schema teaching + worked example + tool-call sequence]
```

The split aligns with the 4 MCP tool categories (workflow / authoring / inspection / qa) plus `scieasy-debug-run` which spans categories ‚Äî slightly finer than #832's 3-skill proposal (workflow/blocks/debug) because the MCP-tool surface has clean 4-way category structure. The base SKILL.md (~50 LOC) keeps existing as the discoverable entry point with YAML frontmatter `name: scieasy`; its body lists "available skills, when to use each" plus the ¬ß3.3 `<!-- project_context:begin/end -->` marker.

**Block-reuse rule (#875)**: the `scieasy-write-block` skill body opens with a mandatory rule:

> Before authoring a new block, you MUST call `mcp__scieasy__list_blocks` and REUSE an existing one if its contract matches. Build a new block ONLY when no existing block satisfies the I/O contract ‚Äî and document why in the new block's docstring.

This is the SKILL-side half of #875; the hook-side half (CLI bypass) lives in ¬ß3.6.

**Port-type selection rule** (companion to ¬ß3.2a soft validation): the same skill body has a "Selecting port types" section:

> Before deciding port types, call `mcp__scieasy__list_types` to see registered `DataObject` subtypes. Pick the **most specific applicable** type for each port ‚Äî narrower types give the user better preview rendering and edge-time type checking.
>
> Use `DataObject` (the abstract base) only when the block legitimately accepts ANY type (e.g. `SubWorkflowBlock` inputs, generic save/load blocks). Generic-typed ports break preview and reduce edge validation to a no-op ‚Äî `scaffold_block` will emit a warning in its result when you choose `DataObject`, and a PostToolUse hook will flag it after direct file writes.
>
> If your block needs a type that isn't in `list_types()`, declare a new `DataObject` subclass in your plugin's `scieasy.types` entry-point ‚Äî do NOT silently widen to `DataObject` to avoid the type registry step.

This is the L1 / L2 (skill + tool description) half of the port-type drift fix; ¬ß3.2a is the L4 MCP-runtime-warning half; ¬ß3.6 `enforce_concrete_port_types.py` is the L5 PostToolUse hook half.

**Wheel packaging (#824)**: relocate `skills/` ‚Üí `src/scieasy/_skills/` and replace `_load_skill_md` with:

```python
from importlib.resources import files
def _load_skill_md() -> str:
    return (files("scieasy") / "_skills" / "scieasy" / "SKILL.md").read_text("utf-8")
```

Add to `pyproject.toml`:

```toml
[tool.setuptools.package-data]
scieasy = ["_skills/scieasy/**/*.md"]
```

Both `compose_system_prompt._load_skill_md` and `install.py::_find_skill_source` switch to `importlib.resources`. Wheel install now ships all 6 skill files inside `site-packages/scieasy/_skills/scieasy/`.

**Backward compatibility**: existing tests reference `skills/scieasy/SKILL.md` at repo root. We keep a thin compatibility symlink (or one-time migration commit) but the canonical location is the packaged path.

**`scieasy install --skill` cross-install**: installs **all 6 skill files** (base + 5 task skills) to the destination, preserving the directory structure. Same for the auto-install path (¬ß3.6).

#### 3.5 Prod-env `CLAUDE.md` + `AGENTS.md` provisioning

Auto-install a single short instruction file (~50 lines) at **both** `<project>/CLAUDE.md` and `<project>/AGENTS.md` at project-creation time. Content is identical on both sides ‚Äî Claude Code reads `CLAUDE.md`, Codex reads `AGENTS.md`, and we want symmetric behavior.

**This is distinct from the SciEasy repo's own `CLAUDE.md`** (~800 lines: gate workflow, ADR process, hotfix protocol). The dev `CLAUDE.md` is irrelevant to end-user agents; copying it wholesale would distract from the user's actual workflow.

Proposed content shape (~50 lines, exact wording finalized in Phase 3):

```markdown
# SciEasy project ‚Äî agent guide

You are an embedded agent inside a SciEasy project workspace. The user
is a researcher building scientific data workflows. The SciEasy GUI is
already running; do NOT start a second backend.

## Identity & non-negotiable rules
- Use mcp__scieasy__* tools for anything touching blocks, workflows, runs,
  or data. Do NOT use the `scieasy` CLI via Bash ‚Äî it bypasses live GUI
  updates and ADR-038 lineage. (Hooks will block such calls ‚Äî exit code 2.)
- Direct Edit/Write on workflows/*.yaml is blocked by hooks. Use
  mcp__scieasy__write_workflow / update_block_config instead.
- BEFORE writing a new block, list existing blocks via
  mcp__scieasy__list_blocks and reuse one if its I/O contract matches.
- All workflow YAML changes are git-tracked (ADR-039); the user sees your
  diffs and can revert.

## Skills available
- scieasy-build-workflow ‚Äî design a new workflow
- scieasy-write-block ‚Äî author a custom block
- scieasy-debug-run ‚Äî diagnose a failed run
- scieasy-inspect-data ‚Äî explore data references / lineage
- scieasy-project-qa ‚Äî answer project structure / docs questions

Invoke the relevant skill before deep work in that area.
```

Both files are written verbatim from a single template living in `src/scieasy/agent_provisioning/templates/`. On `open_project` of an existing project, files are created only if missing (idempotent top-up); user customizations are not overwritten.

#### 3.6 Project-scoped hooks at `<project>/.claude/settings.json`

**Scope reminder: prod env only ‚Äî the user's SciEasy project directory, NOT the SciEasy source repo.** No hooks are added to the source repo's own `.claude/`.

**Hook script language: Python.** Claude Code and Codex both accept any shell-executable command, so Bash / PowerShell / Node would also work, but Python wins on cross-platform consistency, ease of maintenance, and consistency with the rest of SciEasy.

Initial hook set (v1):

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/deny_scieasy_cli.py\"" }]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/protect_workflow_yaml.py\"" }]
      },
      {
        "matcher": "Edit|Write|Bash|mcp__scieasy__scaffold_block",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/enforce_list_blocks_before_block_write.py\"" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "mcp__scieasy__run_workflow",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/remind_poll_status.py\"" }]
      },
      {
        "matcher": "mcp__scieasy__list_blocks",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/mark_list_blocks_called.py\"" }]
      },
      {
        "matcher": "Edit|Write|mcp__scieasy__scaffold_block",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/enforce_concrete_port_types.py\"" }]
      }
    ]
  }
}
```

Hook script behaviors:

- **`deny_scieasy_cli.py`** (PreToolUse / Bash): reads tool input JSON from stdin; if `tool_input.command` matches `^\s*(.*/)?scieasy[\s$]`, prints to stderr `"SciEasy CLI calls bypass the GUI and lineage. Use mcp__scieasy__* tools instead: list_blocks, write_workflow, run_workflow, get_run_status."` and `sys.exit(2)`. Otherwise exit 0. (Closes #875 CLI-vs-MCP half.)
- **`protect_workflow_yaml.py`** (PreToolUse / Edit|Write): reads tool input; if `tool_input.file_path` matches `workflows/.*\.ya?ml$`, prints `"workflows/*.yaml is managed by mcp__scieasy__write_workflow (schema-validated) and mcp__scieasy__update_block_config (preserves comments). Direct Edit/Write bypasses validation."` and exit 2. Otherwise exit 0.
- **`enforce_list_blocks_before_block_write.py`** (PreToolUse / Edit|Write|Bash|mcp__scieasy__scaffold_block): enforces #875 block-reuse rule deterministically. Detects whether the current call is about to write a block file (`tool_input.file_path` matches `blocks/.*\.py$` for Edit/Write; `tool_input.command` contains `> blocks/`, `>> blocks/`, `tee blocks/`, or `cp ... blocks/` for Bash; always-on for `scaffold_block`). When a block write is detected, the hook checks for a session-keyed marker at `<project>/.scieasy/.session-state/<session_id>/list_blocks_called`. Marker absent ‚Üí print to stderr `"Authoring a custom block requires calling mcp__scieasy__list_blocks first to confirm no existing block matches your I/O contract. Call list_blocks now, then retry."` and `sys.exit(2)`. Marker present ‚Üí exit 0. Closes #875 block-reuse half. Known limitation: exotic Bash writes (`python -c '...'`, `mv`, here-doc piping through `sh -c`) bypass the regex; this is hook-layer defense-in-depth, not absolute prevention. Layer 7 (true filesystem ACL on `blocks/`) is the bulletproof escalation path if drift surfaces in production ‚Äî explicitly deferred to a future ADR.
- **`remind_poll_status.py`** (PostToolUse / mcp__scieasy__run_workflow): reads tool response, injects feedback string telling the agent to poll `get_run_status` until terminal. PostToolUse cannot block; always exit 0.
- **`mark_list_blocks_called.py`** (PostToolUse / mcp__scieasy__list_blocks): writes the marker file `<project>/.scieasy/.session-state/<session_id>/list_blocks_called` (the `session_id` comes from the hook stdin payload supplied by Claude Code). The session-keyed path ensures a fresh chat tab does not inherit a previous session's `list_blocks` confirmation. Always exit 0.
- **`enforce_concrete_port_types.py`** (PostToolUse / Edit|Write|mcp__scieasy__scaffold_block): fires after a write that may have touched `blocks/*.py`. For Edit/Write, checks `tool_input.file_path` matches `blocks/.*\.py$` (skips otherwise). For `scaffold_block`, always runs. AST-parses the resulting file and scans for `PortSpec(type="DataObject")` / `type=DataObject` (and references to type names not registered in `TypeRegistry`). Each finding produces a stderr line:
  ```
  Block at {file}:{lineno} declares port '{port_name}' with generic
  DataObject. This degrades preview & edge-time type-check. Call
  mcp__scieasy__list_types and pick a concrete type; DataObject is
  reserved for SubWorkflowBlock and generic AppBlock-class blocks.
  ```
  Always exit 0 (PostToolUse cannot block; the stderr appears in the agent's next-turn context and typically prompts self-correction). Companion to ¬ß3.2a soft validation ‚Äî that catches the MCP-`scaffold_block`-path case via the result envelope; this hook catches the Edit/Write-bypass path via AST. Known limitation: same Bash exotic-write blind spot as `enforce_list_blocks_before_block_write.py` (¬ß7.3).

The marker directory `<project>/.scieasy/.session-state/` is included in the default `.gitignore` written by ADR-039 (it's machine-local, per-session, not portable). Cleanup of stale session-state directories is a Phase 3 task ‚Äî best-effort prune on `open_project` of directories older than 7 days.

#### 3.7 Codex MCP provisioning at `<project>/.codex/config.toml` (newly capable)

[Codex 2026](https://developers.openai.com/codex/config-reference) introduced project-scope `.codex/config.toml` support ‚Äî Codex walks from project root to cwd loading every `.codex/config.toml`. SciEasy can now write a project-scope MCP config, parity with what Claude Code gets via `<project>/.scieasy/mcp.json` + `--mcp-config` argv.

Content at `<project>/.codex/config.toml`:

```toml
[mcp_servers.scieasy]
command = "<sys.executable>"
args = ["-m", "scieasy", "mcp-bridge"]

[mcp_servers.scieasy.env]
SCIEASY_PROJECT_DIR = "<absolute project path>"
```

Exact format mirrors what `install.py::_render_codex_block` already produces for user-scope install ‚Äî same shape, different file. With this file present, `spawn_codex` no longer needs any argv changes; Codex auto-discovers the MCP server.

Note on Codex's "trusted projects only" model: project-scope `.codex/config.toml` is honored only for projects the user has marked trusted. The first time a user opens a SciEasy project after this ADR ships, Codex may prompt for trust acceptance. This is acceptable UX ‚Äî same friction once per project, then permanent.

#### 3.8 Auto-installation at project lifecycle events

A new module `src/scieasy/agent_provisioning/` owns the orchestration:

```python
def install_project_agent_assets(project_dir: Path, *, force: bool = False) -> ProvisionResult:
    """Install all prod-env agent assets into project_dir.

    Writes:
      - <project>/CLAUDE.md          (from template, ¬ß3.5)
      - <project>/AGENTS.md          (identical content, ¬ß3.5)

```

| Caller | Behavior |

#### 3.9 Codex skill cross-install via `scieasy install`

`scieasy install --skill` is the **dev/power-user-brings-own-CLI** path ‚Äî distinct from ¬ß3.8's GUI-project auto-install. It is used by SciEasy contributors or power users who want their personal `~/.claude/skills/` or a project's `.claude/skills/` to know about SciEasy.

Behavior change:

| Scope | Today | After |
|---|---|---|
| `--scope user` | writes `~/.claude/skills/scieasy/` only | writes both `~/.claude/skills/scieasy/` AND `~/.agents/skills/scieasy/`; both contain the 6 skill files |
| `--scope project` | writes `<cwd>/.claude/skills/scieasy/` only | writes both `<cwd>/.claude/skills/scieasy/` AND `<cwd>/.agents/skills/scieasy/` |

Additionally, **`scieasy install --target codex --scope project` gains true project-scope MCP write capability** (writes `<cwd>/.codex/config.toml`) ‚Äî eliminating the "force user-scope" fallback at `install.py:489-498` that's now outdated per ¬ß2.7. User-scope codex install continues to write `~/.codex/config.toml`.

The `--target claude|codex` flag continues to govern only the MCP-config write. The `--skill` flag always cross-installs both providers' skill trees ‚Äî there is no use case for asymmetric skill availability.

`_remove_skill` symmetrically deletes both paths.

#### 3.10 Out of scope (explicit)

Explicitly deferred to keep the ADR scope tight and dev/prod boundaries clean:

- **Dev environment changes** ‚Äî the SciEasy source repo's `.claude/`, the source repo's `CLAUDE.md`, the source repo's hypothetical hooks. Out of scope. Specifically, *making the source repo's `AGENTS.md` mirror its `CLAUDE.md`* is content work ‚Äî tracked as a separate issue, no ADR needed.
- **Per-turn system-prompt re-composition** ‚Äî requires owning the LLM API call stack. The ¬ß3.3 per-project injection + ¬ß3.2 `next_step` mechanism + ¬ß3.4 JIT skills cover the dynamic-context need. (#903 dimension D ‚Äî per-project / per-machine overlay system ‚Äî also out of scope of this ADR but covered by #903's umbrella.)
- **Empirical evaluation framework** for prompt changes (#903 dimension E). Useful but not blocking; tracked under #903.
- **ACP (Agent Client Protocol) adoption** ‚Äî Anthropic's `claude-code` issue #6686 closed as NOT_PLANNED makes upstream support unlikely in 2026.
- **Codex hook coverage** ‚Äî Codex's hook system has documented gaps for `apply_patch` and most MCP calls per Agentic Control Plane research. SciEasy's hooks govern Claude Code only; Codex prod-env behavior is constrained by ¬ß3.5 CLAUDE/AGENTS provisioning + ¬ß3.4 multi-skill + tool description rewrites alone.
- **LLM-call-side guardrails frameworks** (NeMo Guardrails, Guardrails AI, etc.) ‚Äî incompatible with PTY-embed architecture.
- **`scieasy install --hooks` flag** ‚Äî auto-installation at project creation (¬ß3.8) covers every GUI user. Adding an explicit `--hooks` flag for power users is plausible but not necessary for v1.

### 4. Alternatives considered

1. **Stay with hand-rolled MCP server, tighten SKILL.md wording.** Rejected: documented behavior drift is structural; no amount of "NEVER" / "MUST NOT" wording compensates for the model's tendency to fall back to its training distribution when an alternative path is more friction.
2. **Anthropic's official MCP SDK instead of FastMCP.** Rejected: still requires hand-writing `inputSchema` per tool. FastMCP's Pydantic integration is the de facto Python idiom.
3. **System-wide hooks at `~/.claude/settings.json` instead of project-scoped.** Rejected: SciEasy hooks must not affect the user's other Claude Code sessions.
4. **Install hooks via `scieasy install --hooks` only, not auto-install at project creation.** Rejected: GUI users never run `scieasy install`. Auto-install is the only path that reaches every user.
5. **Single SKILL.md covering all task contexts (no split).** Rejected: monolithic SKILL.md loads all 372 lines per session regardless of task; Skills' progressive-disclosure semantics reduce both context pressure and steering noise.
6. **Single skill split target (only Claude Code skills, not Codex).** Rejected: Codex Skills use identical SKILL.md format; cross-installing is one extra `shutil.copytree` per scope.
7. **Use `CLAUDE.md` only (skip `AGENTS.md`).** Rejected: Codex doesn't read `CLAUDE.md`; symmetric provisioning is one file write more.
8. **Generate `next_step` content dynamically via a separate LLM call.** Rejected: cost + latency + reliability worse than static hints encoded in the tool function.
9. **Treat the dev `CLAUDE.md` and prod `CLAUDE.md` as the same file.** Rejected: dev `CLAUDE.md` is 800 lines of contributor process irrelevant to end-user agents.
10. **Keep `skills/` at repo-root and use `[tool.setuptools.package-data]` with relative include.** Rejected: hairier than relocating because the file isn't under the package root; `package_dir` shenanigans are less conventional than `_skills/` inside the package.
11. **Codex user-scope-only MCP (preserve current `install.py` fallback).** Rejected: with Codex 2026 supporting project-scope `.codex/config.toml`, project-scope is now the correct primary path. User-scope remains supported for the `~/.codex/config.toml` install flow.

- Author the ~50-line `claude_agents_md.md` template per ¬ß3.5.
- Author `codex_config.toml` template per ¬ß3.7.
- Implement 6 hook scripts per ¬ß3.6 (3 PreToolUse + 3 PostToolUse).
- Implement `install_project_agent_assets` covering CLAUDE.md + AGENTS.md + hooks + skills (both providers' trees) + codex config.
- Wire into `ApiRuntime.create_project`, `ApiRuntime.open_project` (idempotent), `cli.main.init`.
- Update `terminal.py::spawn_codex` docstring (outdated "asymmetry" comment).
- Tests: create new project ‚Üí all assets present and executable; open old project ‚Üí idempotent top-up; hook behaviors against synthetic stdin; Codex tab in fresh project sees scieasy MCP via auto-discovery.

**Phase 4 ‚Äî Codex parity for `scieasy install` (~2 days, ~130 LOC)**

- Extend `install.py::_install_skill` and `_remove_skill` for cross-install across `.claude/skills/` + `.agents/skills/`.
- Extend `install.py::_install_codex` with project-scope branch writing `<cwd>/.codex/config.toml`; remove the "force user-scope" fallback.
- Tests: cross-install writes both paths for all 6 skills; remove cleans both; codex project-scope install writes correct TOML.

Phases are linearly dependent: 1 ‚Üí 2 ‚Üí 3 ‚Üí 4. After Phase 3 the production-environment agent has the full four-layer reliability stack and Codex tabs work zero-install. Phase 4 brings the dev-power-user-brings-own-CLI flow to parity.

### 7. Consequences

#### 7.1 Wins

- **Behavior drift mitigation**: `PreToolUse` hooks deterministically block known-bad patterns (#875 enforcement half). No more "the agent ignored my SKILL.md rule" in prod env.
- **Codex prod-env unbroken**: GUI codex tabs work zero-install for the first time. Today's silent-failure mode (no MCP, no SKILL) replaced by auto-provisioned project config.
- **Schema validation at MCP boundary**: malformed tool calls rejected by FastMCP before reaching handler code.
- **`next_step` JIT injection**: every write-class tool's result steers the agent's next action. Compounding effect over a session.
- **Per-project dynamic context**: the agent introduces itself with the right project name, workflow count, git state ‚Äî no more cross-project confusion (#825 closed).
- **Single source of truth for tool metadata**: docstring ‚Üí description ‚Üí MCP `tools/list` ‚Üí SKILL.md catalog.
- **Skill discovery**: agent loads task-relevant skill body only, not 372 lines per task. Token savings + sharper steering (#832 closed).
- **Wheel-installable**: SKILL.md ships inside the package (#824 closed); production deploys work day-one.
- **Codex parity**: end-user Codex sessions get the same guidance as Claude Code sessions (CLAUDE.md/AGENTS.md sync + skill cross-install + project-scope MCP config).
- **Auto-provisioning**: GUI users (who never run `scieasy install`) get the full reliability stack from project creation.
- **Dev/prod boundary respected**: changes contained to prod env; SciEasy source repo's `.claude/` and `CLAUDE.md` untouched.

#### 7.2 Losses

- **`fastmcp` is a new dependency**: pinned to `>=3.1,<4`. Third-party but mature (4M+ daily downloads).
- **Hooks add 3 Python files per project**: minor disk footprint. `.claude/` is conventional and git-trackable; opt out by deleting `<project>/.claude/settings.json`.
- **Hooks introduce per-tool-call latency**: ~30-50 ms Python subprocess spawn per Bash/Edit/Write. Invisible in SciEasy's interactive usage pattern.
- **Codex behavior remains weaker than Claude Code on enforcement**: hooks don't fire reliably for codex MCP calls (upstream coverage gap). ¬ß3.5 CLAUDE.md/AGENTS.md + ¬ß3.4 skill content carry the Codex reliability load.
- **More prod-env files in user's project root**: `CLAUDE.md`, `AGENTS.md`, `.claude/`, `.agents/`, `.codex/`. Users opening the project tree see extra noise. Mitigation: a brief note in the auto-generated `CLAUDE.md` explains its purpose.
- **First-time Codex trust prompt**: project-scope `.codex/config.toml` requires user trust acceptance on first open. One-time UX friction per project.

#### 7.3 Risks

- **FastMCP API drift**: 3.x line is recent. Mitigation: pin `<4`; CI matrix runs against `fastmcp-latest` for early warning.
- **`importlib.resources` package-data semantics**: setuptools `package-data` glob patterns can be fiddly. Mitigation: Phase 2 wheel-layout regression test catches breakage in CI.
- **Hook script Python interpreter on PATH**: Claude Code spawns hook commands via shell ‚Äî relies on `python` on PATH. Mitigation: hook command absolute-path fallback documented in operational docs.
- **User customizes provisioned files then `open_project` overwrites**: idempotent top-up must respect existing files. Mitigation: `force=False` semantics; version-marker file governs upgrades.
- **`next_step` strings rot when MCP tools change**: a renamed tool referenced by a `next_step` would mislead. Mitigation: CI check that every `next_step` referenced tool name exists in the FastMCP-discovered tool set.
- **Block-reuse hook bypass via exotic Bash patterns**: `python -c 'open("blocks/x.py","w").write(...)'`, `mv ... blocks/`, `sh -c "..."` indirection, or any heredoc piping the regex misses. The hook covers `>`, `>>`, `tee`, `cp ... blocks/` but cannot cover every legitimate write idiom. Mitigation: deferred ‚Äî if production telemetry shows the gap is exploited frequently, escalate to Layer 7 filesystem ACL on `<project>/blocks/` (out of scope, ¬ß10).
- **Port-type advisory hook is soft, not hard**: ¬ß3.6 `enforce_concrete_port_types.py` runs PostToolUse and outputs stderr but cannot block ‚Äî the agent may proceed with `DataObject` ports against the warning if the user accepts it. Combined with ¬ß3.2a `warnings` field on `scaffold_block` results, the agent gets the signal twice (once at MCP-call time, once after the file write). Hard enforcement requires BlockRegistry-level rejection of `DataObject`-typed ports ‚Äî explicitly out of scope of this ADR (cross-cutting policy affecting human-authored blocks; tracked as ADR-041 if pursued).
- **Dev contributors confusion**: a contributor doing `scieasy init test-project` inside the SciEasy source repo would see hooks land in `test-project/`, not in the source repo itself. Correct behavior; flagged in dev docs.

### 9. Decision summary

**For the production-environment agent only (the claude/codex CLI spawned by the SciEasy GUI inside a user's SciEasy project), move behavior steering from one-layer (single static SKILL.md) to a four-layer stack:** (a) thin base SKILL.md + 5 JIT-loaded task skills with `<!-- project_context -->` per-project metadata injection; (b) `<project>/CLAUDE.md` + `<project>/AGENTS.md` purpose-written for end-user agents; (c) FastMCP-driven MCP server with type-hint-generated schemas, rewritten tool descriptions, and `next_step` hints on every write-class tool result; (d) project-scoped `PreToolUse`/`PostToolUse` hooks deterministically blocking known-bad patterns. All assets auto-installed on project creation (CLI + GUI) and idempotently topped-up on project open, including `<project>/.codex/config.toml` so Codex tabs work zero-install (closes a current prod-env defect). `scieasy install` extends to cross-install Claude Code (`~/.claude/skills/`) and Codex (`~/.agents/skills/`) trees and to gain true project-scope Codex MCP install. Skills relocate from repo-root to packaged `src/scieasy/_skills/` for wheel-install compatibility. The SciEasy source repository's own `CLAUDE.md`, `.claude/`, and dev-env agent configuration are explicitly **out of scope** ‚Äî this ADR governs the prod-env agent only.

The architectural invariant: **SciEasy continues to embed external agents (claude/codex), not implement one. The four-layer reliability stack is purely host-side engineering on the prod-env project tree ‚Äî Claude Code and Codex see standard config files, standard MCP servers, standard hooks, and standard skill trees. No custom protocols are introduced.**

### 10. Out of scope (defer to future ADRs)

- Sub-agent dispatch via Claude Code's `Task` tool from inside SciEasy MCP.
- Custom slash commands (`.claude/commands/`) for SciEasy-specific workflows.
- Per-skill telemetry.
- Per-turn dynamic context refresh (requires owning the LLM API call).
- Project-level / per-machine prompt overlays (#903 dimension D).
- Empirical prompt evaluation framework (#903 dimension E).
- Codex hook governance once OpenAI fills the MCP-call coverage gap.
- **Filesystem ACL (Layer 7) on `<project>/blocks/`** for bulletproof block-write enforcement ‚Äî escalation path if hook-layer regex on Bash writes proves insufficient. Defer until production data shows the gap matters.
- **BlockRegistry runtime validation of port types** ‚Äî rejecting blocks declaring `DataObject`-typed ports unless explicitly opted-in (e.g. `Block.__supports_generic_data__ = True` for `SubWorkflowBlock`, `SaveData`, etc.). The strongest enforcement layer ‚Äî catches the agent's exotic-Bash bypass AND human-authored over-broad blocks. Cross-cutting policy decision: it affects the contract for ALL block authors, not just agent-authored blocks. Deferred to a future ADR-041 if production data motivates it; ADR-040's soft-advisory layers (¬ß3.2a + ¬ß3.6 hook) are expected to cover the dominant cases first.
