# ADR-040: Production-Environment Agent Reliability

## 1. Decision Summary

ADR-040 proposes reliability improvements for the production embedded agent:
FastMCP migration, project-aware prompt/context injection, split skills,
project provisioning for Claude and Codex, hooks, and Codex parity.

### 1.1 Problems Addressed

| Problem | Risk | ADR-040 response | Detailed section |
|---|---|---|---|
| Production agents bypassed MCP and project context | GUI refresh, lineage, and schema-safe writes could be skipped | Move MCP and prompt composition to stricter project-aware contracts | Section 2 |
| Skill and hook provisioning differed across tools | Claude and Codex could behave inconsistently | Cross-install skills, AGENTS/CLAUDE files, hooks, and Codex config | Section 2 |
| MCP server behavior drifted from tool expectations | Agents see malformed schemas or notification failures | Adopt FastMCP and notification-handling fixes | Section 2 |

### 1. Purpose

**This ADR is exclusively about the production-environment agent — the claude/codex CLI spawned by the SciStudio GUI inside a user's SciStudio project, working on the user's workflows and data. It does not govern the development-environment agent (claude/codex used by SciStudio contributors against the SciStudio source repo).** The dev/prod boundary is defined explicitly in §2.1 and must be respected throughout.

Observed drift in production:

1. The embedded agent reads `skills/scistudio/SKILL.md` (372 lines, attached at PTY spawn via `--append-system-prompt @file`) and is told "use MCP tools, never call `scistudio` CLI via Bash", yet routinely falls back to `Bash(scistudio run …)`. The CLI path bypasses live GUI refresh, lineage capture, and the schema-validated MCP write paths (#875).
2. The agent skips listing existing blocks before authoring new ones, bloating `blocks/` with near-duplicates (#875).
3. `compose_system_prompt(project_dir)` accepts a `project_dir` argument but does not use it; every project gets the byte-identical prompt, leaving the agent context-blind across projects (#825).
4. **Codex tabs are entirely broken for any user who has not manually run `scistudio install --target codex`** — `spawn_codex` passes no `--mcp-config` and no `--append-system-prompt`, and the project tree has no `.codex/`, no `.agents/skills/scistudio/`, and no `AGENTS.md`. The user opens a codex tab and sees no SciStudio MCP tools and no behavioral guidance.

These are structural problems, not wording problems — a one-layer prompt-only steering surface cannot enforce constraints once the context window dilutes, and the asymmetric provisioning of Claude Code vs Codex is a configuration gap not a prompt gap.

Six concrete engineering changes, all in production environment:

1. **MCP layer**: migrate the hand-rolled 251-LOC JSON-RPC server to **FastMCP 3.x**; rewrite all 26 tool descriptions; add `next_step` fields to write-class tool result models so the agent gets just-in-time guidance on the next correct MCP call.
2. **Per-project dynamic context injection** (closes **#825**): extend `compose_system_prompt` to splice a `<!-- project_context:begin/end -->` block with per-project facts (project name, workflow count, installed plugins, git branch/SHA, recently-modified workflows). Makes the previously-unused `project_dir` argument do real work.
3. **Skill split** (closes **#832**, **#824**): break the monolithic `skills/scistudio/SKILL.md` into 5 task-scoped skills (`scistudio-build-workflow`, `scistudio-write-block`, `scistudio-debug-run`, `scistudio-inspect-data`, `scistudio-project-qa`) using Claude Code Skills / Codex Skills progressive-disclosure semantics. Base SKILL.md shrinks to a thin identity + skill-pointer document. Skills directory relocates from repo-root `skills/` to packaged `src/scistudio/_skills/` and loads via `importlib.resources` so wheel installs work.
4. **Prod-env CLAUDE.md + AGENTS.md provisioning**: write a single source-of-truth content fragment (~50 lines) at both `<project>/CLAUDE.md` and `<project>/AGENTS.md` so both providers' base-instruction discovery picks it up. Content is purpose-written for end-user agents, distinct from the dev-env `CLAUDE.md` at the SciStudio source repo root.
5. **Project-scoped hooks** (closes **#875** for the CLI-vs-MCP enforcement half): provision `<project>/.claude/settings.json` + `<project>/.claude/hooks/` at project creation time, enforcing deterministic `PreToolUse` matchers that block known-bad patterns (Bash `scistudio *`, direct Edit on `workflows/*.yaml`) before the call reaches the runtime. The "block reuse" half of #875 lands via §3.4 CLAUDE.md content + §3.3 `scistudio-write-block` skill.
6. **Codex MCP provisioning + cross-skill install**: write `<project>/.codex/config.toml` (newly supported in Codex 2026 per [docs](https://developers.openai.com/codex/config-reference)) at project creation so Codex tabs see SciStudio MCP tools with zero manual setup. Extend `scistudio install --skill` to cross-install all 5 task skills to both `~/.claude/skills/` and `~/.agents/skills/`; update `scistudio install --target codex --scope project` to take advantage of Codex's new project-scope config support.

The ADR is incremental: no new subsystem, no contract change. It moves prod-env agent enforcement from prompt-only (Layer 1) to a four-layer stack (prompt + tool description + tool-result hints + deterministic hooks).

### 2. Context

#### 2.1 Dev environment vs production environment — the binding boundary

These two contexts have always existed in the project but were not explicitly documented; conflating them when designing agent reliability is the single largest hazard of this work.

| | **Development environment** | **Production environment** |
|---|---|---|
| What is it | The SciStudio **source repository** (this codebase) | A user's SciStudio **project workspace** (created by `scistudio init` or via the GUI's Create Project) |
| Who is the agent | claude/codex used by SciStudio contributors developing SciStudio itself | claude/codex spawned by the SciStudio GUI's PTY tabs, working in the user's project root |
| Whose code is the agent touching | `src/scistudio/**`, `tests/**`, `docs/**` | `workflows/*.yaml`, `blocks/*.py`, `data/`, etc. |
| Governing instructions | `CLAUDE.md` at SciStudio repo root (≈800 lines: gate workflow, ADR process, etc.) | `skills/scistudio/SKILL.md` (372 lines, injected via `--append-system-prompt`) |
| Hooks today | None | None |
| MCP tools today | None (contributors don't have SciStudio MCP installed against the source repo) | 26 tools for Claude Code via `--mcp-config`; **Codex: zero tools unless user manually ran `scistudio install`** |
| Lifecycle | Git clone + dev install | `scistudio init` / `ApiRuntime.create_project` |
| **ADR-040 scope** | **OUT OF SCOPE.** Only cross-referenced where the prod artifact has a dev analogue (§3.8). | **The entire ADR.** |

**Every "agent reliability" change in this ADR lands inside the user's SciStudio project directory.** The SciStudio source repo's `.claude/`, the source repo's `CLAUDE.md`, and contributors' personal `~/.claude/` configurations are deliberately not touched.

(Making the SciStudio source repo's `AGENTS.md` mirror its `CLAUDE.md` so Codex-using contributors get the same dev-env guidance is a worthwhile separate task. It is called out as out-of-scope in §3.8 and tracked as a follow-up issue. It is fundamentally a docs/content task, not an architecture decision, and does not warrant its own ADR.)

#### 2.2 Prod-env steering today — one layer, statically frozen

`scistudio.ai.agent.system_prompt.compose_system_prompt()` reads `skills/scistudio/SKILL.md`, splices a re-rendered tool catalog between `<!-- tool_catalog:begin -->`/`<!-- tool_catalog:end -->`, writes the result to `<project>/.scistudio/.tmp/scistudio-prompt-*.md`, and `spawn_claude` passes it via `--append-system-prompt @<path>`. The string is **frozen for the lifetime of the claude subprocess**. The user's SciStudio project has no `CLAUDE.md`, no `AGENTS.md`, no `.claude/settings.json`, and no `.claude/skills/` directory. Steering surface = the single injected SKILL.md plus the 26 MCP tools' minimalist descriptions.

This works as long as:
- the conversation stays short enough that SKILL.md retains attention weight at the system-message slot;
- the agent has no equally-valid second path to the same outcome.

Neither holds in practice. SKILL.md says "Never use `Bash` to call the SciStudio CLI" — but as soon as the user asks the agent to do something easier via shell than via the multi-step MCP path, the agent falls back to its trained-distribution default. **#875** documents the two specific drift patterns observed in production.

#### 2.3 `project_dir` is accepted but unused (#825)

`compose_system_prompt(project_dir: Path) -> str` takes `project_dir` purely "for forward compatibility". Today the same byte-for-byte prompt ships to claude/codex regardless of which project the user opened. The agent has no signal for project name, project root path, workflow count, installed block plugins, git branch/SHA, or recently-modified workflows. **#825** filed the proposed mechanism (a `<!-- project_context:begin/end -->` marker mirroring the existing `<!-- tool_catalog -->` pattern) but was queued behind ADR-034 Phase work.

#### 2.4 Prod-env MCP server is hand-rolled

`src/scistudio/ai/agent/mcp/server.py` (251 LOC) implements JSON-RPC 2.0 line-delimited framing, `initialize` / `tools/list` / `tools/call` dispatch, and error envelopes by hand. `_registry.py` (279 LOC) declares 26 `ToolEntry` dataclasses. Every `tools/list` response wraps every tool with `inputSchema: {"type": "object", "additionalProperties": true}` — the **schema accepts anything**; validation happens (if at all) inside each handler. [FastMCP 3.x](https://gofastmcp.com/servers/tools) inverts the model via decorator + type hints → `inputSchema` generation.

#### 2.5 SKILL.md is monolithic, statically loaded, and not wheel-installable (#832, #824)

`skills/scistudio/SKILL.md` is a single 372-line file covering identity, environment, project layout, YAML schema teaching, tool catalog, custom block authoring, 9 working principles, worked example. The agent reads all 372 lines per session regardless of task (#832).

Worse: `_load_skill_md` locates the file by walking up from `__file__` looking for a sibling `skills/` directory — works for source checkout + editable install, **fails for wheel install** because `skills/` is sibling to `src/` and does not land in `site-packages` (#824). For ADR-034 production deployment via wheel this is a release blocker: `compose_system_prompt` raises `FileNotFoundError`, PTY spawns claude without `--append-system-prompt`, claude has no idea it's inside SciStudio.

Both [Claude Code Skills](https://code.claude.com/docs/en/skills) and [Codex Skills](https://developers.openai.com/codex/skills) support progressive-disclosure semantics (frontmatter `description` triggers JIT body load) — the two providers use different paths (`~/.claude/skills/<name>/` vs `~/.agents/skills/<name>/`) but identical SKILL.md structure.

#### 2.6 Prod-env hooks scaffolding does not exist

`grep -r PreToolUse src/` finds one comment in `_registry.py` referencing the obsolete ADR-033 STRICT-mode plan. No hook scripts exist. No `scistudio install --hooks` flag exists. [Claude Code's hooks system](https://code.claude.com/docs/en/hooks) is mature: project-scoped hooks at `<project>/.claude/settings.json` reference scripts at `<project>/.claude/hooks/*` via `$CLAUDE_PROJECT_DIR`, exit code 2 blocks with stderr returned to the agent. The `command` field accepts any executable — Python, Bash, PowerShell, Node, binaries.

#### 2.7 Codex prod-env is broken today (newly discovered)

Audit of `terminal.py::spawn_codex` (verified 2026-05-15):

```python
def spawn_codex(*, project_dir, dangerous, ...):
    argv = ["codex"]
    if dangerous:
        argv.append("--dangerously-bypass-approvals-and-sandbox")
    return PtyProcess(argv, cwd=project_dir, ...)
```

No `--mcp-config`, no `--append-system-prompt` — codex CLI doesn't accept either flag. Codex discovers MCP servers and skills via filesystem-level scanning:

- **MCP**: `~/.codex/config.toml` (user-scope) and now `<project>/.codex/config.toml` (project-scope, [introduced in Codex 2026](https://developers.openai.com/codex/config-reference) — Codex walks from project root to cwd loading every `.codex/config.toml`)
- **Skills**: `~/.agents/skills/<name>/` and `<repo>/.agents/skills/<name>/`
- **Instructions**: `~/.codex/AGENTS.md`, `<project>/AGENTS.md` (hierarchical concatenation)

Today's SciStudio installs **none** of these for the user's project, and `scistudio install --skill` writes only to `~/.claude/skills/` (not `~/.agents/skills/`). Result: a GUI user who never ran `scistudio install --target codex --scope user` manually opens a codex tab and gets **zero SciStudio MCP tools and zero SKILL.md guidance**. This is a current prod-env defect, not introduced by ADR-040 but unaddressed by ADR-040 v2.

Additionally, the existing `install.py::_install_codex` comment ("Codex has no project-scope config file; force user-scope for codex even when the user picked --scope project") is **outdated as of Codex 2026** — project-scope `.codex/config.toml` is supported and this fallback logic should be removed.

#### 2.8 `scistudio install --skill` is Claude-only (#824 paths, plus Codex gap)

`install.py::_install_skill` copies `skills/scistudio/` to either `~/.claude/skills/scistudio/` (user scope) or `<cwd>/.claude/skills/scistudio/` (project scope). There is no equivalent for Codex's `~/.agents/skills/scistudio/`. With Codex Skills (§2.5) using identical SKILL.md format, the gap is one extra `shutil.copytree` per scope.

#### 2.9 Project-creation lifecycle has two entry points (both prod-env)

Both prod-env entry points must learn to install hooks + skills + CLAUDE/AGENTS + codex config:

- `src/scistudio/cli/main.py::init` — CLI path (`scistudio init my_project`).
- `src/scistudio/api/runtime.py::create_project` — GUI path, called by `POST /api/projects`.

ADR-039 already documented that *git auto-init* must happen at both entry points. The same pattern applies here: all prod-env agent assets auto-install at both. Additionally, `open_project` should idempotently top-up any old project missing these assets — necessary because alpha-stage projects on disk pre-date this ADR.

### 3. Decision

#### 3.1 Migrate prod-env MCP server to FastMCP 3.x

Adopt `fastmcp >=3.1, <4.0` as a runtime dependency. Replace the contents of `src/scistudio/ai/agent/mcp/server.py` with a thin FastMCP-backed implementation. Tool handlers move from plain functions in `tools_workflow.py` / `tools_authoring.py` / etc. to `@mcp.tool()`-decorated functions with explicit type hints.

```python
# Before (today)
def write_workflow(path: str, yaml: str) -> dict[str, Any]: ...

# After (FastMCP)
from pydantic import BaseModel, Field

class WriteWorkflowResult(BaseModel):
    path: str
    bytes_written: int
    diff_summary: str
    next_step: str = Field(
        description="Suggested next MCP call to maintain workflow integrity.",
    )

@mcp.tool(name="write_workflow")
async def write_workflow(
    path: str = Field(description="Project-relative path under workflows/."),
    yaml: str = Field(description="Full workflow YAML content."),
) -> WriteWorkflowResult:
    """Persist a workflow YAML to disk with a file lock and pre-write schema validation.

    Use when writing or replacing a workflow YAML. Do NOT use Bash or Edit on
    workflows/*.yaml — this tool validates against the SciStudio schema; direct
    writes can produce malformed YAML the GUI cannot load. Returns a next_step
    field pointing at validate_workflow.
    """
    ...
```

Three deliverables fall out automatically:

1. **`inputSchema` is generated from type hints + Pydantic models**, replacing today's `additionalProperties: true` stub. The model layer (Claude / Codex) rejects malformed calls before reaching the runtime.
2. **Tool descriptions become the function docstring** — one source of truth.
3. **Tool results are typed Pydantic models with `next_step` and `warnings` fields**:
   - `next_step: str` on every write-class tool — names the canonical follow-up MCP call.
   - `warnings: list[str]` on tools that perform soft validation — populated when the call succeeds but with caveats the agent should review (canonical example: `scaffold_block` flagging generic-`DataObject` ports or unregistered type names — see §3.2a).

   FastMCP serializes the result Pydantic model into the MCP `content` text block returned to the agent; both fields land in the agent's next reasoning turn inline with the result. **These fields are unrelated to engine block payload, ADR-031 reference-only contract, or ADR-038 lineage — they are purely string fields inside the MCP response envelope.**

`TOOL_REGISTRY` (`_registry.py`) is deleted; FastMCP discovers tools by decorator. `compose_system_prompt._render_tool_catalog` reads the FastMCP server's `list_tools()` instead.

`finish_ai_block` (ADR-035 §3.5) ports over with the same shape; its error envelopes become Pydantic `Union` returns.

#### 3.2 Tool description + `next_step` content rewrite

A one-time pass across all 26 tool functions rewrites docstrings and adds `next_step` fields. Style guide:

- **First line**: imperative one-sentence summary (becomes MCP `description`).
- **Second paragraph**: "Use when: …" + "Do NOT use to: …" anti-patterns.
- **`next_step`** on write-class tools: name the exact MCP tool the agent should call next + the reason.

The rewrite happens once, lives in code (docstrings), and reflects automatically in both MCP `tools/list` and the rendered system-prompt catalog.

#### 3.2a Soft validation via tool-result `warnings` (port-type narrowness, etc.)

For situations where a hard schema enum would be too strict but the agent's chosen argument deserves a flagged review, tools return a `warnings: list[str]` field on the result. The warning text reaches the agent as part of the MCP `content` text block; the agent's next turn reads it and typically corrects course without user intervention.

**Canonical use case — port type narrowness on `scaffold_block`** (closes part of the agent-drift pattern documented in this section):

Observed: agents authoring new blocks routinely set port `expected_type` to the abstract `DataObject` root because it's the safest default — registration accepts anything. The cost is silent: preview rendering breaks (data viewers can't dispatch on type), and edge-connection type checking degrades to "anything to anything". The user only notices when they try to inspect the block's output and get no view.

A hard schema enum (`expected_type ∈ TypeRegistry.all() − {"DataObject"}`) was considered and rejected because legitimate cases exist:

- a plugin author may declare a new `DataObject` subtype in the same Python module as the block that produces it;
- generic blocks (`SubWorkflowBlock`, certain `AppBlock`s, generic save/load) legitimately accept any `DataObject` subtype.

Soft validation in the tool result is the right shape:

```python
class ScaffoldBlockResult(BaseModel):
    path: str
    bytes_written: int
    next_step: str
    warnings: list[str] = Field(
        default_factory=list,
        description="Soft advisory notes the agent should review in its next turn.",
    )

@mcp.tool(name="scaffold_block")
async def scaffold_block(name, category, input_ports, output_ports, ...) -> ScaffoldBlockResult:
    """..."""
    warnings: list[str] = []
    for port_name, spec in {**input_ports, **output_ports}.items():
        if spec.type == "DataObject":
            warnings.append(
                f"Port '{port_name}' uses generic DataObject. Preview and "
                "edge-time type checking will degrade. Confirm intentional "
                "(e.g. SubWorkflowBlock or generic AppBlock); otherwise pick "
                "a concrete type from mcp__scistudio__list_types()."
            )
        elif not type_registry.has(spec.type):
            warnings.append(
                f"Port '{port_name}' references unregistered type "
                f"'{spec.type}'. Either pick from list_types() or register "
                "the new type via the scistudio.types entry-point in this plugin."
            )
    # write the block file
    return ScaffoldBlockResult(path=..., warnings=warnings, next_step=...)
```

The same pattern applies to other tools where the agent's argument choice has hidden downstream cost (e.g. choosing the wrong `category` for a block, picking a `subcategory` outside the established taxonomy). Tool authors populate `warnings` opportunistically as those patterns surface.

Hook-level enforcement (§3.6 `enforce_concrete_port_types.py`) provides a second layer: PostToolUse on `Edit|Write|mcp__scistudio__scaffold_block` matching `blocks/*.py` AST-parses the written file and re-flags the same issues — catches the case where the agent bypasses `scaffold_block` entirely and writes a block via Edit/Write/Bash. Both layers are advisory (no hard block); BlockRegistry runtime validation of port types is a stronger enforcement worth considering but is **out of scope of this ADR** (cross-cutting policy decision affecting human-authored blocks too — deferred to ADR-041 if pursued).

#### 3.3 Per-project dynamic context injection (closes #825)

Extend `compose_system_prompt(project_dir)` so the previously-unused `project_dir` argument actually shapes the prompt. Add a `<!-- project_context:begin/end -->` marker block to the base SKILL.md (mirroring the `tool_catalog` pattern); the composer splices a rendered project-state section between the markers:

```markdown
## Project context

You are operating inside the **{project_name}** project at `{project_dir}`.

- Workflows directory: `{project_dir}/workflows/` ({workflow_count} workflows present)
- Installed block plugins: scistudio_blocks_imaging, scistudio_blocks_lcms, scistudio_blocks_srs
- Git: branch `{branch}` at commit `{sha[:7]}` (omit section if not a git repo)
- Recently-modified workflows (top 3 by mtime):
  - `microplastics.yaml` (modified 2h ago)
  - `calibration.yaml` (modified 3d ago)

Always reference these instead of inventing paths.
```

Implementation lives in a new helper `_render_project_context(project_dir: Path) -> str`. Each field source:

| Field | Source |
|---|---|
| `project_name` | `project.yaml::project.name` (or `project_dir.name` fallback) |
| `workflow_count` | `len(list((project_dir / 'workflows').glob('*.yaml')))` |
| `installed_plugins` | enumerate via the live `BlockRegistry` |
| `branch`, `sha` | best-effort `git -C <project_dir> rev-parse`; omit section if not a git repo |
| `recent_workflows` | top 3 by `Path.stat().st_mtime`, formatted as "N{h,d,w} ago" |

Performance budget: <100ms even at 1000 workflows (use `os.scandir`). Composer is called once per PTY spawn; no caching needed.

**Note**: the prompt is still frozen-at-spawn (PTY architecture inherits this; per-turn re-composition would require owning the LLM API call). This addition makes the prompt **per-project dynamic** rather than **per-turn dynamic** — sufficient for the user-visible drift cases observed (#825 acceptance criteria).

#### 3.4 Split monolithic SKILL.md into 5 task-scoped skills + relocate for wheel packaging (closes #832 + #824)

Replace `skills/scistudio/SKILL.md` with a directory of 5 task skills + a thin base index file, and **relocate the entire `skills/` tree from repo-root to `src/scistudio/_skills/` so wheel installs find it**:

```
src/scistudio/_skills/                       ← relocated from repo-root skills/
└── scistudio/
    ├── SKILL.md                           ← base: identity + skill index + project_context marker (~50 LOC)
    ├── scistudio-build-workflow/SKILL.md    ← writing new workflows
    ├── scistudio-write-block/SKILL.md       ← authoring custom blocks (covers #875 block-reuse rule)
    ├── scistudio-debug-run/SKILL.md         ← diagnosing failed runs
    ├── scistudio-inspect-data/SKILL.md      ← exploring data refs / lineage
    └── scistudio-project-qa/SKILL.md        ← project structure / docs / data Q&A
```

Each task skill has frontmatter:

```yaml
---
name: scistudio-build-workflow
description: |
  Use when the user wants to design a new workflow, choose block types,
  wire edges between blocks, or convert a verbal pipeline description
  into a valid workflow YAML. NOT for debugging existing runs (use
  scistudio-debug-run) or for writing custom blocks (use scistudio-write-block).
---

[body: YAML schema teaching + worked example + tool-call sequence]
```

The split aligns with the 4 MCP tool categories (workflow / authoring / inspection / qa) plus `scistudio-debug-run` which spans categories — slightly finer than #832's 3-skill proposal (workflow/blocks/debug) because the MCP-tool surface has clean 4-way category structure. The base SKILL.md (~50 LOC) keeps existing as the discoverable entry point with YAML frontmatter `name: scistudio`; its body lists "available skills, when to use each" plus the §3.3 `<!-- project_context:begin/end -->` marker.

**Block-reuse rule (#875)**: the `scistudio-write-block` skill body opens with a mandatory rule:

> Before authoring a new block, you MUST call `mcp__scistudio__list_blocks` and REUSE an existing one if its contract matches. Build a new block ONLY when no existing block satisfies the I/O contract — and document why in the new block's docstring.

This is the SKILL-side half of #875; the hook-side half (CLI bypass) lives in §3.6.

**Port-type selection rule** (companion to §3.2a soft validation): the same skill body has a "Selecting port types" section:

> Before deciding port types, call `mcp__scistudio__list_types` to see registered `DataObject` subtypes. Pick the **most specific applicable** type for each port — narrower types give the user better preview rendering and edge-time type checking.
>
> Use `DataObject` (the abstract base) only when the block legitimately accepts ANY type (e.g. `SubWorkflowBlock` inputs, generic save/load blocks). Generic-typed ports break preview and reduce edge validation to a no-op — `scaffold_block` will emit a warning in its result when you choose `DataObject`, and a PostToolUse hook will flag it after direct file writes.
>
> If your block needs a type that isn't in `list_types()`, declare a new `DataObject` subclass in your plugin's `scistudio.types` entry-point — do NOT silently widen to `DataObject` to avoid the type registry step.

This is the L1 / L2 (skill + tool description) half of the port-type drift fix; §3.2a is the L4 MCP-runtime-warning half; §3.6 `enforce_concrete_port_types.py` is the L5 PostToolUse hook half.

**Wheel packaging (#824)**: relocate `skills/` → `src/scistudio/_skills/` and replace `_load_skill_md` with:

```python
from importlib.resources import files
def _load_skill_md() -> str:
    return (files("scistudio") / "_skills" / "scistudio" / "SKILL.md").read_text("utf-8")
```

Add to `pyproject.toml`:

```toml
[tool.setuptools.package-data]
scistudio = ["_skills/scistudio/**/*.md"]
```

Both `compose_system_prompt._load_skill_md` and `install.py::_find_skill_source` switch to `importlib.resources`. Wheel install now ships all 6 skill files inside `site-packages/scistudio/_skills/scistudio/`.

**Backward compatibility**: existing tests reference `skills/scistudio/SKILL.md` at repo root. We keep a thin compatibility symlink (or one-time migration commit) but the canonical location is the packaged path.

**`scistudio install --skill` cross-install**: installs **all 6 skill files** (base + 5 task skills) to the destination, preserving the directory structure. Same for the auto-install path (§3.6).

#### 3.5 Prod-env `CLAUDE.md` + `AGENTS.md` provisioning

Auto-install a single short instruction file (~50 lines) at **both** `<project>/CLAUDE.md` and `<project>/AGENTS.md` at project-creation time. Content is identical on both sides — Claude Code reads `CLAUDE.md`, Codex reads `AGENTS.md`, and we want symmetric behavior.

**This is distinct from the SciStudio repo's own `CLAUDE.md`** (~800 lines: gate workflow, ADR process, hotfix protocol). The dev `CLAUDE.md` is irrelevant to end-user agents; copying it wholesale would distract from the user's actual workflow.

Proposed content shape (~50 lines, exact wording finalized in Phase 3):

```markdown
# SciStudio project — agent guide

You are an embedded agent inside a SciStudio project workspace. The user
is a researcher building scientific data workflows. The SciStudio GUI is
already running; do NOT start a second backend.

## Identity & non-negotiable rules
- Use mcp__scistudio__* tools for anything touching blocks, workflows, runs,
  or data. Do NOT use the `scistudio` CLI via Bash — it bypasses live GUI
  updates and ADR-038 lineage. (Hooks will block such calls — exit code 2.)
- Direct Edit/Write on workflows/*.yaml is blocked by hooks. Use
  mcp__scistudio__write_workflow / update_block_config instead.
- BEFORE writing a new block, list existing blocks via
  mcp__scistudio__list_blocks and reuse one if its I/O contract matches.
- All workflow YAML changes are git-tracked (ADR-039); the user sees your
  diffs and can revert.

## Skills available
- scistudio-build-workflow — design a new workflow
- scistudio-write-block — author a custom block
- scistudio-debug-run — diagnose a failed run
- scistudio-inspect-data — explore data references / lineage
- scistudio-project-qa — answer project structure / docs questions

Invoke the relevant skill before deep work in that area.
```

Both files are written verbatim from a single template living in `src/scistudio/agent_provisioning/templates/`. On `open_project` of an existing project, files are created only if missing (idempotent top-up); user customizations are not overwritten.

#### 3.6 Project-scoped hooks at `<project>/.claude/settings.json`

**Scope reminder: prod env only — the user's SciStudio project directory, NOT the SciStudio source repo.** No hooks are added to the source repo's own `.claude/`.

**Hook script language: Python.** Claude Code and Codex both accept any shell-executable command, so Bash / PowerShell / Node would also work, but Python wins on cross-platform consistency, ease of maintenance, and consistency with the rest of SciStudio.

Initial hook set (v1):

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/deny_scistudio_cli.py\"" }]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/protect_workflow_yaml.py\"" }]
      },
      {
        "matcher": "Edit|Write|Bash|mcp__scistudio__scaffold_block",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/enforce_list_blocks_before_block_write.py\"" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "mcp__scistudio__run_workflow",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/remind_poll_status.py\"" }]
      },
      {
        "matcher": "mcp__scistudio__list_blocks",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/mark_list_blocks_called.py\"" }]
      },
      {
        "matcher": "Edit|Write|mcp__scistudio__scaffold_block",
        "hooks": [{ "type": "command",
          "command": "python \"$CLAUDE_PROJECT_DIR/.claude/hooks/enforce_concrete_port_types.py\"" }]
      }
    ]
  }
}
```

Hook script behaviors:

- **`deny_scistudio_cli.py`** (PreToolUse / Bash): reads tool input JSON from stdin; if `tool_input.command` matches `^\s*(.*/)?scistudio[\s$]`, prints to stderr `"SciStudio CLI calls bypass the GUI and lineage. Use mcp__scistudio__* tools instead: list_blocks, write_workflow, run_workflow, get_run_status."` and `sys.exit(2)`. Otherwise exit 0. (Closes #875 CLI-vs-MCP half.)
- **`protect_workflow_yaml.py`** (PreToolUse / Edit|Write): reads tool input; if `tool_input.file_path` matches `workflows/.*\.ya?ml$`, prints `"workflows/*.yaml is managed by mcp__scistudio__write_workflow (schema-validated) and mcp__scistudio__update_block_config (preserves comments). Direct Edit/Write bypasses validation."` and exit 2. Otherwise exit 0.
- **`enforce_list_blocks_before_block_write.py`** (PreToolUse / Edit|Write|Bash|mcp__scistudio__scaffold_block): enforces #875 block-reuse rule deterministically. Detects whether the current call is about to write a block file (`tool_input.file_path` matches `blocks/.*\.py$` for Edit/Write; `tool_input.command` contains `> blocks/`, `>> blocks/`, `tee blocks/`, or `cp ... blocks/` for Bash; always-on for `scaffold_block`). When a block write is detected, the hook checks for a session-keyed marker at `<project>/.scistudio/.session-state/<session_id>/list_blocks_called`. Marker absent → print to stderr `"Authoring a custom block requires calling mcp__scistudio__list_blocks first to confirm no existing block matches your I/O contract. Call list_blocks now, then retry."` and `sys.exit(2)`. Marker present → exit 0. Closes #875 block-reuse half. Known limitation: exotic Bash writes (`python -c '...'`, `mv`, here-doc piping through `sh -c`) bypass the regex; this is hook-layer defense-in-depth, not absolute prevention. Layer 7 (true filesystem ACL on `blocks/`) is the bulletproof escalation path if drift surfaces in production — explicitly deferred to a future ADR.
- **`remind_poll_status.py`** (PostToolUse / mcp__scistudio__run_workflow): reads tool response, injects feedback string telling the agent to poll `get_run_status` until terminal. PostToolUse cannot block; always exit 0.
- **`mark_list_blocks_called.py`** (PostToolUse / mcp__scistudio__list_blocks): writes the marker file `<project>/.scistudio/.session-state/<session_id>/list_blocks_called` (the `session_id` comes from the hook stdin payload supplied by Claude Code). The session-keyed path ensures a fresh chat tab does not inherit a previous session's `list_blocks` confirmation. Always exit 0.
- **`enforce_concrete_port_types.py`** (PostToolUse / Edit|Write|mcp__scistudio__scaffold_block): fires after a write that may have touched `blocks/*.py`. For Edit/Write, checks `tool_input.file_path` matches `blocks/.*\.py$` (skips otherwise). For `scaffold_block`, always runs. AST-parses the resulting file and scans for `PortSpec(type="DataObject")` / `type=DataObject` (and references to type names not registered in `TypeRegistry`). Each finding produces a stderr line:
  ```
  Block at {file}:{lineno} declares port '{port_name}' with generic
  DataObject. This degrades preview & edge-time type-check. Call
  mcp__scistudio__list_types and pick a concrete type; DataObject is
  reserved for SubWorkflowBlock and generic AppBlock-class blocks.
  ```
  Always exit 0 (PostToolUse cannot block; the stderr appears in the agent's next-turn context and typically prompts self-correction). Companion to §3.2a soft validation — that catches the MCP-`scaffold_block`-path case via the result envelope; this hook catches the Edit/Write-bypass path via AST. Known limitation: same Bash exotic-write blind spot as `enforce_list_blocks_before_block_write.py` (§7.3).

The marker directory `<project>/.scistudio/.session-state/` is included in the default `.gitignore` written by ADR-039 (it's machine-local, per-session, not portable). Cleanup of stale session-state directories is a Phase 3 task — best-effort prune on `open_project` of directories older than 7 days.

#### 3.7 Codex MCP provisioning at `<project>/.codex/config.toml` (newly capable)

[Codex 2026](https://developers.openai.com/codex/config-reference) introduced project-scope `.codex/config.toml` support — Codex walks from project root to cwd loading every `.codex/config.toml`. SciStudio can now write a project-scope MCP config, parity with what Claude Code gets via `<project>/.scistudio/mcp.json` + `--mcp-config` argv.

Content at `<project>/.codex/config.toml`:

```toml
[mcp_servers.scistudio]
command = "<sys.executable>"
args = ["-m", "scistudio", "mcp-bridge"]

[mcp_servers.scistudio.env]
SCISTUDIO_PROJECT_DIR = "<absolute project path>"
```

Exact format mirrors what `install.py::_render_codex_block` already produces for user-scope install — same shape, different file. With this file present, `spawn_codex` no longer needs any argv changes; Codex auto-discovers the MCP server.

Note on Codex's "trusted projects only" model: project-scope `.codex/config.toml` is honored only for projects the user has marked trusted. The first time a user opens a SciStudio project after this ADR ships, Codex may prompt for trust acceptance. This is acceptable UX — same friction once per project, then permanent.

#### 3.8 Auto-installation at project lifecycle events

A new module `src/scistudio/agent_provisioning/` owns the orchestration:

```python
def install_project_agent_assets(project_dir: Path, *, force: bool = False) -> ProvisionResult:
    """Install all prod-env agent assets into project_dir.

    Writes:
      - <project>/CLAUDE.md          (from template, §3.5)
      - <project>/AGENTS.md          (identical content, §3.5)
      - <project>/.claude/settings.json    (hooks config, §3.6)
      - <project>/.claude/hooks/*.py       (3 hook scripts, §3.6)
      - <project>/.claude/skills/<6 dirs>/SKILL.md  (§3.4 — flat, see Addendum 1)
      - <project>/.agents/skills/<6 dirs>/SKILL.md  (Codex mirror — flat, see Addendum 1)
      - <project>/.codex/config.toml       (Codex project-scope MCP, §3.7)

    Idempotent: when force=False, existing files are not overwritten.
    Failures are non-fatal — logged at WARNING. The project still opens
    even in degraded mode (per ADR-039 §3.9 precedent).
    """
```

Wired into three lifecycle points (all prod-env):

| Caller | Behavior |
|---|---|
| `ApiRuntime.create_project` | `install_project_agent_assets(project_path, force=False)` after git auto-init |
| `cli.main.init` | Same call after CLI git auto-init |
| `ApiRuntime.open_project` | Same call with `force=False` — idempotent top-up for alpha-stage projects created before this ADR |

#### 3.9 Codex skill cross-install via `scistudio install`

`scistudio install --skill` is the **dev/power-user-brings-own-CLI** path — distinct from §3.8's GUI-project auto-install. It is used by SciStudio contributors or power users who want their personal `~/.claude/skills/` or a project's `.claude/skills/` to know about SciStudio.

Behavior change:

| Scope | Today | After |
|---|---|---|
| `--scope user` | writes `~/.claude/skills/scistudio/` only | writes both `~/.claude/skills/scistudio/` AND `~/.agents/skills/scistudio/`; both contain the 6 skill files |
| `--scope project` | writes `<cwd>/.claude/skills/scistudio/` only | writes both `<cwd>/.claude/skills/scistudio/` AND `<cwd>/.agents/skills/scistudio/` |

Additionally, **`scistudio install --target codex --scope project` gains true project-scope MCP write capability** (writes `<cwd>/.codex/config.toml`) — eliminating the "force user-scope" fallback at `install.py:489-498` that's now outdated per §2.7. User-scope codex install continues to write `~/.codex/config.toml`.

The `--target claude|codex` flag continues to govern only the MCP-config write. The `--skill` flag always cross-installs both providers' skill trees — there is no use case for asymmetric skill availability.

`_remove_skill` symmetrically deletes both paths.

#### 3.10 Out of scope (explicit)

Explicitly deferred to keep the ADR scope tight and dev/prod boundaries clean:

- **Dev environment changes** — the SciStudio source repo's `.claude/`, the source repo's `CLAUDE.md`, the source repo's hypothetical hooks. Out of scope. Specifically, *making the source repo's `AGENTS.md` mirror its `CLAUDE.md`* is content work — tracked as a separate issue, no ADR needed.
- **Per-turn system-prompt re-composition** — requires owning the LLM API call stack. The §3.3 per-project injection + §3.2 `next_step` mechanism + §3.4 JIT skills cover the dynamic-context need. (#903 dimension D — per-project / per-machine overlay system — also out of scope of this ADR but covered by #903's umbrella.)
- **Empirical evaluation framework** for prompt changes (#903 dimension E). Useful but not blocking; tracked under #903.
- **ACP (Agent Client Protocol) adoption** — Anthropic's `claude-code` issue #6686 closed as NOT_PLANNED makes upstream support unlikely in 2026.
- **Codex hook coverage** — Codex's hook system has documented gaps for `apply_patch` and most MCP calls per Agentic Control Plane research. SciStudio's hooks govern Claude Code only; Codex prod-env behavior is constrained by §3.5 CLAUDE/AGENTS provisioning + §3.4 multi-skill + tool description rewrites alone.
- **LLM-call-side guardrails frameworks** (NeMo Guardrails, Guardrails AI, etc.) — incompatible with PTY-embed architecture.
- **`scistudio install --hooks` flag** — auto-installation at project creation (§3.8) covers every GUI user. Adding an explicit `--hooks` flag for power users is plausible but not necessary for v1.

### 4. Alternatives considered

1. **Stay with hand-rolled MCP server, tighten SKILL.md wording.** Rejected: documented behavior drift is structural; no amount of "NEVER" / "MUST NOT" wording compensates for the model's tendency to fall back to its training distribution when an alternative path is more friction.
2. **Anthropic's official MCP SDK instead of FastMCP.** Rejected: still requires hand-writing `inputSchema` per tool. FastMCP's Pydantic integration is the de facto Python idiom.
3. **System-wide hooks at `~/.claude/settings.json` instead of project-scoped.** Rejected: SciStudio hooks must not affect the user's other Claude Code sessions.
4. **Install hooks via `scistudio install --hooks` only, not auto-install at project creation.** Rejected: GUI users never run `scistudio install`. Auto-install is the only path that reaches every user.
5. **Single SKILL.md covering all task contexts (no split).** Rejected: monolithic SKILL.md loads all 372 lines per session regardless of task; Skills' progressive-disclosure semantics reduce both context pressure and steering noise.
6. **Single skill split target (only Claude Code skills, not Codex).** Rejected: Codex Skills use identical SKILL.md format; cross-installing is one extra `shutil.copytree` per scope.
7. **Use `CLAUDE.md` only (skip `AGENTS.md`).** Rejected: Codex doesn't read `CLAUDE.md`; symmetric provisioning is one file write more.
8. **Generate `next_step` content dynamically via a separate LLM call.** Rejected: cost + latency + reliability worse than static hints encoded in the tool function.
9. **Treat the dev `CLAUDE.md` and prod `CLAUDE.md` as the same file.** Rejected: dev `CLAUDE.md` is 800 lines of contributor process irrelevant to end-user agents.
10. **Keep `skills/` at repo-root and use `[tool.setuptools.package-data]` with relative include.** Rejected: hairier than relocating because the file isn't under the package root; `package_dir` shenanigans are less conventional than `_skills/` inside the package.
11. **Codex user-scope-only MCP (preserve current `install.py` fallback).** Rejected: with Codex 2026 supporting project-scope `.codex/config.toml`, project-scope is now the correct primary path. User-scope remains supported for the `~/.codex/config.toml` install flow.

### 7. Consequences

#### 7.1 Wins

- **Behavior drift mitigation**: `PreToolUse` hooks deterministically block known-bad patterns (#875 enforcement half). No more "the agent ignored my SKILL.md rule" in prod env.
- **Codex prod-env unbroken**: GUI codex tabs work zero-install for the first time. Today's silent-failure mode (no MCP, no SKILL) replaced by auto-provisioned project config.
- **Schema validation at MCP boundary**: malformed tool calls rejected by FastMCP before reaching handler code.
- **`next_step` JIT injection**: every write-class tool's result steers the agent's next action. Compounding effect over a session.
- **Per-project dynamic context**: the agent introduces itself with the right project name, workflow count, git state — no more cross-project confusion (#825 closed).
- **Single source of truth for tool metadata**: docstring → description → MCP `tools/list` → SKILL.md catalog.
- **Skill discovery**: agent loads task-relevant skill body only, not 372 lines per task. Token savings + sharper steering (#832 closed).
- **Wheel-installable**: SKILL.md ships inside the package (#824 closed); production deploys work day-one.
- **Codex parity**: end-user Codex sessions get the same guidance as Claude Code sessions (CLAUDE.md/AGENTS.md sync + skill cross-install + project-scope MCP config).
- **Auto-provisioning**: GUI users (who never run `scistudio install`) get the full reliability stack from project creation.
- **Dev/prod boundary respected**: changes contained to prod env; SciStudio source repo's `.claude/` and `CLAUDE.md` untouched.

#### 7.2 Losses

- **`fastmcp` is a new dependency**: pinned to `>=3.1,<4`. Third-party but mature (4M+ daily downloads).
- **Hooks add 3 Python files per project**: minor disk footprint. `.claude/` is conventional and git-trackable; opt out by deleting `<project>/.claude/settings.json`.
- **Hooks introduce per-tool-call latency**: ~30-50 ms Python subprocess spawn per Bash/Edit/Write. Invisible in SciStudio's interactive usage pattern.
- **Codex behavior remains weaker than Claude Code on enforcement**: hooks don't fire reliably for codex MCP calls (upstream coverage gap). §3.5 CLAUDE.md/AGENTS.md + §3.4 skill content carry the Codex reliability load.
- **More prod-env files in user's project root**: `CLAUDE.md`, `AGENTS.md`, `.claude/`, `.agents/`, `.codex/`. Users opening the project tree see extra noise. Mitigation: a brief note in the auto-generated `CLAUDE.md` explains its purpose.
- **First-time Codex trust prompt**: project-scope `.codex/config.toml` requires user trust acceptance on first open. One-time UX friction per project.

#### 7.3 Risks

- **FastMCP API drift**: 3.x line is recent. Mitigation: pin `<4`; CI matrix runs against `fastmcp-latest` for early warning.
- **`importlib.resources` package-data semantics**: setuptools `package-data` glob patterns can be fiddly. Mitigation: Phase 2 wheel-layout regression test catches breakage in CI.
- **Hook script Python interpreter on PATH**: Claude Code spawns hook commands via shell — relies on `python` on PATH. Mitigation: hook command absolute-path fallback documented in operational docs.
- **User customizes provisioned files then `open_project` overwrites**: idempotent top-up must respect existing files. Mitigation: `force=False` semantics; version-marker file governs upgrades.
- **`next_step` strings rot when MCP tools change**: a renamed tool referenced by a `next_step` would mislead. Mitigation: CI check that every `next_step` referenced tool name exists in the FastMCP-discovered tool set.
- **Block-reuse hook bypass via exotic Bash patterns**: `python -c 'open("blocks/x.py","w").write(...)'`, `mv ... blocks/`, `sh -c "..."` indirection, or any heredoc piping the regex misses. The hook covers `>`, `>>`, `tee`, `cp ... blocks/` but cannot cover every legitimate write idiom. Mitigation: deferred — if production telemetry shows the gap is exploited frequently, escalate to Layer 7 filesystem ACL on `<project>/blocks/` (out of scope, §10).
- **Port-type advisory hook is soft, not hard**: §3.6 `enforce_concrete_port_types.py` runs PostToolUse and outputs stderr but cannot block — the agent may proceed with `DataObject` ports against the warning if the user accepts it. Combined with §3.2a `warnings` field on `scaffold_block` results, the agent gets the signal twice (once at MCP-call time, once after the file write). Hard enforcement requires BlockRegistry-level rejection of `DataObject`-typed ports — explicitly out of scope of this ADR (cross-cutting policy affecting human-authored blocks; tracked as ADR-041 if pursued).
- **Dev contributors confusion**: a contributor doing `scistudio init test-project` inside the SciStudio source repo would see hooks land in `test-project/`, not in the source repo itself. Correct behavior; flagged in dev docs.

### 9. Decision summary

**For the production-environment agent only (the claude/codex CLI spawned by the SciStudio GUI inside a user's SciStudio project), move behavior steering from one-layer (single static SKILL.md) to a four-layer stack:** (a) thin base SKILL.md + 5 JIT-loaded task skills with `<!-- project_context -->` per-project metadata injection; (b) `<project>/CLAUDE.md` + `<project>/AGENTS.md` purpose-written for end-user agents; (c) FastMCP-driven MCP server with type-hint-generated schemas, rewritten tool descriptions, and `next_step` hints on every write-class tool result; (d) project-scoped `PreToolUse`/`PostToolUse` hooks deterministically blocking known-bad patterns. All assets auto-installed on project creation (CLI + GUI) and idempotently topped-up on project open, including `<project>/.codex/config.toml` so Codex tabs work zero-install (closes a current prod-env defect). `scistudio install` extends to cross-install Claude Code (`~/.claude/skills/`) and Codex (`~/.agents/skills/`) trees and to gain true project-scope Codex MCP install. Skills relocate from repo-root to packaged `src/scistudio/_skills/` for wheel-install compatibility. The SciStudio source repository's own `CLAUDE.md`, `.claude/`, and dev-env agent configuration are explicitly **out of scope** — this ADR governs the prod-env agent only.

The architectural invariant: **SciStudio continues to embed external agents (claude/codex), not implement one. The four-layer reliability stack is purely host-side engineering on the prod-env project tree — Claude Code and Codex see standard config files, standard MCP servers, standard hooks, and standard skill trees. No custom protocols are introduced.**

- Custom slash commands (`.claude/commands/`) for SciStudio-specific workflows.

## Addendum 3 (2026-05-17) — MCP server must silently drop JSON-RPC notifications

**Status**: accepted (Phase 4 hotfix)

**What changed**: `MCPServer.dispatch` now returns `None` for any
request without an `id` field; the per-client connection loop skips
writing a response in that case. Previously, dispatch returned a
JSON-RPC error envelope (`-32601 "unknown method
'notifications/initialized'"`) for every notification, regardless of
the absent `id`.

**Why**: Phase 4 e2e on the Codex provider (vcodex-cli 0.130.0) failed
to bring up any scistudio MCP tools — the chat tab showed `MCP client
for "scistudio" failed to start: MCP startup failed: Transport closed`.
End-to-end frame inspection of the bridge handshake showed the MCP
server sending an error response to the MCP-mandatory
`notifications/initialized` notification right after the
`initialize` response. JSON-RPC 2.0 §4.1 forbids responding to
notifications. Claude Code v2.1.143 silently tolerates the bogus
error envelope; Codex 2026 treats any unexpected frame on the
transport as fatal and tears down the connection — leaving the Codex
tab with no MCP tools and no path to MCP teaching content.

**Impact**:
- `src/scistudio/ai/agent/mcp/server.py`:
  - `dispatch()` returns `dict | None`. Returns `None` for any frame
    without an `id` field, regardless of whether the method is known
    or unknown.
  - `_handle_client()` skips writing when dispatch returns `None`.
- `tests/ai/test_mcp_fastmcp.py` gains a regression test
  `test_dispatch_silently_drops_notifications` covering
  `notifications/initialized`, `notifications/cancelled`, unknown
  notification methods, missing-method notifications, and the
  positive case that unknown methods WITH an `id` still surface
  `METHOD_NOT_FOUND`.

**Scope discipline**: this is a strict spec-compliance fix on the MCP
server. No changes to FastMCP, the bridge, the tool registry, or any
provisioning surface. Pre-existing notification handling in any
other code path was already correct.

## Addendum 4 (2026-05-17) — Codex hook parity (reverses §3.10 Codex deferral)

**Status**: accepted (Phase 4 hotfix, user-directed)

**What changed**: The auto-provisioned `<project>/.codex/config.toml`
now declares the same six hook matcher groups Claude Code's
`.claude/settings.json` carries. Codex 0.130+ honours these via its
PreToolUse / PostToolUse hook surface
(https://developers.openai.com/codex/config-reference). All six hook
scripts are SHARED between providers — both Codex and Claude Code
invoke the same Python files at `<project>/.claude/hooks/<name>.py`.

| Matcher group | Hook script | Stage | Behavior |
|---|---|---|---|
| `^Bash$` | `deny_scistudio_cli.py` | Pre | Block `Bash(scistudio …)` invocations |
| `^(Edit|Write|MultiEdit|apply_patch)$` | `protect_workflow_yaml.py` | Pre | Block direct edits of `workflows/*.yaml` |
| `^(Edit|Write|MultiEdit|apply_patch|Bash|mcp__scistudio__scaffold_block)$` | `enforce_list_blocks_before_block_write.py` | Pre | #875 — require `list_blocks` before authoring |
| `^mcp__scistudio__run_workflow$` | `remind_poll_status.py` | Post | Stderr advisory to poll `get_run_status` |
| `^mcp__scistudio__list_blocks$` | `mark_list_blocks_called.py` | Post | Set per-session marker that #875 was satisfied |
| `^(Edit|Write|MultiEdit|apply_patch|mcp__scistudio__scaffold_block)$` | `enforce_concrete_port_types.py` | Post | §3.2a — AST-scan for `DataObject` / empty `accepted_types` |

`apply_patch` is added to every Edit/Write group because Codex routes
some edit operations through its `apply_patch` tool rather than the
Claude-style `Edit`/`Write` tools. Both names are valid simultaneously
— Codex emits one or the other depending on the operation shape.

**Why**: Phase 4 e2e confirmed Codex MCP integration works end-to-end
(tests a–e + g all pass on Codex), but test (f) was the only documented
gap because §3.10 had deferred Codex hook coverage. User directive
overrode the deferral: Codex and Claude Code must be treated identically
for the cascade to ship. Codex 0.130 ships a hook surface with the same
stdin-JSON contract (`session_id`, `cwd`, `tool_name`, `tool_input`,
exit code 2 + stderr blocks the call) — so a single Python script set
covers both providers without duplication.

**Impact**:
- `src/scistudio/agent_provisioning/codex_config.py` now emits
  `features.hooks = true` + 6 hook matcher groups after the
  `[mcp_servers.scistudio]` block.
- `tests/agent_provisioning/test_codex_config.py` gains
  `test_codex_config_emits_six_hooks` covering matcher count, feature
  flag, and the cross-provider script-path invariant.
- The Codex 0.130 hook command lines use `git rev-parse --show-toplevel`
  to locate the project root (Codex has no equivalent of Claude's
  `$CLAUDE_PROJECT_DIR` env var; ADR-039 git auto-init guarantees the
  toplevel resolves correctly).
- ADR-040 §3.10's Codex hook bullet is no longer deferred. Issue #1015
  (the placeholder follow-up for Codex hooks) is closed by this
  addendum.

**Scope discipline**: change is confined to one writer module + one
test file + this addendum. No changes to the hook scripts themselves —
they were already provider-agnostic. No changes to the Claude-side
provisioning. No changes to the MCP server, the bridge, or skill
content.

**Known upstream gap — Codex 0.130 on Windows (2026-05-17)**:
End-to-end testing showed Codex 0.130 does NOT fire any PreToolUse /
PostToolUse hook on Windows, regardless of:
  - inline `[hooks]` in `config.toml` vs separate `hooks.json`
  - project-scope vs user-scope (`~/.codex/config.toml`) location
  - matcher specificity (wildcard `.*` fails identically to `^Bash$`)
  - `features.hooks = true` (which is stable + default-true anyway)

A trivial debug hook that logs to a file and exits 0 — invoked at
`.*` matcher, both inline TOML and standalone `hooks.json` forms,
project-scope and user-scope — never produced a log entry, never
fired. Same hook scripts work correctly when invoked directly with
the documented stdin-JSON payload. Therefore the runtime gap is
upstream in Codex 0.130's Windows hook executor, NOT in our config.

SciStudio ships the correct hook declarations regardless. Three reasons:

  1. The TOML we emit IS correct per
     https://developers.openai.com/codex/config-reference and the
     associated Hooks guide; the implementation matches the documented
     schema verbatim.
  2. Non-Windows platforms may already honor it; the cascade
     pre-positions parity rather than waiting for upstream.
  3. When Codex resolves the Windows hook runtime, our config
     "just works" — no SciStudio change required for users to gain
     parity.

A follow-up issue should be opened against the upstream Codex CLI
project with a minimal reproducer (TOML + always-deny hook + the
exact `codex exec --dangerously-bypass-approvals-and-sandbox …`
command that bypasses the hook). Until that lands, Windows Codex
users effectively retain ADR §3.10's documented gap — the cascade
ships the correct architecture, with the gap noted in this addendum
rather than silently in the ADR §3.10 list.
