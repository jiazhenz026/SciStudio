# ADR-019: ProcessHandle, ProcessRegistry, And Cross-Platform Process Lifecycle

## 1. Decision Summary

SciStudio manages every runtime subprocess through a small engine-layer process
lifecycle abstraction. `ProcessHandle` represents one subprocess,
`ProcessRegistry` tracks active handles, `ProcessMonitor` polls for unexpected
exits, and `PlatformOps` hides operating-system differences. ADR-017 requires
subprocess isolation; ADR-018 requires reliable cancellation. ADR-019 is the
process-management layer that makes those decisions operational.

The historical ADR status was `Proposed`; this rewrite preserves that status
instead of inventing a later acceptance event. The implementation surface now
exists, so the ADR records auditable contracts for the current modules.

### 1.1 Problems Addressed

| Problem | Risk | ADR-019 response | Detailed section |
|---|---|---|---|
| Raw subprocess handles are easy to lose | External apps and worker processes could become orphaned | Wrap processes in `ProcessHandle` and register them | Section 3 |
| Process-tree termination differs by OS | Killing only the root process can leave R, Julia, Python, or GUI children alive | Put platform logic behind `PlatformOps` | Section 4 |
| Unexpected process death needs detection | Crashes, OOM kills, and task-manager kills would not update workflow state | Poll active handles through `ProcessMonitor` | Section 5 |
| Shutdown must clean active work | Engine shutdown could leave block subprocesses running | Provide registry-level `terminate_all` | Section 6 |
| Process contracts must be auditable | Lifecycle prose could drift from code | Govern process handle, monitor, platform, and factory signatures | Section 7 |

## 2. Context

Subprocess isolation only works if launched processes remain observable and
terminable. Early AppBlock behavior launched an external process but did not
keep a durable lifecycle handle. Worker subprocesses need the same discipline:
the runtime must know which block owns the process, whether it is alive, how it
ended, and how to terminate it on cancellation or shutdown.

The hard part is not only starting a process. Scientific blocks may spawn
children: R scripts, multiprocessing pools, GUI helper processes, or nested
workflow workers. POSIX process groups and Windows process-tree mechanics do
not share one API, so the rest of the engine must not open-code platform
details.

## 3. ProcessHandle And ProcessRegistry

`ProcessHandle` is the per-process control object. It exposes non-blocking
status and termination operations and carries the owning `block_id`, PID,
start time, and resource request.

`ProcessRegistry` is the active-process map. It registers a handle when a
process is launched, supports lookup by block id for cancellation, provides a
snapshot for monitors, and offers `terminate_all` for shutdown cleanup.

The registry is intentionally simple: one active handle per block id. ADR-020
removed engine-level item iteration, so the runtime no longer needs multiple
simultaneous subprocesses for the same block id as a normal block-execution
case.

## 4. Platform Operations

`PlatformOps` isolates OS behavior:

| Operation | POSIX behavior | Windows behavior |
|---|---|---|
| Process group creation | Launch with a new session/process group | Launch with a new process group flag |
| Graceful termination | Send SIGTERM, wait, then SIGKILL | Terminate tree with `psutil`-backed behavior; Job Objects are available for nested cleanup |
| Forced termination | Kill process group | Kill process tree |
| Alive check | PID existence check via OS signal semantics | PID existence check via Windows/psutil semantics |
| Exit info | Decode wait status when available | Report process disappearance or status when available |
| Nested cleanup | Process groups | Job Object helper methods plus process-tree fallback |

The accepted boundary is that callers use `ProcessHandle.terminate()`,
`ProcessHandle.kill()`, `ProcessHandle.is_alive()`, and
`ProcessHandle.exit_info()` rather than reaching into platform-specific code.

## 5. ProcessMonitor

`ProcessMonitor` is a background coroutine. It polls active handles, detects
processes that are no longer alive, emits `PROCESS_EXITED`, and deregisters the
handle. Subscribers such as the scheduler and resource manager then update
block state, release resources, and write checkpoint or lineage evidence.

This closes a gap in interactive and external-application workflows. If an
external app exits or crashes while the block is paused, the framework can
notice the process death and decide whether files were produced, whether the
block should finish, or whether it should enter error state.

## 6. Lifecycle Factory

Subprocess creation is centralised. `spawn_block_process` builds the worker
payload, launches the worker with platform-appropriate process isolation,
creates a handle, registers it, emits `PROCESS_SPAWNED`, and returns the
handle. `register_async_process` covers subprocesses launched through
`asyncio.create_subprocess_exec`, which the local runner uses to avoid
fork-related deadlocks on macOS.

This design makes code review easier: direct process launch points should be
rare and justified. Normal block execution goes through the factory or the
async-registration helper.

## 7. Verification And Tooling Impact

Tests should verify:

- process handles expose alive, exit-info, terminate, and kill behavior;
- registry lookup, deregistration, active snapshots, and shutdown cleanup work;
- monitor polling emits `PROCESS_EXITED` and deregisters handles;
- platform implementations return the expected operation objects and do not
  treat invalid PIDs as alive;
- `Block.terminate_grace_sec` defaults to five seconds and can be overridden;
- AppBlock and file-watcher paths clean up process resources and surface
  process-exited-without-output failures.

## 8. Consequences

- Cancellation can terminate real operating-system processes and not just
  scheduler bookkeeping.
- Unexpected process death becomes observable and can drive terminal runtime
  events.
- Platform-specific process code is concentrated in one module.
- Direct raw `subprocess.Popen` use becomes exceptional and should be justified
  by a narrower contract.
- A background monitor adds a small polling cost, accepted for lifecycle
  correctness.

## 9. Alternatives Considered

Using `psutil` alone was rejected because Windows Job Object behavior and
launch isolation still need explicit handling. Relying on `Popen.terminate()`
or `Popen.kill()` was rejected because those calls only target the root process
and can leave children alive. Detecting death only when the scheduler next
checks a block was rejected because external-app workflows can wait in paused
states. Combining handle, registry, and monitor into one class was rejected
because they have different lifetimes and are easier to test separately.
