# ADR-021: MergeCollection And Collection Operation Blocks

> **Update (#1781, ADR-051):** The interactive **DataRouter** block now covers
> arbitrary collection filtering, slicing, and splitting through direct
> item-to-output routing, so the standalone `FilterCollection`,
> `SliceCollection`, and `SplitCollection` blocks were retired. `MergeCollection`
> remains the merge primitive but is now a **variadic-input** block (2–8 input
> ports) in the `routing` subcategory, so users merge multiple collections in
> one block instead of chaining pairwise merges. Sections 3–7 below describe the
> original four-block design and are retained for history; the live contract is
> `MergeCollection` (variadic) plus `DataRouter`/`PairEditor` (ADR-051).

## 1. Decision Summary

SciStudio provides built-in process blocks for collection-level operations:
merge, split, filter, and slice. These are ordinary blocks, not special
scheduler behavior. They operate on the `Collection` wrapper and preserve the
contained `DataObject` type whenever possible.

This ADR preserves the 2026-04-03 decision and records the current audit
surface: the operation blocks are discoverable through `scistudio.blocks`
entry-points, are implemented under `scistudio.blocks.process.builtins`, and are
tested through collection block tests.

### 1.1 Problems Addressed

| Problem | Risk | ADR-021 response | Detailed section |
|---|---|---|---|
| Users need to combine or subset collection outputs | Workflows would need custom script blocks for common graph operations | Provide built-in `MergeCollection`, `SplitCollection`, `FilterCollection`, and `SliceCollection` blocks | Section 3 |
| Multi-input collection handling could pressure the static port model | Dynamic ports would complicate registry, validation, and GUI behavior | Keep fixed ports and compose operations by chaining blocks | Section 4 |
| Collection utility behavior must stay auditable | ADR text could describe helpers without importable contracts | Govern concrete block classes and entry-points | Section 7 |

## 2. Context

ADR-020 moved inter-block data transport to `Collection` values. Once
collections became the normal boundary shape, users needed simple graph
operators for collection manipulation. Examples include merging two processed
LC-MS batches before opening them in one application, splitting a dataset into
review and downstream branches, filtering by metadata, and slicing a quick
subset for testing.

These operations are workflow structure, not engine scheduling. They should be
visible on the canvas and testable like any other block.

## 3. Collection Operation Blocks

The originally-accepted built-ins were (see the update banner above for the
current, post-#1781 surface):

| Block | Purpose | Input ports | Output ports | Status |
|---|---|---|---|---|
| `MergeCollection` | Concatenate same-typed collections | variadic `input_1..N` (was `input_a`, `input_b`) | `output` | Active (now variadic, `routing`) |
| `SplitCollection` | Split one collection at an index | `input` | `output_a`, `output_b` | Retired #1781 (use DataRouter) |
| `FilterCollection` | Keep items matching metadata or a safe expression predicate | `input` | `output` | Retired #1781 (use DataRouter) |
| `SliceCollection` | Extract a contiguous `[start:end]` range | `input` | `output` | Retired #1781 (use DataRouter) |

Each block works at the collection level. It does not transform the contained
scientific object payloads. Output collections preserve the source
`item_type`; `MergeCollection` rejects mismatched item types.

## 4. Static Port Constraint

Collection operation blocks have fixed ports. The framework does not introduce
runtime-added ports for "merge N inputs" or "split N outputs".

Users who need to merge more than two collections chain merge blocks:

```text
Merge(A, B) -> AB
Merge(AB, C) -> ABC
```

This keeps port validation, block schemas, frontend rendering, and workflow
serialization inside the static port model used by the rest of the block
system.

## 5. Predicate Filtering Boundary

`FilterCollection` supports the original metadata equality predicate and a
later expression mode. Expression filtering is evaluated through the
AST-whitelisted `ExpressionEvaluator`. It is still a collection utility block;
it is not a general Python `eval` hook and does not create a trusted bypass.

## 6. Scope

In scope:

- collection-level concatenate, split, filter, and slice operations;
- fixed input and output ports for each operation;
- preservation of collection item type;
- entry-point registration as built-in process blocks.

Out of scope:

- dynamic port count;
- sorting, deduplication, sampling, grouping, or joins;
- item payload transformation;
- scheduler-level collection fan-out.

## 7. Verification And Tooling Impact

Tests should verify same-type merge, mismatch rejection, non-collection input
errors, split boundaries, default midpoint behavior, empty filter results,
legacy metadata predicates, expression predicates, and slice range behavior.

### 7.1 Signature-Level Contracts

```python
class MergeCollection:
    def run(self, inputs: dict[str, Any], config: BlockConfig) -> dict[str, Any]: ...
```

## 8. Consequences

- Common collection graph operations become first-class and visible.
- The scheduler remains simpler because collection manipulation is ordinary
  block execution.
- The GUI can show these operations in the same palette and node model as other
  process blocks.
- Adding future collection utilities should create new blocks rather than
  hidden engine behavior.

## 9. Alternatives Considered

Dynamic input ports on downstream blocks were rejected because they would break
the static port contract and force every receiving block to implement merge
logic.

Engine-level collection helpers were rejected because they hide important graph
steps from lineage, review, and user control.
