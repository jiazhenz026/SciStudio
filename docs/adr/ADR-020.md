# ADR-020: Collection-Based Data Transport Eliminates Engine-Level Batch Iteration

## 1. Decision Summary

SciEasy treats inter-block values as `Collection` transport units. The engine
does not unpack, iterate, batch, or inspect collection items during scheduling.
Each block invocation receives one collection per input port and returns one
collection per output port. Item-level strategy belongs to the block, because
only the block knows whether a tool should process items independently, as a
whole collection, through GPU batching, through an external GUI session, or by
some domain-specific iteration scheme.

This ADR preserves the historical 2026-04-03 ADR-020 decision and folds in the
lazy-loading and boundary-conversion addenda that became part of the current
runtime: `Collection` is a transport wrapper, empty collections need explicit
item type, CodeBlock presents multi-item input as a memory-safe `LazyList`, and
worker/checkpoint/lineage paths preserve the collection wire format.

### 1.1 Problems Addressed

| Problem | Risk | ADR-020 response | Detailed section |
|---|---|---|---|
| Engine-level batch modes conflicted with process tracking | A single block id could spawn multiple simultaneous item processes and break `ProcessRegistry` assumptions | Make one block invocation equal one subprocess and one `ProcessHandle` | Section 3 |
| Parallel-to-serial item pipelines were ambiguous | The engine could buffer or reorder items without domain knowledge | Move item iteration and buffering into block code | Section 4 |
| External tools often want whole datasets | The engine would open many app instances when users expect one session | Pass whole collections to blocks and let AppBlocks decide their exchange format | Section 5 |
| Batch abstractions added scheduler complexity | Batch modes, batch error strategies, and batch result records complicated cancellation and lineage | Delete engine-level batch iteration and use collection transport | Section 6 |
| CodeBlock users should not need framework internals | Scripts would need to learn `Collection` and `DataObject` APIs | Auto-convert collections to native single values or `LazyList` | Section 7 |
| Collection contracts need audit evidence | ADR text could describe transport behavior without importable symbols | Govern `Collection`, block utilities, port checks, worker serialization, and collection utility blocks | Section 8 |

## 2. Context

Earlier architecture defined batch behavior as an engine concern. A block could
declare parallel, serial, or adaptive execution, and a scheduler or batch
executor would decide how items move through the graph. That model created four
architectural tensions:

1. Process lifecycle tracking wants one process handle per block invocation.
   Engine-managed item fan-out would require one-to-many handle mappings.
2. Scientific tools often have native collection semantics. ElMAVEN can load
   many files into one session; Cellpose can batch internally; napari can
   present multiple images in one UI.
3. Item-level buffering between parallel and serial stages is domain-specific.
   A general scheduler cannot know when early item release is correct.
4. Cancellation and error reporting become much more complex when the engine
   owns partial item state.

The collection transport model makes the engine simpler and gives domain blocks
the authority to choose item strategy.

## 3. Core Decision

All values crossing a block boundary are represented as `Collection` objects or
as their wire-format equivalent. A single item is represented as a
length-one `Collection`; a multi-file or multi-object input is represented as a
longer `Collection`; an empty result is allowed only when the item type is
explicit.

The engine-level invariant is:

```text
one scheduled block node = one block invocation = one subprocess = one ProcessHandle
```

The engine may store, checkpoint, serialise, and route collections, but it does
not iterate their items to schedule per-item runs.

## 4. Collection Semantics

`Collection` is not a seventh data type in the base type hierarchy. It is a
transport wrapper around homogeneous `DataObject` instances. Its type identity
for port validation is the item type, not the wrapper class.

Required semantics:

- items are ordered;
- item type is immutable after construction;
- non-empty collections infer item type from the first item when omitted;
- empty collections require explicit `item_type`;
- all items must be instances of the item type;
- `Collection[Image]` annotation syntax is allowed for readability, but the
  runtime wrapper remains the same class;
- `storage_refs` exposes per-item storage references for persistence and
  serialization paths.

## 5. Port And Constraint Behavior

Ports declare item types. A port that accepts `Image` accepts a
`Collection[Image]`; it does not declare `Collection` as the accepted runtime
type. The wrapper is transparent to type compatibility.

Input constraints operate on the collection value. Constraint helpers that need
per-item validation iterate the collection and short-circuit on failure. This
keeps a constraint such as "every image has axes y/x" collection-aware without
forcing the scheduler to inspect item contents.

## 6. Removed Engine-Level Batch Concepts

ADR-020 supersedes ADR-010 and ADR-016. The following concepts are not part of
the runtime contract:

| Removed concept | Replacement |
|---|---|
| `BatchMode` enum | Blocks choose their own item strategy. |
| `BatchErrorStrategy` enum | Blocks decide whether to save partial outputs or fail. |
| `BatchResult` | Terminal block events and collection outputs carry run evidence. |
| `Block.batch_mode` | Block implementation uses `map_items`, `parallel_map`, or custom logic. |
| `BatchExecutor` | `DAGScheduler` schedules block nodes only. |
| `NodeDef.batch_mode` | Workflow nodes do not carry engine batch policy. |
| `LineageRecord.batch_info` | ADR-038 lineage records object and block-execution relationships. |
| `InputDelivery` as an engine handoff mode | CodeBlock and worker conversion handle boundary materialization. |

The deletion is architectural, not only cleanup: the scheduler must not grow a
new generic item iteration path under another name.

## 7. Boundary Conversion

### 7.1 Worker Boundary

Worker serialization preserves collections explicitly:

- incoming wire payloads with `_collection: true` reconstruct as
  `Collection(item_type=...)`;
- each item is reconstructed through typed DataObject serialization;
- output collections serialize to the same envelope with item payloads and
  `item_type`;
- checkpoint and lineage paths preserve the collection envelope rather than
  flattening it into untyped lists.

### 7.2 CodeBlock Boundary

CodeBlock keeps script authors away from framework transport details:

- a length-one collection becomes one native value loaded from the item;
- a multi-item collection becomes a `LazyList`;
- `LazyList` loads one item at a time during iteration or index access;
- explicit `LazyList.to_list()` materializes all items and warns for large
  item counts;
- outputs are repacked into collections before leaving the block.

This keeps old single-input scripts natural while making multi-item inputs
memory-safe by default.

### 7.3 AppBlock And External Tool Boundary

AppBlock exchange code may present a collection as a manifest, a directory, a
set of files, or a single whole-dataset argument, depending on the tool. The
engine does not decide that shape. External tool blocks own the exchange
contract because the appropriate format is domain-specific.

## 8. Block Authoring Model

Block authors get three levels of control:

| Pattern | Intended use |
|---|---|
| `unpack_single` | Blocks that require exactly one item. |
| `map_items` | Simple per-item sequential transforms. |
| `parallel_map` | Block-internal parallelism when the block author accepts memory and process-pool tradeoffs. |
| Custom collection logic | Whole-dataset tools, GPU batching, external apps, or domain-specific grouping. |

The framework does not promise per-item partial-result preservation on
subprocess cancellation. Blocks that require partial output durability should
write incrementally to storage and return a clear terminal state when possible.

## 9. Verification And Tooling Impact

Tests should cover:

- `Collection` construction, homogeneity, empty-item-type validation, indexing,
  iteration, representation, and `storage_refs`;
- port type checks against collection item types;
- block utility pack/unpack/map helpers;
- CodeBlock single-item native conversion and multi-item `LazyList` behavior;
- worker reconstruction and serialization of `_collection` wire payloads;
- checkpoint and lineage preservation of collection envelopes;
- built-in collection operations such as split and merge.

## 10. Consequences

- The scheduler is simpler: it schedules block nodes, not item batches.
- Process lifecycle stays simple: one block invocation has one process handle.
- Block authors get control over scientific iteration semantics.
- CodeBlock remains friendly to users who expect native Python values.
- Partial item-level recovery is no longer an engine guarantee.
- Collection support becomes a shared runtime contract across worker,
  checkpoint, lineage, ports, and block helper APIs.

## 11. Alternatives Considered

Keeping engine-level iteration with a `BatchHandling` flag was rejected because
it leaves the scheduler responsible for iteration semantics. Adding buffers
between parallel and serial stages was rejected because it creates a new
runtime component for a domain-specific policy. Fixing `ProcessRegistry` to
support one block id with many handles was rejected because it solves only the
handle collision and not the broader mismatch between generic batch modes and
scientific tool behavior.
