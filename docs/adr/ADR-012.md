# ADR-012: Checkpoint-Based Pause And Resume

## 1. Decision Summary

SciStudio persists checkpoint state so a workflow can pause, recover, or resume
without rerunning completed blocks.

### 1.1 Problems Addressed

| Problem | Risk | ADR-012 response | Detailed section |
|---|---|---|---|
| Workflows can run for hours or wait on users | Restarting from the beginning wastes time | Save checkpoint state after terminal block events | Section 3 |
| Checkpoints must stay lightweight | Copying data at every checkpoint is too expensive | Store references and workflow state, not payload copies | Section 4 |

## 2. Decision

The checkpoint subsystem records block states, intermediate output references,
pending interactive state, and config snapshots. Current checkpoint files live
under `<project>/.scistudio/pause/` and serve pause/resume plus latest-run "run
from here" behavior.

## 3. Scope

ADR-038 separates checkpoint state from immutable run lineage. Checkpoints are
single-slot latest-run artifacts; lineage records historical recipes but does
not preserve intermediate data for old runs.

## 5. Alternatives Considered

No checkpointing was rejected for expensive and interactive workflows. In-memory
state only was rejected because crashes lose it. Full data duplication was
rejected as storage-prohibitive.
