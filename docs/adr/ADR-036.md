# ADR-036: Embedded Code Editor For Project Files

## 1. Decision Summary

ADR-036 accepts an in-browser project-file editor for reading and editing
workflow-adjacent files such as `blocks/*.py`, scratch scripts, and configs,
with backend validation and reload-on-save hooks for block code.

### 1.1 Problems Addressed

| Problem | Risk | ADR-036 response | Detailed section |
|---|---|---|---|
| Users had to leave SciStudio to edit project files | Non-programmers face avoidable IDE and filesystem friction | Add a lightweight embedded code editor | Section 2 |
| Saving block files can affect runtime registries | Edits may not reload or may reload broken code | Route saves through lint and reload hooks | Section 2 |
| Project file access needs boundaries | The editor could read or write outside the project | Use backend project-file APIs for path resolution | Section 2 |

## ADR-036: Embedded code editor for project files

### 1. Purpose

Add a lightweight in-browser code editor to SciStudio so users can read and edit project-local files (`blocks/*.py`, scratch scripts, configs) without leaving the GUI. The existing main-tab abstraction extends to support two kinds of content per tab: a workflow **canvas** (current behaviour) or a **code editor** wired to a file.

Driving intuition: not all SciStudio users are programmers; making them open a separate IDE just to tweak a custom block parameter is friction we can remove.

---

### 2. Context

#### 2.1 Current state

- Frontend `TabState` (`frontend/src/store/types.ts`) is workflow-only: `workflowId`, `workflowNodes`, `workflowEdges`, `workflowHistory`, etc.
- `App.tsx` renders `<WorkflowCanvas .../>` hardcoded as the only tab content.
- `TabBar.tsx` is content-agnostic — uses `tab.workflowName` + dirty indicator.
- `ProjectTree.tsx` already recognises `.py` files in `blocks/` and triggers `reload_blocks`, but never opens them for editing.
- Backend has `filesystem.browse` for directory listing but no per-file R/W endpoints scoped to a project.
- An auto-save loop exists in `App.tsx:478-487` — watches `workflowDirty + workflowPayload`, debounces 800 ms, calls `saveWorkflow()`. The editor will reuse this pattern.

#### 2.2 What users want

- Double-click `blocks/my_filter.py` → edit in a new tab, Ctrl+S to save, palette refreshes when valid.
- Open `scratch.py` for ad-hoc analysis (no run — see §5.6).
- Open a `.yaml` workflow's **source** view side-by-side with its canvas (read-only).
- Syntax checking that matches what `ruff` (already a dev dep) reports.

---

### 3. Decisions

#### 3.1 Editor library — **Monaco Editor**

Bundle ~600 KB gzip. Heavier than CodeMirror 6 (~80 KB), but accepted for: native IntelliSense, real VSCode behaviour out of the box, mature React integration, future LSP path. Lazy-chunked the same way `xterm.js` is so cold-start of the workflow canvas is unaffected.

#### 3.2 File API surface

New backend routes (project-scoped, `_safe_under` path protection from `routes/filesystem.py`):

- `GET /api/projects/{project_id}/file?path=<rel>` → `{content: str, mtime: float, size: int, encoding: "utf-8"}`. 404 if missing, 403 if path escapes project root, 413 if file exceeds size cap, 415 if extension not in allowlist.
- `PUT /api/projects/{project_id}/file?path=<rel>` body `{content: str}` → `{mtime, size}`. Atomic write (tempfile + rename); coordinates self-write suppression with `workflow_watcher` so the user's own save doesn't echo as an external-change event.

**Allowlist (initial)**: `.py .txt .md .yaml .yml .json .csv .log`. Anything else returns 415 — explicit rejection rather than silently treating binary as text. Extending the list is a one-line change later if a use case emerges.

**Size cap (initial)**: 10 MB. Files larger than that are not editable; the editor shows a "file too large for in-browser editing" placeholder. The cap is a single constant; tune up if users hit it.

#### 3.3 Lint pipeline — **server-side `ruff`**

- `POST /api/lint/python` body `{content: str, filename: str}` → `{diagnostics: [{line, column, end_line, end_column, code, severity, message}]}`. Shells out to `ruff check --stdin-filename=<filename> -` and parses JSON output. Typical round-trip ~50 ms.
- Frontend debounces 600 ms idle before sending content for lint; Monaco renders `diagnostics` as squiggles via its model markers API.
- No client-side WASM linter (Pyodide). Backend is already serving the user; one more endpoint is cheaper than a 10 MB asset load on every editor mount.

#### 3.4 `.yaml` editor scope — **canvas-primary, read-only source-view as separate tab**

- `.yaml` files in `workflows/` always open in the canvas (current behaviour preserved).
- A "View source" affordance (toolbar button when a workflow tab is active) opens a **new read-only tab** whose `tabId` is `source:<workflow_id>` and whose display title is `<workflow_name> (source)`.
- Re-clicking "View source" focuses the existing source-view tab instead of opening a duplicate (the `source:` prefix in `tabId` is the dedup key).
- No two-way binding. Editing the source view is disabled. Users wanting to hand-edit a workflow YAML use an external editor or wait for a future "promote to editable" toggle (not in v1).

#### 3.5 Auto-reload on save for `blocks/*.py` — **gated on lint pass**

- File save → server checks the saved file's path. If it falls under `<project>/blocks/` and the file is `.py`, call the existing `reload_blocks` registry refresh.
- **But only if lint returned zero errors** for the saved content. A file with parse errors won't get loaded into the registry — keeps the block palette stable.
- Frontend shows a passive "Reloaded blocks" toast on success and a "Save succeeded; lint errors present — blocks not reloaded" toast on lint-fail.

#### 3.6 Run-from-editor — **NOT in v1**

Spawning a subprocess to execute an arbitrary `.py` from the editor introduces a new "off-graph code execution" semantic that's worth its own ADR (output capture, isolation, lineage, etc.). v1 is editor-only.

#### 3.7 Toolbar behaviour — **content-swap by `activeTab.kind`**

Top toolbar splits into a shared section + a kind-specific section:

| Button | `kind="workflow"` | `kind="file"` |
|---|---|---|
| New (see below) | shown | shown |
| Import | shown | shown |
| Save | shown (Ctrl+S → save workflow) | shown (Ctrl+S → save file) |
| Run / Pause / Stop / Reset / Reload | shown | **hidden** |
| Delete | shown (delete workflow) | **hidden** |
| Note / Group | shown (canvas-only) | **hidden** |
| Find / Format / View source / Goto-line | **hidden** | shown (v1: only Find via Ctrl+F; rest deferred) |

v1 simplification: file-tab toolbar shows only New / Import / Save; Find / Format / Goto-line work via Monaco built-in keybindings (`Ctrl+F`, `Shift+Alt+F`) without dedicated buttons. The kind-swap mechanism is wired so adding visible buttons later is one line of JSX.

**"New" button is constrained to a fixed menu** — to avoid users dumping arbitrary files into the project, the toolbar's "New" exposes exactly three options:

- **New workflow** — opens an empty `kind=workflow` tab; saves to `workflows/<name>.yaml`.
- **New custom block** — prompts for a filename, scaffolds `blocks/<name>.py` from the template (§3.12), opens it in a `kind=file` editor tab.
- **New note** — prompts for a filename, creates an empty `notes/<name>.md` (or `<name>.md` at project root if `notes/` doesn't exist), opens it in a `kind=file` editor tab with markdown highlighting. For project READMEs, design notes, scratch ideas, meeting summaries — content that wants to live next to the work.

No "New text file" / "New arbitrary YAML" / "New JSON" / "New CSV" entries. Users with a more general need (e.g. editing a `requirements.txt`) reach the file through the Project tree, which uses the file's existing extension to decide the editor language. The toolbar's bias is "guide the user toward one of the three file types that mean something to SciStudio."

#### 3.8 Agent / MCP exposure — **no new MCP tools**

- Claude Code TUI already has built-in Read / Write / Edit tools that operate on the filesystem directly — no new MCP work required.
- Codex users without native fs MCP tools fall back to Bash (`cat`, `printf > file`). Accepted friction; codex parity work tracked separately if it becomes a real complaint.
- Rationale: every new MCP write tool is a new schema-validation surface to maintain. `write_workflow` is justified because YAML workflows are a typed contract; arbitrary `.py` files are not, so a `write_file` MCP tool would just be a thin wrapper around `os.write` with no value-add over the agent's native Write tool.

#### 3.9 Unsaved-state UX — **no dirty-state recovery; rely on auto-save**

- Closing a dirty tab triggers the existing `ConfirmDialog` ("This tab has unsaved changes…"). Same as workflow tabs.
- Closing the browser / refreshing the page **does not** persist or recover dirty content. Users are responsible for their own data.
- However, **auto-save is enabled by default** — same 800 ms debounce that `App.tsx:478-487` already runs for workflow tabs. Edit a character → 800 ms idle → save fires. In practice losing more than 1 keystroke of work is impossible without backend downtime.
- `beforeunload` hint is not used (treated as too intrusive given auto-save is on).

#### 3.10 `TabState` shape — **discriminated union**

```ts
// types.ts
export interface WorkflowTab {
  kind: "workflow";
  id: string;
  workflowId: string;
  workflowName: string;
  workflowDescription: string;
  workflowVersion: string;
  workflowMetadata: Record<string, unknown>;
  workflowNodes: WorkflowNode[];
  workflowEdges: WorkflowEdge[];
  workflowDirty: boolean;
  workflowHistory: WorkflowHistoryEntry[];
  workflowFuture: WorkflowHistoryEntry[];
  selectedNodeId: string | null;
}

export interface FileTab {
  kind: "file";
  id: string;            // "file:<path>" or "source:<workflow_id>"
  filePath: string;      // project-relative
  displayName: string;   // basename, with "(source)" suffix for source views
  language: "python" | "yaml" | "json" | "text" | "markdown";
  content: string;
  contentLoadedAt: number;  // server mtime at last fetch
  dirty: boolean;
  readOnly: boolean;     // true for source-view tabs
}

export type TabState = WorkflowTab | FileTab;
```

Every read of `TabState` that previously assumed `workflowId` etc. must guard on `tab.kind === "workflow"`. The TypeScript exhaustiveness check makes this a one-pass conversion across `App.tsx` + components — annoying but bounded.

#### 3.11 Tab persistence — **path only, content re-fetched on restore**

`localStorage` (via Zustand `persist`) stores `{kind, id, filePath, displayName, language, readOnly}` for file tabs — **not `content`**. On rehydrate, file tabs are restored in a "loading" state and content is re-fetched from the backend. Workflow tabs continue to persist their structured payload.

If the file disappeared from disk between sessions, the tab is restored to a "this file is gone — close?" state rather than silently dropped. Avoids silent data loss when users move files around outside SciStudio.

#### 3.12 New-file action — **scaffolds a BlockBase template under `blocks/`**

The Project tree adds a "New .py file" affordance (right-click menu inside the `blocks/` folder; future: a top-level "New file" button). The action:

1. Prompts for a filename (default `my_block.py`, validated as a Python identifier-friendly stem).
2. Creates the file from a built-in **block template** that already wires up `BlockSpec`, named input/output ports, a minimal `config_schema`, and a `run()` method whose body is a `raise NotImplementedError("…")` placeholder.
3. Opens the new file in an editor tab focused on the placeholder body so the user can start typing immediately.

The template lives at `src/scistudio/blocks/_templates/block_base_template.py` (shipped with the package, served via a new `GET /api/blocks/template?kind=basic` endpoint). It includes:

- The exact import statements the project's `BlockRegistry` expects (`from scistudio.blocks.base import Block, BlockSpec, PortSpec`).
- A docstring block with one-line examples of `config_schema` JSON Schema shapes and how to reference `inputs[port_name]` inside `run`.
- A comment marker `# >>> EDIT THIS <<<` on the `run` body so users find the right line.

For the broader-scope case (a whole **package** of related blocks with `pyproject.toml`, tests, entry-point registration) the existing `scistudio init-block-package` CLI is the right tool — surfacing that from the GUI is a separate item, not part of this ADR.

For markdown notes from "New note", the file is created **empty** (no template — the user's first line is the title, doing template insertion would just be noise to delete). Markdown highlighting + preview support is provided by Monaco's built-in `markdown` language and matches the editor's behaviour for an existing `.md` opened from the Project tree.

Outside these three constrained creation paths, the editor only opens files that already exist (via Project tree double-click). It never offers to create arbitrary `.txt` / `.csv` / `.json` — those are project tree concerns.

### 4. Implementation plan

Each phase = one PR. No phase implements features from the next.

**Phase 1 — Store + types refactor** (frontend-only, ~80 LOC + tests)

- Convert `TabState` to discriminated union; add `kind: "workflow"` to all existing tab creations.
- Add `openFileTab(filePath: string, opts?: {readOnly?: boolean})` / `saveFileTab(id: string)` / `updateFileTabContent(id, content)` store actions.
- Update all read sites in App.tsx + tests to type-guard on `tab.kind`. No new UI yet.

**Phase 2 — Backend file R/W + lint** (backend-only, ~160 LOC + tests)

- `GET / PUT /api/projects/{project_id}/file` with allowlist + size cap + `_safe_under` path check.
- `POST /api/lint/python` calling `ruff check --stdin --output-format json`.
- Workflow_watcher self-write coordination so file PUTs don't echo back as external-change events.
- Tests: path traversal rejection, allowlist enforcement, size cap, lint diagnostic shape.

**Phase 3 — Editor component + Save UX** (frontend, ~300 LOC + tests)

- `CodeEditor.tsx` Monaco wrapper, lazy-imported. Props: `tab: FileTab`, `onContentChange`, `onSave`, diagnostics rendered via Monaco's `setModelMarkers`.
- Lint debounce (600 ms) + save debounce (800 ms, same as canvas) wired through the store.
- `App.tsx` content-area switch on `tab.kind`. Toolbar split per §3.7.
- `Ctrl+S` saves the active tab regardless of kind.

**Phase 4 — ProjectTree wire-up + View source** (frontend + small backend, ~80 LOC + tests)

- `ProjectTree.tsx` double-click on `.py / .txt / .md / .json / .csv` → `openFileTab(path)`.
- Workflow tab toolbar adds a "View source" button → `openFileTab("workflows/<id>.yaml", {readOnly: true})` with `id` prefixed `source:`.
- `blocks/*.py` save → server triggers `reload_blocks` (gated on lint pass per §3.5) → palette refreshes via existing block-registry-changed event.

**Total**: ~620 LOC + ~220 test LOC. Each phase ships green and is independently usable (you can stop after Phase 2 and use the lint endpoint via curl; you can stop after Phase 3 and lose only the convenience of double-click-to-open).

---

### 5. Out of scope (v1)

- Run-from-editor (executing `.py` from a tab) — separate ADR.
- New MCP `read_file` / `write_file` tools — agents use native fs.
- Two-way YAML ↔ canvas sync — source view is read-only.
- Multi-file workspace search (`Ctrl+Shift+F`).
- Vim / Emacs keybinding modes.
- Block authoring scaffolding (insert `BlockBase` skeleton on new `.py` in `blocks/`).
- Dirty-content recovery across page reloads — auto-save makes this near-impossible to need.
- IPynb / Jupyter notebook editing.
- Image / parquet / zarr preview in the editor surface (those have / will have dedicated preview panes).

---

### 6. Risks

| Risk | Mitigation |
|---|---|
| Monaco bundle inflates frontend cold-start. | Lazy-chunk; load only when first file tab opens. Match the existing xterm.js pattern. |
| `ruff` not installed in the env the backend runs from. | Fail soft: `POST /api/lint/python` returns `{diagnostics: [], note: "ruff unavailable"}` plus a backend WARN log. Editor renders without squiggles; saves still work. |
| Auto-save fires while user is mid-edit on a syntactically broken file → file on disk transiently broken. | Acceptable: the file was broken in the editor anyway; auto-save is just a snapshot. `reload_blocks` is gated on lint so the registry isn't poisoned. |
| Allowlist too strict; users want to edit `.toml` / `.ini`. | One-line patch to the allowlist constant; ship v1 with the conservative set, expand on first complaint. |
| Size cap too low for some users' YAMLs. | Same — single constant. 10 MB chosen because Monaco's perf degrades on larger files anyway. |

## Addendum 1 (2026-05-22) — Editor visual theme: soft-dark island

**Status**: accepted

**What changed**: The Monaco editor pane uses a custom soft-dark theme
`scistudio-soft-dark` instead of the built-in light `vs` theme. The rest
of the SciStudio UI shell (project tree, toolbar, side panels, bottom
panels) keeps its cream/light palette.

| Surface | Before | After |
|---|---|---|
| `CodeEditor.tsx` `theme` prop | `"vs"` (Visual Studio light) | `"scistudio-soft-dark"` |
| Editor background | `#ffffff` | `#282c34` (One Dark warm gray) |
| Outer UI shell | light cream `#f5f1e8` | unchanged |

**Why**: Owner feedback that the white-background editor "feels 15 years
old" and is visually disconnected from the rest of the workflow surface.
The two off-the-shelf alternatives evaluated:

1. **Switch entire UI to dark mode** — requires a token-system audit and
   dark variants for every component; design surface too large for a
   single PR. Punted to a future ADR if the demand materializes.
2. **Pure `vs-dark` (`#1e1e1e`) editor island** — too harsh against the
   surrounding cream UI on owner inspection.

The accepted direction is **option 3, the editor-island pattern with a
softened warm-dark palette**: the editor pane carries its own dark theme
(the same visual metaphor GitHub / Notion / Linear use for embedded code
blocks), warmed to `#282c34` so the contrast against the cream shell
reads as "this is a code surface" rather than "this is broken".

**Implementation**:
- `frontend/src/components/CodeEditor.tsx` defines theme
  `scistudio-soft-dark` via a `beforeMount` hook:
  - `base: "vs-dark"`, `inherit: true` — inherits all syntax token
    colors from vs-dark unchanged (markdown headers, strings, keywords,
    etc.); no custom token rules.
  - Overrides only the chrome colors via `colors`:
    `editor.background` `#282c34`, `editorLineNumber.foreground`
    `#4b5263`, `editor.lineHighlightBackground` `#2c313a`,
    `editor.selectionBackground` `#3e4452`, etc.
- The Editor's `theme` prop is set to `"scistudio-soft-dark"`.

**Out of scope** (tracked separately if needed):
- A user-toggleable light/dark theme switch (would require persistence,
  a token system audit, and is a separate ADR-scale feature).
- Dark-mode skinning of the surrounding shell, project tree, side
  panels, or bottom panels — the editor island is intentional.
- Custom syntax token color rules; vs-dark's inherited palette is the
  baseline.

**Test**: `frontend/src/components/__tests__/CodeEditor.test.tsx`
asserts the theme prop is `scistudio-soft-dark`, that `defineTheme`
registers it with `base: "vs-dark"`, `inherit: true`, and
`editor.background: "#282c34"`.
