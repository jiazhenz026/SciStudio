# ADR-052: Public API Contract And Stability Policy

## 1. Decision Summary

SciStudio asks users to write code. They write custom blocks, custom plots,
custom data types, and previewers; they publish those as installable packages.
The embedded AI agent does the same on their behalf. Yet there is no statement
anywhere of which symbols those authors may rely on, which ones may change
without warning, and which version a symbol first appeared in. The public/private
boundary today is the underscore convention and nothing else, and there is no
pipeline that turns the code's own docstrings into reference documentation an
author can read.

The cost is visible in the existing domain packages. A block author who needs to
build a `DataFrame` from rows, or coerce inputs to a domain type, finds no
documented helper and reaches into a package's private `_support` module
(`scistudio-blocks-spectroscopy`). An author who needs to iterate an image over
its axes imports `scistudio.utils.axis_iter`, a protected internal layer; one who
needs MIME guessing or a process adapter imports `_guess_mime` and
`_PopenProcessAdapter` outright (`scistudio-blocks-imaging`). None of this is
malice — it is the absence of a public surface to aim at. Heading into alpha,
that absence becomes a promise we cannot keep: anything an author imports today
can break in any release, because nothing was ever declared stable.

This ADR defines the **public API contract**: which surfaces are public, how a
symbol's stability and first-available version are declared in the code itself,
how author-facing reference docs are generated from those declarations and
versioned, and the symmetric rules a package must follow to expose its own public
API. It does **not** rewrite any internal; it draws the boundary and sets the
policy. The actual `__all__` declarations, the stability module, the doc
pipeline, and the package migrations are later phases tracked by #1817, and the
existing reach-through paths keep working until each has a public replacement.

The ADR turns on five decisions, each addressed by a later section: a public
symbol is one reachable through a **canonical root import path**, not a deep
module path (Section 2); the **core** exposes a named author surface — including
a public-only ergonomic accessor layer so authors read pandas/numpy instead of
raw Arrow (Section 3); a **package** exposes its own surface under the same rules
(Section 4); every public symbol carries a machine-readable **stability tier and
`Since` version** through a decorator (Section 5); and today's reach-through into
internals is **classified, not presumed generic**, and resolved case by case
(Section 6). Documentation generation and versioning follow from these
(Section 7).

### 1.1 Problems Addressed

| Problem | Risk | ADR response | Detailed section |
|---|---|---|---|
| There is no statement of which symbols an author may import and rely on | Any author/AI import can break in any release; alpha ships a promise we cannot keep | Define a public boundary anchored on canonical root import paths | Section 2 |
| Block/plot/type/previewer authors have no documented core surface to aim at | Authors guess, copy, or reach into internals; the surface drifts silently | Name the core public surface and its canonical import paths | Section 3 |
| Tabular `to_memory()` returns Arrow, which authors who are fluent in pandas/numpy (most custom-block authors) find unusable, so each package reinvents a `to_pandas` | Authors convert by hand ad hoc, or copy a divergent helper per package; the boundary fragments | Add public-only ergonomic accessors (`to_pandas`/`to_numpy`) on core types, inherited by package types | Section 3 |
| Packages have no rule for exposing their own public API; domain helpers hide behind `_support` | Users reach into another package's privates; each package reinvents the boundary | Define a symmetric package public contract | Section 4 |
| Stability and first-available version are nowhere recorded, and hand-written docs drift from code | Authors cannot tell stable from experimental; deprecation has no defined path | Declare tier + `Since` in code via decorators; define deprecation policy | Section 5 |
| Authors reach into core internals (`utils._*`, `_guess_mime`, `_PopenProcessAdapter`, previewer internals) | Removing or refactoring internals silently breaks installed packages | Classify reach-through into three kinds and provide public homes case by case | Section 6 |
| There is no docstring→reference-docs pipeline and no doc versioning | Reference docs are hand-maintained, incomplete, and unversioned | Generate reference docs from docstrings + decorators; version the contract | Section 7 |

## 2. What Counts As Public

A symbol is public when it is reachable through a **canonical root import path**
that this contract names. Everything else — deeper module paths, underscore-named
modules, and underscore-named members — is internal, carries no stability
promise, and does not appear in generated documentation.

This is the load-bearing decision, because today the packages import from deep
paths such as `scistudio.core.types.dataframe`, `scistudio.blocks.base.ports`,
and `scistudio.previewers.models`. Deep paths couple the author to the internal
file layout: moving a class between files, splitting a module, or renaming a file
breaks author code even when the class itself never changed. A contract whose
whole purpose is to let internals evolve cannot be expressed in terms of internal
file structure. The public path must therefore be the curated root the contract
controls, not the file a symbol happens to live in.

Concretely:

- **Public** is `from scistudio.core.types import DataFrame`,
  `from scistudio.blocks.base import InputPort, ProcessBlock`, and the other root
  paths named in Sections 3 and 4. Each such root declares an explicit `__all__`;
  membership in that `__all__` is what makes a symbol public.
- **Internal** is `from scistudio.core.types.dataframe import DataFrame`, any
  `scistudio.*._private` module, and any `_name` member. These remain importable
  — this ADR breaks nothing — but they are unsupported and excluded from docs.

The contract is declared with `__all__` lists on the root modules, not with a new
re-export façade module (for example a single `scistudio.api` or `scistudio.sdk`
package). A façade was considered and rejected; see Section 10. `__all__` keeps
the public path identical to today's recommended imports for most symbols and
leaves every template and package import working, which a façade would not.

## 3. The Core Public Surface

The core exposes one author surface, grouped by what an author is doing. The
exhaustive per-symbol `__all__` is produced in the implementation phase; this
section names the surface and its canonical roots so the boundary is unambiguous.

- **Core data types** — `scistudio.core.types`: `DataObject`, `Array`,
  `DataFrame`, `Series`, `Text`, `Artifact`, `CompositeData`, `Collection`,
  `StorageReference`, `TypeSignature`, and the metadata facilities an author
  needs to subclass a type (`with_meta` updates, the `Meta` model pattern).
- **Block authoring** — `scistudio.blocks.base` for the shared contract (`Block`,
  `BlockConfig`, `InputPort`, `OutputPort`, `ExecutionMode`, `PackageInfo`, and
  the interactive facilities `InteractiveMixin`, `InteractivePrompt`,
  `PanelManifest` from ADR-051), and the concrete base classes an author
  subclasses from their packages: `ProcessBlock` (`scistudio.blocks.process`),
  `IOBlock` and the simple loader/saver helpers (`scistudio.blocks.io`),
  `AppBlock` and its file-exchange/watcher facilities (`scistudio.blocks.app`), and
  `CodeBlock` for packaged Python/R/Julia scripts (`scistudio.blocks.code`).
  `AIBlock` and `SubWorkflowBlock` are deliberately **not** part of this surface:
  they are runtime base classes the engine and the embedded agent compose, not an
  author extension point, so they remain internal and carry no stability promise.
- **Previewer authoring** — `scistudio.previewers.models`: `PreviewerSpec`,
  `FrontendManifest`, the owner-kind and API-version constants, and the
  preview-error reporting types, per ADR-048.
- **Plot authoring** — the import-free `render(collection)` contract. A plot
  script imports nothing from `scistudio`; its public surface is the shape of the
  `collection` object the harness passes (`collection.types`,
  `collection.items.open()/open_one()`, `item.type/metadata/open()`) and the
  return contract (a figure object or artifact path). This is a small surface but
  a real one, and the contract documents it alongside the import-based surfaces as
  one supported authoring usage.

Symbols an author legitimately needs that live only behind internal paths today
(for example axis iteration, used heavily by the imaging package) are handled
under Section 6, not silently blessed here.

### 3.1 Ergonomic Accessors (Public-Only)

The canonical in-memory form of a type — what `to_memory()` returns — is chosen
for the type's storage and cross-language role, not for an author's comfort
(ADR-031 settled this): `Array` returns a numpy `ndarray`, but
`DataFrame` and `Series` return a `pyarrow.Table`, because Arrow is the
zero-copy, cross-language (Python↔R) interchange and Parquet-backed storage form.
The cost is that a `pyarrow.Table` is awkward for authors who think in pandas or
numpy — which is most custom-block authors. Leaving each package to add its own
`to_pandas` produces N divergent helpers and is exactly the fragmentation this
contract exists to prevent. Core therefore owns one ergonomic accessor layer.

These accessors are **additive and public-only**. They wrap `to_memory()` and
never replace it; the canonical form and the `data=` constructor from ADR-031
are unchanged. This extends ADR-031's thin-type model with a convenience layer for
authors; it does not reopen the storage/interchange decision.

These accessors are **read-only**, and the asymmetry is deliberate: core adds an
ergonomic way to read a type *out* to pandas/numpy, but **not** a mirror way to
construct a type *from* pandas/numpy. Reading out is generic and unambiguous, so
core can own it; construction needs domain metadata and conventions (units, axis
names, ids, which column is which) that core does not have, so it belongs to the
package that owns the type (Section 4.2).

| Type | Canonical `to_memory()` | Ergonomic accessor(s) to add |
|---|---|---|
| `Array` | numpy `ndarray` (already ergonomic) | `to_numpy()` — explicit, discoverable alias of the inherited reader |
| `DataFrame` | `pyarrow.Table` | `to_pandas() -> pandas.DataFrame`; `to_numpy() -> ndarray` |
| `Series` | `pyarrow.Table` (one column) | `to_pandas() -> pandas.Series`; `to_numpy() -> ndarray` |
| `Text` | `str` | none — already ergonomic |
| `Artifact` | `pathlib.Path` | none — already ergonomic |
| `CompositeData` | `dict[str, native]` | none — already ergonomic |

`pandas` and `numpy` are already core runtime dependencies, so these add no new
dependency. The accessors are marked `@stable` (Section 5) from the core version
that ships them.

Two rules keep this layer from leaking into the runtime:

- **Core internal data flow must not use the ergonomic accessors.** Loaders,
  savers, auto-flush, worker serialization/reconstruction, checkpointing, the
  scheduler/engine, and previewers' bounded reads operate on the canonical form
  via `to_memory()` (Arrow/`ndarray`) only. The accessors exist for author code
  (custom blocks, plots, scripts), never for the workflow data path. A guard or
  audit enforces this (Section 8).
- **Packages inherit, never redefine.** A package type subclasses a core type and
  inherits these accessors; it MUST NOT define its own `to_pandas`/`to_numpy`.
  This is what removes the per-package divergence above.

**One sanctioned exception.** The Excel (`.xlsx`) reader/writer (#1810, PR #1815) must
use pandas internally, because `pyarrow` cannot read or write Excel. It is the single
core data-flow component allowed to depend on pandas as a *format reader/writer*; it still
returns a canonical Arrow-backed `DataObject` to the rest of the pipeline, so the
rule "the data path carries the canonical form" holds downstream.

### 3.2 Large-Data Access

Scientific datasets routinely exceed memory — a single mass-spec image can be
100 GB — so the canonical-form accessors of Section 3.1, which materialize the
whole object, are not enough. The contract therefore also exposes methods that
read and write data without materializing it whole, so an author handling such
data has a supported path instead of OOM or reaching into internals.

The exposed large-data methods (all `@stable`):

| Method | Type | Semantics |
|---|---|---|
| `sel(**axes)` | `Array` | partial read selecting along named axes, served by the Zarr backend without full materialization |
| `slice(...)` | `DataObject` | backend partial read: array sub-region (Zarr) / row range (Arrow) / byte range (filesystem) |
| `iter_chunks(chunk_size)` | `DataObject` | backend streaming read: array chunks along axis 0 (Zarr) / Parquet row batches (Arrow) / fixed-size byte chunks (filesystem) |
| `persist_array(...)` | `Block` | streaming array write to the Zarr backend |
| `persist_table(...)` | `Block` | streaming table write to the Arrow/Parquet backend |

ADR-007 (lazy by default) and ADR-031 (reference-only `DataObject`) govern this
surface.

## 4. The Package Public Surface

A package exposes to two different audiences, and conflating them is what left the
boundary unclear. It exposes a **registration surface to core** — the block,
type, and previewer classes core needs to populate the palette, the type system,
and the preview router — and it exposes a **reuse surface to other authors** — the
symbols a user, or the AI agent on their behalf, imports when writing their own
block, plot, or script against the package. The reuse surface follows the same
rules as core's, so an author writing against any installed package meets one
boundary rather than several ad-hoc ones.

### 4.1 Registration Surface (To Core)

Unchanged from ADR-025 and already checked by the shared
`scripts/validate_contract.py`: three entry-point callables —
`scistudio.blocks` → `(PackageInfo, [Block, ...])`, `scistudio.types` →
`[type, ...]`, and `scistudio.previewers` → `[PreviewerSpec, ...]`. This is how
core discovers a package's blocks, types, and previewers. It is a handshake with
the runtime, not an author-facing import surface.

### 4.2 Reuse Surface (To Authors)

What an author imports from a package, in order of importance:

- **Types — required.** Every type a package's blocks consume or produce is
  public, so an author can name it on a port (`accepted_types=[Spectrum]`) and
  build it. The canonical import path is the package's top level
  (`from scistudio_blocks_spectroscopy import Spectrum`), not a deep module path,
  for the reason in Section 2. A public type is hollow unless the author can also
  construct and read it, which is the next point.
- **Construction and reading — on the type, following the core idiom.** Packages
  build their types the way core builds its own (Section 3) rather than inventing
  a parallel mechanism. Core has no helper classes and no `from_*` factory
  functions: it constructs through a rich keyword-only constructor that takes the
  in-memory payload via `data=`, and it reads through instance methods
  (`to_memory()`, `sel()`, `with_meta()`), plus the ergonomic accessors added in
  Section 3.1 (`to_pandas()`, `to_numpy()`). A package type inherits all of this. `Spectrum`
  subclasses the core `Series`, so it already inherits `to_memory()` and the
  Section 3.1 accessors and needs no new reader. Construction is the asymmetric
  half (Section 3.1): core deliberately does not add domain construction, so a
  package **SHOULD** provide a public constructor on the type that accepts
  domain-native inputs and packs to the canonical form internally — moving today's
  private `_support.build_spectrum(...)` onto `Spectrum` (illustrative; not yet
  implemented). If a package does not, the custom-block author must construct via
  the canonical `data=` constructor by hand — building the Arrow payload and
  setting metadata themselves — which is exactly the friction the package is
  expected to remove.

  Reading is solved at the core layer (Section 3.1), not per package: the "give me
  an ndarray/pandas, not Arrow" need that `_support.spectrum_arrays(...)` exists to
  serve is met by the inherited `to_numpy()`/`to_pandas()`. A package MUST NOT
  define its own `to_pandas`/`to_numpy`. It MAY still add a domain-named reader
  that returns named components, but built on the inherited accessors, never on
  Arrow internals or `_transient_data`.

  **A separate helper class is not required** — construction and reading belong on
  the type (above), which keeps the type self-contained. But a package is
  **encouraged** to add public helpers wherever they make an author's life easier:
  the more a package exposes, well-named and documented, the more comfortable its
  users are. Cross-type or coerce/validate utilities — such as the existing
  `coerce_spectra`, which accepts a Collection and normalizes it — are a natural
  home for a public helper module or class. The only hard rules are that such
  helpers carry no leading underscore (they are public, Section 4.3) and do not
  shadow the inherited `to_pandas`/`to_numpy`.
- **Blocks — not author-facing by default.** Blocks are composed in the workflow
  graph, not instantiated in author code, so a package's block classes are exposed
  to core for registration but are not part of the reuse surface. A package MAY
  publish a block class for programmatic reuse, but only as an opt-in marked
  `stable`; the default is that authors interoperate with a package's blocks
  through the types on their ports, not by importing the block. Logic worth sharing
  between blocks belongs in a public helper or on a type, not in an imported block
  class that carries engine lifecycle.
- **Previewers — not author-facing.** A previewer is registered to core and
  invoked by the preview router; authors do not import it. The protocol an author
  needs to *write* a previewer is core's surface (`scistudio.previewers.models`,
  Section 3), not the package's.

### 4.3 Cross-Cutting Package Rules

- **`__all__` on everything public.** The package top level and any public
  submodule declare `__all__`; membership defines the reuse surface, exactly as
  for core (Section 2).
- **No underscore-named author-facing helper.** Anything an author is expected to
  call has a public home — a type method or a public function. `_support`-style
  modules are legitimate only for genuinely package-internal code no author should
  call. This is the direct fix for the gap in #1817.
- **Same stability policy.** Public package symbols carry the Section 5 decorators
  and follow the same `Since`/deprecation rules, versioned against the package's
  own version line.

### 4.4 Discovery

So an author or the AI agent can find a package's public types and helpers without
reading its source — the failure mode behind #1817 — the contract calls for a
public discovery surface: a way to enumerate installed packages and the
types/helpers each exports. The mechanism (a `scistudio.plugins`-style API and/or
an MCP tool) is specified in the implementation phase; this ADR fixes only that it
must be public and author-facing.

The reference repositories for this surface are `scistudio-package-template` (the
canonical layout), `scistudio-blocks-spectroscopy`, and
`scistudio-blocks-imaging`.

## 5. Stability Tiers, `Since`, And Deprecation

Every public symbol carries two facts an author needs and cannot infer: how much
they may rely on it, and when it became available. The contract records both **in
the code, on the symbol**, rather than in a separate hand-maintained table that
would drift the moment a symbol changed.

The mechanism is a small set of decorators in a new `scistudio.stability` module —
decorators only, not a re-export surface, so it does not reopen the façade
question from Section 2. They are no-ops at runtime: each attaches metadata (the
tier and the `Since` version) to the symbol and returns it unchanged, so they add
no runtime cost and change no behavior.

```python
from scistudio.stability import stable, provisional, internal

@stable(since="0.3.0")
class ProcessBlock(Block):
    ...

@provisional(since="0.3.0")
def new_helper(...):
    ...
```

The three tiers:

- **`stable`** — supported. Will not change incompatibly within a major version;
  removal or breaking change requires deprecation first.
- **`provisional`** — usable but still settling. May change in a minor release;
  changes are called out in the changelog.
- **`internal`** — no promise. May change or vanish in any release without
  deprecation. Internal symbols are not part of the contract and are excluded
  from generated docs; the tier exists so a symbol that must remain importable
  for compatibility can still be labelled honestly.

`Since` records the version a symbol first became public on its surface (core's
version for core symbols, the package's version for package symbols).

**Deprecation policy.** A `stable` symbol is removed only after it has been
marked deprecated for at least one minor release within the same major line, with
a changelog entry and, where practical, a runtime `DeprecationWarning` pointing
at the replacement. A `provisional` symbol may change in a minor release without
this guarantee but with a changelog entry. `internal` symbols carry no policy.

These decorators are author-facing too: a package marks its own public symbols
with the same tiers and the same policy, against its own version line (Section 4).

## 6. Handling Today's Internal Reach-Through

The packages already reach into internals, and this ADR deliberately does **not**
presume which of those reaches should become public core API. Genericity is
asserted by evidence, not by appearance: a helper that looks general
(`dataframe_from_rows`) may carry domain assumptions, and one that looks like an
implementation detail (`axis_iter`) may be genuinely cross-cutting. Each reach is
therefore classified, and the disposition of each is decided case by case under
the #1817 migration, not decreed here.

- **(a) Core-internal reach-through** — author code importing core internals:
  `scistudio.utils.axis_iter`, `scistudio.utils.constraints.has_axes`,
  `scistudio.blocks.app.bridge._guess_mime`,
  `scistudio.blocks.app.app_block._PopenProcessAdapter`, previewer internals such
  as `scistudio.previewers.data_access`. These signal an incomplete core public
  surface (notably AppBlock authoring and axis iteration). The disposition is to
  give each a public home or a public alternative, evaluated individually. Axis
  iteration is already tracked for relocation into core by #1729 and is handled
  there.
- **(b) Package-owned domain helpers** — helpers like `build_spectrum`,
  `spectrum_arrays`, `coerce_spectra`. These belong to their package. The
  disposition is for the package to expose them publicly (Section 4), not to pull
  them into core.
- **(c) "Looks generic" builders** — helpers like `dataframe_from_rows`,
  `dataframe_from_pandas`, `dataframe_collection` that several packages
  re-implement in `_support`. The disposition is **not presumed**: such a helper
  is promoted to a core public helper only when it is shown to be used identically
  across packages with no domain assumptions; otherwise it stays in the package's
  public surface. Each candidate is evaluated and recorded under #1817.

Crucially, none of these paths break when this ADR lands. They remain importable;
each is migrated only once its public replacement exists, so author code has a
continuous path forward rather than a flag day. The existing `_support` modules
and deep imports are retired per surface, tracked by #1817, never wholesale.

## 7. Generated Documentation And Versioning

Author-facing API reference is **generated from docstrings and the stability
decorators**, so the reference cannot drift from the code and the stability tiers
render automatically rather than being transcribed by hand.

- **Toolchain.** `mkdocstrings` with the `griffe` collector. `griffe` can read
  the `scistudio.stability` metadata off each symbol, so the generator renders a
  tier badge and the `Since` version next to each entry without any manual table.
  `mkdocstrings` was chosen over `pdoc` because of this decorator-driven rendering
  and because its versioning story (via `mike`) aligns with the policy below; the
  trade-off is more configuration than `pdoc`'s zero-config default.
- **What is generated.** Only the public surface (Sections 3 and 4): symbols in a
  declared `__all__`. `internal` symbols are excluded even if a docstring exists.
  Generated reference lives under the repository's generated-docs path and is not
  hand-edited, consistent with ADR-042's generated-doc rule.

Two distinct things are versioned, and the distinction matters:

- **The contract is a hard, versioned promise.** Each public symbol's tier and
  `Since`, and the stability guarantees in Section 5, are versioned commitments,
  not presentation. This is the part that actually protects authors across
  releases.
- **The generated documentation is version-stamped, single-version for alpha.**
  Each generated reference set is stamped with the core (or package) version it
  came from. Because the reference is generated from the shipped snapshot,
  regenerating it per release is automatic and the docs a user sees match their
  installed version. Hosting multiple browsable versions side by side (via `mike`)
  is deferred until more than one version line is supported; the policy is fixed
  here, the multi-version machinery is a tracked follow-up.

```text
# TODO(#1817): multi-version published reference docs (mike) once >1 core line is supported.
#   Out of scope per ADR-052 §7 (single-version generated docs for alpha).
#   Followup: https://github.com/jiazhenz026/SciStudio/issues/1817
```

Each domain package generates and version-stamps its own reference the same way,
against its own version line.

## 8. Verification And Tooling Impact

This ADR is a planning decision; its verification obligations land with the
implementation phases, which are tracked by #1817 and will carry their own tests.
The expected tooling impact this ADR commits to:

- A `scistudio.stability` module with `stable`/`provisional`/`internal`
  decorators, unit-tested for no-op runtime behavior and correct metadata
  attachment (`tests/stability/test_stability.py`).
- Explicit `__all__` declarations on the core public roots in Section 3, with a
  test that asserts the declared public surface and that no `internal` symbol
  leaks into it (`tests/api/test_public_surface.py`).
- An `mkdocstrings`/`griffe` documentation build wired into the existing docs
  generation, emitting only the declared public surface with rendered tier/`Since`
  badges.
- An extension of the shared `scripts/validate_contract.py` to check the
  package-side rules in Section 4 (public `__all__` present; no underscore-named
  author-facing helpers in the documented surface; no package-defined
  `to_pandas`/`to_numpy` that shadows the inherited core accessor).
- The Section 3.1 ergonomic accessors (`Array.to_numpy`, `DataFrame.to_pandas`,
  `DataFrame.to_numpy`, `Series.to_pandas`, `Series.to_numpy`), unit-tested for
  correct round-trips from the canonical form
  (`tests/api/test_ergonomic_accessors.py`).
- A guard or audit asserting the ergonomic accessors do not appear in the core
  data-flow path (loaders, savers, auto-flush, worker serialization, checkpoint,
  scheduler/engine, previewer bounded reads), with the Excel `.xlsx` reader/writer
  (#1810, PR #1815) recorded as the single sanctioned pandas-using exception.

No protected-core behavior changes in this ADR's own change set, which is the ADR
document only.

## 9. Consequences

- Authors and the AI agent get one boundary to aim at: import from the canonical
  root, rely on `stable`, read the generated reference. The `_support`
  reach-through has a defined path to extinction rather than being load-bearing.
- The boundary is expressed as `__all__` on existing roots, so today's
  recommended imports keep working and most author code needs no change. Deep-path
  and underscore imports keep working too, now explicitly unsupported, and migrate
  per surface under #1817.
- Core gains the freedom to refactor anything not in a public `__all__` without
  breaking authors — the freedom the contract exists to create.
- The cost is ongoing discipline: every new public symbol must be placed in an
  `__all__` and decorated with a tier and `Since`, and the package template and
  domain packages must adopt the same rules. The shared contract validator and the
  public-surface test enforce this so it does not erode.

## 10. Alternatives Considered

- **A single re-export façade (`scistudio.api` / `scistudio.sdk`).** A dedicated
  package re-exporting the whole public surface gives one obvious import root.
  Rejected: it forces a churn of every template, every domain package, and every
  existing author's imports onto a new path, and it creates a second source of
  truth (the façade vs. each symbol's home) that can fall out of sync. `__all__`
  on the existing roots achieves the same boundary while keeping current imports
  valid.
- **A hand-maintained stability/version table in docs.** Listing every public
  symbol with its tier and `Since` in a Markdown table. Rejected: it drifts from
  the code the instant a symbol changes — the exact failure mode this ADR exists
  to remove. Per-symbol decorators keep the facts on the code and let the
  generator aggregate them.
- **`pdoc` for generation.** Zero-config and fast to stand up. Rejected as the
  primary choice because rendering stability badges from decorator metadata and
  supporting multi-version publishing are both weaker than `mkdocstrings`+`griffe`;
  the small alpha-stage config cost is worth the alignment with the versioning
  policy.
- **Presuming the generic `_support` builders into core now.** Tempting because
  several packages duplicate them. Rejected: genericity is unproven and asserting
  it risks freezing domain assumptions into the core public API. Section 6 gates
  promotion on evidence instead.
- **Keeping core types thin and leaving Arrow→pandas/numpy conversion to the
  author** (the strict reading of ADR-031's posture). Rejected for the author
  audience: most custom-block authors are fluent in pandas/numpy, not Arrow, and
  leaving the conversion unowned produces a divergent `to_pandas` in every package.
  Section 3.1 instead adds one public-only accessor layer on the core types, kept
  out of the internal data path by guard, which extends ADR-031 additively rather
  than reopening the storage/interchange choice.
- **A reading helper that returns numpy arrays as the canonical reader** (an
  earlier framing where a Series-derived type's reader returned component arrays).
  Rejected: it conflates the canonical `to_memory()` form (Section 3.1) with a
  convenience. `to_memory()` stays the canonical reader; `to_numpy()`/`to_pandas()`
  are additive accessors; a domain reader returning named components is optional
  and builds on them.
