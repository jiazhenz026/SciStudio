# ADR-022: OS-Level Memory Monitoring Replaces Estimated Memory Budgets

## 1. Decision Summary

SciEasy resource scheduling does not use static per-block memory estimates.
`ResourceManager` gates dispatch using live operating-system memory usage while
continuing to count discrete CPU and GPU resources from declarations.

This ADR preserves the 2026-04-04 decision. Later ADR-027 refinements extend
CPU accounting and GPU slot detection, but the ADR-022 memory boundary remains:
system RAM is monitored at the OS level and is not reserved or released in a
ledger.

### 1.1 Problems Addressed

| Problem | Risk | ADR-022 response | Detailed section |
|---|---|---|---|
| Static `estimated_memory_gb` cannot predict collection-sized workloads | The scheduler can be too conservative for flushed work or too aggressive for huge collections | Remove memory estimates from `ResourceRequest` and monitor OS memory | Section 3 |
| Memory controls exist inside blocks | `ResourceManager` cannot know `LazyList`, auto-flush, or internal batching behavior | Treat block-internal memory control as a separate layer | Section 4 |
| GPU and CPU are not the same kind of resource as RAM | Runtime could pretend to monitor VRAM portably when it cannot | Keep declared counting for CPU/GPU while monitoring system memory | Section 5 |
| Resource contracts need audit evidence | ADR text could drift from the active scheduler API | Govern `ResourceRequest`, `ResourceSnapshot`, and `ResourceManager` | Section 8 |

## 2. Context

The previous model asked blocks to declare `estimated_memory_gb`. That model
fit a simpler per-item batch architecture where the engine controlled
concurrency and could guess the memory footprint of each subprocess.

ADR-020 changed the runtime shape. A block may receive a collection of 10 items
or 10,000 items. It may process one item at a time, use `LazyList`, flush
outputs as it goes, call a parallel library, or load a whole collection into an
external application. A static class-level memory number cannot represent those
runtime choices.

## 3. Core Decision

`ResourceRequest` no longer carries `estimated_memory_gb`. System memory is not
reserved on acquisition and not released on terminal events. Instead,
`ResourceManager.can_dispatch()` reads live memory pressure through
`psutil.virtual_memory().percent` and pauses new dispatch when the high
watermark is exceeded.

The manager still tracks:

- GPU slot demand, because GPU availability is discrete;
- GPU memory hints, because cross-platform VRAM monitoring is not reliable;
- CPU core demand, including later ADR-027 internal-worker refinement.

## 4. Three-Layer Defense

Layer 1 is `ResourceManager` dispatch gating. It blocks new work when discrete
resources are exhausted or system memory is above threshold.

Layer 2 is block-internal memory control. `LazyList`, auto-flush, and
block-specific iteration policies decide how much data one block process loads
at a time.

Layer 3 is operating-system process isolation. If a subprocess still exhausts
memory, the OS may terminate it; `ProcessMonitor` and scheduler terminal-state
handling keep the engine process alive.

## 5. GPU Boundary

System RAM can be monitored portably with `psutil`. GPU VRAM cannot be
monitored consistently across NVIDIA, AMD, Apple Silicon, and CPU-only
systems. SciEasy therefore keeps GPU handling declaration-based. A block may
declare `requires_gpu` and `gpu_memory_gb`; the scheduler counts slots rather
than treating VRAM as a universal live metric.

## 6. Scope

In scope:

- removing static system RAM estimates from resource requests;
- dispatch gating by OS memory watermarks;
- GPU and CPU allocation counters;
- terminal-state release for discrete resources only.

Out of scope:

- per-subprocess memory polling and kill limits;
- cross-vendor VRAM monitoring;
- block-specific memory prediction APIs;
- moving block-internal flush or batching policies into `ResourceManager`.

## 7. Compatibility Notes

ADR-017, ADR-018, ADR-019, and ADR-020 continue to apply. `ProcessHandle`
records the resource request for release of discrete resources. Event-driven
terminal handling still calls `release()`, but the release operation does not
attempt to free memory; memory returns to the OS when the subprocess exits.

ADR-027 later adds `max_internal_workers`, `effective_cpu`, and GPU
auto-detection. Those additions extend CPU/GPU behavior without changing the
ADR-022 memory decision.

## 8. Verification And Tooling Impact

Tests should mock `psutil.virtual_memory()` at low, high-watermark, and
critical thresholds; verify CPU and GPU slot counting; verify acquisition and
release counters; verify no `estimated_memory_gb` field exists; and verify
available snapshots report live memory percentage.

### 8.1 Signature-Level Contracts

```python
class ResourceRequest:
    requires_gpu: bool
    gpu_memory_gb: float
    cpu_cores: int
    max_internal_workers: int

    @property
    def effective_cpu(self) -> int: ...


class ResourceSnapshot:
    available_gpu_slots: int
    available_cpu_workers: int
    system_memory_percent: float


class ResourceManager:
    def can_dispatch(self, request: ResourceRequest, active_count: int = 0) -> bool: ...

    async def acquire(self, request: ResourceRequest, block_id: str = "") -> bool: ...

    def release(self, request: ResourceRequest, block_id: str = "") -> None: ...

    @property
    def available(self) -> ResourceSnapshot: ...
```

## 9. Consequences

- Block authors no longer need to invent static RAM estimates.
- Scheduling becomes reactive for memory and predictive only for discrete
  resources.
- The first heavy block can still start if the system is below the critical
  threshold; later dispatches are throttled by observed memory pressure.
- `psutil` becomes a runtime dependency for resource monitoring and process
  support.

## 10. Alternatives Considered

Keeping static estimates with collection multipliers was rejected because item
memory varies by data size and block strategy.

Adding `estimate_memory_gb(inputs)` was rejected as the default because it
pushes difficult prediction work onto block authors.

Removing memory management entirely was rejected because it would allow the
scheduler to launch many heavy blocks at once without back-pressure.

Per-subprocess memory polling remains a possible future enhancement, but it is
not part of the ADR-022 baseline.
