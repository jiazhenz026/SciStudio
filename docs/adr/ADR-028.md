# ADR-028: IOBlock Architectural Refactor - Plugin-Owned IO Pattern

## 1. Decision Summary

SciStudio IO is owned by concrete blocks, not by a central adapter registry.
`IOBlock` is an abstract loader/saver base class; plugin packages expose
domain loaders and savers as normal blocks through `scistudio.blocks`.

ADR-028 Addendum 1 is folded into this standalone ADR. The addendum supersedes
the main ADR's seven loader/saver pairs with two dynamic core IO blocks:
`LoadData` and `SaveData`. Their effective port type is driven by a `core_type`
config enum and surfaced to the API/frontend through `dynamic_ports`.

### 1.1 Problems Addressed

| Problem | Risk | ADR-028 response | Detailed section |
|---|---|---|---|
| Central IO adapters conflict with plugin ownership | Core must know every domain format and type | Delete adapter registry and make IO ordinary blocks | Section 3 |
| Core still needs generic load/save capability | Users lose basic file ingress and egress | Provide core `LoadData` and `SaveData` dynamic IO blocks | Section 4 |
| Port type can depend on config | Static `input_ports` and `output_ports` mislead validation and UI | Add effective-port hooks and `dynamic_ports` descriptors | Section 5 |
| Frontend had hardcoded IO assumptions | New IO block names and directions would break node rendering | Expose direction and dynamic ports through block schemas | Section 6 |
| IO APIs need audit anchors | Loader/saver contracts can drift from docs | Govern IOBlock, dynamic core IO blocks, registry, and API schema symbols | Section 9 |

## 2. Context

ADR-027 moved domain types out of core. The old adapter registry assumed the
opposite: central code knew about many formats and returned core-owned domain
types. That design does not scale when imaging, LC-MS, flow, and single-cell
packages own their types and dependencies.

ADR-028 therefore moves IO behavior to the same extension mechanism as all
other blocks. A plugin that owns `SRSImage` also owns `LoadSRSImage` and
`SaveSRSImage`.

## 3. IOBlock And Adapter Registry Removal

`IOBlock` is an abstract block base with two subclass roles:

- input-direction loaders implement `load(config, output_dir="")`;
- output-direction savers implement `save(obj, config)`.

The default `run(inputs, config)` dispatches by `direction`. Input blocks wrap
loaded data in a `Collection`. Output blocks read their active input port,
call `save`, and return a path receipt.

The historical `scistudio.adapters` entry-point group and central adapter
registry are deleted as an active design. New packages must not add
`FormatAdapter` implementations for current SciStudio. They should register IO
blocks under `scistudio.blocks`.

## 4. Core IO Blocks After Addendum 1

The main ADR originally proposed one loader and one saver pair per base type.
Addendum 1 replaces that with two concrete core blocks:

| Block | Direction | Dynamic side | Driving config | Purpose |
|---|---|---|---|---|
| `LoadData` | input | output port | `core_type` | Load one of the core `DataObject` base types |
| `SaveData` | output | input port | `core_type` | Save one of the core `DataObject` base types |

The active core type enum covers `Array`, `DataFrame`, `Series`, `Text`,
`Artifact`, and `CompositeData`. Format-specific behavior remains inside the
concrete block implementation or plugin-owned IO blocks.

## 5. Dynamic Port Mechanism

`Block.get_effective_input_ports()` and
`Block.get_effective_output_ports()` are the runtime hooks for per-instance
port computation. Static blocks return their class-level ports. Dynamic IO
blocks override the relevant hook so validation, runtime execution, and UI
schemas can see the port type implied by `config["core_type"]`.

`Block.dynamic_ports` is a declarative descriptor copied into `BlockSpec` and
`BlockSchemaResponse`. It names the driving config key and maps config enum
values to the effective accepted type names for a fixed port.

This mechanism is for type changes on a fixed port count. ADR-029 separately
defines variadic port count.

## 6. API And Frontend Consequences

The backend schema exposes:

- `direction`: `input`, `output`, or null for non-IO blocks;
- `dynamic_ports`: the descriptor used by the frontend to recompute port
  accepted types when the driving config changes.

The frontend must not hardcode a single `io_block` type name. File and
directory browse behavior follows schema fields and IO direction. Port colors
and connection validation follow effective ports instead of static class ports.

Plugin IO blocks remain static unless they explicitly opt into the dynamic
descriptor. A plugin loader that always emits `FluorImage` does not need
dynamic ports.

## 7. Scope

In scope:

- `IOBlock` abstract base semantics;
- deletion of central adapter-registry architecture;
- plugin-owned IO blocks registered as normal blocks;
- core `LoadData` and `SaveData` dynamic IO blocks;
- dynamic port descriptors for enum-driven type selection;
- API/frontend schema fields needed to render dynamic IO safely.

Out of scope:

- ADR-029 variadic port count;
- ADR-030 config-schema MRO merge;
- ADR-031 reference-only loader enforcement;
- ADR-043 format capability normalization beyond the basic IOBlock extension
  declaration;
- plugin-specific loader/saver implementation details.

## 8. Addendum 1: Dynamic Port Override Mechanism

Addendum 1 adds the port override machinery needed by `LoadData` and
`SaveData`:

- `Block.get_effective_input_ports()` and
  `Block.get_effective_output_ports()` become the required read path for code
  that needs instance port shape.
- `Block.dynamic_ports` describes enum-driven type changes.
- `BlockRegistry._validate_dynamic_ports()` validates descriptor shape at scan
  time.
- `BlockSpec.dynamic_ports`, `BlockSpec.direction`,
  `BlockSchemaResponse.dynamic_ports`, and `BlockSchemaResponse.direction`
  expose the descriptor to clients.
- Workflow validation constructs block instances and reads effective ports.
- `LoadData` and `SaveData` replace the earlier seven loader/saver pairs.

The addendum does not introduce variable port counts; that remains ADR-029.

## 9. Verification And Tooling Impact

Tests should verify IOBlock load/save dispatch, unsupported direct use of the
abstract methods, extension detection, dynamic-port descriptor validation,
`LoadData` and `SaveData` effective ports, workflow validator effective-port
checks, API schema serialization, and frontend effective-port computation.

### 9.1 Signature-Level Contracts

```text
class IOBlock:
    direction: ClassVar[str]
    supported_extensions: ClassVar[dict[str, str]]

    def load(self, config: BlockConfig, output_dir: str = "") -> DataObject | Collection: ...

    def save(self, obj: DataObject | Collection, config: BlockConfig) -> None: ...

    def _detect_format(self, path: Path) -> str | None: ...

    def run(self, inputs: dict[str, Collection], config: BlockConfig) -> dict[str, Collection]: ...


class Block:
    dynamic_ports: ClassVar[dict[str, Any] | None]

    def get_effective_input_ports(self) -> list[InputPort]: ...

    def get_effective_output_ports(self) -> list[OutputPort]: ...


class LoadData:
    dynamic_ports: ClassVar[dict[str, Any] | None]

    def get_effective_output_ports(self) -> list[OutputPort]: ...

    def load(self, config: BlockConfig, output_dir: str = "") -> DataObject | Collection: ...


class SaveData:
    dynamic_ports: ClassVar[dict[str, Any] | None]

    def get_effective_input_ports(self) -> list[InputPort]: ...

    def save(self, obj: DataObject | Collection, config: BlockConfig) -> None: ...


class BlockSpec:
    direction: str
    dynamic_ports: dict[str, Any] | None


class BlockSchemaResponse:
    direction: str | None
    dynamic_ports: dict[str, Any] | None
```

## 10. Consequences

- Domain packages own their IO dependencies and typed loaders.
- Core still provides general-purpose load/save blocks for base types.
- The frontend and validator must use effective ports where instance config can
  affect type.
- Historical `scistudio.adapters` material is superseded and should not be used
  for new work.

## 11. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Keep central `AdapterRegistry` | Requires core to know plugin domains and dependencies |
| One loader/saver pair per base type | Creates palette clutter and duplicates config schemas |
| Frontend hardcodes IO block names | Breaks plugin-owned and dynamic IO blocks |
| Make every plugin IO block dynamic | Static plugin loaders are simpler when their type is fixed |
