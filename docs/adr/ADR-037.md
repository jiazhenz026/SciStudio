## ADR-037: Desktop Application Packaging, Plugin Distribution, and First-Run Dependency Management

**Status**: **proposed** — design accepted in session 2026-05-15; implementation tracked in phased issues TBD. Promote to "accepted" when Phase 1 PoC produces a runnable bootstrap.
**Date**: 2026-05-15
**Related**: ADR-017 (subprocess isolation), ADR-019 (ProcessHandle), ADR-024 (frontend bundling + `scieasy gui`), ADR-025 (entry-points plugin distribution), ADR-029 (variadic ports), ADR-031 (reference-only data contract), ADR-032 (project SQLite metadata), ADR-034 (PTY + xterm.js), ADR-035 (AI Block on PTY), ADR-036 (embedded code editor)

---

### 1. Purpose

Package SciEasy as a double-click-installable desktop application for the non-technical scientist demographic (bench biologists, mass-spec operators, imaging users) who do not know `pip install`, do not know what Node is, and will not configure a Python environment. The current `scieasy gui` developer CLI is unchanged; this ADR adds a parallel desktop distribution layer **on top of** the existing runtime, without rewriting it.

This ADR covers three coupled concerns:

1. **Packaging technology** — how Python, frontend, and bundled assets ship as a single double-click artifact across Windows / macOS / Linux.
2. **Plugin distribution** — how third-party block packages (e.g. `scieasy-imaging` with cellpose) reach users without forcing them to touch `pip`.
3. **First-run dependency management** — how the app detects and helps users install external CLIs (claude / codex / git-bash) when needed, while keeping core functionality fully usable offline.

The driving principle: **the inclusivity principle in CLAUDE.md §2.4 ("everything connectable, not everything native") must survive the packaging boundary**. Bundling the app must not break the plugin model, and the plugin model must not require the user to understand the bundling.

### 2. Context

#### 2.1 Current state

- `scieasy gui` launches uvicorn + FastAPI on a localhost port and opens the system browser at the SPA (ADR-024).
- Users install SciEasy via `pip install scieasy` or `git clone + pip install -e .`.
- Block plugins are Python packages discovered via the `scieasy.blocks` entry-point group (ADR-025); users install them with `pip install scieasy-<plugin>`.
- AI features (ADR-034, ADR-035) require the user's locally installed `claude` or `codex` CLI; `scieasy install` writes `.mcp.json` and `SKILL.md` to register SciEasy with that CLI.
- Project files (workflows, intermediate zarr/arrow, `metadata.db` from ADR-032) live in a user-chosen project directory.

All of the above assumes the user is comfortable with a terminal. **This eliminates ~70% of the target demographic for a scientific workflow platform.**

#### 2.2 User feedback (2026-05)

Bench scientists report that the platform is good "from a developer's perspective" but unusable for them. The friction is uniformly at the entry point: they do not know what `pip` is, they will not install Node, they do not know how to set `PATH`, and they will not open a terminal. They want **double-click to install, double-click to run** — the same experience as Fiji, GraphPad Prism, or FlowJo.

#### 2.3 Why this needs an ADR rather than just a packaging script

The naive answer is "use Electron, run electron-builder, done." But four substantive questions cannot be answered by a build script:

1. The plugin model assumes `pip install` into a writable Python environment. Frozen-exe distributions are read-only. How do users add cellpose?
2. ADR-017 subprocess isolation re-invokes `sys.executable` per block. In a frozen bundle, what is `sys.executable`?
3. ADR-034 spawns the user's `claude` binary via PATH discovery. GUI apps launched from the Dock have a stripped PATH. How does discovery survive?
4. ADR-036 hot-reloads `blocks/*.py`. Plugins live outside the bundle. Does hot-reload still work?

Each of these requires an architectural decision, not a build choice. Decisions made here freeze contracts that downstream packaging work depends on.

#### 2.4 Reference precedents reviewed

- **Fiji / ImageJ Updater** — JAR plugins, central update sites, manifest via `db.xml.gz`, **restart required to pick up plugins** (JVM classloader limit). Twenty-year-old model; the restart requirement is the biggest UX liability.
- **ClaudePrism (Tauri-based desktop wrapper for claude CLI)** — per-dependency check + install commands, inline setup cards rather than blocking modal, streaming subprocess output via Tauri events with synthetic step progress, `osascript` for macOS admin prompts. Source-verified.
- **Langflow** — drop-in component directory (`LANGFLOW_COMPONENTS_PATH`), no marketplace. Simplest but no discovery.
- **ComfyUI + ComfyUI-Manager** — Python plugins as GitHub repos, in-app manager with channel DB (1-day cache), automatic `pip install -r requirements.txt` on user machine. Plugin lifecycle is good UX but pip-on-user-machine is a CUDA hell vector.
- **Dify** — plugin manifest declares `max_memory`, `permission_requests`, `data_handling`; plugin runtime is a **separate subprocess** with stdin/stdout IPC; marketplace has tiered review. The most architecturally aligned with SciEasy's existing ADR-017 model.
- **n8n** — community nodes are npm packages with naming convention `n8n-nodes-*` + keyword tag; piggybacks on **npm's existing supply chain** (versioning, CDN, provenance attestation); two tiers (verified for Cloud, unverified for self-hosted). From 2026-05-01, verified nodes must be published via GitHub Actions with provenance statement. **This is the most strategically sound distribution pattern.**

The pattern adopted in this ADR is **n8n-style supply chain reuse (via PyPI) + Dify-style isolated runtime (already in ADR-017) + ClaudePrism-style first-run dependency UX**.

### 3. Decisions

#### 3.1 Packaging technology

**D1. Stack = Electron + python-build-standalone + Vite-built static frontend.**

The Electron main process is a thin coordinator:
1. On launch, spawn the bundled Python interpreter as a child process: `<bundle>/resources/python/python(.exe) -m scieasy.gui --port 0 --bundled`.
2. Read uvicorn's stdout for the assigned port.
3. Open a `BrowserWindow` at `http://127.0.0.1:<port>/`, which loads the Vite-built SPA.
4. On window close, SIGTERM the Python child and any worker subprocesses tracked in ProcessRegistry (ADR-019).

`python-build-standalone` is chosen because:
- It is a **relocatable** full CPython distribution. Unzip-and-run; no system Python dependency; no registry writes; clean uninstall.
- `sys.executable` is a real Python interpreter, not a PyInstaller bootloader. ADR-017 subprocess workers can re-invoke it directly without `freeze_support` rituals.
- Bundled `pip` and `uv` are available for plugin installs (D15).

**D2. SciEasy runtime is decoupled from the packaging technology.**

No SciEasy core module imports Electron, calls Electron APIs, or makes assumptions about being inside a bundle. The only adaptation point is a `--bundled` flag passed to `scieasy.gui` that:
- Reads paths from `platformdirs` (D3) instead of `~/...`.
- Suppresses opening the system browser (Electron opens its own BrowserWindow).
- Emits structured stdout for the Electron main process to consume (port, ready signal).

This means: the `scieasy gui` developer CLI continues to work unchanged; the desktop app is **one consumer** of that CLI, not a replacement.

#### 3.2 Data directories

**D3. All user-scoped paths go through `platformdirs`. Frozen mapping:**

| Category | Windows | macOS | Linux (XDG) |
|---|---|---|---|
| Config | `%APPDATA%\SciEasy\` | `~/Library/Application Support/SciEasy/` | `~/.config/scieasy/` |
| Cache | `%LOCALAPPDATA%\SciEasy\Cache\` | `~/Library/Caches/SciEasy/` | `~/.cache/scieasy/` |
| Logs | `%LOCALAPPDATA%\SciEasy\Logs\` | `~/Library/Logs/SciEasy/` | `~/.local/state/scieasy/` |
| Plugins | `%LOCALAPPDATA%\SciEasy\plugins\` | `~/Library/Application Support/SciEasy/plugins/` | `~/.local/share/scieasy/plugins/` |
| Model weight cache (shared) | `%PROGRAMDATA%\SciEasy\models\` | `/Library/Application Support/SciEasy/models/` | `/var/cache/scieasy/models/` |

Project directories remain user-chosen (existing behavior). `metadata.db` (ADR-032) stays with the project, not user-scoped — lineage must travel with the data.

#### 3.3 First-run dependency management

**D4. External CLIs (claude / codex / git-bash) use their official installers; SciEasy never reimplements them.**

- claude: `claude.ai/install.sh` (Unix) / `claude.ai/install.ps1` (Windows, via `irm | iex`).
- codex: OpenAI's official installer.
- git-bash on Windows: link the user to git-for-windows.org with an "I've installed Git" recheck button.

SciEasy spawns these installer commands as child processes, streams stdout/stderr line-by-line to the frontend with ANSI stripping, and reports completion via IPC events. Pattern lifted verbatim from ClaudePrism (verified source: `apps/desktop/src-tauri/src/claude.rs`).

**D5. Bundled and offline-installable dependencies require no network.**

The following are written from app resources on first launch, offline:
- `SKILL.md` → `<config>/scieasy/skill/SKILL.md` (and copied into `~/.claude/skills/scieasy/SKILL.md` when claude is detected).
- `.mcp.json` → `<project>/.scieasy/mcp.json` and/or user-scoped equivalent.
- Bundled Python, frontend assets, default sample project — already in the bundle.

**This means SciEasy has no required-online dependency on first launch.** Core workflow editing, execution, and data viewing all work air-gapped.

**D6. UX = silent background check + notification drawer; not a blocking modal.**

On app start:
1. Main window renders immediately (current web first-screen UX per user request).
2. Background tasks: write bundled offline assets (D5); `check_claude_status` / `check_codex_status` / `check_plugins` in parallel.
3. If any AI dep is missing, a non-intrusive notification appears in the top-right drawer: *"AI features available — install claude or codex"*.
4. Clicking the notification opens a setup drawer with one card per dep (claude / codex / git-bash if Windows). Each card has its own state machine (`checking → not-installed | missing-git | not-authenticated | installing → ready | error`).
5. The drawer also has a top-level **"Install all missing AI dependencies"** button that chains the per-dep installs (git-bash → claude → auth) for users who want one click.

Disabled AI buttons in the canvas show a tooltip *"Install claude or codex to enable"*, linking to the drawer. They are **not hidden** — visibility advertises the feature even when unavailable.

**D7. Per-dep architecture: IPC pair + store + card.**

For each external dep, a uniform contract:

```
backend (Electron IPC handler in Node, calling into bundled Python helper):
  check_<dep>_status()  →  { installed, authenticated, version, version_compat, ... }
  install_<dep>()        →  emit("<dep>-install-output", line) | "<dep>-install-complete"

frontend (Zustand store + React card):
  store: { status, isInstalling, installSteps[], installLogs[], error }
  listener: append logs, advance synthetic step progress
  card: state-specific copy + action button + collapsible log
```

Synthetic step progress (`downloading → installing → verifying → complete`) is driven by keyword detection in installer stdout. The underlying installer is opaque; the steps exist for user confidence, not for accurate progress.

**D8. `check_<dep>_status` returns version compatibility, not just install state.**

```typescript
type ClaudeStatus = {
  installed: boolean;
  authenticated: boolean;
  binary_path: string | null;
  version: string | null;
  version_compat: "compatible" | "outdated_too_old" | "newer_untested" | "unknown";
  account_email: string | null;
  missing_git: boolean;  // Windows only
}
```

UI rendering:
- `compatible` → green checkmark with version + email.
- `outdated_too_old` → red, "claude N.M is too old; minimum required X.Y; click to run `claude --upgrade`".
- `newer_untested` → yellow, "claude N.M is newer than SciEasy's tested version X.Y; features should work; please report issues".

Min/max tested versions live in a SciEasy-side static manifest and are bumped per release.

**D9. macOS permission-sensitive operations go through `osascript with administrator privileges`.**

Touching `~/.local/` on macOS sometimes requires admin (typical when `pip` previously ran as root). The bundled Python helper detects EACCES and invokes the system authorization dialog via `osascript`. No custom elevation UI.

#### 3.4 Plugin system

**D10. Primary plugin distribution channel = PyPI.**

Plugins are published as PyPI packages with two requirements:
- **Naming convention**: package name matches `scieasy-plugin-*` or `scieasy-blocks-*` (regex enforced by the in-app browser).
- **Trove classifier**: `Framework :: SciEasy` declared in package metadata.

This piggybacks on PyPI's existing infrastructure: versioning, CDN, trusted publishing, PEP 740 attestations, deletion / yanking flows. SciEasy operates no plugin registry of its own.

**D11. Plugin manifest = `pyproject.toml` with `[tool.scieasy]` extension section.**

```toml
[project]
name = "scieasy-plugin-imaging"
version = "1.4.2"
requires-python = ">=3.11"

[project.entry-points."scieasy.blocks"]
cellpose_segment = "scieasy_imaging.blocks.cellpose:CellposeSegmentBlock"
tiff_loader = "scieasy_imaging.io.tiff:TiffLoader"

[tool.scieasy]
api_version = "0.5"                    # SciEasy plugin API (independent of app version)
scieasy_min_version = "0.8.0"
scieasy_max_version = "1.x"

[tool.scieasy.requirements]
gpu = "optional"                       # required | optional | no
disk_cache_mb = 500                    # rough estimate
ram_estimate_mb = 2000                 # rough estimate
heavy_deps_url = "https://github.com/foo/scieasy-plugin-imaging/releases/download/v1.4.2/heavy-deps.zip"  # optional

[tool.scieasy.permissions]
filesystem = ["project_dir", "user_temp"]   # whitelist relative to home
network = "none"                            # none | api_only | unrestricted
subprocess = ["cellpose"]                   # spawn whitelist (empty = forbidden)
data_egress = "no"                          # explicit declaration for compliance
```

PEP 621 compliance keeps the manifest compatible with standard Python build tooling. The `[tool.scieasy]` namespace prevents collision with other tools.

**D12. Three plugin tiers: Verified / Community / Sideload.**

- **Verified**: hand-reviewed by SciEasy maintainers; listed first in the in-app browser; shown with a "Verified" badge. Source code review + manifest audit + smoke test. Verified status is revocable.
- **Community**: any PyPI package matching the naming convention + classifier; shown with an "Unverified" warning + author / GitHub link / PyPI attestation status.
- **Sideload**: user-provided local zip or directory; never appears in the browser; placed under `<plugins>/sideload/<name>/`. Used for air-gapped labs, plugin development, and private institutional plugins.

The browser shows Verified by default; Community is one tab over; Sideload is a separate "Install from file" button.

**D13. Heavy native deps (>60MB PyPI single-file cap) ship via plugin author's GitHub Releases.**

Plugins that need PyTorch / CUDA wheels / large model weights declare a `heavy_deps_url` pointing to a release artifact. On install, SciEasy:
1. Downloads the PyPI wheel (small; Python code + manifest only).
2. Reads `heavy_deps_url` from the manifest.
3. Downloads + verifies + extracts heavy deps into the plugin's venv (D14).

This sidesteps PyPI's per-file size limit without us building our own large-file CDN. Plugin authors use GitHub's free release hosting (2GB per file, 10GB total per release).

**D14. Plugin install uses the bundled `uv` into a per-plugin venv.**

Each plugin gets its own venv at `<plugins>/<plugin-name>-<version>/venv/`. This isolates dependency conflicts across plugins — `scieasy-plugin-imaging` can pin numpy 2.0 while `scieasy-plugin-ms` pins 1.26 without conflict.

`uv` is preferred over `pip` for: resolution speed (10-100×), correctness on edge-case constraints, native lockfile support. `uv` ships inside the Python bundle (D1).

Plugin block execution (D17) injects the plugin venv's `site-packages` into the worker subprocess's `sys.path` at spawn time.

**D15. Permission/resource declarations: v1 UI-only, v2 enforce.**

- **v1**: manifest declarations (`[tool.scieasy.permissions]`, `[tool.scieasy.requirements]`) are shown to the user pre-install in a confirmation dialog ("This plugin will: access project_dir and user_temp; spawn cellpose; use up to 2GB RAM; not send data externally"). The user clicks Install or Cancel. No runtime enforcement.
- **v2**: same declarations are converted to OS-level constraints on the worker subprocess at spawn time (Windows Job Objects, Linux cgroups + seccomp, macOS sandbox-exec). Out of scope for this ADR; tracked as a follow-up.

The v1 / v2 split lets us ship the UX immediately while deferring the harder enforcement engineering.

**Plugin threat model and defense-in-depth.**

| Threat | Likelihood | Primary defense |
|---|---|---|
| Malicious plugin author publishes to PyPI under our naming convention | Medium | PyPI provenance attestation (D10); Verified tier audit gate; in-app browser flags Unverified |
| Supply-chain attack on a legitimate plugin's transitive dep | Medium-High | Per-plugin venv isolation (D14); SBOM check on install (Phase 5); revocation list checked at app start |
| Sideloaded plugin from untrusted source | Medium (user-driven) | Pre-install permission confirmation modal (this D15 v1); Sideload tier visibly marked Unverified; user explicitly accepts risk |
| Verified plugin compromised post-review | Low | Verified-list revocation feed (`verified.json` at fixed URL, 24h cache); compromised plugins auto-disabled on next launch until user reconfirms |
| Plugin lies about declared permissions | High | v1: not defended — user is the trust boundary. v2: OS-level enforcement (Job Object / cgroups + seccomp / sandbox-exec) closes this |
| Cross-plugin access (plugin A reads plugin B's data) | Low | Per-plugin venv (D14) + per-block worker subprocesses (ADR-017) means no shared in-process state |
| Plugin escapes worker subprocess | Low | ADR-017 subprocess isolation already required for all blocks; v2 adds OS-level capability dropping |

**Defense layers, ordered by reliability:**

1. **Distribution gate** — PyPI naming convention + classifier + provenance attestation. First-pass supply-chain check; not perfect (PyPI namespace squatters exist) but raises attacker cost.
2. **Verified-tier code review** — manual review by maintainers before listing as Verified. High-confidence for Verified-only users.
3. **Per-plugin venv isolation (D14)** — a compromised plugin's deps cannot poison core or sibling plugins via import shadowing.
4. **Subprocess isolation (ADR-017)** — each block runs in its own short-lived worker.
5. **User consent at install (D15 v1)** — pre-install confirmation modal surfaces manifest claims; user is the final gate.
6. **OS-level enforcement (D15 v2)** — deferred; provides defense against permission lying.
7. **Revocation** — Verified list publishes a revocation feed; checked at app start with 24h cache.

**What v1 explicitly does NOT defend against:**

- A user installing a Sideloaded plugin that lies about permissions and exfiltrates project data. The user bypassed all distribution gates by choosing Sideload; we ship the mode because air-gapped labs need it, with the explicit understanding that **Sideload = "you accept responsibility."**
- A Verified plugin that compromises post-review. v1 relies on revocation as the only post-deployment lever; v2's runtime enforcement provides defense-in-depth.
- Network-level attacks on PyPI / GitHub itself. Out of scope.

**D16. In-app plugin browser from v1.**

A "Plugins" panel in Settings shows:
- Search box (queries PyPI `simple` index filtered by classifier + naming regex).
- Tabs: Verified / Community / Installed / Sideload.
- Per-plugin card: name, version, author, manifest summary, permission/resource preview, Install/Update/Uninstall buttons.
- Click Install → permission confirmation modal (D15) → progress UI streaming `uv pip install` output → entry-points rescan → palette refresh.

Approximate scope: ~500 LOC frontend + ~300 LOC backend (PyPI query, manifest parse, install orchestration). Sideload entry is a separate button on this same panel.

**D17. Plugin block execution reuses ADR-017 subprocess isolation.**

No new isolation primitive. Workers spawned for plugin-provided blocks differ from core-block workers in exactly one way: the plugin venv's `site-packages` is prepended to `sys.path` via a `PYTHONPATH` environment variable. Everything else — argument marshaling, StorageReference transport, cancellation semantics — is unchanged.

This means plugin developers do not learn a new contract: a plugin block is a normal SciEasy block.

#### 3.5 Updates and hot-load

**D18. Engine = whole-bundle replacement; plugins = independent.**

- SciEasy core auto-update replaces the entire app bundle on next launch. Electron's `electron-updater` handles download + atomic swap + restart. No partial updates of core.
- Plugin updates are independent of core updates. A plugin can be updated in-place without restarting the app (D19).

**D19. Plugin hot-load via registry rescan + immutable per-version directories.**

This is SciEasy's architectural advantage over Fiji / ComfyUI (which both require restart):

- **New plugin install**: extract to `<plugins>/<name>-<version>/`, call `BlockRegistry.rescan()`, emit `block-registry-changed` event (already used by ADR-036 §3.5), palette refreshes. Next time a block from this plugin is dragged onto the canvas, the spawned worker subprocess sees the new entry-point and imports it. **Engine never restarts; running workflows are not interrupted.**
- **Plugin upgrade**: install new version to `<plugins>/<name>-<new-version>/` alongside the old. Wait for any in-flight workers using the old version to finish (queryable via ProcessRegistry from ADR-019). Once idle, registry rescans to point at the new version; delete old directory. If the user forces immediate switch: dialog "N workers running; cancel them / wait / postpone".
- **Plugin removal**: same as upgrade with empty target.
- **SciEasy core update**: restart required (D18).

The immutable per-version directory is the trick: old and new can coexist transiently, eliminating the "classloader can't unload" problem Fiji has.

**D20. The engine MUST NOT import plugin code.**

CLAUDE.md §7 boundary: SciEasy core never imports anything from `<plugins>/`. Plugin code is loaded only inside worker subprocesses. Violation of this rule reintroduces the Fiji restart requirement and is rejected at code review.

#### 3.6 Signing and distribution

**D21. Phased signing approach.**

| Phase | macOS | Windows | Linux |
|---|---|---|---|
| **v0** (internal testers, developer audience) | Unsigned `.dmg`; users right-click → Open to bypass Gatekeeper. `identity: null` in electron-builder. | Unsigned `.exe`; users click "More info" → "Run anyway" past SmartScreen. | Unsigned AppImage (no warning). |
| **v0.5** (early public, open-source audience) | Still unsigned. | **SignPath.io** open-source code signing (free for qualifying projects). Eliminates SmartScreen "unknown publisher" warning. | Unchanged. |
| **v1** (general scientist audience, target ICP) | **Apple Developer Program** ($99/yr) + notarization via `notarytool` in CI. | **EV Authenticode certificate** (~$500-700/yr). Instantly eliminates SmartScreen warnings (no reputation ramp-up). | AppImage + optional Flathub submission. |

Until v1, the ADR-037 install experience does not meet the non-technical-user bar. v0 and v0.5 are explicitly for advanced users willing to click past warnings.

**Installer format choices (apply to all signing phases)**

| Platform | Format | Rationale | Alternatives rejected for v1 |
|---|---|---|---|
| Windows | **NSIS `.exe`**, per-user, allow custom install path | Per-user install needs no admin password — critical for non-technical users on shared lab machines. NSIS supports our uninstall data-prompt (D22) via custom `.nsh` script. | **MSI**: requires admin or per-machine; better for GPO/IT-managed enterprise. Deferred to v2 for institutional deployment. **MSIX**: Store-only path; not until Microsoft Store listing is in scope. **Squirrel**: lighter but less mature; harder to ship a custom uninstall prompt. |
| macOS | **`.dmg`** with drag-to-Applications + universal2 binary (`arm64` + `x86_64`) | Familiar UX; universal2 means one DMG covers both Intel and Apple Silicon hardware. | **`.pkg`**: invasive installer flow; needed only for system-level writes we do not perform. **Mac App Store**: review process + sandboxing constraints conflict with spawning the user's `claude` binary. Deferred indefinitely. |
| Linux | **AppImage** (primary), with `--appimage-extract-and-run` fallback for FUSE-missing systems | Single file, no install, no root, works across distros. Extract-and-run fallback handles systems missing FUSE (modern containers, minimal Ubuntu). | **`.deb` / `.rpm`**: distro-specific; needs apt/dnf integration; admin required. Deferred to v2 as a convenience for Debian/Ubuntu/Fedora power users. **Snap**: Canonical-controlled review + slower cold start + auto-update gymnastics. Deferred. **Flatpak / Flathub**: sandboxing benefits but harder integration with claude/codex discovery (D4). Deferred to v2 with explicit sandbox bypass for host-CLI calls. |

All installers ship via GitHub Releases as the primary download URL. Update channels (D28) and auto-update (D29) use the same artifacts.

**Note on the "unsigned dev build works fine" trap**: macOS Gatekeeper only triggers when `com.apple.quarantine` extended attribute is set, which happens when the file is downloaded via a browser. Locally built artifacts have no quarantine bit, so developers testing their own builds see no warning. This must not be confused with "signing is unnecessary." Reference: a sibling project (`zjzcpj/ihhdmodel-desktop`) appears warning-free only because no actual distribution has occurred.

**D22. Uninstall semantics.**

- App bundle removed by the OS-native uninstaller (NSIS on Windows, drag to Trash on macOS, package manager on Linux).
- Prompt during uninstall: *"Also delete user data (projects, plugins, cache, logs)?"* Default = unchecked (preserve user data). If checked, deletes everything under the D3 paths.
- Plugin and cache directories are deleted by default if the prompt is checked; project directories (user-chosen) are listed but never auto-deleted — the uninstaller cannot reliably know all project locations.

#### 3.7 First-run onboarding

**D23. Same web first-screen + sample project + silent CLI detection.**

- First launch shows the current web first-screen (no new onboarding wizard).
- A bundled sample project is offered: "Open example project" button on the start screen. The example demonstrates 2-3 blocks producing a visible result within 30 seconds of launch.
- Silent claude/codex detection runs in background. If both are missing, a one-time notification (not modal) introduces AI features as optional + links to the setup drawer. After the user dismisses it, only the drawer is the entry point.

#### 3.8 Offline behavior

**D24. Core functionality is fully offline; AI / plugin install / auto-update degrade gracefully.**

| Functionality | Offline behavior |
|---|---|
| Workflow editing | ✅ Full |
| Workflow execution (core blocks) | ✅ Full |
| Data viewing / IO | ✅ Full |
| Plugin execution (already installed) | ✅ Full |
| AI (AIBlock, embedded chat) | ❌ Disabled with tooltip *"Requires claude/codex login"* |
| Plugin browser (Verified/Community) | ❌ Disabled with tooltip *"Requires network"*; Sideload still available |
| Auto-update check | ❌ Silently skipped |

UI elements that need network are **disabled, not hidden** — visibility of the feature is preserved even when unusable.

#### 3.9 Telemetry and privacy

**D25. Telemetry is opt-in; airgapped build variant exists.**

- Default: no telemetry, no crash reporting, no update-check pings.
- Settings → Privacy: opt-in checkboxes for (a) anonymous crash reports, (b) anonymous usage telemetry. Each shows a verbatim sample of what would be sent.
- A separate `-airgapped` build variant is released alongside standard builds. It compiles with telemetry / update-check / plugin update-site / AI features hard-disabled. Targets regulated environments (clinical research, classified facilities).

#### 3.10 Multi-user and shared resources

**D26. Plugins are per-user; model weights are per-machine.**

- Plugin installs under `<LocalAppData>/SciEasy/plugins/` (per-user). Two researchers sharing a lab Windows machine have independent plugin sets.
- Model weight cache (cellpose models, embedding model weights, etc.) lives under a per-machine shared directory (`%PROGRAMDATA%` on Windows, `/Library/Application Support/` on macOS, `/var/cache/` on Linux) — symbolic link or env-var indirection from each user's plugin venv.
- The first user to download a 2GB cellpose weight does so once for the machine; subsequent users reuse it.

**D27. Project metadata.db (ADR-032) stays with the project, not user-scoped.**

Lineage must travel with data. A project copied to a USB drive and opened on another machine must retain its full history. ADR-032 already places `metadata.db` at `<project>/metadata.db`; this ADR reaffirms it.

#### 3.11 Release, update, rollback, and compatibility policy

**D28. Three release channels.**

| Channel | Audience | Cadence | Stability bar |
|---|---|---|---|
| `stable` | Default for all users | Monthly | Full test matrix (§7) green; ≥1 week of `beta` soak |
| `beta` | Opt-in early-access (Settings → Update channel) | Weekly | CI green + smoke test on all OS; known issues documented |
| `nightly` | Opt-in developers; not shown by default | Per-merge to `main` | CI green only |

Channel choice persists across upgrades. Users can switch channels; the next install pulls from the selected channel.

**D29. Update atomicity: download-aside, swap-on-restart.**

`electron-updater` downloads the new bundle to a sibling directory (e.g. `SciEasy.app.update/`) while the user keeps using the running app. On user-initiated restart (or next launch), the old bundle is replaced atomically by directory swap. If the swap fails partway, the launcher falls back to the previous bundle.

**No mid-session restart prompts.** Updates never interrupt running workflows; the user is notified via a non-blocking app-menu badge and decides when to apply.

**D30. Rollback: keep the previous version on disk for one-click revert.**

After a successful update, the previous bundle stays under `<install>/versions/` for at least **2 minor versions or 30 days**, whichever is shorter. Settings → About → "Roll back to <version>" performs the reverse atomic swap. Plugin venvs and project data are untouched.

Rationale: scientific users can have a workflow break under a new core version with no quick fix in sight; rollback is a critical safety valve.

**D31. SemVer for core; independent SemVer for the plugin API.**

- **Core version** (`major.minor.patch`): SemVer over user-visible behavior + IPC contracts. Breaking changes only at major bumps.
- **Plugin API version** (declared in `[tool.scieasy] api_version`): independent SemVer for the contract between core and plugins. Plugins declare `scieasy_min_version` / `scieasy_max_version`. Core release notes publish the supported plugin API range.

If core ships X→X+1, plugins built against X may continue to load via a compatibility shim documented in release notes; otherwise the plugin shows *"Incompatible — update plugin or roll back core"* and is disabled until resolved.

**D32. Workflow YAML and `metadata.db` schema migration.**

- **Workflow YAML**: backward compatible within a core major version. Migration script (in-place, with `.bak` copy) on first open under a new major version. Forward-only.
- **`metadata.db` (ADR-032)**: schema version stamped in the DB. On project open, if DB schema is older than the running core, run forward-only migration with a `.bak` snapshot. If newer (project last touched by future SciEasy version), refuse to open and show *"This project requires SciEasy ≥X.Y; please update or use the matching version"*.

Migrations tested in CI on a representative project corpus across the last 3 versions.

**D33. Deprecation policy.**

- Public APIs (block contracts, plugin API surface, REST routes, IPC events) deprecate at minor versions with a warning and continue to work for at least **2 minor versions**. Removal allowed only at the next major bump.
- Plugin API deprecations honor the same window measured in **plugin API SemVer**, not core SemVer.
- Deprecation warnings emit to (a) backend log, (b) `block-registry-changed` event metadata when a block uses a deprecated API, and (c) the plugin browser's "Health" tab for installed plugins.

**D34. Plugin lockfile per project (opt-in).**

When enabled (Settings → Project → "Pin plugin versions"), `<project>/scieasy-plugins.lock` records the exact version of each plugin used by any block in the project's workflows. On project reopen:
- Pinned version not installed → prompt to install that exact version into the per-plugin venv directory (D14).
- Different version installed → warn; offer to switch the project to the installed version or install the pinned one alongside (per-version directories per D19 make coexistence cheap).

This lets two researchers share a project across machines with reproducible block behavior despite plugin updates.

### 4. Alternatives considered

1. **Tauri instead of Electron.** Smaller bundle (~10MB vs ~150MB Chromium). Rejected for v1: Windows depends on WebView2 runtime (preinstalled on Windows 11, requires install on Windows 10), and cross-platform rendering parity is weaker than Chromium. **Re-evaluate for v2.**

2. **PyInstaller single-file bundle (no Electron).** Smaller distribution, but: bootloader self-extracts on every launch (slow first start, antivirus false positives); `sys.executable` indirection complicates ADR-017 subprocess re-invocation; harder to debug. Rejected.

3. **Native desktop rewrite (PyQt / wxWidgets).** Months of work to recreate the existing React canvas in native widgets, with significantly worse UX for graph editing. The web stack is the right one; we just need to bundle it. Rejected.

4. **Curated SKU strategy ("SciEasy Imaging Edition" with cellpose preinstalled, "SciEasy Mass Spec Edition" with pyopenms preinstalled).** Eliminates plugin install UX entirely but breaks §2.4 inclusivity (matrix explosion: 3 OS × N SKUs × M versions = unmaintainable). Rejected.

5. **Fiji-style self-built update site for plugins.** Reinvents what PyPI already does, without provenance attestation or trusted publishing. Rejected per §2.4 reference research.

6. **ComfyUI-style git clone + pip install -r requirements.txt on user machine.** User-side dependency resolution is the exact CUDA-hell vector we want to avoid for cellpose-class plugins. Rejected.

7. **Hot-load SciEasy core itself.** Possible in theory (Python `importlib.reload`), but reliability is poor for a long-running engine with cached state. Whole-bundle replace on update is more predictable. Rejected.

#### 5.4 What this ADR does NOT change

To bound scope, the following architectural surfaces are **explicitly untouched**:

- **ADR-031 reference-only data contract** — DataObject, StorageReference, IOBlock loaders all unchanged. Plugin blocks must comply with the existing contract.
- **ADR-017 subprocess isolation** — worker spawn semantics, IPC marshaling, cancellation all preserved. Plugin block workers differ only in injected `PYTHONPATH`.
- **ADR-019 ProcessHandle / ProcessRegistry** — used as-is for plugin worker tracking and hot-upgrade quiescence detection.
- **ADR-029 variadic ports** — plugin blocks can use them; mechanism is unchanged.
- **ADR-032 project SQLite metadata** — `metadata.db` location and schema unchanged; lineage continues to travel with project.
- **ADR-034 PTY + xterm.js** — entire AIChat / embedded agent surface unchanged. SetupDrawer is a sibling, not a replacement.
- **ADR-035 AIBlock on PTY** — AI Block runtime untouched; SetupScreen subcomponent is shared via `depSetupSlice` (D6) instead of duplicating dep-check logic.
- **ADR-036 embedded code editor** — Monaco editor, `TabState` discriminated union, file R/W API all unchanged. Plugin install does not go through the file editor.
- **No core files deleted.** Unlike ADR-034 (which deleted ~3000 LOC of stream-json UI), this ADR is **purely additive**.

**Invariant**: The `scieasy gui` developer CLI continues to work after this ADR ships. Developers who do `pip install scieasy` + `scieasy gui` get the same experience they have today; the desktop bundle is a parallel distribution channel layered on top.

### 8. Consequences

#### 8.1 Wins

- Reaches the ~70% of the target ICP that the current pip-install model excludes.
- Plugin model **preserved** and arguably stronger than the developer-CLI model (Verified tier, manifest-declared permissions, in-app browser).
- Hot-load for plugins, which Fiji / ComfyUI / Langflow / n8n all require restart for — SciEasy's subprocess isolation makes this free.
- Supply chain security via PyPI provenance, without operating our own registry.
- Existing runtime untouched; `scieasy gui` developer CLI continues to work in parallel.

#### 8.2 Losses

- Bundle size grows to ~200-300MB (Electron + Chromium + python-build-standalone + frontend + uv + bundled SKILL.md). Mitigation: compression, lazy chunk loading where possible, accept the size as cost-of-business for non-technical users.
- Per-plugin venvs duplicate common deps (numpy, scipy) across plugins. Tradeoff against dependency conflicts is worth it. Disk cost ~50-100MB per plugin.
- v1 signing cost: $99/yr Apple + ~$500-700/yr Windows EV. Operational expense, not an engineering blocker.
- Plugin developers must publish to PyPI (PEP 517 build + trusted publishing setup). Higher bar than "drop a Python file." Mitigation: provide `scieasy create-plugin` CLI scaffolding (future enhancement).

#### 8.3 Risks

- **GUI-launched PATH stripping** (macOS, Linux Dock). Plugin venv `sys.path` injection works regardless, but claude / codex discovery is sensitive. Hotfix #886 already addressed one instance; Phase 1 PoC must validate end-to-end.
- **Windows EV cert procurement timeline.** Issuing process is 1-4 weeks. Schedule v1 release planning accordingly.
- **uv resolution failures on user machine for plugins with complex deps.** Mitigation: plugin author tests their plugin against bundled-uv version in CI; SciEasy ships a known-good uv version per release.
- **Anti-virus false positives** on the spawned `python.exe` child. python-build-standalone is generally well-recognized but not perfectly. Mitigation: code signing (v0.5+) substantially reduces this.
- **Plugin authors abusing permission declarations.** v1 has no enforcement; a malicious manifest could declare `network=none` and still call out. Mitigation: Verified tier audits exist for this reason; v2 enforcement closes the gap.

### 9. Open questions

1. **Sample project content.** What does the bundled sample demonstrate? Imaging is the broadest, but a 2GB image is too large for a bundled sample. Tentatively: a small synthetic dataset + a 2-block workflow producing a visible chart in <30s. Resolve in Phase 5.

2. **PyPI rate limits for in-app browser search.** PyPI's `simple` index is large; we may need to cache the SciEasy-filtered subset on a SciEasy-hosted JSON endpoint. Decide based on PoC measurements.

3. **GPU detection in the plugin browser.** Should the in-app browser detect CUDA / Metal availability and hide GPU-required plugins on machines without GPU? Tentatively yes; gracefully indicate "needs GPU" rather than hiding silently. Resolve in Phase 3.

4. **Plugin venv Python version pinning.** Bundled python-build-standalone is one version. If a plugin requires a different Python version, we cannot accommodate it. Tentatively: plugins MUST be compatible with the bundled Python version range declared in SciEasy's release notes. Plugins requiring other versions are rejected.

5. **Verified tier review process and SLA.** Who reviews? How long does review take? What's the appeal path for rejected plugins? Out of scope for this ADR but must be defined before announcing Verified tier publicly.

6. **Auto-update notification UX.** Force-update? Notify-only? Skippable for N versions? Tentatively: notify-only with "Update now / Later / Skip this version" buttons; mandatory update only for security fixes.

7. **i18n.** The current frontend is English-only. The non-technical scientist demographic includes significant non-English populations. Not required for v1 release but must be planned for; tracked separately.

- Plugin v2 permission enforcement (deferred per D15).
- Multi-tenant cloud SaaS deployment (separate ADR if pursued).
- Auto-translation of UI / docs (i18n infrastructure only).
- Custom Python version per plugin (single bundled Python version per release).
- Cluster / distributed workflow execution from desktop client (existing engine is single-machine).
- Plugin signing distinct from PyPI provenance (we rely on PyPI's attestation chain).

---

### 11. Decision summary

**SciEasy ships as an Electron desktop app bundling python-build-standalone + the existing FastAPI runtime, with no changes to the SciEasy core.** Plugins distribute through PyPI (named `scieasy-plugin-*`, classified `Framework :: SciEasy`), with manifest declarations in `pyproject.toml [tool.scieasy]`, in three tiers (Verified / Community / Sideload), installed via bundled `uv` into per-plugin venvs, with heavy native deps fetched from plugin authors' GitHub Releases. Plugin block execution reuses ADR-017 subprocess isolation; plugin hot-load is free because of subprocess + immutable-per-version-directory design. First-run dependency UX (claude / codex / git-bash) follows ClaudePrism's per-dep card pattern in a setup drawer, not a blocking modal — core features remain usable offline and without AI deps installed. Signing rolls out in three phases (v0 unsigned for testers → v0.5 SignPath for Windows open-source → v1 Apple Developer + Windows EV for general release). Telemetry is opt-in; an airgapped build variant exists for regulated environments.

The core invariant: **packaging is additive, not transformative**. The existing `scieasy gui` developer CLI and the existing plugin model continue to work; the desktop app is a distribution channel layered on top, not a fork of the architecture.
