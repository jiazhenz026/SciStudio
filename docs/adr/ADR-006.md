# ADR-006: External Software Integration Via File-Exchange Bridge

## 1. Decision Summary

SciEasy integrates standalone scientific tools through `AppBlock` and a
file-exchange directory. The framework exports inputs, launches or waits for
the external tool, watches for outputs, and then resumes the workflow.

### 1.1 Problems Addressed

| Problem | Risk | ADR-006 response | Detailed section |
|---|---|---|---|
| Scientific workflows depend on existing GUI tools | Replacing those tools would exclude real lab practice | Integrate them through file exchange | Section 3 |
| App APIs are inconsistent or absent | Per-app API integrations would be brittle | Use native files as the common boundary | Section 4 |

## 2. Decision

`AppBlock` is the external-software block category. `FileExchangeBridge`
prepares inputs, launches or coordinates the external process, watches output
locations, and collects resulting files back into SciEasy data objects.

## 3. Scope And Consequences

File exchange is slower than an in-process API but works across GUI tools,
CLI tools, and manual review steps. Human-in-the-loop AppBlocks can pause
workflow throughput; that is expected for interactive scientific tools.

### 3.1 Signature-Level Contracts

```python

    ...

class FileExchangeBridge:
    ...
```

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- AppBlock tests verify file-exchange preparation and output reconstruction.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Stable per-app APIs were rejected as unavailable for many tools. Screen
scraping was rejected as fragile. Ignoring external tools was rejected because
SciEasy must wrap existing lab software rather than replace it.
