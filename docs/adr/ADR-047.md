# ADR-047: BlockRegistry Sub-Package Decomposition + Legacy IO Finder Removal

## 1. Decision Summary

`src/scistudio/blocks/registry.py` (1708 LOC) becomes the sub-package
`src/scistudio/blocks/registry/`, in the **Path D pattern** (per umbrella
#1427 Phase 2 PR #1460). As part of the same PR, the pre-ADR-043 "legacy
IO finder" API (`find_loader` / `find_saver` / `find_io_blocks_for_type`
plus the internal `_find_io_block` / `_class_accepts_dtype` /
`_best_specificity` helpers) is **deleted** — these methods exist for
backward-compat during ADR-043 migration; production callers all reach
the capability-based equivalents first, and the legacy fallback paths
are dead code. Net effect: ~250 LOC delete + ~1500 LOC decompose.

### 1.1 Problems Addressed

| Problem | Current pain | Response |
|---|---|---|
| `registry.py` is 1708 LOC, 30+ methods | hardest registry file to navigate; multiple concerns coupled | Decompose into 4 responsibility-grouped sibling modules |
| Legacy IO finder API has zero hot-path callers | dead weight + parallel-to-capability surface confuses readers | Delete `find_loader` / `find_saver` / `find_io_blocks_for_type` |
| ADR-043 migration was scoped to Phase-11 timeframe | legacy API was kept for "old extension-only callers continue to run during migration" — migration is done | Cleanup completes the ADR-043 transition |
| God-file advisory (#1427 Phase 1) tracks this as Bucket D | last Bucket D protected-core file alongside scheduler | Closes the remaining decomposition target |

## 3. Decomposition layout

```
src/scistudio/blocks/registry/
├── __init__.py        # BlockRegistry class body + bindings, BlockSpec, 5 error classes  (~260 LOC)
├── _scan.py           # scan loops (builtin / tier1 / tier2 / monorepo) + _register_spec  (~400 LOC)
├── _capability.py     # capability registry: find_*_capability + matching primitives     (~500 LOC)
└── _spec.py           # _spec_from_class + config_schema MRO merge + dist version, etc.  (~500 LOC)
```

Method assignment per group:

| Group | Items | Module |
|---|---|---|
| Public API + class definition (stays in `__init__.py`) | `BlockRegistry.__init__`, `add_scan_dir`, `scan`, `get_spec`, `instantiate`, `hot_reload`, `packages`, `specs_by_package`, `all_specs`; `BlockSpec` dataclass; 5 error classes | — |
| Scan / registration | `_scan_builtins`, `_scan_tier1`, `_scan_tier2`, `_scan_monorepo_packages`, `_register_spec`, `_validate_capability_registration` | `_scan.py` |
| Capability registry (ADR-043) | `list_format_capabilities`, `find_loader_capability`, `find_saver_capability`, `_find_format_capability`, `_capability_satisfies_query`, `_iter_capability_specs`, `_resolve_capability_class`, `_resolve_first_capability_class`, `_validate_dynamic_ports`, `_resolve_class` + module-level `_capability_matches_type`, `_capability_type_specificity`, `_capability_error_message`, `_validate_capability_id`, `_format_capabilities_from_class`, `_iter_compound_to_single_suffix`, `_exact_ext_in_mapping`, `_ext_in_mapping` | `_capability.py` |
| Spec construction (ADR-009 + ADR-030 MRO merge) | `_spec_from_class`, `_infer_category`, `_type_name_for_class`, `_merge_config_schema`, `_resolve_distribution_version`, `_packages_distributions_cached`, `_validate_simple_extension_declaration`, `_validate_class_capability`, `_subclass_declares_field` | `_spec.py` |

**Deleted entirely** (not relocated):

- `find_loader(dtype, ext)` — line 981 — replaced by `find_loader_capability`
- `find_saver(dtype, ext)` — line 1048 — replaced by `find_saver_capability`
- `find_io_blocks_for_type(type, direction)` — line 1082 — replaced by `list_format_capabilities` enumeration
- `_find_io_block`, `_class_accepts_dtype`, `_best_specificity`, `_matching_capabilities_for_legacy_io`, `_resolve_legacy_capability_class` — all internal-only to the deleted methods

## 4. Legacy IO finder deletion rationale

The legacy finder methods exist for the ADR-043 migration window. Today they have zero hot-path callers:

| Caller | Usage | Migration |
|---|---|---|
| `src/scistudio/blocks/code/validation.py:291` (`legacy_method_name = ...`) | Fallback when `find_loader_capability` not callable. Production `BlockRegistry` always defines the capability methods, so this branch is **dead in production** — only test mocks reach it. | Delete the fallback branch in `validation.py`. Update test mocks to provide `find_*_capability` methods. |
| `src/scistudio/blocks/io/materialisation.py:174` (`_format_supported_savers`) | Builds a human-readable error string listing all savers for a type. Not on hot path. | Rewrite using `list_format_capabilities(direction="save", data_type=type(obj))` enumeration. Equivalent output. |

`tests/blocks/test_registry_find.py` is removed in its entirety (its tests exercise the deleted API); coverage of the capability equivalents already lives in `tests/blocks/test_block_registry_capabilities.py`.

The `find_*_capability` methods raise `MissingCapabilityError` and `AmbiguousCapabilityError` (per ADR-043) — that's a stricter contract than the legacy finders, which selected "by registration order" on ambiguity. Removing the legacy methods means callers must handle the new exceptions; both production callsites already do this on the capability path (their legacy path was the silent-fallback safety net, not a behavior they depend on).

## 5. Path D binding sketch

Same shape as ADR-046. `__init__.py` defines `BlockRegistry` with method bindings to sibling-module functions. `BlockSpec` and the 5 error classes also live in `__init__.py` so import paths like `from scistudio.blocks.registry import BlockSpec, BlockRegistrationError` continue to work unmodified.

## 6. Scope

In scope:

- The 4-way sub-package split (Section 3).
- Deletion of the legacy IO finder API (Section 4) including `validation.py` + `materialisation.py` caller migration and `test_registry_find.py` removal.
- `scripts/check_god_files.py` waiver removal once new files all measure <750 LOC.
- Frontmatter `governs.files` glob updates on ADRs that reference `registry.py` (this ADR + any others surfaced by full-audit closure).
- One new test file under `tests/blocks/test_registry_package_layout.py` AST-asserting zero `class` definitions in `_scan.py` / `_capability.py` / `_spec.py`.

Out of scope:

- Any change to capability matching semantics, scan-order rules, hot-reload behavior, or `BlockSpec` schema (those are ADR-009 / ADR-043 contract surfaces).
- ADR-030 `_merge_config_schema` semantic edits.
- ADR-038 lineage interactions with the registry.

## 8. Consequences

Positive:

- Registry navigability dramatically improves (each sibling is one concern).
- Legacy IO finder dead code removed — the ADR-043 migration is finally complete on the registry side.
- `_capability.py` and `_spec.py` are good homes for any future capability/spec-related additions.

Negative / accepted:

- The `find_loader` / `find_saver` / `find_io_blocks_for_type` names become unresolved imports for any external consumer that snuck past the audit. The migration period was long; no current consumer is known.
- `_spec.py` is ~500 LOC of module-level helpers — close to the threshold. Future growth there triggers another split.

## 9. Alternatives considered

| Alternative | Reason rejected |
|---|---|
| Decompose without deleting legacy IO finder | Carries dead code into a refactor; defeats one of the wins |
| Delete legacy IO finder in a separate PR before this one | Loses cohesion — the deletion frees ~250 LOC that determines the sub-module sizes |
| Keep legacy IO finder as `@deprecated` instead of deleting | `@deprecated` adds boilerplate for a code path no production caller reaches; warning noise without benefit |
| One mega-PR combining ADR-046 + ADR-047 implementation | Audit burden too high; the two refactors are independent of each other |
