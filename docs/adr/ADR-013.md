# ADR-013: AI As A Service Layer, Not Embedded In Core

## 1. Decision Summary

SciEasy is AI-native, but the core runtime must remain usable without AI,
network access, or API keys. AI features call into core contracts; core logic
does not depend on AI.

### 1.1 Problems Addressed

| Problem | Risk | ADR-013 response | Detailed section |
|---|---|---|---|
| AI can assist blocks, types, workflows, and parameters | Embedding AI in core would make normal runtime unpredictable | Keep AI as an optional service layer | Section 3 |
| AI-generated artifacts need safety checks | Generated code could bypass contracts | Route AI output through validation and review | Section 4 |

## 2. Decision

AI capabilities live in a service layer for block authoring, workflow
synthesis, inspection, and runtime assistance. Validated AI-generated artifacts
become normal blocks, workflows, or docs after they pass the same contracts as
hand-authored work.

## 3. Scope

In scope: optional AI tools, AIBlock behavior, MCP/agent assistance, and
validation boundaries. Out of scope: requiring AI for workflow execution or
automatic AI recovery inside core runtime decisions.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- AI/MCP tests verify AI tooling uses runtime contracts.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Deep AI integration was rejected because it makes base behavior unpredictable.
No AI was rejected because SciEasy should expose generation-friendly contracts
from the start.
