# ADR-007: Lazy Loading By Default

## 1. Decision Summary

SciStudio data objects should cross block boundaries as lightweight references,
not eagerly materialized payloads. ADR-031 later removed the original
`ViewProxy` intermediary; the current contract is typed `DataObject`
instances with `StorageReference` and direct access methods.

### 1.1 Problems Addressed

| Problem | Risk | ADR-007 response | Detailed section |
|---|---|---|---|
| Scientific datasets can exceed memory | Eager block inputs can OOM | Pass references and load only on demand | Section 3 |
| Blocks need slices or metadata without full reads | Every block would over-read data | Put access methods on the data object contract | Section 4 |

## 2. Decision

The accepted decision is lazy loading by default. Blocks receive typed data
objects that point to storage and explicitly call access methods when they
need data in memory. The historical `ViewProxy` name is not current; ADR-031
moved that behavior onto `DataObject`.

## 3. Scope

In scope: reference-backed data objects, explicit materialization, and storage
abstraction. Out of scope: changing file-format capability selection,
collection transport, or loader persistence rules defined by later ADRs.

### 3.1 Signature-Level Contracts

```python
class DataObject:
    ...

class StorageReference:
    ...
```

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- Worker reconstruction and block base tests verify reference-backed transport.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Eager loading was rejected for large datasets. Memory mapping alone was too
local-storage-specific. Dask as the core abstraction was rejected as too heavy
for the base block-author contract.
