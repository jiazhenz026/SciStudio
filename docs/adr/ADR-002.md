# ADR-002: Named Axes On Array Types

## 1. Decision Summary

SciStudio `Array` values carry named axes so blocks can reason about scientific
dimensions by semantic name rather than by positional index. Axis names such
as `y`, `x`, `z`, `c`, and `lambda` are metadata on the `Array` object. They
do not define a storage backend, chunk layout, file format, or domain subtype.

This ADR preserves the historical ADR-002 decision from 2026-04-02. ADR-027
later strengthened the implementation from optional axis metadata to
explicit per-instance axes plus class-level schema constraints. That later
refinement does not replace the ADR-002 decision; it makes named axes
machine-checkable.

### 1.1 Problems Addressed

| Problem | Risk | ADR-002 response | Detailed section |
|---|---|---|---|
| Array dimensions need scientific meaning | Blocks would treat `dim 0` or `dim 2` as implicit domain knowledge | Store axis labels on `Array` instances | Section 3 |
| Multi-modal alignment needs more than rank | A 2D mask and 3D image cube can share spatial axes but differ in extra axes | Align arrays by axis name instead of position | Section 4 |
| Port constraints need semantic validation | Blocks would accept arrays with the right rank but wrong dimension meaning | Use axis-aware constraints and signatures | Section 5 |
| Axis metadata must not dictate storage | Storage and chunk layout would become coupled to user-facing semantics | Keep axes as metadata independent of storage | Section 6 |

## 2. Context

Scientific arrays often have the same rank but different meaning. A 3D array
may be `(z, y, x)`, `(c, y, x)`, `(lambda, y, x)`, or `(y, x, mz)`. A
position-only model makes every block encode assumptions such as "the third
axis is the spectral axis" or "the first two axes are spatial." Those
assumptions are fragile when workflows combine microscopy, spectroscopy, MSI,
and tabular outputs.

ADR-001 establishes `Array` as the semantic base for N-dimensional numeric
data. ADR-002 adds named axes to that base so the same `Array` type can carry
dimension meaning without forcing each domain into a new core root type.

## 3. Decision

`Array` instances carry axis labels as object metadata. The current
implementation requires callers to construct arrays with explicit axes:

```python
array = Array(axes=["y", "x"], shape=(512, 512))
```

The historical ADR-002 text allowed `axes = None` for arrays where axis
semantics were irrelevant. ADR-027 D1 superseded that implementation detail by
requiring instance-level axes and leaving the base `Array` class permissive:
it declares no required axes, no allowed-axis bound, and no canonical order.

Domain subclasses may tighten the schema by declaring class-level axis rules.
For example, an image-like subtype may require `y` and `x`, while a spectral
image subtype may also require `lambda` or `c`. Those subclasses remain plugin
or project-owned under ADR-001 and ADR-027.

## 4. Axis Alignment

Axis-aware logic compares names, not positions. A block that applies a 2D
spatial mask to a higher-dimensional spectral image should check that both
objects share `y` and `x` rather than assuming those axes occupy the same
positions.

ADR-003 builds on this decision for explicit broadcast utilities. ADR-027
later adds `Array.sel(...)` and `Array.iter_over(...)` for named-axis slicing
and iteration. Those utilities are refinements of the same architectural
decision: scientific code should use axis names where dimension meaning
matters.

## 5. Port And Type Compatibility

Axis labels are part of semantic validation but not a replacement for type
compatibility. A port still declares accepted data types through the normal
`DataObject` inheritance model from ADR-001. Axis checks add a second layer
when rank alone is insufficient.

The current implementation supports this through:

- `Array.required_axes`, `Array.allowed_axes`, and `Array.canonical_order` for
  class-level schema;
- `TypeSignature.required_axes` for capturing subclass axis requirements in
  type facts;
- `scistudio.utils.constraints.has_axes` and `has_exact_axes` for block-level
  input predicates.

## 6. Storage Independence

Named axes are metadata. They must not choose a storage backend, chunk shape,
file format, or IO handler. Storage-backed lazy loading is governed by
ADR-007, ADR-027, and ADR-031. File-format capability selection is governed
by ADR-043.

This separation keeps the type system small: `Array` says "N-dimensional
numeric data with axis labels," while storage and IO decisions remain in
their own contracts.

## 7. Scope

### 7.1 In Scope

- axis labels on `Array` instances;
- axis schema tightening by `Array` subclasses;
- axis-aware port constraints;
- axis-aware alignment as a prerequisite for ADR-003 broadcast behavior;
- preservation of storage independence.

### 7.2 Out Of Scope

- adding domain image or spectrum roots to core;
- adopting xarray as the internal data model;
- file-format capability selection;
- storage backend and chunk-layout policy;
- changing runtime code as part of this rewrite.

## 10. Consequences And Risks

Named axes make array workflows more explicit and safer. Blocks can say they
require `y` and `x` rather than relying on the shape length or a positional
convention. This also gives ADR-003 broadcast behavior a stable semantic
foundation.

The tradeoff is that array producers must provide axis labels, and plugin
subclasses must be careful not to over-constrain legitimate data. The base
`Array` class remains permissive so general numeric arrays are still possible.

Axis labels are metadata, so they can drift from real payload semantics if
loaders set them incorrectly. That risk is addressed by loader tests, package
contracts, and domain plugin validation rather than by changing the core type
hierarchy.

## 11. Alternatives Considered

### 11.1 Xarray As The Core Data Model

Making xarray-style dimension labeling the core data model was rejected. It
would force all `Array` users through one library's conventions and add
conversion overhead for scientific packages that expect plain NumPy-like
arrays.

### 11.2 Positional Dimensions Only

Keeping dimensions positional was rejected. It leaves semantics implicit and
forces every multi-modal block to duplicate assumptions about which dimension
means space, channel, time, or spectrum.

### 11.3 Domain Roots For Every Axis Pattern

Adding core roots such as `Image`, `Volume`, `Spectrum`, or `MSImage` was
rejected by ADR-001. Named axes let those meanings live in plugin subtypes and
metadata without expanding the core root taxonomy.
