# ADR-017: Subprocess Isolation For All Block Execution

## 1. Decision Summary

SciStudio executes block logic outside the engine process. The engine is an
orchestrator: it schedules work, serialises lightweight references, starts a
worker subprocess, observes its lifecycle, and reads a structured result
envelope. Scientific data does not cross the process boundary as in-memory
payloads; the boundary carries configuration and storage-backed references.

This ADR preserves the historical 2026-04-03 decision. Later ADRs refined the
transport model: ADR-020 removed engine-level batch iteration, ADR-027 and
ADR-031 refined typed worker reconstruction and lazy loading, and ADR-041
narrowed CodeBlock execution semantics. The subprocess isolation decision
remains the runtime baseline.

### 1.1 Problems Addressed

| Problem | Risk | ADR-017 response | Detailed section |
|---|---|---|---|
| User and plugin code can hang | The API server and scheduler would freeze if block code runs in-process | Run block logic in a worker subprocess | Section 3 |
| Cancellation needs an OS kill boundary | Python threads and cooperative flags cannot stop native extensions reliably | Make every execution cancellable through a process handle | Section 4 |
| Scientific data can be huge | Copying arrays or tables through stdin/stdout would be slow and fragile | Pass storage references and reconstruct typed objects in the worker | Section 5 |
| Block runners need one execution model | CodeBlock, ProcessBlock, AppBlock, IOBlock, and SubWorkflowBlock would each need bespoke lifecycle handling | Use one subprocess baseline with category-specific adapters | Section 6 |
| Isolation must be auditable | ADR text could claim subprocess isolation without importable symbols | Govern LocalRunner, worker entry points, and process factory functions | Section 7 |

## 2. Context

The early architecture allowed block logic to run in the same Python process as
the engine. That was attractive for small examples, but it did not survive the
scientific runtime requirements:

- inline Python scripts can call blocking code or import libraries that never
  yield back to the event loop;
- native libraries can segfault, leak memory, or deadlock outside Python's
  control;
- R, Julia, external applications, and user scripts are process-oriented tools;
- cancellation must work even when the block author did not cooperate;
- a large dataset should remain in its backing store rather than being copied
  through an IPC channel.

The original ADR rejected a trusted in-process fast path. The overhead of a
worker subprocess is small relative to scientific block durations, while the
cost of maintaining two execution models would leak into scheduling,
cancellation, resource management, and testing.

## 3. Decision

All normal block execution uses an isolated subprocess. The engine process
prepares a small invocation payload, launches the worker, records a process
handle, and awaits the subprocess result. The worker imports the block class,
reconstructs typed inputs, calls `run()`, serialises outputs, and exits.

The engine does not execute user block logic directly. It may still coordinate
interactive runtime flows that need frontend round-trips, but those are
explicit runtime exceptions handled by the scheduler and not a general
in-process block execution path.

## 4. Cancellation And Fault Isolation

Subprocess isolation creates a hard boundary around block execution:

- a hung block can be terminated by the process lifecycle layer;
- a crashing block becomes a block failure rather than an engine crash;
- cancellation is expressed as a runtime event and resolved through the process
  handle and scheduler semantics defined in ADR-018 and ADR-019;
- engine shutdown can terminate outstanding worker processes instead of
  leaving them orphaned.

This decision intentionally rejects thread-only cancellation. Threads can be
useful inside a worker subprocess, but the cancellation guarantee is the death
of the worker process, not cooperative interruption of every thread or library
call.

## 5. Cross-Process Data Transfer

The worker payload is reference-oriented. Inputs are serialised as JSON-friendly
wire records: storage backend, storage path, metadata, collection envelope, and
configuration. The worker reconstructs typed `DataObject` instances from those
records and preserves lazy loading at the method level. A worker only loads
scientific data when the block calls APIs such as `to_memory()`, `sel()`, or
`iter_over()`.

Outputs follow the reverse path. The worker auto-flushes persisted objects when
an output directory is available, serialises typed outputs back to the wire
format, and prints one JSON result envelope. The envelope may also include
environment and final-state sidecars consumed by scheduler and lineage logic.

The invariant is: no large scientific payload is copied through the engine to
run a block. The engine passes references; the worker touches the data store.

## 6. Block Category Impact

| Category | Runtime consequence |
|---|---|
| `ProcessBlock` | The block class is imported in the worker and receives typed reconstructed inputs. |
| `CodeBlock` | Python/R/Julia-style execution is treated as subprocess-backed code execution. ADR-041 further narrows CodeBlock v2 behavior. |
| `AppBlock` | External application subprocesses remain external, but lifecycle tracking is brought under the process-management contract. |
| `IOBlock` | File and storage IO runs with the same isolation boundary as other blocks. |
| `AIBlock` | AI execution is represented as a block-level runtime surface; latency-sensitive or PTY-backed behavior is refined by later ADRs. |
| `SubWorkflowBlock` | Nested workflows run under the same subprocess lifecycle principle instead of recursive in-process execution. |

## 7. Verification And Tooling Impact

The auditable implementation surface is the local runner, worker entry point,
and process factory. Tests should cover:

- worker payload parsing and typed reconstruction;
- worker output serialisation and final-state envelopes;
- real subprocess round-trips for representative blocks;
- Tier-1 drop-in block re-import in a fresh worker process;
- cancellation and cleanup through the process lifecycle layer.

## 8. Consequences

- Engine stability improves because block crashes, hangs, and memory faults are
  isolated from the orchestrator.
- Block cancellation becomes practical because the runtime can terminate an OS
  process instead of asking arbitrary user code to cooperate.
- The startup cost of a worker process is accepted as the price of a uniform
  lifecycle model.
- All future runner optimisations must preserve the architectural default:
  block logic does not become ambient engine-process logic by convenience.

## 9. Alternatives Considered

Keeping trusted in-process execution was rejected because it creates two
runtime models. Threads with cancellation tokens were rejected because native
scientific libraries and external tools will not honor framework-specific
flags. `multiprocessing.Process` was rejected as the primary abstraction
because it inherits too much parent-process state and gives less explicit
control than a clean worker interpreter. Containers remain a future deployment
option, but are too heavy as the local runtime baseline.
