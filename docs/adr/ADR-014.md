# ADR-014: ReactFlow And FastAPI Frontend-Backend Stack

## 1. Decision Summary

SciEasy uses a React/ReactFlow frontend over a FastAPI backend. The workflow
file remains the source of truth; the frontend is an editor and live monitor.

### 1.1 Problems Addressed

| Problem | Risk | ADR-014 response | Detailed section |
|---|---|---|---|
| Workflows need a rich visual editor | Generic UI stacks cannot handle graph editing well | Use ReactFlow for the canvas | Section 3 |
| Scientific runtime belongs in Python | Browser-only execution would lose the Python ecosystem | Use FastAPI as backend runtime boundary | Section 4 |

## 2. Decision

The frontend uses React, ReactFlow, and supporting UI state libraries. The
backend uses FastAPI with REST, WebSocket, and SSE surfaces. ADR-039 later
replaces the old in-memory revision counter with git-backed project state.

## 3. Scope

In scope: frontend/backend stack choice, API boundary, live status transport,
and GUI editing. Out of scope: making frontend state runtime truth.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- API tests verify FastAPI app setup and live project behavior.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Desktop-native UI was rejected for distribution and future web scenarios.
Streamlit or Panel were rejected for graph-editor limitations. Flask or Django
were rejected in favor of FastAPI's async, WebSocket, and Pydantic support.
