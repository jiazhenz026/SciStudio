# ADR-018: Block Cancellation, Graceful Workflow Degradation, And Event-Driven Runtime

## 1. Decision Summary

SciEasy represents workflow execution as an event-driven runtime rather than a
synchronous topological loop. The runtime adds explicit terminal states for
user cancellation and downstream skip propagation, and it routes scheduler,
resource, process, checkpoint, WebSocket, and lineage reactions through an
`EventBus`.

This ADR also incorporates the later scheduler-concurrency clarification that
was appended to the historical mega-document: independent ready branches must
run concurrently as `asyncio.Task` instances, while cancellation and shutdown
still use the process lifecycle contract from ADR-019.

### 1.1 Problems Addressed

| Problem | Risk | ADR-018 response | Detailed section |
|---|---|---|---|
| Cancellation was not represented in block state | A cancelled block would look like a generic error or disappear from the run record | Add `CANCELLED` as a first-class terminal state | Section 3 |
| Downstream blocks needed a non-error terminal state | Blocks that never ran would be misreported as failed | Add `SKIPPED` with deterministic propagation | Section 4 |
| Runtime components needed to react to changes | Scheduler, resources, WebSocket, checkpoint, and lineage would drift into separate state machines | Use `EventBus` as the coordination backbone | Section 5 |
| The scheduler could serialize independent branches | Parallel scientific workflows would run one block at a time | Dispatch long-running work in tracked `asyncio.Task` instances | Section 6 |
| Termination evidence must survive audit | Cancellation and skipped work need durable run records | Emit terminal events consumed by checkpoint and lineage systems | Section 7 |

## 2. Context

The original runtime model described the happy path: a scheduler walks the DAG,
blocks transition from ready to running to done, and checkpoints capture the
state. That model was incomplete for real user operations. Users need to stop a
long-running block, cancel a whole workflow, recover from an external process
crash, and see why downstream nodes did not run.

The problem is semantic as much as technical. `ERROR` means "this block ran and
failed." A downstream block that cannot run because an upstream output is
missing did not fail in its own logic. A user-cancelled block also differs from
an unexpected crash. The runtime needs separate states so UI, lineage,
checkpoint, and rerun behavior can be truthful.

## 3. Extended Block States

ADR-018 adds two terminal states to `BlockState`:

| State | Value | Meaning |
|---|---|---|
| `CANCELLED` | `cancelled` | The user or workflow controller explicitly terminated the block. |
| `SKIPPED` | `skipped` | The block did not execute because required upstream work did not produce usable outputs. |

The terminal set for completion is therefore `DONE`, `ERROR`, `CANCELLED`, and
`SKIPPED`. Reset and rerun operations may move terminal blocks back to `IDLE`
under scheduler control, but normal execution does not move from a terminal
state back into running work.

## 4. Cancellation And Skip Propagation

Block cancellation follows the process model:

1. A cancel request reaches the runtime through API or WebSocket surfaces.
2. The scheduler emits or receives a cancellation event.
3. If a `ProcessHandle` exists, the subprocess is terminated.
4. If no handle exists yet, the scheduler cancels the active setup task.
5. The block enters `CANCELLED`.
6. Downstream blocks whose inputs cannot be satisfied enter `SKIPPED`.

Workflow cancellation applies the same rule to all running or paused blocks and
marks not-yet-started blocks as `SKIPPED`. This is deterministic; users are not
asked to decide for each downstream block because the missing input makes the
runtime choice unambiguous.

## 5. Event-Driven Runtime

`EventBus` is the runtime coordination backbone. Components publish and
subscribe to typed event names rather than calling each other directly for every
state change.

| Event family | Examples | Runtime consumers |
|---|---|---|
| Block lifecycle | `BLOCK_RUNNING`, `BLOCK_DONE`, `BLOCK_ERROR`, `BLOCK_CANCELLED`, `BLOCK_SKIPPED` | Scheduler, WebSocket, checkpoint, lineage, resources |
| Cancellation requests | `CANCEL_BLOCK_REQUEST`, `CANCEL_WORKFLOW_REQUEST` | Scheduler |
| Process lifecycle | `PROCESS_SPAWNED`, `PROCESS_EXITED` | Scheduler, resource manager, process registry |
| Workflow lifecycle | `WORKFLOW_STARTED`, `WORKFLOW_COMPLETED`, `WORKFLOW_CHANGED` | API/WebSocket, UI, watchers |
| Checkpoint and interaction | `CHECKPOINT_SAVED`, interactive prompt/complete events | Checkpoint manager, interactive blocks |

The bus isolates subscriber failures: one callback error is logged and does not
stop later subscribers from observing the event. This matters because terminal
events are shared evidence for multiple systems.

## 6. Scheduler Concurrency Clarification

The accepted behavior is not "one block at a time." When multiple DAG branches
are independently ready and resources allow dispatch, each branch starts
without waiting for unrelated branches to finish.

The scheduler implements this by splitting dispatch into a short prelude and a
long-running task:

- `_dispatch` checks pause/resource gates, emits `BLOCK_RUNNING`, gathers
  inputs, instantiates the block, creates an `asyncio.Task`, stores it in
  `_active_tasks`, and returns.
- `_run_and_finalize` awaits the runner, records outputs or terminal errors,
  emits terminal events, saves checkpoint evidence, and removes the task from
  `_active_tasks`.
- `_check_completion` requires all block states to be terminal and
  `_active_tasks` to be empty before completing the workflow.
- shutdown calls `_cancel_active_tasks_on_shutdown` to terminate process
  handles and cancel pre-subprocess tasks.

Resource throttling does not block the event loop. If `can_dispatch` refuses a
block, the block remains `READY` and is retried when terminal or process events
free capacity.

## 7. Lineage, Checkpoint, And UI Evidence

Terminal events are the shared evidence source. Checkpoints store terminal
states. Lineage records store termination semantics and environment data.
WebSocket messages expose cancellation and skip state to the UI. ADR-038 later
superseded the original flat lineage record shape with a normalized store, but
the event semantics from this ADR remain intact: each terminal block event
still drives durable lineage evidence.

## 8. Verification And Tooling Impact

Tests should verify:

- `BlockState` includes `CANCELLED` and `SKIPPED`;
- `EventBus` supports async and sync subscribers and isolates callback errors;
- cancelling a running block emits `BLOCK_CANCELLED` and skips dependents;
- cancelling a whole workflow cancels running work and skips pending work;
- independent ready branches run concurrently;
- resource throttling leaves blocks ready and retries dispatch later;
- checkpoint and lineage consumers observe terminal events;
- WebSocket cancel messages emit runtime cancellation requests.

## 9. Consequences

- Runtime state is more honest: cancelled and skipped work no longer masquerade
  as generic failures.
- Scheduler behavior becomes concurrent and event-driven, so tests must avoid
  relying on artificial ordering between independent branches.
- Resource management becomes load-bearing; bad resource gates are visible
  because the scheduler can now attempt parallel dispatch.
- Debugging gets more asynchronous, but task naming and terminal event evidence
  provide better operational traces.
- Cancellation and shutdown behavior is safer because active tasks and process
  handles are cleaned explicitly.

## 10. Alternatives Considered

Automatic timeout states were rejected because scientific runtime duration is
too workload-dependent. Downstream `ERROR` propagation was rejected because it
misstates what happened. Prompting users for each downstream skip was rejected
because the data dependency result is deterministic. A synchronous for-loop was
rejected because it cannot respond promptly to cancellation, process death, or
parallel branch readiness.
