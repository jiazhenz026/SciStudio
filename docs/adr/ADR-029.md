# ADR-029: Variadic Port Count And Per-Instance Port Editor

## 1. Decision Summary

SciStudio supports blocks whose input and output port counts are configured per
workflow node. Variadic port lists are stored in node config as JSON-compatible
`input_ports` and `output_ports` arrays, converted to effective ports by the
existing block hooks, exposed through `BlockSpec` and API schemas, and
validated exactly like static ports.

ADR-029 Addendum 1 is folded into this standalone ADR. The addendum adds
optional min/max ClassVars so authors can constrain variadic port counts.

### 1.1 Problems Addressed

| Problem | Risk | ADR-029 response | Detailed section |
|---|---|---|---|
| Some blocks need per-instance port counts | Users chain blocks or overload one port name for many logical values | Store node-specific port lists in config | Section 3 |
| Dynamic type and dynamic count are different concerns | ADR-028 dynamic ports could be stretched beyond its design | Keep `dynamic_ports` and variadic ports independent | Section 4 |
| Validation must inspect instance ports | Graph checks would validate stale class-level ports | Use effective-port hooks for validator and runtime read paths | Section 5 |
| UI must let users add, remove, and type ports | Variadic capability is invisible or unpersisted | Add canvas and config-panel port editing contract | Section 6 |
| Blocks need type and count constraints | Any-type, any-count editors can create invalid blocks | Add allowed type lists and min/max ClassVars | Section 7 |
| Variadic APIs need audit anchors | Port schema drift would break workflows silently | Govern Block, BlockSpec, API schemas, and validator symbols | Section 9 |

## 2. Context

ADR-028 Addendum 1 handles a fixed number of ports whose accepted type changes
with configuration. ADR-029 handles the separate count dimension: a block
instance may have a user-edited number of ports, and each port has a name and a
declared type list.

Concrete motivating cases include AI analysis with N inputs and M outputs,
script blocks that declare multiple named artifacts, app/file exchange blocks
with configurable slots, and utility blocks such as `PairEditor` and
`DataRouter`.

## 3. Per-Instance Port Storage

Variadic port lists are stored in the existing block config object:

```json
{
  "input_ports": [
    {"name": "field_1", "types": ["Image"]},
    {"name": "field_2", "types": ["Image"]}
  ],
  "output_ports": [
    {"name": "counts", "types": ["DataFrame"]}
  ]
}
```

This keeps workflows single-file and uses the existing `nodes[].config`
serialization path. No new per-node sidecar file, `PortConfig` object, or
per-instance `BlockSpec` is introduced.

## 4. Relationship To ADR-028 Dynamic Ports

ADR-028 dynamic ports and ADR-029 variadic ports are independent:

| Dimension | ADR-028 dynamic ports | ADR-029 variadic ports |
|---|---|---|
| Port count | Fixed | Variable per node |
| Port name | Fixed by class | User-declared |
| Port type | Selected from enum mapping | User-declared from allowed type list |
| Storage | Existing config enum | `config["input_ports"]` and `config["output_ports"]` |

A block may use neither, either, or both, but combining both should be rare and
must keep the fixed-type and variable-count semantics explicit.

## 5. Validation And Runtime Behavior

Variadic ports are treated exactly like static ports after effective-port
resolution. Call sites that need port shape must call:

- `Block.get_effective_input_ports()`;
- `Block.get_effective_output_ports()`.

For variadic blocks, those hooks read the configured port lists and convert
dict entries into `InputPort` and `OutputPort` objects. The workflow validator
then performs the same edge and type compatibility checks it already performs
for static ports.

The scheduler and worker payload format do not need special variadic handling.
Payload items are keyed by port name and typed payloads remain self-describing.

## 6. Editor And Frontend Contract

The UI contract has two entry points:

- a compact canvas affordance for quick add/remove;
- a full config-panel port editor for renaming ports and selecting types.

Removing a connected port must disconnect or reject affected edges through
normal workflow graph mutation rules; the backend remains the source of truth
through persisted config and validation.

Type dropdown options come from allowed type lists on the block spec. Empty
allowed lists mean "any registered `DataObject` subclass".

## 7. Author Controls And Addendum 1 Limits

Variadic block authors declare:

- `variadic_inputs`;
- `variadic_outputs`;
- `allowed_input_types`;
- `allowed_output_types`;
- `min_input_ports`;
- `max_input_ports`;
- `min_output_ports`;
- `max_output_ports`.

The optional min/max fields are Addendum 1. `None` means unbounded in that
direction. Limits are meaningful only when the corresponding variadic flag is
true. They are enforced by runtime validation, workflow validation, and exposed
to the frontend so controls can disable add/remove actions at the limits.

## 8. CodeBlock And Later ADRs

The original ADR-029 discussion included CodeBlock source introspection as a
possible way to populate port lists. Later CodeBlock-specific architecture is
governed by ADR-041. This ADR still governs the general per-instance port list
mechanism that CodeBlock-compatible schemas may use, but CodeBlock execution,
file exchange, and script contract details belong to ADR-041.

## 9. Scope

In scope:

- variadic port flags on block classes;
- config-backed per-instance port lists;
- allowed type lists;
- optional min/max port count constraints;
- block registry and API schema exposure;
- workflow validation against effective ports;
- frontend editor contract at the schema level.

Out of scope:

- ADR-028 enum-driven fixed-port type changes;
- CodeBlock v2 execution and exchange semantics;
- scheduler payload format changes;
- per-block custom React nodes as the default editor model.

## 10. Verification And Tooling Impact

Tests should verify registry serialization of variadic flags, allowed type
lists, and limits; `Block` effective port conversion; workflow validation of
configured ports and min/max constraints; representative variadic blocks such
as `PairEditor` and `DataRouter`; and frontend effective-port computation.

### 10.1 Signature-Level Contracts

```text
class Block:
    variadic_inputs: ClassVar[bool]
    variadic_outputs: ClassVar[bool]
    allowed_input_types: ClassVar[list[type]]
    allowed_output_types: ClassVar[list[type]]
    min_input_ports: ClassVar[int | None]
    max_input_ports: ClassVar[int | None]
    min_output_ports: ClassVar[int | None]
    max_output_ports: ClassVar[int | None]

    def get_effective_input_ports(self) -> list[InputPort]: ...

    def get_effective_output_ports(self) -> list[OutputPort]: ...


class BlockSpec:
    variadic_inputs: bool
    variadic_outputs: bool
    allowed_input_types: list[str]
    allowed_output_types: list[str]
    min_input_ports: int | None
    max_input_ports: int | None
    min_output_ports: int | None
    max_output_ports: int | None


class BlockSummary:
    variadic_inputs: bool
    variadic_outputs: bool


class BlockSchemaResponse:
    allowed_input_types: list[str]
    allowed_output_types: list[str]
    min_input_ports: int | None
    max_input_ports: int | None
    min_output_ports: int | None
    max_output_ports: int | None


def validate_workflow(workflow: WorkflowDefinition, registry: BlockRegistry | None = None) -> None: ...
```

## 11. Consequences

- Blocks can model natural fan-in and fan-out without fake shared port names.
- Workflow YAML stores topology-like data inside node config; this is accepted
  to avoid introducing a second persistence surface.
- Validators and runtimes must consistently use effective-port hooks.
- Frontend editing and backend validation must stay aligned on type names and
  count limits.

## 12. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Separate `PortConfig` field | Requires every persistence path to learn another node field |
| Sidecar port JSON files | Makes workflows multi-file and harder to checkpoint |
| Per-block custom React nodes | Fragments common port editing behavior |
| Scheduler-level variadic payload contract | Port-keyed payloads are already sufficient |
