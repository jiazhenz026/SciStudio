## ADR-038: Unified Run Lineage Database — Reproducibility through Recipe, Not Storage

**Status**: **proposed** — design accepted in session 2026-05-15; implementation tracked in phased post-merge issues. Promotes to "accepted" after Phase 1 wires the LineageRecorder into the scheduler end-to-end with a green smoke test.
**Date**: 2026-05-15
**Related**: ADR-007 (lazy loading), ADR-012 (checkpoint pause/resume), ADR-017 (subprocess isolation), ADR-018 (terminal events), ADR-019 (ProcessHandle / ProcessRegistry), ADR-020 (Collection transport), ADR-025 (entry-points plugin), ADR-027 D5 (three-slot DataObject metadata), ADR-031 (reference-only contract), ADR-032 (project SQLite metadata — **superseded by this ADR**), ADR-035 (AIBlock on PTY), ADR-039 (git versioning — sibling)
**Supersedes**: ADR-032

---

### 1. Purpose

Establish a single coherent run-tracking system answering four user questions: **which run, which workflow, which blocks ran, with what parameters and what input/output data**. Achieved by collapsing two existing-but-disjoint subsystems (`metadata.db` from ADR-032 and `lineage.db` from ADR-007 §4.4) into a unified `lineage.db` with four normalized tables, and by wiring the existing-but-dormant `LineageRecorder` into the production scheduler path.

The deeper architectural intent: **stop conflating source version control with run tracking**. The companion ADR-039 covers source version control (workflows / blocks / notes via git). This ADR covers run tracking (immutable execution records). They share one join key: every run records the git commit SHA it ran against.

---

### 2. Context

#### 2.1 Current state — three problems

**(a) `LineageStore` is dormant scaffolding.** Investigation (2026-05-15) confirmed:

- `src/scieasy/engine/scheduler.py:102` accepts a `lineage_recorder` kwarg but both production callsites (`api/runtime.py:969-978` and `cli/main.py:187-194`) omit it. `self._lineage_recorder` is always `None`.
- BLOCK_DONE event data is `{"workflow_id", "outputs"}` only (`scheduler.py:425-431`, `:549-555`). The keys `LineageRecorder` reads (`config`, `block_version`, `input_hashes`, `output_hashes`, `environment`) are absent — even if wired up it would write empty records.
- No content hashing exists in the engine path. `output_hashes` / `input_hashes` are aspirational fields that nothing computes.
- `api/deps.py::get_lineage_store` is declared but invoked from zero routes (orphan dependency).
- `block_version` field defaults to `"unknown"` because BlockRegistry has no version-injection logic.
- Tests pass because they construct `LineageStore` and `LineageRecord` directly, bypassing the scheduler — but **no production codepath has ever written a single lineage record**.

**(b) `metadata.db` is misnamed.** Per ADR-032, `<project>/metadata.db` stores DataObject identity (object_id, derived_from, storage_path, wire_payload). The word "metadata" collides with at least four other concepts in this codebase: `DataObject.meta` (typed scientific metadata, ADR-027 D5), `DataObject.user` (user annotations, ADR-027 D5), plugin manifest metadata (ADR-037), workflow YAML `metadata:` block. The actual content of the database is provenance / lineage, not metadata. The naming itself drives user confusion when they encounter the term in a workflow context.

**(c) The two databases have no join key.** `metadata.db` uses `object_id` (FrameworkMeta UUID). `lineage.db` (the schema) uses content hashes for input/output identification. There is no SQL path from "this DataObject" (object_id) to "the block execution that produced it" (lineage record). Each system answers a partial question; together they answer none of the user's four questions.

#### 2.2 What the user needs

Four questions, all asked verbatim by users 2026-05-15:

1. **Which run was it?** — identify a specific execution by time / workflow / outcome
2. **Which workflow was being run?** — get the exact YAML topology + version at run time
3. **Which blocks executed?** — list of blocks + their order + their outcome
4. **What were the parameters and the input/output data per block?** — for methods sections and debugging

Plus one reverse query: **given a figure file on disk, find the run that produced it + its methods**.

#### 2.3 Why a refactor, not a patch

Patching (wire LineageRecorder + extend BLOCK_DONE event + leave metadata.db alone) leaves two databases with no join, two terminology surfaces ("metadata" vs "lineage"), and no answer to the reverse query. Investigation showed the two stores are doing the same job at different granularity — the cleanest fix is to collapse them.

#### 2.4 Reference precedents

- **MLflow run tracking**: per-execution runs table with parameters / metrics / artifacts. Good model for the runs table; we adapt without the artifact-store complexity (no per-run dirs — see §3.5).
- **Snakemake / Nextflow lineage**: file-based provenance, recompute-on-stale. We do less: record-and-overwrite, re-run-on-demand.
- **Fiji / ImageJ / GraphPad Prism**: no run history at all; recipe (macro / settings) is the only persisted artifact. The user explicitly invoked this as the scientific-software norm; we adopt it for intermediate data while preserving full recipe history.
- **The current `<project>/.scieasy/`** convention (ADR-034, ADR-035, ADR-036): user-scoped runtime state already lives there; lineage.db joins that family.

### 3. Decisions

#### 3.1 Single `<project>/.scieasy/lineage.db` with four tables

```sql
-- Table 1: runs — one row per workflow execution
CREATE TABLE runs (
    run_id                  TEXT PRIMARY KEY,         -- UUIDv4
    workflow_id             TEXT NOT NULL,            -- logical name e.g. "image_pipeline"
    workflow_git_commit     TEXT,                     -- SHA at run start (NULL if git unavailable)
    workflow_yaml_snapshot  TEXT NOT NULL,            -- full YAML; safety net if git is lost
    workflow_dirty          INTEGER NOT NULL,         -- 0/1 worktree-clean flag at run start
    started_at              TEXT NOT NULL,            -- ISO-8601
    finished_at             TEXT,
    status                  TEXT NOT NULL,            -- running | completed | failed | cancelled
    environment_snapshot    TEXT NOT NULL,            -- JSON: full uv pip freeze
    triggered_by            TEXT NOT NULL,            -- user | ai_block | execute_from
    parent_run_id           TEXT REFERENCES runs(run_id),  -- for re-runs / retries
    execute_from_block_id   TEXT,                     -- if this run started mid-DAG via "Run from here", which block (NULL = full DAG)
    user_notes              TEXT                      -- mutable, user-editable
);
CREATE INDEX idx_runs_workflow ON runs(workflow_id, started_at DESC);

-- Table 2: block_executions — one row per block per run
CREATE TABLE block_executions (
    block_execution_id      TEXT PRIMARY KEY,         -- UUIDv4
    run_id                  TEXT NOT NULL REFERENCES runs(run_id),
    block_id                TEXT NOT NULL,            -- DAG node id, e.g. "load_image_1"
    block_type              TEXT NOT NULL,            -- class name, e.g. "LoadImage"
    block_version           TEXT NOT NULL,            -- auto-injected from importlib.metadata
    block_config_resolved   TEXT NOT NULL,            -- JSON, post-template-expansion params
    started_at              TEXT NOT NULL,
    finished_at             TEXT,
    duration_ms             INTEGER,
    termination             TEXT NOT NULL,            -- completed | error | cancelled | skipped
    termination_detail      TEXT,
    UNIQUE (run_id, block_id)
);
CREATE INDEX idx_be_run ON block_executions(run_id);
CREATE INDEX idx_be_block ON block_executions(block_id);

-- Table 3: data_objects — DataObject identity catalog (subsumes the old metadata.db role)
CREATE TABLE data_objects (
    object_id               TEXT PRIMARY KEY,         -- FrameworkMeta UUID
    type_name               TEXT NOT NULL,            -- "Image" / "DataFrame" / ...
    backend                 TEXT,                     -- "zarr" / "arrow" / "file"
    storage_path            TEXT,                     -- best-effort; may be stale after overwrite
    size_bytes              INTEGER,                  -- captured at write; used by re-run check
    mtime_at_write          TEXT,                     -- ISO-8601; same purpose
    created_at              TEXT NOT NULL,
    wire_payload            TEXT NOT NULL,            -- full wire-format JSON for reconstruction
    derived_from            TEXT REFERENCES data_objects(object_id),       -- data-level lineage
    produced_by_execution   TEXT REFERENCES block_executions(block_execution_id)  -- run-level
);
CREATE INDEX idx_do_storage ON data_objects(storage_path);
CREATE INDEX idx_do_derived ON data_objects(derived_from);
CREATE INDEX idx_do_producer ON data_objects(produced_by_execution);

-- Table 4: block_io — port-to-DataObject edges per execution
CREATE TABLE block_io (
    block_execution_id      TEXT NOT NULL REFERENCES block_executions(block_execution_id),
    direction               TEXT NOT NULL,            -- 'input' | 'output'
    port_name               TEXT NOT NULL,
    object_id               TEXT NOT NULL REFERENCES data_objects(object_id),
    position                INTEGER NOT NULL,         -- index within port (Collections)
    PRIMARY KEY (block_execution_id, direction, port_name, position)
);
CREATE INDEX idx_io_object ON block_io(object_id);
```

**SQLite WAL mode** for multi-workflow concurrent writes. Connection lifetime tied to `ApiRuntime`; switching projects closes/reopens.

**Collection unrolling.** A `Collection[Image]` output with N items is **not** stored as a single row. Each item is unrolled into one `data_objects` row (one `object_id`, one `wire_payload` ≈ 1-2 KB per ADR-031 reference-only contract) plus one `block_io` row with the corresponding `position` index. The `_collection: true` wrapper produced by `worker.py::serialise_outputs` is **not** persisted; it can be reconstructed at read time by grouping `block_io` rows for `(block_execution_id, direction='output', port_name)` ordered by `position` and observing that all referenced `data_objects.type_name` values are homogeneous (a Collection invariant). A 1000-item Collection therefore costs ~MB of storage, not GB.

The four user questions map to one SQL query each (see §3.7).

#### 3.2 Write timing — engine-owned, block-transparent

```
ApiRuntime.start_workflow(workflow_id):
    run_id = uuid4()
    yaml = read(workflows/<workflow_id>.yaml)
    git_commit, dirty = git_engine.head_state()    # per ADR-039 (None if no git)
    env = EnvironmentSnapshot.capture(full=True)
    INSERT runs (...)
    construct LineageRecorder(lineage_store)
    pass to DAGScheduler

DAGScheduler._run_block(node_id, block, inputs, config):
    block_execution_id = uuid4()
    INSERT block_executions (started_at=now, ...)
    for each input port + position:
        INSERT OR IGNORE data_objects (wire_payload, ...)
        INSERT block_io (direction='input', ...)
    result = await runner.run(block, inputs, config)
    for each output port + position:
        INSERT data_objects (produced_by_execution=block_execution_id,
                              derived_from=obj.framework.derived_from, ...)
        INSERT block_io (direction='output', ...)
    UPDATE block_executions SET finished_at=now, duration_ms, termination=...

ApiRuntime end-of-run:
    UPDATE runs SET finished_at=now, status=...
```

All writes happen in the engine process. Worker subprocesses do not connect to lineage.db. The scheduler reads each output's `wire_payload` from the existing wire-format envelope produced by `runners/local.py` and writes it directly to data_objects.

**Block authors see zero contract change.** Blocks continue to write `Block.run(inputs, config) → outputs`. The lineage system observes externally.

#### 3.3 `block_version` is force-injected at registry time

`BlockRegistry.scan()` resolves each entry-point to its source distribution and reads `importlib.metadata.version(distribution_name)`. The resolved version is stamped onto the `BlockSpec.version` field. In-tree blocks read `scieasy.__version__`; plugin blocks read their package version per ADR-037 D11 (`[tool.scieasy] version` in `pyproject.toml`).

Default `"unknown"` is removed. Registration fails loudly if version cannot be resolved — this is a reproducibility hard requirement.

#### 3.4 No content hashing

The schema does **not** include `content_hash` columns on `data_objects`. Rationale evaluated:

- For the four user questions, content hash provides no new information that `object_id` + `storage_path` does not already provide.
- For methods export, the recipe (config + version + env) suffices.
- For replay, re-running with the same recipe is the user-facing affordance, not a hash-keyed cache lookup.
- Real hashing on large zarr/arrow stores is expensive (seconds per GB). Pseudo-hash (size + mtime + path) is just rephrased non-information.

A future caching / `-resume` feature may introduce hashing as a v2 optimization — separate ADR if and when it becomes a real ask.

#### 3.5 No per-run data isolation, no retention, no GC

The engine does not write outputs to per-run directories. Intermediate block outputs land at their natural backend-managed paths and are **overwritten freely** by subsequent runs.

`lineage.db.data_objects.storage_path` is recorded as it was at write time and is documented as **best-effort** — may point to overwritten data on subsequent reads. UI surfaces this honestly: "Data may have been overwritten since this run. Re-run to regenerate."

The user-facing affordance is **re-run with recorded settings** (§3.6), not "browse old data". This matches the dominant scientific-software pattern (Fiji, ImageJ, GraphPad). Disk usage stays bounded by one workflow's worth of intermediate data regardless of run count.

`data_objects.wire_payload` preserves full type / `meta` / `user` metadata even after the underlying file is overwritten, so the methods-export use case works without the raw data.

**Explicit trade-off — "Run from here" cannot operate against historical runs.** Because intermediate data is overwritten by subsequent runs, the only run whose intermediate outputs are guaranteed loadable is the most recent one (preserved by the ADR-012 single-slot checkpoint). To reproduce an older run, the user re-runs the **full workflow** from start with the recorded recipe (§3.6a). This is the deliberate price of the no-per-run-isolation decision and matches scientific-software norms where "the recipe is the artifact". Users who need mid-DAG reruns against arbitrary historical state should pin their critical intermediate outputs via explicit `SaveData` blocks to dedicated user-named paths.

#### 3.6 Re-run validates inputs and environment

When the user clicks "Re-run this run" on a historical run record, the engine performs two checks before re-executing:

```python
# Check 1: input files unchanged
for io in past_run.input_objects_at_workflow_boundary():
    p = Path(io.storage_path)
    if not p.exists():
        warn(f"Input {p} no longer exists")
    elif p.stat().st_size != io.size_bytes:
        warn(f"Input {p} size changed: {io.size_bytes} → {p.stat().st_size}")
    elif iso(p.stat().st_mtime) > io.mtime_at_write:
        warn(f"Input {p} modified after this run")

# Check 2: environment not drifted
current = EnvironmentSnapshot.capture(full=True)
for pkg, old_ver in past_run.environment_snapshot["key_packages"].items():
    new_ver = current.key_packages.get(pkg)
    if new_ver != old_ver:
        warn(f"{pkg}: {old_ver} → {new_ver}; result may differ")
```

Warnings are surfaceable + dismissible. Both checks are advisory, not blocking — the user can proceed.

The new run gets a new `run_id` and `parent_run_id` pointing at the source. The re-run chain is queryable: `SELECT * FROM runs WHERE parent_run_id = ?`.

#### 3.6a "Run from here" (partial DAG execution) — schema semantics

The existing `execute_from` parameter on `ApiRuntime.start_workflow` and `DAGScheduler.execute_from(node_id)` lets the user re-execute starting at an arbitrary block B in workflow `A → C → B → D → E`. Under this ADR's lineage model:

- A **new** `run_id` is created, with `execute_from_block_id = "B"` and `parent_run_id` pointing at the historical run whose outputs are reused.
- `block_executions` rows are written **only for blocks that actually execute** in this run (B, D, E). Skipped upstream blocks (A, C) get **no row** in the new run.
- B's `block_io` rows of `direction='input'` reference the `object_id` of A's outputs from the prior run (`data_objects` rows already exist; their `produced_by_execution` still points to the prior run's A execution). No data is re-recorded.
- D and E follow the normal flow: their inputs reference the new run's B outputs.
- UI rendering of the DAG for this run greys out A and C with a tooltip indicating "Reused from run <SHA> (<timestamp>)", inferred by walking `block_io.input → data_objects.produced_by_execution → block_executions.run_id`.

**Relation to ADR-012 checkpoint**: per ADR-012 and verified in `scheduler.py:1292-1408`, the checkpoint is a single-slot artifact (one file per `workflow_id`, overwritten on every `BLOCK_DONE/ERROR/CANCELLED/SKIPPED` event). It records `intermediate_refs` — wire-format dicts (not reconstructed objects) containing `path` strings pointing at the data files on disk. `execute_from(block_id)` (line 1322) hard-requires that all DAG ancestors of `block_id` have entries in `checkpoint.intermediate_refs` (line 1334-1336); the wire-format dicts are assigned directly to `_block_outputs` and shipped to worker subprocesses where `_reconstruct_one()` re-loads the typed DataObject from the path.

The implication: the checkpoint mirrors **the on-disk state at the moment of the most recent terminal block event**. "Run from here" works against that on-disk state. Once a subsequent run overwrites the intermediate data files at their natural paths (per §3.5), the older checkpoint is also overwritten with the new state — both move forward together. There is no slot in which an older intermediate state survives.

This ADR does **not** change that scope. Lineage records every run's recipe, but the intermediate data needed to start mid-DAG is preserved only in the single-slot checkpoint, not in lineage.db. Consequences:

- "Run from here" on the latest completed run → checkpoint resolves; works as today.
- "Run from here" on an older historical run → **not supported**. The intermediate data has been overwritten by subsequent runs; lineage.db's `storage_path` values for that older run are stale; the upstream DataObjects cannot be loaded back.
- "Reproduce an older historical run" → run the **full workflow from start** using the recorded recipe (workflow YAML snapshot + per-block resolved params + environment + git commit SHA). The original raw inputs in `data/inputs/` are user-managed and unaffected by runs.

The lineage `runs.execute_from_block_id` field records which block the user clicked "Run from here" on — semantic provenance for the UI. It does not enable execute-from against arbitrary historical runs.

#### 3.7 Four-question SQL

```sql
-- Q1: Which run?
SELECT * FROM runs WHERE run_id = ?

-- Q2: Which workflow was running?
SELECT workflow_id, workflow_yaml_snapshot, workflow_git_commit, workflow_dirty
FROM runs WHERE run_id = ?

-- Q3: Which blocks ran?
SELECT block_id, block_type, block_version, started_at, duration_ms, termination
FROM block_executions WHERE run_id = ? ORDER BY started_at;

-- Q4a: Per-block params?
SELECT block_config_resolved FROM block_executions WHERE block_execution_id = ?

-- Q4b: Per-block I/O DataObjects?
SELECT bio.direction, bio.port_name, bio.position,
       do.object_id, do.type_name, do.storage_path, do.wire_payload
FROM block_io bio
JOIN data_objects do USING (object_id)
WHERE bio.block_execution_id = ?
ORDER BY bio.direction, bio.port_name, bio.position;

-- Reverse: file → run
SELECT r.run_id, r.workflow_id, be.block_id, be.block_config_resolved,
       r.environment_snapshot, r.workflow_yaml_snapshot, r.workflow_git_commit
FROM data_objects do
JOIN block_executions be ON do.produced_by_execution = be.block_execution_id
JOIN runs r ON be.run_id = r.run_id
WHERE do.storage_path = ?;
```

### 4. Alternatives considered

1. **Patch instead of refactor** — wire LineageRecorder, leave metadata.db alone. Rejected: two databases with no join key, terminology confusion persists, reverse query (file → run) requires brittle SQL across two stores.

2. **Per-run data directories with retention policy** — `data/runs/<run_id>/...` + GC + pin UX. Rejected per user direction 2026-05-15: violates the "scientific software keeps the recipe, not the artifacts" norm; introduces disk bloat, GC complexity, pin/SaveData double-write that the user explicitly does not want.

3. **Content hashing for cache / replay** — `sha256` on every input/output, used to detect "same inputs → skip block". Rejected as scope creep: independent feature (Nextflow `-resume` style) that doesn't answer the four user questions; defer to potential ADR-040 if cache becomes a real ask.

4. **Keep metadata.db separate** (ADR-032 stays alive) — Rejected: maintaining two stores with overlapping ownership is the source of the current confusion. The unification is the architectural point.

5. **Object-store / append-only log** (instead of SQLite) — overkill for KB-MB scale; SQLite is in stdlib and already used by ADR-032 / lineage. Rejected.

6. **Tie run identity to git commit SHA + input hash** (no separate run_id UUID) — Rejected: runs need to exist even without git (newly opened project pre-init), and need to exist independently of input hashing (which we are not doing). Explicit UUID is also easier to surface in UI.

7. **Wire payload compression by default** — premature optimization. Defer; see §8 OQ-3.

### 6. Phased implementation

Each phase = one PR. Each phase ships green and is independently useful.

### 7. Consequences

#### 7.1 Wins

- The four user questions become one-SQL queries.
- The reverse query (figure → methods) works.
- Single source of truth for run history; single name ("lineage") with unambiguous meaning.
- `LineageStore` and `LineageRecorder`, dormant since their introduction, finally serve their design purpose.
- Block author contract preserved; zero block-side migration.
- Methods export, the longest-standing reproducibility ask from scientific users, becomes a one-click affordance.
- Disk usage stays bounded by the recipe, not the run count.

#### 7.2 Losses

This ADR is **net-additive** relative to the existing implementation: no user-visible capability is removed. The latest run's intermediate data continues to live at its natural paths and remain viewable in the data preview exactly as today, and the ADR-012 checkpoint's "Run from here" against the latest run is unchanged. Historical run intermediate data was already not preserved before this ADR (no per-run directories ever existed); §3.5 codifies that pattern as an intentional architectural decision rather than introducing a new constraint.

Actual losses are limited to:

- `metadata.db` programmatic consumers break during the 6-month deprecation window. No in-tree consumers expected — this was an internal store. External code (if any) imports `scieasy.core.metadata_store` and must migrate to `scieasy.core.lineage.store`.
- Wire_payload column grows linearly with DataObject count, but each row is KB-scale (FrameworkMeta + Pydantic meta + user dict + StorageReference; no data payload, just references per ADR-031). A 1000-item Collection = ~MB of rows. No practical scaling concern at v1.
- Existing `tests/engine/test_lineage_recorder.py` and `tests/core/test_lineage*.py` need schema migration to align with the new four-table layout (Phase 1).
- `block_version = "unknown"` default is removed. In-tree blocks and plugins lacking discoverable distribution metadata will fail registration. Mitigation: in-tree blocks read `scieasy.__version__`; plugin blocks read their PyPI package version (ADR-037 D11). The migration affects only blocks registered through irregular paths.

#### 7.3 Risks

- **SQLite WAL contention** under high parallelism (multiple workflows + multiple block executions per second writing concurrently). Mitigation: bulk INSERT in transactions per block; benchmark in Phase 1.
- **Storage growth** is bounded by `O(total_DataObjects_ever_produced × KB)`. A heavy user with 100 runs × 1000-item Collections = ~100MB lineage.db over months. SQLite handles GB; no compression infrastructure needed for v1.
- **Force-inject `block_version` failures** for legacy in-tree blocks lacking version. Mitigation: in-tree blocks read `scieasy.__version__` as a uniform default; only mis-configured plugins should fail. Document recovery path.
- **Re-run validation false alarms**: file system mtime resolution is OS-dependent (1s on FAT, 1ns on ext4); same file copied may have different mtime. Treat mtime drift as warning, not error.

Tentatively: one row per block execution; AI Block transcript stays in the existing `<project>/.scieasy/ai-block-runs/<block_execution_id>/transcript.jsonl`.

6. **OQ-6**: Methods export — markdown only for v1, or also LaTeX / Word? Markdown only for v1; user can convert with pandoc.

### 9. Out of scope

- Content hashing for cache or replay (potential ADR-040).
- Per-run data directory isolation (rejected per §3.5).
- Retention policy / GC / pin (rejected per §3.5).
- Git-style versioning of lineage.db itself (handled by user editing `.gitignore` if desired; default keeps it out of git per ADR-039).
- Run sharing / collaboration (lineage.db is local per ADR-039 default; collaboration is a separate future ADR).
- Lineage diff across runs UI (compute is trivial via SQL; UI element deferred to a v2 polish phase).
- Migration of pre-existing projects' historical metadata.db data (per user direction 2026-05-15 — project is pre-release).
- AI Block transcript ingestion into lineage.db (transcript files stay separate, referenced by path).
- **MCP tool extensions for lineage / git access**. Agents query `lineage.db` via SQL and use bash `git` inside their PTY — both already work without new MCP tools. A small `skills/scieasy/SKILL.md` update documenting the new schema + git availability is a **separate post-merge issue** filed alongside this PR. The MCP server is not extended.

### 10. Decision summary

**Collapse `metadata.db` and `lineage.db` (the schema) into a single `<project>/.scieasy/lineage.db` with four normalized tables — `runs`, `block_executions`, `data_objects`, `block_io` — and wire the dormant `LineageRecorder` into the production scheduler path.** Force-inject `block_version` from `importlib.metadata`. Capture full `uv pip freeze` as `environment_snapshot`. Block authors see zero contract change. No per-run data isolation, no content hashing, no retention policy — overwrite is fine because the recipe (workflow YAML snapshot + resolved params + environment) is preserved and re-runnable. Re-run validates inputs (size+mtime) and environment drift advisory-only. UI: implement the existing-but-empty Lineage tab as a two-pane runs-list + run-detail surface; delete the Jobs tab. Status flips to `accepted` after Phase 1 smoke test green. ADR-032 status flips to `superseded by ADR-038` when Phase 2 ships.

The architectural invariant: **history is two layers — git (source) per ADR-039, lineage.db (runs) per this ADR**. The join key is `runs.workflow_git_commit`. The conflation of these two concerns into one "metadata" subsystem ends here.
