# ADR-001: Six Base Data Types With Inheritance

## 1. Decision Summary

SciEasy defines a small semantic data-object hierarchy rooted at
`DataObject`. The accepted base data types are `Array`, `Series`,
`DataFrame`, `Text`, `Artifact`, and `CompositeData`. Domain-specific data
types extend those bases through normal Python inheritance instead of adding
more first-class roots to core.

This ADR preserves the historical ADR-001 decision from 2026-04-02 and
normalizes it under the ADR-042 document structure. Later ADRs refine storage,
collection transport, metadata, and lazy loading, but they do not replace the
six-base-type decision.

### 1.1 Problems Addressed

| Problem | Risk | ADR-001 response | Detailed section |
|---|---|---|---|
| Scientific workflows exchange many data shapes | A workflow type system with too many roots becomes hard to learn and hard to validate | Use six broad semantic base types under `DataObject` | Section 3 |
| Domain data types need extensibility | Core would need to own every image, spectrum, omics, or report subtype | Let plugins and users define subclasses of the six bases | Section 4 |
| Ports need predictable compatibility checks | Blocks may accept incompatible inputs or reject valid subtypes | Use inheritance-aware type signatures and registry checks | Section 5 |
| Multi-modal containers need a first-class model | Heterogeneous containers such as AnnData or SpatialData would be forced into one flat table or array shape | Model heterogeneous named slots with `CompositeData` | Section 6 |

## 2. Context

SciEasy workflows move scientific data between typed blocks. The data may be
numeric arrays, spectra, tables, text, generated reports, opaque binary files,
or multi-modal containers that group several data objects together.

The type system therefore needs two qualities at the same time:

- a small learning surface for block authors and workflow users;
- enough extensibility for scientific domains that SciEasy core does not own.

ADR-001 chooses a stable set of semantic base classes and lets domain packages
extend them. The decision is intentionally about semantic shape, not storage
format. ADR-043 later keeps file formats and IO capabilities out of the data
type hierarchy.

## 3. Decision

SciEasy uses `DataObject` as the root for first-class data values that cross
block boundaries. The accepted concrete base types are:

| Base type | Semantic role | Examples |
|---|---|---|
| `Array` | N-dimensional numeric data | Images, image stacks, tensors, hyperspectral cubes |
| `Series` | One-dimensional indexed data | Spectra, chromatograms, time series |
| `DataFrame` | Columnar tabular data | Peak tables, cell metadata, feature matrices |
| `Text` | Textual content | Plain text, Markdown, JSON text payloads |
| `Artifact` | Opaque file-oriented output | PDFs, reports, binary blobs, external app artifacts |
| `CompositeData` | Named heterogeneous slots | AnnData-like or SpatialData-like containers |

SciEasy core owns these six bases. Domain-specific types, such as microscopy
images, Raman spectra, mass-spectrometry images, or single-cell containers,
must be subclasses supplied by plugins or project code unless a later accepted
ADR explicitly moves a type into core.

## 4. Extension Model

Users and plugin packages extend the type hierarchy with ordinary Python
inheritance. A domain type should inherit from the narrowest appropriate base:

- image-like and tensor-like data inherit from `Array`;
- spectrum-like and chromatogram-like data inherit from `Series`;
- table-like data inherit from `DataFrame`;
- textual reports or prompts inherit from `Text` when a specialized type is
  needed;
- file-backed opaque outputs inherit from `Artifact`;
- heterogeneous containers inherit from `CompositeData`.

Core must not add fine-grained domain roots merely to improve palette names,
file-format dispatch, or one package's ergonomics. Those concerns belong in
plugin metadata, IO capabilities, block manifests, and typed metadata models.

## 5. Port Compatibility

Port compatibility is inheritance-aware. A block that accepts a base type must
also accept registered subtypes of that base. A block that requires a domain
subtype may reject a broader base instance.

The type registry and type-signature model preserve this behavior by recording
the chain from `DataObject` through the concrete subclass. Compatibility is a
semantic type decision; it must not depend on frontend labels, file extensions,
or storage backend names.

## 6. Composite Data

`CompositeData` is the base type for named collections of heterogeneous
`DataObject` slots. It exists because some scientific data models are not a
single table, a single array, or a list of homogeneous items.

Each slot in a `CompositeData` instance is itself a typed `DataObject`.
Slot-level expectations may be added by domain subclasses, but the root
decision remains that heterogeneous multi-modal containers are first-class
data values rather than disguised `DataFrame` or `Array` objects.

## 7. Scope

### 7.1 In Scope

- the six semantic base data types;
- `DataObject` as the root of first-class block data values;
- inheritance-based extension by users and plugins;
- type-signature and registry compatibility for block ports;
- `CompositeData` as the heterogeneous-slot model.

### 7.2 Out Of Scope

- storage backend selection and lazy materialization, refined by ADR-007,
  ADR-027, and ADR-031;
- collection transport and batch semantics, governed by ADR-020;
- file format capability registration, governed by ADR-043;
- domain package ownership for concrete scientific subtypes.

## 9. Verification

Verification for this ADR is satisfied when:

- frontmatter lint accepts this ADR under the ADR-042 schema;
- the built-in type registry exposes `DataObject` and the six base types;
- registry compatibility tests confirm inheritance-aware matching;
- worker reconstruction tests preserve type chains across process and storage
  boundaries;
- plugin SDK integration tests demonstrate third-party `DataObject` subtype
  registration.

## 10. Consequences And Risks

The six-base-type model gives users a compact vocabulary and gives plugin
authors a stable extension target. It also limits core churn because new
scientific domains normally add plugin subtypes instead of new core roots.

The tradeoff is that some domain distinctions are not visible at the root
level. Blocks that require those distinctions must declare a domain subtype,
typed metadata requirement, or IO capability rather than asking core to add a
new base class.

`CompositeData` introduces extra validation complexity because slot-level
constraints may matter. That complexity is accepted because forcing
heterogeneous scientific containers into a flat table or single array would
misrepresent their structure.

## 11. Alternatives Considered

### 11.1 Fewer Base Types

A smaller model with only `Array`, `DataFrame`, and a blob-like type was
rejected. It would collapse series-like data, text, and opaque artifacts into
types that do not describe their semantics well enough for block contracts.

### 11.2 Many Domain Roots In Core

A larger model with first-class roots such as `Image`, `Volume`,
`TimeSeries`, `Spectrum`, or `PeakTable` was rejected. It would create a
large and unstable core taxonomy. Inheritance lets domain packages add these
types without changing the core contract.

### 11.3 No Composite Base

Treating AnnData-like or SpatialData-like values as `DataFrame` subclasses was
rejected. These objects are heterogeneous containers with multiple named
slots, not a single columnar table.
