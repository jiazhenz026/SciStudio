# ADR-005: CodeBlock Supports Inline And Script Execution Modes

## 1. Decision Summary

ADR-005 accepted CodeBlock as the bridge for user-authored analysis code. The
historical decision allowed both inline snippets and script files. Current
CodeBlock v2 is narrowed by ADR-041 toward project-local script file exchange;
inline fields now exist only for explicit migration diagnostics.

### 1.1 Problems Addressed

| Problem | Risk | ADR-005 response | Detailed section |
|---|---|---|---|
| Users already have scripts | Requiring a custom block API would block adoption | Provide CodeBlock as a user-code boundary | Section 3 |
| Short snippets and larger scripts have different ergonomics | One execution style is either too heavy or too loose | Preserve the historical inline/script distinction while noting ADR-041 narrowing | Section 4 |

## 2. Decision

CodeBlock is the block category for user-provided code. The original accepted
contract supported inline code for short snippets and script mode for existing
`.py`, `.R`, or `.jl` files. The current runtime contract is script-oriented:
ADR-041 rejects inline execution in CodeBlock v2 and directs framework-shaped
logic to `ProcessBlock`.

## 3. Scope

In scope: CodeBlock as user-code integration, script execution, migration
diagnostics for legacy inline configs, and runner abstraction. Out of scope:
multi-user sandboxing, notebook state management, and replacing reusable typed
blocks with scripts.

### 3.1 Signature-Level Contracts

```python
    ...

class CodeBlockConfig:
    ...

class CodeRunner:
    ...

class PythonRunner:
    ...
```

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- CodeBlock exchange and workflow-validator tests cover the current v2 script
  contract and legacy inline diagnostics.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Inline-only mode was rejected for real scripts. Script-only mode was too much
friction for the original design, but ADR-041 later narrowed the implemented
runtime for reliability. Notebook-style cells were rejected as duplicating
Jupyter's state model.
