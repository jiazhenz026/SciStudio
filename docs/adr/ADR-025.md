# ADR-025: Block Package Distribution Protocol With Entry Points

## 1. Decision Summary

SciEasy block packages are distributed as ordinary Python packages that
register through Python entry-points. External packages can expose block
classes under `scieasy.blocks` and custom `DataObject` subtypes under
`scieasy.types`. Block packages may also provide package-level metadata through
`PackageInfo`, allowing the GUI to group blocks by package and category.

This ADR preserves the accepted 2026-04-05 package-distribution decision. The
historical `scieasy.adapters` portion of ADR-025 is no longer active: ADR-028
supersedes the central adapter registry and moves concrete IO behavior into
`IOBlock` subclasses registered as blocks.

### 1.1 Problems Addressed

| Problem | Risk | ADR-025 response | Detailed section |
|---|---|---|---|
| External block packages need a standard registration path | Domain packages would rely on ad hoc filesystem scanning | Use Python entry-points for package discovery | Section 3 |
| The GUI needs package-level grouping | Blocks from multiple domains would appear as one flat list | Introduce `PackageInfo` and `package_name` on block specs | Section 4 |
| Custom data types must be known to the runtime | Port validation and reconstruction would lose plugin type identity | Add `scieasy.types` type discovery | Section 5 |
| IO adapter protocol changed after acceptance | ADR-025 could point implementers at deleted adapter APIs | Record ADR-028 supersession of `scieasy.adapters` | Section 6 |
| Entry-point behavior needs auditable contracts | Distribution rules could drift from registry code | Govern registry, package metadata, and type registry symbols | Section 9 |

## 2. Context

ADR-009 established a registry that stores block specs instead of live class
objects. ADR-008 introduced tiered discovery. To make `pip install
scieasy-blocks-srs` work, SciEasy needs a package-level protocol that does not
depend on users manually copying files into a project directory.

Entry-points are the accepted Python packaging mechanism for this purpose. They
let SciEasy discover installed block packages while keeping domain-specific
dependencies outside the core package.

## 3. Entry-Point Groups

The active entry-point groups are:

| Group | Purpose | Active status |
|---|---|---|
| `scieasy.blocks` | Discover block classes and optional package metadata | Active |
| `scieasy.types` | Discover custom `DataObject` subclasses | Active |

The historical group below is not active:

| Group | Historical purpose | Current status |
|---|---|---|
| `scieasy.adapters` | Discover format adapters | Superseded by ADR-028; plugin IO is registered as `IOBlock` subclasses under `scieasy.blocks` |

External packages should not implement new `FormatAdapter` registries for
current SciEasy.

## 4. Block Package Metadata

An external package may expose a callable that returns either:

```python
(PackageInfo(...), [BlockClassA, BlockClassB])
```

or, for backward compatibility:

```python
[BlockClassA, BlockClassB]
```

When `PackageInfo` is provided, the registry records the package display name
and stores that name on each resulting `BlockSpec.package_name`. When only a
plain list is returned, the entry-point name becomes the package name.

The GUI groups palette entries as package, then block category, then block.
Block category remains the block author's domain label and is not a fixed core
taxonomy beyond the base block category.

## 5. Type Registration

External packages may register custom `DataObject` subclasses through
`scieasy.types`. The type registry loads each entry-point callable, expects a
list or tuple of type classes, validates each class, and registers valid
subtypes.

Type registration supports:

- port compatibility checks across plugin types;
- reconstruction from persisted type chains;
- frontend type hierarchy metadata for preview and port rendering.

Domain-specific metadata persistence remains the plugin author's
responsibility. Small JSON metadata may live on the data object's metadata
surface; larger or specialized metadata may use plugin-owned sidecar files or
IO blocks.

## 6. Adapter Supersession

The original ADR-025 included a `scieasy.adapters` group for custom IO
adapters. ADR-028 later removed the central adapter registry and replaced that
design with `IOBlock` subclasses. Plugin-owned loaders and savers now register
as normal blocks.

This ADR keeps the historical context but does not govern new adapter-registry
work. Active implementations should follow ADR-028 and later IO ADRs.

## 7. Built-In And External Blocks

`pip install scieasy` ships general-purpose core blocks. Domain-specific
blocks, such as imaging, SRS, and LC-MS packages, should live in separate
Python packages with their own dependencies and entry-points.

This balances first-run usability with dependency containment.

## 8. Scope

In scope:

- Python packaging entry-point protocol for blocks and types;
- `PackageInfo` metadata;
- two-level GUI package/category grouping;
- backward-compatible block entry-point returns;
- ADR-028 correction for the removed adapter group.

Out of scope:

- central `FormatAdapter` resurrection;
- plugin dependency installation UX, which belongs to later packaging ADRs;
- block SDK scaffolding, which belongs to ADR-026;
- frontend implementation details beyond the package/category grouping
  contract.

## 9. Verification And Tooling Impact

Tests should verify tuple and plain-list block entry-point returns, package
metadata capture, `specs_by_package()` grouping, `packages()` metadata
response, type entry-point registration, invalid type entry-point warnings, and
the SDK scaffold's generated package metadata.

### 9.1 Signature-Level Contracts

```python
class PackageInfo:
    name: str
    description: str
    author: str
    version: str


class BlockSpec:
    package_name: str


class BlockRegistry:
    def packages(self) -> dict[str, PackageInfo]: ...

    def specs_by_package(self) -> dict[str, list[BlockSpec]]: ...


class TypeRegistry:
    def _scan_entrypoint_types(self) -> None: ...
```

## 10. Consequences

- External developers can distribute blocks with normal Python packaging.
- SciEasy core stays smaller because domain dependencies live in plugins.
- The GUI can present package-level grouping instead of a flat block list.
- Type identity can survive plugin boundaries when packages register custom
  types.
- Implementers must follow ADR-028 for IO adapters and not the superseded
  `scieasy.adapters` text.

## 11. Alternatives Considered

Shipping all domain blocks in core was rejected because it would bloat the main
package with specialized dependencies.

Making every block external was rejected because a fresh SciEasy installation
should still provide useful built-ins.

Custom plugin YAML manifests were rejected because Python entry-points already
solve installed package discovery.

A single entry-point group for blocks, types, and adapters was rejected because
these surfaces have different lifecycles and registry behavior.
