# ADR-008: Two-Tier Block And Type Distribution

## 1. Decision Summary

SciEasy supports quick local extension and packaged community extension.
Drop-in project files and installed packages feed the same block and type
registries.

### 1.1 Problems Addressed

| Problem | Risk | ADR-008 response | Detailed section |
|---|---|---|---|
| Bench scientists need low-friction local blocks | Packaging-only extension would be too heavy | Support drop-in project files | Section 3 |
| Community packages need versioned distribution | Local files alone do not scale to ecosystems | Support entry-point packages | Section 3 |

## 2. Decision

Tier 1 is drop-in files in project or user extension directories. Tier 2 is
standard Python packages with `scieasy.blocks` and `scieasy.types`
entry-points. Both tiers register through `BlockRegistry` and `TypeRegistry`.

## 3. Scope And Consequences

The two-tier model gives a five-minute local path and a durable package path.
Registry scanning costs are accepted. Installed packages are the authoritative
distribution mechanism for reusable modality-specific blocks.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- Registry tests verify descriptor storage and entry-point importability.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Package-only extension was rejected as too much friction. A marketplace tier
was deferred as product infrastructure, not required for the base
architecture. Database or cloud-loaded code was rejected for inspectability and
versioning reasons.
