# ADR-016: Per-Port InputDelivery For CodeBlock Data Handoff

## 1. Decision Summary

ADR-016 recorded the original `InputDelivery` design for CodeBlock inputs. It
is superseded by ADR-020, which removed the `InputDelivery` enum and uses
collection auto-unpack/repack for CodeBlock handoff.

### 1.1 Problems Addressed

| Problem | Risk | ADR-016 response | Detailed section |
|---|---|---|---|
| CodeBlock needed a bridge from lazy data to user scripts | Always materializing large inputs could OOM | Historical `InputDelivery` modes were proposed | Section 3 |
| The historical modes complicated the runtime contract | PROXY and CHUNKED leaked framework knowledge into scripts | ADR-020 replaces them with Collection handoff | Section 4 |

## 2. Historical Decision

The original design had `MEMORY`, `PROXY`, and `CHUNKED` per input port.
Inline mode defaulted to memory delivery. Script mode could opt into proxy or
chunked behavior.

## 3. Supersession

ADR-020 removed `InputDelivery`. CodeBlock now uses collection auto-unpack and
native in-memory handoff. Users needing lazy or chunk-aware behavior should
write a framework-aware `ProcessBlock`.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- State enum and CodeBlock tests verify `InputDelivery` remains removed.

## 5. Alternatives Considered

Always materializing data was too risky for large inputs. Always passing lazy
framework objects broke normal scripts. Auto-detection by size was rejected as
unpredictable.
