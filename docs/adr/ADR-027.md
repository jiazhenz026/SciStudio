# ADR-027: Phase 10 Core Type System And Block Runtime Refinements

## 1. Decision Summary

Phase 10 refines SciEasy's core type system and block runtime so plugin-owned
scientific domains can handle real 5D and 6D data without moving domain logic
back into core. `Array` carries instance axes, domain subtypes move to plugins,
metadata becomes stratified, process blocks gain setup/teardown hooks, resource
scheduling learns block-internal worker counts and GPU auto-detection, and
workers scan registered plugin types before reconstruction.

The ADR-027 addendum is preserved in this standalone file as historical
context. ADR-031 later supersedes that addendum's remaining ViewProxy language
and restates the active reference-only data contract.

### 1.1 Problems Addressed

| Problem | Risk | ADR-027 response | Detailed section |
|---|---|---|---|
| Fixed class-level axes do not represent microscopy and spectral stacks | Every axis combination would require a new core subclass | Move axes to instances and keep class-level axis constraints | Section 3 |
| Domain subtypes leak into core | Core grows domain dependencies and plugin packages cannot own their types | Keep only base types in core; move domain subtypes to plugins | Section 4 |
| Block authors need slice iteration and metadata propagation | Blocks materialize whole arrays and drop provenance | Add axis iteration, `Array.sel`, `Array.iter_over`, and stratified metadata | Section 5 |
| Expensive setup is repeated per Collection item | GPU/model blocks reload resources for every item | Add `ProcessBlock.setup` and `teardown` lifecycle hooks | Section 6 |
| Resource scheduling lacks thread and GPU policy | Blocks oversubscribe CPU or never dispatch on GPU defaults | Add `max_internal_workers` and GPU auto-detection | Section 7 |
| Worker subprocesses must reconstruct plugin types | Moved domain types become unknown in isolated workers | Warm the `TypeRegistry` and reconstruct typed DataObjects | Section 8 |
| Addendum behavior was partly superseded | ViewProxy and in-memory transport claims diverged from later decisions | Preserve addendum but mark ADR-031 as the active successor | Section 9 |

## 2. Context

ADR-001 through ADR-026 define the first SciEasy runtime architecture. Phase 10
introduced domain plugin packages, starting with imaging, and exposed gaps in
the original core type and runtime surfaces. The original core still contained
domain types such as `Image`, had fixed class-level axes, lacked typed metadata
propagation, and did not scan plugin type entry-points in workers.

These concerns are bundled because they form one coherent plugin-readiness
package. Moving domain types to plugins requires worker type discovery.
Instance axes need slice helpers. Slice helpers need metadata propagation.
GPU imaging blocks need setup/teardown and resource policy.

## 3. Array Axes And Core Type Boundary

`Array` instances carry their own `axes: list[str]`. Classes declare only
constraints:

- `required_axes`: the minimum axes required by the class;
- `allowed_axes`: the accepted axis alphabet, or `None` for unrestricted;
- `canonical_order`: the preferred axis ordering for display and transforms.

The Phase 10 scientific axis alphabet is `{"t", "z", "c", "lambda", "y",
"x"}`. `c` is a discrete channel axis; `lambda` is a continuous spectral axis.
They may coexist in one instance.

Core contains only base data types. Domain subtypes such as image, spectral,
mass-spec, single-cell, and spatial-omics types belong to plugin packages and
are discovered through `scieasy.types`.

## 4. Plugin-Owned Domain Types

The accepted boundary is:

| Core keeps | Plugins own |
|---|---|
| `DataObject`, `Array`, `Series`, `DataFrame`, `Text`, `Artifact`, `CompositeData`, `Collection` | `Image`, `FluorImage`, `SRSImage`, `MSImage`, `Spectrum`, `PeakTable`, `AnnData`, `SpatialData`, and other domain subtypes |

Port compatibility should continue to rely on Python type relationships and
registered type metadata. Built-in generic blocks should depend on base types
unless a plugin-specific block is explicitly imported from a plugin package.

## 5. Array Iteration, Selection, Metadata, And Units

SciEasy exposes two complementary axis helpers:

- `iterate_over_axes(source, operates_on, func)`: an explicit utility for
  applying a function to every slice outside the axes a block operates on.
- `Array.sel(**kwargs)` and `Array.iter_over(axis)`: object-level helpers for
  named-axis selection and one-axis iteration.

Selection and iteration preserve metadata:

- framework metadata is derived from the parent object;
- typed domain metadata is shared or copied according to the current
  immutable metadata contract;
- user metadata is shallow-copied;
- axes and shape are updated for the resulting slice.

`PhysicalQuantity` records small, fast unit-bearing values for common scientific
metadata. The original ADR rejected `pint` for this layer because import cost
matters in isolated worker subprocesses.

## 6. ProcessBlock Lifecycle

`ProcessBlock` gains per-run lifecycle hooks. `setup(config)` is called once
before item iteration and returns an opaque state object. `process_item(item,
config, state=None)` receives that state. `teardown(state)` runs in a `finally`
block.

This keeps common blocks simple while allowing expensive resources such as
models, database handles, or compiled transforms to be loaded once per block
run rather than once per item.

## 7. Threads, Fan-Out, And Resources

Threads inside a block subprocess are allowed as an escape hatch, especially
for libraries that release the GIL or manage GPU kernels internally. The
recommended Collection-level parallelism remains graph-level L2 fan-out:
split the Collection, run parallel branches, and merge the result. That pattern
keeps cancellation, resource accounting, and future distribution at workflow
graph level.

`ResourceRequest.max_internal_workers` declares internal parallelism. The
scheduler uses `effective_cpu = cpu_cores * max_internal_workers` for CPU
gating.

`ResourceManager(gpu_slots=None)` auto-detects GPU count once, using torch when
available and then `nvidia-smi`, with `0` fallback. Explicit integer overrides
are respected. If no GPU is available and a block requires GPU, the manager
emits a one-time warning.

## 8. Worker Type Reconstruction

Worker subprocesses must scan registered types before reconstructing payloads.
The worker resolves type chains through `TypeRegistry`, reconstructs typed
`DataObject` instances, and serializes outputs symmetrically.

The active implementation delegates per-object reconstruction and serialization
to `scieasy.core.types.serialization` helpers. `Array` provides extra metadata
hooks so axes, shape, dtype, and chunk shape round-trip through the worker wire
format.

## 9. Addendum 1: Typed Reconstruction And ADR-031 Supersession

The original ADR-027 Addendum 1 changed worker reconstruction from ViewProxy
injection to typed `DataObject` reconstruction. That direction remains
important, but ADR-031 supersedes the addendum as the active architecture for
lazy loading and reference-only transport.

Current reading:

- Active: worker reconstruction returns typed `DataObject` instances.
- Active: per-class hooks such as `Array._reconstruct_extra_kwargs` and
  `Array._serialise_extra_metadata` round-trip subtype metadata.
- Active: `TypeRegistry` resolves plugin-provided types.
- Superseded by ADR-031: ViewProxy as an opt-in lazy helper.
- Superseded by ADR-031: any implication that raw data payloads may cross block
  boundaries as ordinary `DataObject` instance state.

## 10. Scope

In scope:

- core base type boundary;
- instance-level array axes;
- axis iteration and selection helpers;
- stratified metadata and physical quantities;
- ProcessBlock lifecycle hooks;
- internal thread policy and L2 fan-out recommendation;
- GPU auto-detection and internal-worker resource accounting;
- worker type registry scan and typed reconstruction;
- historical addendum preservation.

Out of scope:

- domain subtype implementations inside plugin packages;
- full reference-only storage enforcement, which belongs to ADR-031;
- frontend UX for type hierarchy rendering;
- changing the workflow graph language.

## 11. Verification And Tooling Impact

Tests should cover axis validation, `sel`, `iter_over`, axis utility behavior,
metadata propagation, `PhysicalQuantity` conversion and Pydantic integration,
ProcessBlock setup/teardown behavior, resource accounting and GPU defaults,
type registry resolution, and worker typed reconstruction/serialization.

## 12. Consequences

- Core remains small while plugin packages own domain semantics.
- Block authors can work with named axes without creating subclass explosions.
- Runtime reconstruction can preserve plugin type identity across subprocesses.
- Some later decisions, especially ADR-031, override parts of the original
  addendum; this ADR must be read with those supersession notes.

## 13. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Keep class-level axes only | Cannot represent routine 5D and 6D data without many subclasses |
| Keep domain subtypes in core | Violates plugin boundary and grows core dependencies |
| Add many axis-specific block base classes | Encodes a function argument as inheritance hierarchy |
| Use `pint` for all units immediately | Import cost is high for per-block subprocess startup |
| Require block-internal thread pools for Collection parallelism | Hides resource use from the workflow scheduler |
