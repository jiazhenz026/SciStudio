# ADR-009: Registry Stores Specs, Not Class References

## 1. Decision Summary

SciStudio registries store descriptors, not live class objects. Reloading a
drop-in file updates specs for future instantiation without mutating running
workflow instances.

### 1.1 Problems Addressed

| Problem | Risk | ADR-009 response | Detailed section |
|---|---|---|---|
| Drop-in files can change during a session | Cached class references become stale after reload | Store `BlockSpec` and `TypeSpec` descriptors | Section 3 |
| Running workflows must stay stable | Hot reload could change in-flight objects | Instantiate fresh only for future runs | Section 4 |

## 2. Decision

`BlockRegistry` stores `BlockSpec` records. `TypeRegistry` stores `TypeSpec`
records. Specs include names, module paths, class names, metadata, and reload
metadata. Runtime code imports a fresh class when it needs to instantiate.

## 3. Scope And Consequences

Reload is predictable: scan files, update specs, notify users, and leave
running instances alone. The cost is a small import-time overhead when
instantiating from descriptors.

### 3.1 Signature-Level Contracts

```python
class BlockSpec:
    ...

class BlockRegistry:
    ...

class TypeSpec:
    ...

class TypeRegistry:
    ...
```

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- `tests/architecture/test_registries.py` verifies specs do not store class
  references and registries store spec objects.
- Signature drift resolves section 3.1 to implementation symbols.

## 5. Alternatives Considered

Storing class references with `importlib.reload()` was rejected because old and
new class objects diverge. Server restart on every file change was rejected
for usability. Hot-swapping running instances was rejected as unsafe.
