# ADR-030: Config Schema MRO Merge And Base-Class Field Injection

## 1. Decision Summary

SciStudio block registration merges `config_schema` properties along a block
class's MRO instead of reading only the leaf class. Base classes such as
`IOBlock` and `AppBlock` can inject common config fields, while subclasses can
override a property by declaring the same property name in their own schema.

### 1.1 Problems Addressed

| Problem | Risk | ADR-030 response | Detailed section |
|---|---|---|---|
| `ClassVar` schemas do not merge through inheritance | Subclasses lose base fields such as IO path or AppBlock output directory | Merge schema properties in the registry | Section 3 |
| IO path fields differ by direction | Savers may render file pickers instead of directory pickers | Post-process inherited IO `path` by `direction` | Section 4 |
| Future base-class fields need one injection path | Every plugin subclass would need manual updates | Make registry-time MRO merge the general mechanism | Section 5 |
| Schema behavior needs audit anchors | Frontend config forms can drift from registry output | Govern `BlockRegistry.scan()` and `BlockRegistry.get_spec()` as the public registry surface | Section 7 |

## 2. Context

Every block class declares a JSON-schema-like `config_schema`. Before this ADR,
the registry read that class variable with a simple lookup. That meant a
subclass declaring its own schema replaced the base schema entirely.

This broke two recurring needs:

- IO blocks need a common `path` field whose widget differs by direction.
- App blocks need a common output directory field that every concrete external
  app block inherits.

## 3. MRO Merge Rule

`_merge_config_schema(cls)` walks `cls.__mro__` from base to leaf and merges
the `properties` dict from each class's own `__dict__`. It intentionally does
not use `getattr`, because `getattr` would follow the MRO itself and repeat an
inherited schema as if it were declared on every class.

Semantics:

- base properties are added first;
- child properties override parent properties on the same name;
- `required` lists are unioned and deduplicated;
- each property is treated as an atomic JSON Schema object, not deep-merged.

## 4. Direction-Aware IO Path Injection

`IOBlock.config_schema` declares a base `path` property suitable for
input-direction blocks: file browser, string-or-array type, and string items.

After the MRO merge, if the class has `direction == "output"` and the leaf
class did not declare its own `path`, the registry converts the inherited field
to a single string with `directory_browser` and removes array `items`.

This gives loaders multi-file selection and savers directory selection without
forcing every saver subclass to repeat a path field.

## 5. Base-Class Field Injection

The merge rule is the general mechanism for future base-class config fields.
`AppBlock` can declare fields such as an output directory once and have every
subclass inherit it through registration. Subclasses that need a custom field
can still override by declaring the same property name locally.

Frontend forms, path portability, and runtime config use the schema emitted by
the backend registry. The frontend should not independently inject base-class
fields.

## 6. Scope

In scope:

- registry-time `config_schema` MRO merge;
- child-wins property override semantics;
- direction-aware inherited IO `path` adjustment;
- base-class field injection for AppBlock and future block bases;
- path portability verification against merged schemas.

Out of scope:

- deep-merge of individual JSON Schema properties;
- client-side schema injection;
- cleanup of every redundant subclass `path` declaration in this ADR;
- changing widget semantics beyond the inherited fields.

## 7. Verification And Tooling Impact

Tests should verify base and child schema merging, child override precedence,
deduplicated `required` fields, IO input/output path widget selection,
AppBlock inherited fields, registry `BlockSpec.config_schema` output, and path
portability for merged path-like fields.

### 7.1 Signature-Level Contracts

```text
def _merge_config_schema(cls: type) -> dict[str, Any]: ...


def _spec_from_class(cls: type, source: str = "") -> BlockSpec: ...


class BlockSpec:
    config_schema: dict[str, Any]


class IOBlock:
    direction: ClassVar[str]
    config_schema: ClassVar[dict[str, Any]]


class AppBlock:
    config_schema: ClassVar[dict[str, Any]]
```

## 8. Consequences

- Base-class config fields become reliable and visible to the API/frontend.
- Plugin authors write less boilerplate.
- Subclass override behavior is simple but atomic; changing only one nested key
  requires the subclass to provide the full property object.
- Debugging schema origin requires understanding the MRO merge order.

## 9. Alternatives Considered

| Alternative | Reason rejected |
|---|---|
| Frontend-side injection | Creates a second source of truth and does not help backend runtime config |
| Manual `_base_config_schema()` calls | Depends on every block author remembering to call the helper |
| Decorator-based composition | Adds class-creation complexity when the registry already scans classes |
