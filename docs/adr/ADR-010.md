# ADR-010: Batch Execution Mode Declared Per Block

## 1. Decision Summary

ADR-010 recorded the original per-block `batch_mode` design. It is superseded
by ADR-020, which removes engine-level item iteration and moves collection
handling into blocks.

### 1.1 Problems Addressed

| Problem | Risk | ADR-010 response | Detailed section |
|---|---|---|---|
| Different blocks need different batch behavior | A global batch mode is too coarse | Historical design used per-block `batch_mode` | Section 3 |
| The historical design conflicted with process and collection semantics | Engine-level item iteration became too complex | ADR-020 supersedes this ADR | Section 4 |

## 2. Historical Decision

The original design had `PARALLEL`, `SERIAL`, and `ADAPTIVE` batch modes,
plus resource throttling. That design is not the current runtime contract.

## 3. Supersession

ADR-020 replaces `batch_mode`, `BatchResult`, `BatchMode`, and
`BatchErrorStrategy` with collection-based transport. The engine passes
collections; blocks decide whether and how to iterate internally.

## 4. Verification

- `frontmatter_lint` accepts this ADR.
- State/config tests verify removed batch enums and result types stay removed.

## 5. Alternatives Considered

The rejected alternatives were a global batch mode, no batch support, and
per-run UI batch strategy. ADR-020 records the accepted replacement.
